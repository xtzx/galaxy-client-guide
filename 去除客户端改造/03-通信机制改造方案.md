## 一、概述

```
│  │  前端页面   │◄────────────►│ galaxy-client │◄────────────►│ 逆向程序 │ │
│  │  (galaxy)  │ 127.0.0.1   │  (Electron)  │  IPC 管道    │ (C++)  │ │
│  └─────┬─────┘              └──────┬───────┘              └────────┘ │
│        │                           │                                   │
│   HTTP │ API                  MQTT │                                   │
│        │                           │                                   │
│        ▼                           ▼                                   │
│  ┌──────────────────────────────────────────┐                         │
│  │              云端后台                      │                         │
│  │         (瓦力云 / MQTT Broker)            │                         │
│  └──────────────────────────────────────────┘                         │
│                                                                        │
│  另外在前端内部：                                                       │
│  ┌──────────┐  Electron IPC  ┌──────────┐                             │
│  │  menu 页面 │◄──────────────►│ sub 页面  │  (通过 webview IPC)        │
│  └──────────┘                └──────────┘                             │

│  └──────────┘                └──────────┘                             │
└────────────────────────────────────────────────────────────────────────┘
```

### 1.2 改造后通信链路

```
┌────────────────────────────────────────────────────────────────────────┐
│                        目标通信链路                                     │
│                                                                        │
│  ┌───────────┐                                      ┌────────────────┐│
│  │  前端页面   │              ┌──────────┐  管道/直调  │ 逆向程序        ││
│  │  (galaxy)  │     HTTP     │  云端后台  │◄──MQTT───►│ (C++ 客户端)   ││
│  │  浏览器中   │◄────────────►│          │            │                ││
│  └─────┬─────┘              └──────────┘            └────────────────┘│
│        │                                                               │
│   前端内部：                                                            │
│  ┌──────────┐  postMessage   ┌──────────┐                             │
│  │  主页面    │◄──────────────►│ iframe   │  (sub / EmbedSys)          │
│  └──────────┘                └──────────┘                             │
└────────────────────────────────────────────────────────────────────────┘
```

### 1.3 通信方式变化对照

| 通信方式 | 当前 | 目标 |
|---------|------|------|
| 前端 ↔ 云端 | HTTP API | HTTP API（保持不变） |
| 前端 ↔ galaxy-client | WebSocket（本地） | **废弃** |
| galaxy-client ↔ 逆向 | Named Pipe（IPC 管道） | **前端不再涉及** |
| 云端 ↔ 逆向 | MQTT（经 galaxy-client 中转） | MQTT（直连） |
| menu ↔ sub | Electron webview IPC | **postMessage**（iframe） |
| menu ↔ EmbedSys | postMessage（iframe）/ webview IPC | **统一 postMessage**（iframe） |
| 前端 ↔ Electron 主进程 | IPC（electron-better-ipc） | **废弃** |

### 2.2 WebSocket 命令分析与处置

#### 2.2.1 getAllConfig

| 项目 | 说明 |
|------|------|
| **当前功能** | 获取所有微信账号配置、登录状态、用户信息 |
| **消息格式** | 发送：`{ cmdId: 'getAllConfig' }`<br>接收：`{ cmdId: 'getAllConfig', body: [{ channelId, wxInfo, workWx, ... }] }` |
| **使用场景** | 页面初始化时获取所有账号列表 |

### 2.3 WebSocket 废弃后的替代通信策略

#### 2.3.1 方案 A：纯 HTTP 轮询

```
前端 ──── HTTP API ────► 云端后台
          定时轮询
          (getAllConfig: 30s)
          (status: 10s)
```

**优点**：
- 实现简单，前端改造成本低
- 无需维护长连接
- 兼容所有浏览器和网络环境

**缺点**：
- 实时性差（最多延迟一个轮询周期）
- 服务端压力：多客户端频繁轮询
- 带宽浪费：大部分请求返回无变化

**适用场景**：
- 状态查询（账号列表、登录状态等）
- 低频数据（标签、配置等）

#### 2.3.2 方案 B：SSE（Server-Sent Events）

```
前端 ──── SSE 长连接 ────► 云端后台
          服务端推送
          (状态变化时推送)
```

**优点**：
- 实时性好（服务端主动推送）
- 单向通信，比 WebSocket 轻量
- 浏览器原生支持，自动重连

**缺点**：
- 只支持服务端到客户端的单向推送
- 每个 SSE 连接占用一个 HTTP 连接
- 部分代理/防火墙可能不支持长连接

**适用场景**：
- 账号状态变化通知
- 任务执行进度推送
- 系统通知推送

#### 2.3.3 方案 C：MQTT over WebSocket（mqtt.js）

```
前端 ──── MQTT over WS ────► MQTT Broker ────► C++ 客户端
                              (阿里云 MQTT)
```

**优点**：
- 前端可直接订阅逆向程序的状态变化
- 实时性最好
- 可复用现有 MQTT 基础设施

**缺点**：
- 前端增加 MQTT 连接管理复杂度
- 需要处理重连、心跳等
- 安全性：前端暴露 MQTT 凭证

**适用场景**：
- 需要极高实时性的场景
- 前端需要直接与逆向程序交互的场景

#### 2.3.4 推荐方案

**建议采用「HTTP 轮询 + SSE」组合**：

| 数据类型 | 通信方式 | 频率/时机 |
|---------|---------|----------|
| 账号列表（getAllConfig） | HTTP API + SSE 通知 | 初始加载 + SSE 推送变化 |
| 登录状态（login） | SSE 推送 | 状态变化时推送 |
| 好友/群列表（userlist） | HTTP API | 按需请求 |
| 标签列表（labels） | HTTP API | 按需请求 |
| 任务执行结果 | SSE 推送 | 任务完成时推送 |
| 系统通知（sendNotice） | SSE 推送 | 有通知时推送 |
| 文件上传 | HTTP API（直传 OSS） | 用户操作时 |

**SSE 连接设计**：

```javascript
const eventSource = new EventSource('/api/events/stream?token=xxx');

eventSource.addEventListener('account-update', (event) => {
  const data = JSON.parse(event.data);
  dispatch(updateAllConfig(data));
});

eventSource.addEventListener('login-status', (event) => {
  const data = JSON.parse(event.data);
  dispatch(updateLoginStatus(data));
});

eventSource.addEventListener('task-result', (event) => {
  const data = JSON.parse(event.data);
  dispatch(handleTaskResult(data));
});

eventSource.addEventListener('notification', (event) => {
  const data = JSON.parse(event.data);
  dispatch(updateNoticeList(data));
});

eventSource.onerror = () => {
  // SSE 自动重连
};
```

### 3.3 postMessage 通信协议设计

#### 3.3.1 消息格式

```typescript
interface SubMessage {
  source: 'galaxy-sub' | 'galaxy-menu';
  type: string;         // 消息类型（对应原 IPC 通道名）
```

### 3.4 通信工具类封装

建议封装统一的通信工具，降低迁移成本：

```javascript
// src/common/bridge.js - 替代 ipc.js

class MessageBridge {
  constructor(role) {
    this.role = role; // 'menu' | 'sub'
    this.handlers = new Map();
    this.pendingRequests = new Map();
    
    window.addEventListener('message', this._handleMessage.bind(this));
  }

  // 发送消息（替代 ipc.send / ipc.sendToHost）
  send(type, payload) {
    const message = {
      source: `galaxy-${this.role}`,
      type,
      payload,
      timestamp: Date.now(),
    };
    
    if (this.role === 'sub') {
      window.parent.postMessage(message, MENU_ORIGIN);
    }
    // menu 端通过 sendToSub 方法指定目标 iframe
  }

  // 发送到指定 sub iframe（menu 端使用）
  sendToSub(iframeEl, type, payload) {
    const message = {
      source: 'galaxy-menu',
      type,
      payload,
      timestamp: Date.now(),
    };
    iframeEl.contentWindow.postMessage(message, SUB_ORIGIN);
  }

  // 广播到所有 sub iframe（替代 webviewBroadcast）
  broadcastToSubs(type, payload) {
    const iframes = document.querySelectorAll('.sub-iframe');
    iframes.forEach(iframe => {
      this.sendToSub(iframe, type, payload);
    });
  }

  // 监听消息（替代 ipc.on）
  on(type, handler) {
    if (!this.handlers.has(type)) {
      this.handlers.set(type, []);
    }
    this.handlers.get(type).push(handler);
  }

  // 请求-响应模式（替代 ipc.callMain）
  async request(type, payload) {
    const requestId = `${Date.now()}-${Math.random()}`;
    return new Promise((resolve) => {
      this.pendingRequests.set(requestId, resolve);
      this.send(type, { ...payload, requestId });
    });
  }

  _handleMessage(event) {
    const expectedSource = this.role === 'menu' ? 'galaxy-sub' : 'galaxy-menu';
    if (event.data?.source !== expectedSource) return;
    
    const { type, payload, requestId } = event.data;
    
    // 处理响应
    if (requestId && this.pendingRequests.has(requestId)) {
      this.pendingRequests.get(requestId)(payload);
      this.pendingRequests.delete(requestId);
      return;
    }
    
    // 分发消息
    const handlers = this.handlers.get(type) || [];
    handlers.forEach(handler => handler(event.data));
  }
}

export default MessageBridge;
```

## 八、问题与风险

### 8.1 实时性问题

**问题**：当前 WebSocket 是持续连接，消息实时送达。改为 HTTP + SSE 后：
- HTTP 轮询有延迟
- SSE 连接可能断开需要重连

**应对**：
- 关键状态使用 SSE 推送
- SSE 断线自动重连（`EventSource` 内置支持）
- 非关键数据使用 HTTP 按需请求

### 8.2 数据一致性问题

**问题**：当前好友列表等数据从逆向程序实时获取，改造后从云端获取可能有延迟。

**应对**：
- 逆向程序定期（5-10s）同步数据到云端
- 前端加载时从云端获取，操作后触发强制同步
- 展示时标注数据更新时间

### 8.3 文件上传路径变化

**问题**：当前上传使用本地文件路径，浏览器环境只有 File 对象。

**应对**：
- `upload` 命令改为 File 对象直传
- `send.js`、`upload.js` 中所有 `path` 参数改为 `File` 对象
- 上传组件统一使用 `<input type="file">` + File API

### 8.4 操作类指令的响应

**问题**：踢人、退出登录等操作类指令，当前通过 WebSocket 等待结果返回。改为云端 API 后，结果如何获取？

**应对**：
- 云端 API 返回任务 ID
- SSE 推送任务执行结果
- 或前端轮询任务状态：`GET /api/tasks/{taskId}/status`
