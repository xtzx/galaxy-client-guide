## 一、概述

### 1.1 当前页面架构

galaxy 前端采用 **多入口 + 多 webview 子进程** 的 Electron 架构：

### 1.2 关键特性

| 特性 | 当前实现 |
|------|---------|
| **多账号隔离** | 每个微信账号一个独立 webview 进程，完全隔离 |
| **状态独立** | 每个 sub webview 有独立的 Redux store |
| **通信方式** | webview.send / ipc.sendToHost（Electron IPC） |
| **显示控制** | CSS `display: none` 隐藏非当前账号的 webview |
| **外部系统嵌入** | iframe（AISOP）+ webview（IMIS Chat） |
| **构建产物** | menu.html + sub.html 两个独立 HTML |

### 1.3 改造目标

将 Electron 多 webview 架构改造为**纯浏览器可用的方案**，需要解决：

1. Sub 工作台多实例问题：如何在浏览器中实现多账号工作台
2. Menu-Sub 通信问题：如何替代 Electron IPC
3. EmbedSys 嵌入问题：webview 改为 iframe

## 二、Sub 工作台多实例方案

### 2.1 方案概述

| 方案 | 描述 |
|------|------|
| **A. iframe 多实例** | 每个账号一个 iframe，加载 sub 页面 |
| **B. 单页面 + Tab 切换** | 单个 React 组件，Tab 切换不同账号数据 |
| **C. 动态路由 + 状态缓存** | 单个路由组件，URL 参数区分账号，状态缓存 |

---

### 2.2 方案 A：iframe 多实例

#### 架构设计

```
┌────────────────────────────────────────────────────────┐
│  menu 主页面（浏览器 Tab）                               │
│                                                        │
│  ┌──────┐  ┌──────────────────────────────────────┐   │
│  │ 左侧栏 │  │              右侧内容                 │   │
│  │      │  │                                      │   │
│  │      │  │  ┌──────────────────────────────┐   │   │
│  │      │  │  │ iframe#sub-1 (账号1) [显示]   │   │   │
│  │      │  │  └──────────────────────────────┘   │   │
│  │      │  │  ┌──────────────────────────────┐   │   │
│  │      │  │  │ iframe#sub-2 (账号2) [隐藏]   │   │   │
│  │      │  │  └──────────────────────────────┘   │   │
│  │      │  │  ┌──────────────────────────────┐   │   │
│  │      │  │  │ iframe#sub-3 (账号3) [隐藏]   │   │   │
│  │      │  │  └──────────────────────────────┘   │   │
│  │      │  │                                      │   │
│  └──────┘  └──────────────────────────────────────┘   │
│                                                        │
│  每个 iframe 加载：                                      │
│  /sub.html?channelId=xxx&qyFlag=0                      │
└────────────────────────────────────────────────────────┘
```

#### 通信方式

```javascript
// menu → sub: postMessage
const iframe = document.getElementById(`sub-${channelId}`);
iframe.contentWindow.postMessage({
  source: 'galaxy-menu',
  type: 'parent-message',
  payload: data
}, SUB_ORIGIN);

// sub → menu: postMessage
window.parent.postMessage({
  source: 'galaxy-sub',
  type: 'sub-message',
  channelId: currentChannelId,
  payload: data
}, MENU_ORIGIN);
```

#### 详细评估

| 维度 | 评分 | 说明 |
|------|------|------|
| **迁移成本** | ★★★★★ | 最低 - 架构与当前 webview 方案最接近 |
| **状态隔离** | ★★★★★ | 每个 iframe 独立 window/document/Redux store |
| **代码改动** | ★★★★☆ | 主要改 webview → iframe + IPC → postMessage |
| **内存占用** | ★★☆☆☆ | 每个 iframe 独立加载 JS bundle，占用较多内存 |
| **性能** | ★★★☆☆ | 多个 iframe 共享主进程线程，可能互相影响 |
| **首屏速度** | ★★☆☆☆ | 多个 iframe 并行加载，网络开销大 |
| **通信复杂度** | ★★★★☆ | postMessage 比 IPC 更标准，但需严格校验 origin |
| **可维护性** | ★★★★☆ | 结构清晰，sub 代码基本不需要改 |
| **SEO/SSR** | ★☆☆☆☆ | iframe 不利于 SEO（但此项目不需要 SEO） |

#### 优化策略

| 优化项 | 说明 |
|--------|------|
| **惰性加载** | 仅加载当前激活的账号 iframe，切换时再加载其他 |
| **LRU 淘汰** | 超过 N 个 iframe 时，销毁最久未使用的 |
| **共享 JS bundle** | iframe 加载同一个 sub.html，浏览器会缓存 JS 资源 |
| **预加载** | 当前账号加载完成后，后台预加载下一个 |

#### 关键代码改动

```javascript
// 当前 menu/App.js（webview 版本）
allConfig.map((item) => (
  <webview
    id={item.channelId}
    className={`subWebview ${currentAccount === item.channelId ? '' : 'hidden'}`}
    src={`${subHTMLUrl}?channelId=${item.channelId}&qyFlag=${toNum(item.workWx)}`}
    nodeintegration="true"
  />
));

// 改造后 menu/App.js（iframe 版本）
allConfig.map((item) => (
  <iframe
    id={`sub-${item.channelId}`}
    key={item.channelId}
    className={`sub-iframe ${currentAccount === item.channelId ? '' : 'hidden'}`}
    src={`/sub.html?channelId=${item.channelId}&qyFlag=${toNum(item.workWx)}`}
    title={`工作台-${item.channelId}`}
    sandbox="allow-scripts allow-same-origin allow-forms allow-popups"
  />
));
```

### 2.3 方案 B：单页面 + Tab 切换

#### 架构设计

```
┌────────────────────────────────────────────────────────┐
│  单个 React 应用                                        │
│                                                        │
│  ┌──────┐  ┌──────────────────────────────────────┐   │
│  │ 左侧栏 │  │              右侧内容                 │   │
│  │      │  │                                      │   │
│  │      │  │  ┌────────────────────────────────┐  │   │
│  │      │  │  │ Tab: [账号1] [账号2] [账号3]    │  │   │
│  │      │  │  ├────────────────────────────────┤  │   │
│  │      │  │  │                                │  │   │
│  │      │  │  │   SubWorkbench                 │  │   │
│  │      │  │  │   (单实例，props.channelId)     │  │   │
│  │      │  │  │                                │  │   │
│  │      │  │  │   Nav + RouteView              │  │   │
│  │      │  │  │   (当前账号的功能页面)           │  │   │
│  │      │  │  │                                │  │   │
│  │      │  │  └────────────────────────────────┘  │   │
│  │      │  │                                      │   │
│  └──────┘  └──────────────────────────────────────┘   │
│                                                        │
│  所有账号共享同一个 React 组件树                          │
│  通过 channelId 区分当前活跃账号                          │
│  状态按 channelId 分区存储在 Redux 中                     │
└────────────────────────────────────────────────────────┘
```

#### 状态管理

```javascript
// Redux store 结构改造
{
  menu: { ... },          // menu 状态
  accounts: {             // 按账号分区
    [channelId_1]: {
      base: { ... },
      broadcast: { ... },
      smartReply: { ... },
      contacts: { ... },
      // ... 所有 sub reducer
    },
    [channelId_2]: {
      base: { ... },
      // ...
    }
  },
  currentChannelId: 12345, // 当前激活账号
}
```

#### 详细评估

| 维度 | 评分 | 说明 |
|------|------|------|
| **迁移成本** | ★★☆☆☆ | 高 - 需要大量重构 Redux、路由、组件 |
| **状态隔离** | ★★☆☆☆ | 需要手动管理状态分区，容易出现状态污染 |
| **代码改动** | ★☆☆☆☆ | 所有 sub 组件需要改为接收 channelId 参数 |
| **内存占用** | ★★★★★ | 最优 - 只有一个 React 实例 |
| **性能** | ★★★★☆ | 好 - 但账号切换时可能触发大量重渲染 |
| **首屏速度** | ★★★★★ | 最快 - 只加载一个应用 |
| **通信复杂度** | ★★★★★ | 无需跨 iframe 通信，直接 dispatch |
| **可维护性** | ★★★☆☆ | 状态管理复杂度增加 |
| **切换体验** | ★★★☆☆ | 切换账号时组件需要重新初始化 |

#### 主要改造工作

1. **Redux 重构**：将 sub 的 Redux store 改为按 channelId 分区

```javascript
// 需要改造所有 sub reducer 和 thunks
// 每个 action 都需要带 channelId 参数
dispatch(setUserList({ channelId: 123, data: [...] }));

// selector 也需要按 channelId 查询
const userList = useSelector(state => state.accounts[currentChannelId]?.base.userList);
```

2. **组件改造**：所有 sub 组件需要接收 channelId

```javascript
// 当前：sub 组件直接从 store 读取（每个 webview 独立 store）
const userInfo = useSelector(state => state.base.userInfo);

// 改造后：需要从分区 store 读取
const currentChannelId = useSelector(state => state.currentChannelId);
const userInfo = useSelector(state => state.accounts[currentChannelId]?.base.userInfo);
```

3. **路由改造**：嵌套路由支持 channelId

```javascript
// /workspace/:channelId/broadCast
// /workspace/:channelId/smartReply
// /workspace/:channelId/contacts
```

4. **生命周期改造**：账号切换时需要正确处理副作用

```javascript
useEffect(() => {
  // 账号切换时需要重新初始化数据
  dispatch(initAccountData(channelId));
  return () => {
    // 清理副作用
    dispatch(cleanupAccount(channelId));
  };
}, [channelId]);
```

#### 风险与挑战

| 风险 | 说明 |
|------|------|
| **状态污染** | 不同账号的操作可能互相影响（全局变量、定时器、事件监听等） |
| **改造量巨大** | sub 目录下 19 个功能模块全部需要改造 |
| **兼容性** | 部分组件可能依赖全局状态或单例模式 |
| **回归风险** | 改造范围太大，回归测试工作量极大 |

### 2.4 方案 C：动态路由 + 状态缓存（keep-alive）

#### 架构设计

```
┌────────────────────────────────────────────────────────┐
│  单个 React 应用                                        │
│                                                        │
│  ┌──────┐  ┌──────────────────────────────────────┐   │
│  │ 左侧栏 │  │              右侧内容                 │   │
│  │      │  │                                      │   │
│  │      │  │  KeepAlive 容器                       │   │
│  │      │  │  ┌────────────────────────────────┐  │   │
│  │      │  │  │ Cache: 账号1 [激活]             │  │   │
│  │      │  │  │   SubWorkbench (完整组件树)      │  │   │
│  │      │  │  └────────────────────────────────┘  │   │
│  │      │  │  ┌────────────────────────────────┐  │   │
│  │      │  │  │ Cache: 账号2 [缓存/隐藏]        │  │   │
│  │      │  │  │   SubWorkbench (DOM 保留)       │  │   │
│  │      │  │  └────────────────────────────────┘  │   │
│  │      │  │                                      │   │
│  └──────┘  └──────────────────────────────────────┘   │
│                                                        │
│  使用 react-activation 或自定义 KeepAlive 实现            │
│  切换账号时保留 DOM 和状态，仅切换显示                     │
└────────────────────────────────────────────────────────┘
```

#### 详细评估

| 维度 | 评分 | 说明 |
|------|------|------|
| **迁移成本** | ★★★☆☆ | 中等 - 需引入 keep-alive 方案 + Redux 分区 |
| **状态隔离** | ★★★☆☆ | 通过 keep-alive 保留组件状态，但仍需 Redux 分区 |
| **代码改动** | ★★☆☆☆ | sub 组件需要改造，但比方案 B 少 |
| **内存占用** | ★★★☆☆ | 中等 - 缓存的组件树占用内存 |
| **性能** | ★★★★☆ | 好 - 切换不重渲染，复用 DOM |
| **首屏速度** | ★★★★★ | 快 |
| **通信复杂度** | ★★★★★ | 无需跨 iframe 通信 |
| **可维护性** | ★★★☆☆ | keep-alive 引入额外复杂度 |
| **切换体验** | ★★★★☆ | 好 - 切换瞬间，保留滚动位置等状态 |

#### 主要改造工作

1. 引入 `react-activation` 或类似 keep-alive 库
2. Redux store 按 channelId 分区（与方案 B 相同）
3. KeepAlive 容器包裹 sub 组件

```javascript
import { KeepAlive } from 'react-activation';

function SubWorkbenchContainer({ allConfig, currentChannelId }) {
  return (
    <>
      {allConfig.map((item) => (
        <div
          key={item.channelId}
          style={{ display: currentChannelId === item.channelId ? 'block' : 'none' }}
        >
          <KeepAlive id={`sub-${item.channelId}`} name={`sub-${item.channelId}`}>
            <SubWorkbench channelId={item.channelId} config={item} />
          </KeepAlive>
        </div>
      ))}
    </>
  );
}
```

#### 风险与挑战

| 风险 | 说明 |
|------|------|
| **keep-alive 可靠性** | React 社区的 keep-alive 方案不如 Vue 成熟 |
| **内存泄漏** | 缓存的组件可能导致内存泄漏（事件监听、定时器等未清理） |
| **状态同步** | 缓存组件激活时需要同步最新数据 |
| **第三方库兼容** | Ant Design 等组件可能与 keep-alive 有兼容性问题 |

### 2.5 三方案对比总表

| 维度 | 方案 A：iframe | 方案 B：单页面+Tab | 方案 C：keep-alive |
|------|---------------|-------------------|-------------------|
| **迁移成本** | ★★★★★ 最低 | ★★☆☆☆ 最高 | ★★★☆☆ 中等 |
| **状态隔离** | ★★★★★ 天然隔离 | ★★☆☆☆ 需手动 | ★★★☆☆ 需手动 |
| **内存占用** | ★★☆☆☆ 高 | ★★★★★ 最低 | ★★★☆☆ 中等 |
| **首屏性能** | ★★☆☆☆ 慢 | ★★★★★ 快 | ★★★★★ 快 |
| **切换体验** | ★★★☆☆ 中等 | ★★★☆☆ 需初始化 | ★★★★☆ 快速 |
| **代码改动量** | ★★★★☆ 小 | ★☆☆☆☆ 巨大 | ★★☆☆☆ 大 |
| **通信方案** | postMessage | 直接 dispatch | 直接 dispatch |
| **生产可靠性** | ★★★★★ 高 | ★★★★☆ 高 | ★★★☆☆ 中等 |
| **技术风险** | ★★★★☆ 低 | ★★★☆☆ 中 | ★★☆☆☆ 较高 |

### 2.6 推荐与建议

**推荐方案 A：iframe 多实例**

理由：
1. **迁移成本最低**：架构与当前 webview 方案一一对应，sub 代码几乎不需要改动
2. **状态隔离最好**：每个 iframe 独立 window，天然隔离，不会出现状态污染
3. **风险最低**：iframe 是浏览器原生特性，行为确定，不依赖第三方库
4. **渐进式迁移**：可以逐个功能模块迁移，不需要一次全部改完
5. **sub 代码复用**：sub.html 作为独立页面继续使用，内部逻辑基本不变

如果后期发现 iframe 内存占用过高，可以考虑将热点功能改为方案 B/C，作为**性能优化阶段**的工作。

## 三、EmbedSys 嵌入系统改造

### 3.1 当前 EmbedSys 架构

```
EmbedSys/
├── index.tsx          # 主入口：TabProvider + TabNav + TabView
├── TabView/index.tsx  # 根据 embedType 渲染不同组件
├── TabNav/index.tsx   # 导航标签
├── TabContext.tsx      # 标签状态管理
├── menu.ts            # 菜单配置
├── types.ts           # 类型定义
```

## 四、Menu-Sub 通信方案对比

### 4.1 方案概述

如果选择方案 A（iframe 多实例），需要选择 Menu-Sub 通信方式：

| 方案 | 描述 |
|------|------|
| **A1. 纯 postMessage** | 所有通信通过 postMessage |
| **A2. postMessage + 共享 Worker** | postMessage + SharedWorker 作为消息总线 |
| **A3. postMessage + BroadcastChannel** | postMessage + BroadcastChannel 广播 |

### 4.2 方案 A1：纯 postMessage

```
menu ──── postMessage ────► sub iframe
     ◄─── postMessage ─────
```

#### 优点
- 最简单、最标准
- 浏览器原生支持，无第三方依赖
- 点对点通信，性能好
- 安全性好（可校验 origin）

#### 缺点
- 广播需要遍历所有 iframe
- 需要自己管理消息路由
- 大数据量传输时需要考虑序列化性能

#### 适用性
- **推荐** - 完全满足需求

### 4.3 方案 A2：postMessage + SharedWorker

```
menu ──── postMessage ────► sub iframe
          SharedWorker (消息总线)
menu ──── SharedWorker.port ────► 广播到所有 sub
```

#### 优点
- SharedWorker 可作为多 iframe 间的消息总线
- 支持高效广播

#### 缺点
- 增加复杂度
- SharedWorker 兼容性不如 postMessage（Safari 有限制）
- 调试困难

#### 适用性
- **不推荐** - 过度设计，postMessage 已足够

### 4.4 方案 A3：postMessage + BroadcastChannel

```
menu ──── BroadcastChannel ────► 所有同源页面
sub  ──── BroadcastChannel ────► 所有同源页面
```

#### 优点
- 简化广播逻辑
- 同源页面自动接收

#### 缺点
- 无法精确定向发送（广播给所有监听者）
- Safari 14.1+ 才支持
- 消息过滤需要额外逻辑

#### 适用性
- **可选** - 可配合 postMessage 使用，用于广播场景

### 4.5 通信方案推荐

**推荐 A1（纯 postMessage），广播场景可选 A3 配合**：

| 场景 | 通信方式 | 说明 |
|------|---------|------|
| menu → 特定 sub | postMessage（指定 iframe） | 业务消息、配置下发 |
| menu → 所有 sub | 遍历 iframe + postMessage | 登出通知、全局数据 |
| sub → menu | parent.postMessage | 业务请求、状态上报 |
| sub → 其他 sub | 不直接通信 | 通过 menu 中转 |

## 八、FAQ

### Q1: iframe 会不会有跨域问题？

**A**: 如果 menu.html 和 sub.html 部署在同一个域名下（推荐方案），则是同源的，不存在跨域问题。postMessage 也可以跨域工作，但建议同源部署。

### Q2: iframe 内的 sub 页面能否访问父页面的数据？

**A**: 同源情况下可以通过 `parent.window` 访问父页面，但**不推荐**这种方式。建议通过 postMessage 进行通信，保持清晰的接口边界。

### Q3: 多个 iframe 的内存占用如何控制？

**A**: 
- 惰性加载：仅加载当前激活和最近使用的 2-3 个 iframe
- LRU 淘汰：超过阈值时销毁最久未使用的 iframe
- 监控：通过 `Performance API` 监控内存使用

### Q4: iframe 内的页面刷新会影响主页面吗？

**A**: 不会。iframe 是独立的浏览上下文，刷新 iframe 不影响主页面。但需要注意 iframe 刷新后需要重新建立 postMessage 通信。

### Q5: sub 页面的 URL 参数如何传递？

**A**: 通过 iframe 的 `src` 属性传递：
```html
<iframe src="/sub.html?channelId=123&qyFlag=0" />
```
sub 页面通过 `URLSearchParams` 解析参数。

### Q6: 如何处理 iframe 内的弹窗和遮罩层？

**A**: 
- iframe 内的弹窗（Modal）只在 iframe 内显示，不会遮挡主页面
- 如果需要全局弹窗，通过 postMessage 通知主页面显示
- 可以考虑使用 CSS `pointer-events` 和 `z-index` 处理层级关系
