# 01 - 去除客户端改造总览与架构设计

> 文档状态：讨论稿  
> 创建时间：2026-03-20  
> 适用项目：galaxy（前端）、galaxy-client（客户端，待废弃）

---

## 一、改造背景

### 1.1 当前架构概述

当前系统由三个核心部分组成：

```
┌─────────────────────────────────────────────────────────────────┐
│                        当前架构                                  │
│                                                                  │
│  ┌──────────┐    ┌──────────────┐    ┌────────────────────────┐ │
│  │  galaxy   │    │ galaxy-client│    │    逆向程序             │ │
│  │ (前端页面) │◄──►│  (Electron   │◄──►│ (BasicService.exe +    │ │
│  │  React    │ WS │   客户端)    │IPC │  PipeCore.dll +        │ │
│  │  渲染进程  │    │   主进程     │    │  Galaxy.dll)           │ │
│  └──────────┘    └──────┬───────┘    └────────────────────────┘ │
│                         │                                        │
│                    MQTT │                                        │
│                         ▼                                        │
│                  ┌──────────────┐                                │
│                  │   云端后台    │                                │
│                  │ (瓦力云后台)  │                                │
│                  └──────────────┘                                │
└─────────────────────────────────────────────────────────────────┘
```

**数据流向（当前）：**

1. 用户在 galaxy 前端页面配置任务（群发、加好友、进群回复等）
2. 配置提交到云端后台
3. 云端通过 MQTT 下发任务给 galaxy-client（Electron 客户端）
4. galaxy-client 通过 Named Pipe（IPC 管道）将任务发送给逆向程序
5. 逆向程序操控微信/企微执行任务
6. 执行结果沿反向路径回传

**关键角色：**

| 组件 | 技术栈 | 职责 |
|------|--------|------|
| galaxy | React 16 + Electron 7 | 前端页面渲染、用户交互、配置管理 |
| galaxy-client | Electron 20 + Node.js | MQTT 消息收发、IPC 管道通信、进程管理、文件操作 |
| 逆向程序 | C++ | 注入微信/企微进程、执行自动化操作 |
| 云端后台 | Java | 任务管理、MQTT Broker、数据存储 |

### 1.2 当前架构的问题

1. **中间层冗余**：galaxy-client 本质上是一个消息中转站，将 MQTT 消息转换为 IPC 管道消息，增加了不必要的复杂度
2. **技术债务**：两个 Electron 应用（galaxy 自带 Electron 壳 + galaxy-client），维护成本高
3. **部署复杂**：需要在每台 Windows 机器上安装 Electron 客户端，更新升级繁琐
4. **调试困难**：消息链路 `前端 → WS → 客户端 → IPC → 逆向` 过长，问题定位困难
5. **资源浪费**：Electron 进程占用大量内存和 CPU

### 1.3 改造目标

去除 galaxy-client 客户端中间层，简化整体架构：

- galaxy 前端**从 Electron 应用变为纯 Web 应用**，部署在 Nginx/CDN
- 逆向程序**直接接收 MQTT 消息**，不再依赖 galaxy-client 中转
- **一对一关系**：一个前端页面管理一台 Windows 机器上的逆向程序

### 2.2 核心变化

| 方面 | 当前 | 目标 |
|------|------|------|
| **前端部署** | Electron 桌面应用 | 纯 Web 应用（Nginx/CDN） |
| **前端访问** | 本地 Electron 窗口 | 浏览器访问 |
| **MQTT 消费者** | galaxy-client（Electron） | C++ 客户端（逆向程序集成） |
| **逆向通信** | galaxy-client → Named Pipe → 逆向 | C++ 客户端内部直接调用 |
| **前端↔逆向** | WS → galaxy-client → IPC | HTTP/云端中转 |
| **本地文件** | Electron Node.js 操作 | 逆向程序处理 / 云端处理 |
| **进程管理** | galaxy-client 管理 | C++ 客户端自管理 |
| **本地存储** | SQLite（galaxy-client） | 逆向程序 / 云端数据库 |

### 2.3 各组件职责重新划分

#### 2.3.1 galaxy（前端 - 纯 Web 应用）

**保留的职责：**
- 用户界面渲染（React 组件）
- 用户交互和表单处理
- 任务配置和提交（通过 HTTP API 提交到云端）
- 数据展示（从云端 API 获取数据）
- 状态管理（Redux）
- 路由管理（React Router）

**新增的职责：**
- 浏览器原生文件选择（替代 Electron dialog）
- 浏览器原生 Cookie 管理（替代 Electron session）
- 浏览器端配置管理（替代 IPC 获取配置）

**废弃的职责：**
- Electron 主进程相关代码（`electron/` 目录）
- IPC 通信（`ipc.callMain`、`ipc.sendSync` 等）
- WebSocket 与 galaxy-client 通信
- WebView 子进程管理
- 本地文件系统操作
- 进程注入控制
- 本地 SQLite 操作

#### 2.3.2 C++ 客户端（新逆向主程序）

**承接自 galaxy-client 的职责：**
- MQTT 消息接收和发送
- 与云端后台的通信
- 逆向程序注入和管理
- 本地进程扫描和监控
- 本地文件操作（文件上传 OSS 等）
- 本地存储管理

**新增的职责：**
- 直接 MQTT 连接（不再需要 galaxy-client 中转）
- 自启动和自管理

#### 2.3.3 云端后台

**保留的职责：**
- MQTT Broker 管理
- 任务管理和调度
- 数据存储和查询
- API 服务

**可能新增的职责：**
- 文件转换服务（替代客户端的 ffmpeg、sharp）
- 跨域代理服务（替代 Electron 的 cross-origin-request）
- 配置下发服务（替代 IPC 的 get-app-config）

## 七、风险评估

### 7.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| Sub 工作台多实例性能问题 | 多个 iframe 可能导致内存占用过高 | 中 | 惰性加载、及时销毁、性能监控 |
| 跨域通信安全问题 | iframe postMessage 可能存在安全隐患 | 低 | 严格校验 origin、加密敏感数据 |
| 文件操作功能缺失 | 部分文件操作浏览器无法完成 | 中 | 提前确认哪些功能可以废弃 |
| 浏览器兼容性 | 不同浏览器行为差异 | 低 | 限定 Chrome 最新版 |

### 7.2 业务风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 功能回归 | 改造过程中功能缺失或异常 | 高 | 完善测试用例、分阶段上线 |
| 用户习惯变化 | 从桌面应用变为浏览器应用 | 中 | 保持 UI 一致性、提供使用指南 |
| 数据迁移 | 本地 SQLite 数据需迁移到云端 | 中 | 制定数据迁移方案 |

### 7.3 项目风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 与逆向团队协作 | 逆向程序 MQTT 改造进度依赖 | 中 | 接口先行定义、Mock 开发 |
| 改造周期过长 | 影响正常业务迭代 | 中 | 分阶段、并行开发 |
| 云端服务压力增加 | 文件处理等原本本地完成的任务转到云端 | 低 | 评估服务容量 |
