# 日志系统优化实施清单

> 本文档为日志系统优化的具体实施步骤，面向开发者逐步执行。

---

## 一、方案分析与决策记录

### 1.1 关于 JSON 对象改造

**你的方案**：保留 `slsLogUtil.customLog`（纯字符串），新增一个 JSON 对象上报能力。在私聊/群聊发消息链路中，移除旧的 SLS 上报，全部改用新的 JSON 方法。

**分析结论：方案可行，建议微调**

- 新增 `slsLogUtil.customLogV2(jsonObj)` 作为 JSON 上报方法，和旧方法并存，**这是正确的渐进策略**
- 在私聊/群聊链路中"移除旧 SLS 改用新方法"的思路是对的，但建议同时保留 **本地 electron-log 写入**（与现有行为一致，error 级别写文件）
- ali-sls SDK 支持 `createField(key, value)` 多字段上报，可以直接使用，无需 hack SDK
- 新的 `logUtil.customLogV2(jsonObj)` 内部同时做两件事：(1) SLS JSON 上报 (2) 将 JSON 格式化为字符串后写入 electron-log

**一个风险提示**：移除旧日志后，如果新的 JSON schema 遗漏了某些信息，排查问题时会有盲区。建议在设计 JSON schema 时，对照旧日志的所有信息点逐一确认覆盖。本清单在每个步骤中会列出需要覆盖的信息点。

### 1.2 关于日志太多太杂

**你的方案**：执行日志聚合/压缩、心跳日志优化、空操作日志静默。按任务链路逐步处理，不一次性全改。

**分析结论：按链路处理是最佳策略**

- "按链路处理"可以把日志降噪和 JSON 改造合并在同一次代码修改中，减少重复工作
- 链路内的噪音（如 DispatchCenter 中间步骤日志、完整 JSON 重复输出）在链路改造时一并处理
- 链路外的噪音（如 `IPC-CONNECT-SUCCESS`、`HeartBeatTimer`、`ping`/`pong`、空列表推送）单独用一个步骤处理
- 建议先不动 `msgHandleBase.js` 中与当前链路无关的日志（如企微、旧版微信逻辑），只改私聊/群聊链路涉及的路径

### 1.3 关于 taskId 链路贯通

**你的方案**：通过逆向执行通知微信助手时携带任务 ID 来串联。

**分析结论：方案正确，需要注意具体实现**

**关键发现**：逆向返回的 `sendmessage` 响应**已经携带了 taskId**，真正缺失 taskId 的是 `recvmsg` 响应。具体情况：

| 逆向消息类型 | taskId 是否存在 | 说明 |
|------------|----------------|------|
| sendmessage（第一条回执） | 有 | 逆向原样返回了下发时的 taskId |
| recvmsg（第二条回执，微信4.0） | 无 | 逆向生成的消息确认，不包含 taskId |

当前的 `wxMsgHandle.isWx4RecvmsgComplete()` 通过"三段式匹配"从 sendmessage 继承 taskId 到 recvmsg，但依赖 key 匹配（`from + to + msgType + content`），有失败风险。

如果能让逆向在 recvmsg 中也带上 taskId，就不再依赖 key 匹配，可靠性大幅提升。建议：

1. **短期**：让逆向在 recvmsg 中加入 taskId 字段（C++ DLL 修改）
2. **兜底**：保留现有的三段式匹配逻辑作为 fallback（当 recvmsg 中 taskId 为空时走旧逻辑）
3. **日志增强**：在 taskId 匹配成功/失败时都记录结构化日志，方便排查

## 三、JSON 日志 Schema 设计

### 3.1 通用字段（每条日志必带）

| 字段名 | 类型 | 说明 | 来源 |
|--------|------|------|------|
| `ts` | number | 毫秒级时间戳 | `Date.now()` |
| `seq` | number | 全局自增序列号，解决同毫秒排序 | 全局计数器 |
| `logLevel` | string | 日志级别：info / warn / error | 调用者传入 |
| `module` | string | 模块名 | 调用者传入 |
| `action` | string | 具体动作/事件 | 调用者传入 |
| `user` | string | CAS 用户名 | `store.getUserInfo().user` |
| `gid` | string | 设备标识 | `getStoreGid()` |
| `version` | string | 主版本号 | `clientVersion` |
| `sonVersion` | string | 子版本号 | `clientSonVersion` |
| `registryCount` | number | 号数量 | `registryList.length` |

### 3.2 业务字段（按需携带）

| 字段名 | 类型 | 说明 | 使用场景 |
|--------|------|------|---------|
| `wxid` | string | 当前微信号 | 几乎所有链路日志 |
| `taskId` | string/number | MQTT 任务 ID | 任务链路日志 |
| `traceId` | string | 链路追踪 ID（本次新增） | 任务链路日志 |
| `msgType` | string | 消息类型 | 消息处理日志 |
| `channelId` | string | 通道 ID / processId | IPC 通信日志 |
| `reportId` | string | 逆向上报 ID | 逆向返回日志 |
| `target` | string | 消息目标（to） | 发消息链路 |
| `status` | number | 状态码 | 执行结果日志 |
| `duration` | number | 耗时（ms） | 性能相关日志 |
| `errorKey` | string | 错误标识 | error 级别日志 |
| `detail` | string | 补充详情（简短） | 需要额外信息时 |

### 3.3 traceId 生成规则

格式：`{taskId}_{timestamp_ms}_{random4}`

示例：`914895608057201063_1773048240123_a3f2`

生成时机：MQTT 收到任务时生成，贯穿整条链路。非 MQTT 任务触发的操作（如缓存刷新），traceId 留空。

## 四、实施步骤

### 第一步：JSON SLS 日志上报基础设施

**目标**：创建新的 JSON 日志上报方法，与旧方法并存，不影响现有功能。

**预计工作量**：0.5-1 天

#### 1.1 修改 `src/init/slsLog.js`

新增 `customLogV2(jsonObj)` 方法：

- 接收一个 JSON 对象作为参数
- 将 JSON 对象的每个字段通过 `SlsLogger.createField(key, value)` 转为 SLS 字段
- 通用字段（user、gid、version、sonVersion、registryCount）在方法内部自动注入，调用者不需要传
- `ts` 和 `seq` 在方法内部自动生成
- 调用 `slsClient.log(message, ...fields)` 上报（其中 message 为 `action` 字段的值，fields 为所有 createField）
- `seq` 使用模块级全局计数器，每次调用自增

需要注意的实现细节：
- 确保 `slsClient` 已初始化，未初始化时走 `initSlsLog()` 流程（与现有 `customLog` 一致）
- JSON 对象中值为 `undefined` 或 `null` 的字段不上报，避免无效字段
- 对 `detail` 字段做长度截断（如最大 500 字符），防止单条日志过大

#### 1.2 修改 `src/init/log.js`

新增 `customLogV2(jsonObj)` 方法：

- 调用 `slsLogUtil.customLogV2(jsonObj)` 上报 SLS
- 同时将 JSON 对象格式化为可读字符串，调用 `log[level](formattedMessage)` 写入本地 electron-log
- 本地日志格式建议：`[${module}] [${action}] ${关键字段} ${detail}`
- `level` 从 `jsonObj.logLevel` 读取，默认为 `'info'`
- 当 `logLevel === 'error'` 时，electron-log 写入 `log.error(errorKey, formattedMessage)`（与现有行为一致）

#### 1.3 新增 `src/init/traceIdGenerator.js`（可选，也可以内联在 slsLog.js）

traceId 生成工具：

- 提供 `generateTraceId(taskId)` 函数
- 格式：`{taskId}_{Date.now()}_{4位随机字符}`
- 提供 `traceIdStore` 对象，用于在链路中传递 traceId（后续步骤使用）

#### 1.4 验证

- 写一个简单的测试调用 `logUtil.customLogV2({ module: 'Test', action: '测试', wxid: 'test_wxid' })`
- 确认 SLS 上能看到独立字段（ts、seq、module、action、wxid 等）
- 确认本地日志文件中有格式化后的文本
- 确认旧的 `logUtil.customLog()` 不受影响

**mqttClientBase.js — MQTT 接收任务**

**dispatchInBound.js — 任务分发与缓存**

当前日志：

- 各种错误分支日志

改造要求：
1. 入口日志：module=DispatchCenter, action=DISPATCH_IN, wxid, taskId, traceId, msgType
2. **降噪**：移除完整 JSON 输出，只记录关键字段

4. 错误分支保留，改为 JSON 格式

**reverseSend.js — 发送给逆向**

当前日志：
- `ReverseSendService message: wxId: [wxid-xxx], message:{完整JSON}`
- `[发送逆向type]=xxx [wxid-xxx] channelId:xxx taskId:xxx`
- `[发送消息给逆向] wxid: [wxid-xxx] type=[xxx] message: {完整JSON}`
- taskId 锁相关日志

改造要求：
1. 将当前的 3 条发送日志**合并为 1 条**：module=ReverseSend, action=SEND, wxid, taskId, traceId, msgType, channelId
2. **降噪**：不再输出完整 message JSON，只记录 type、taskId、target 等关键字段
3. taskId 锁冲突日志保留：action=SEND_BLOCKED, logLevel=warn, taskId, traceId

**clibrary.js — IPC 发送**

当前日志：
- `[发送逆向type]=xxx [wxid-xxx] channelId:xxx taskId:xxx`（error 级别，写入本地文件）

改造要求：
1. 替换为 JSON 日志：module=IPC, action=SEND, wxid, taskId, traceId, msgType, channelId, logLevel=info
2. 注意：当前此处 level 是 error（会写入本地文件），改造后应改为 info（因为发送成功不是 error）。如果仍需本地文件记录，可通过 logLevel=info 配合后续本地日志级别调整实现
3. ping 类型消息的日志可以跳过不打（降噪）

#### 2.3 traceId 传递方式

在链路 1 中，traceId 需要从 MQTT 接收处一直传递到 IPC 发送处。传递路径：

```
mqttClientBase（生成 traceId，附加到 serverTaskBO）
  → mqttChatService（从 serverTask 取 traceId，写入 clientTask）
    → clientTaskFactory（将 traceId 写入 clientTask 对象）
      → cloudFlowInBound（从 message 中解析 traceId）
        → dispatchInBound（从 jsonObject 中取 traceId）
          → reverseSend（从 message 中取 traceId）
            → clibrary（从 jsonObj 中取 traceId）
```

实现方式：在 `clientTask` / `serverTaskBO` 等数据对象中增加 `traceId` 字段，沿链路透传。

当前日志（sendmessage/recvmsg 路径上的主要日志）：
- `[DispatchCenter] [msgHandleBase]: outBoundMsg sendType = xxx, jsonObject: {完整JSON}`
- `[DispatchCenter] [msgHandleBase] 执行消息转化处理responseMsgHandler：`
- `[DispatchCenter] [msgHandleBase]: responseMsgHandler jsonObject.type = xxx`
- `[DispatchCenter] [msgHandleBase] businessHandler init`
- `[DispatchCenter] [msgHandleBase] outBoundByBroadcast`

- `[msgConvert][微信4.0] recvmsg继承第一条taskId: xxx`

1. **入口日志**（outBoundMsg）：module=MsgHandler, action=MSG_RECV, wxid, msgType(=sendType), taskId(如有), reportId

   - taskId 继承时：action=TASKID_INHERIT, taskId, traceId, fromType=sendmessage, toType=recvmsg
4. **thirdMsgHandler** 中的日志：action=MSG_COMPLETE, taskId, traceId, msgType, status

注意事项：
- msgHandleBase.js 代码非常复杂，改造时只动 sendmessage/recvmsg 的 `if` 分支
- 其他消息类型的日志暂时不动
- 所有 `logUtil.customLog` 的 error 级别日志保留，改为 JSON 格式

**wxMsgHandle.js — Wx4.0 recvmsg 匹配**

当前日志：
- `[msgConvert][微信4.0] recvmsg继承第一条taskId: xxx`

改造要求：
1. taskId 继承成功时，已在 msgHandleBase 中通过 action=TASKID_INHERIT 记录，此处可以不重复记录
2. 如果 `node.first.taskId` 为空（继承失败），记录 warn 日志：action=TASKID_INHERIT_FAIL, wxid, matchKey

**cloudFlowOutBound.js — 云端出站**

当前日志：
- `[CloudFlowHandler]:outBound执行convert-service:{完整JSON}`

改造要求：
1. 替换为 JSON 日志：module=CloudFlowHandler, action=OUTBOUND, wxid, msgType, taskId, traceId
2. **关键降噪**：不输出完整消息 JSON

**wx4RecvmsgResponse.js — recvmsg 回执处理**

当前日志可能包含 taskId 和 status 信息。

改造要求：
1. 记录一条：module=RecvmsgResponse, action=TASK_REPORT_PREPARE, wxid, taskId, traceId, status, msgType

**baseConvertResponse.js — 604 消息构造**

当前日志：
- `[msgConvert]taskId is null:xxx`（taskId 为空时）
- 构造 604 消息相关日志

改造要求：
1. taskId 为空时：module=TaskReport, action=TASKID_MISSING, logLevel=error, wxid, msgType, detail(=消息摘要)
2. 604 消息构造成功：module=TaskReport, action=TASK_REPORT_SEND, wxid, taskId, traceId, status, msgType

**mqttSend.js + mqExcuteMsg.js — MQTT 上报**

当前日志：
- `[wxid-xxx] 上报MQTT任务结束,taskId: xxx, messsage: {完整JSON}`

改造要求：
1. 替换为 JSON 日志：module=MqttSend, action=TASK_REPORT_COMPLETE, wxid, taskId, traceId, type=604, status
2. **降噪**：不输出完整 clientMsg JSON

#### 3.3 链路 2 中 traceId 的获取方式

链路 2 中 traceId 的来源：

- **sendmessage 响应**：逆向返回的 sendmessage 可能不包含 traceId（取决于是否传给了逆向），但包含 taskId。可通过 taskId 从 `GalaxyTaskCache` 中查回 traceId
- **建议**：在第二步的 `dispatchInBound.js` 中，将 `{taskId → traceId}` 映射存入一个新的缓存（如 `GalaxyTaskCache.TASK_TRACE_MAP`）
- 在链路 2 的各环节中，有 taskId 时就从缓存中取 traceId

- [ ] 在 SLS 上通过 `taskId: xxx` 可以查到从 MQTT 接收到云端上报的完整链路
- [ ] 一个私聊/群聊发消息任务的完整链路日志从当前的 20-30 条减少到 8-10 条
- [ ] `module: MsgHandler AND action: MSG_MATCH_FAIL` 可以快速定位三段式匹配失败
- [ ] `action: TASKID_MISSING` 可以快速发现 taskId 丢失的情况
- [ ] DispatchCenter 的中间过程日志（`businessHandler init`、`outBoundByBroadcast` 等）不再出现在 SLS 中
- [ ] 完整 JSON 对象不再出现在 SLS 日志中（msgHandleBase、cloudFlowOutBound 等处）

### 第四步：taskId 全链路贯通（依赖逆向 DLL 修改）

**目标**：让逆向的 recvmsg 响应携带 taskId，并在微信助手侧适配，使 taskId 在整条链路中不再断裂。

**预计工作量**：1-2 天（Node.js 侧，不含 C++ DLL 修改）

**前置条件**：逆向团队已完成 recvmsg 中增加 taskId 字段的修改

#### 4.1 逆向侧改造要求（提供给逆向团队的需求）

当前 sendmessage 响应格式（已包含 taskId）：
```
{
  "type": "sendmessage",
  "taskId": 914895608057201063,
  "status": 0,
  "ownerWxId": "wxid_xxx",
  "data": { "to": "wxid_xxx", "content": "..." },
  ...
}
```

当前 recvmsg 响应格式（缺少 taskId）：
```
{
  "type": "recvmsg",
  "status": 2,
  "from": "wxid_xxx",
  "to": "wxid_xxx",
  "content": "...",
  "msgtype1": 1
}
```

需要改为（增加 taskId 字段）：
```
{
  "type": "recvmsg",
  "status": 2,
  "taskId": 914895608057201063,    ← 新增
  "from": "wxid_xxx",
  "to": "wxid_xxx",
  "content": "...",
  "msgtype1": 1
}
```

逆向侧实现思路：逆向在处理 sendmessage 任务时，将 taskId 暂存；当收到对应的 recvmsg 事件时，将暂存的 taskId 附加到 recvmsg 消息中。

#### 4.2 Node.js 侧适配

**wxMsgHandle.js — isWx4RecvmsgComplete()**

修改 taskId 继承逻辑：

1. 优先使用 recvmsg 自身的 taskId（逆向新增的字段）：`if (msgObj.taskId) { /* 直接使用 */ }`
2. 如果 recvmsg 自身没有 taskId（兼容旧版逆向），再走现有的继承逻辑：`if (node.first.taskId && !msgObj.taskId) { msgObj.taskId = node.first.taskId; }`
3. 记录 JSON 日志：
   - 使用自身 taskId 时：action=TASKID_DIRECT, source=recvmsg
   - 走继承逻辑时：action=TASKID_INHERIT, source=sendmessage
   - 都没有 taskId 时：action=TASKID_MISSING, logLevel=warn

**clibrary.js — IPC 接收**

修改接收日志，增加 taskId 字段的提取：

1. 对于 recvmsg 类型，检查是否包含 taskId 字段
2. 在 JSON 日志中记录 taskId（如果有）

#### 4.3 traceId 关联优化

有了 recvmsg 中的 taskId 后，traceId 的获取变得更可靠：

1. recvmsg 有 taskId → 从 `TASK_TRACE_MAP` 取 traceId → 所有日志都能带上 traceId
2. 不再依赖三段式匹配来获取 taskId，三段式匹配仅用于消息聚合（将 sendmessage 和 recvmsg 关联起来）

#### 4.4 验证清单

- [ ] 逆向返回的 recvmsg 消息中包含正确的 taskId
- [ ] SLS 中 recvmsg 相关日志都有 taskId 和 traceId
- [ ] 通过 `traceId: xxx` 可以查到完整的"MQTT 接收 → 逆向发送 → 逆向返回 → 云端上报"全链路
- [ ] 旧版逆向（不带 taskId 的 recvmsg）仍然正常工作（通过 fallback 继承逻辑）

### 第五步：通用日志降噪

**目标**：处理非任务链路的高频噪音日志，如心跳、IPC 连接状态、空操作推送等。

**预计工作量**：1 天

**说明**：此步骤修改的是所有任务类型共享的公共代码，不限于私聊/群聊。

#### 5.1 IPC-CONNECT-SUCCESS 降噪

**文件**：`src/msg-center/core/reverse/asyncSelectTask.js`（或实际打印此日志的位置）

**当前问题**：每次 IPC 轮询成功都打印，1 个号每秒可产生多条。

**改造方案**：
- 首次连接成功：打印一条 module=IPC, action=CONNECT_SUCCESS, wxid
- 断开后重连成功：打印一条 module=IPC, action=RECONNECT_SUCCESS, wxid
- 连续成功时：不打印
- 实现方式：维护一个 `lastConnectStatus` 状态，仅在状态变化时打印

#### 5.2 ping/pong 日志降噪

**文件**：涉及 `reverseSend.js`（发送 ping）、`clibrary.js`（接收 pong）、`dispatchOutBound.js`（分发 pong）、`msgHandleBase.js`（处理 pong）、`cloudFlowOutBound.js`（pong 出站）

**当前问题**：每 10 秒一个心跳周期，每次产生约 10 条日志（发送 ping + 接收 pong + DispatchCenter 5-6 条中间日志 + CloudFlowHandler）。

**改造方案**：
- ping 发送：不记录 SLS 日志（仅本地）
- pong 接收：不记录 SLS 日志（仅本地）
- pong 在 DispatchCenter 和 CloudFlowHandler 中的中间日志：全部不记录 SLS
- 心跳超时：记录 module=Heartbeat, action=PONG_TIMEOUT, logLevel=warn, wxid
- 实现方式：在各处理节点判断 `type === 'pong' || type === 'ping'` 时跳过 SLS 上报

#### 5.3 HeartBeatTimer 日志降噪

**文件**：心跳定时器相关文件

**当前问题**：每 10 秒记录 `[HeartBeatTimer] [timestamp | timestamp] registry:[processId]`。

**改造方案**：
- 正常心跳：不记录 SLS 日志
- 异常情况（registry 数量变化、心跳检测异常）：记录 JSON 日志

#### 5.4 空操作日志静默

**文件**：涉及群信息推送、好友列表推送、内存队列等

需要静默的日志：
- `[chatroomUpdateFront] wxId = [wxid-xxx] sendMsgs = []`（空列表）
- `[chatroomUpdateFront] start time` / `end time`（无实际操作时不记录）
- `[friendUpdateFront] keys = [[]]`（空列表）
- `[friendUpdateFront] start time` / `end time`（无实际操作时不记录）
- `[MemoryQueueExecute] no task sleep`

**改造方案**：
- 在输出前判断是否有实际数据，为空则跳过
- `chatroomUpdateFront`：`if (sendMsgs.length === 0) return;`（在日志输出前判断）
- `friendUpdateFront`：`if (keys.flat().length === 0) return;`
- `MemoryQueueExecute`：移除 `no task sleep` 日志

#### 5.5 验证清单

- [ ] 空闲状态下（无 MQTT 任务），SLS 日志量相比改造前减少 80% 以上
- [ ] ping/pong 不再出现在 SLS 中
- [ ] IPC-CONNECT-SUCCESS 在持续连接状态下不再出现
- [ ] 空列表推送日志不再出现
- [ ] 心跳超时等异常情况仍能在 SLS 中查到
- [ ] 本地 electron-log 不受影响（仍然只记录 error 级别）

## 七、注意事项

### 7.1 兼容性

- 新的 `customLogV2` 与旧的 `customLog` 并存，不影响其他模块
- 只有私聊/群聊发消息链路使用新方法，其他链路保持原样
- 旧版逆向（recvmsg 不带 taskId）通过 fallback 逻辑兼容

### 7.2 风险控制

- 每完成一个步骤就上线验证，不要攒到一起上线
- 第一步完成后，在测试环境验证 SLS 多字段查询是否正常工作
- 第二步和第三步分别上线，确保各自独立可回滚
- msgHandleBase.js 改动要特别谨慎，只改 sendmessage/recvmsg 分支

### 7.3 后续扩展

本次只处理私聊发消息和群聊发消息两种任务类型。后续可按相同模式扩展到：

- 好友验证通过（mqttFriendPassService）
- 群邀请（mqttJoinChatroomService）
- 群踢人（mqttKickOutService）
- 好友列表上传（mqttFriendListService）
- 企业微信相关任务（mqttWorkWxChatService 等）
- ...

每种任务类型的改造模式相同：识别链路涉及的文件 → 替换日志方法 → 聚合中间日志 → 确保 taskId/traceId 贯穿。
