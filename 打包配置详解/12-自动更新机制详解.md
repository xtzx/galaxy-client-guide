# 12 - 自动更新机制详解

## 一、结论先行

**项目存在完整的自动更新逻辑**，但实现方式较为特殊：

| 更新方案 | 是否存在 | 是否实际使用 | 说明 |
|---------|---------|-------------|------|
| electron-updater 标准流程 | ✅ 有代码 | ❌ 未被调用 | `setUpdateUrl` + `checkForUpdate` 注册了但渲染进程从未调用 |
| 自定义下载更新流程 | ✅ 有代码 | ✅ 实际使用 | 通过 Apollo 配置获取下载 URL，自行下载 .exe/.zip 安装包 |

> **核心发现**：项目引入了 `electron-updater` 包，注册了标准更新事件，但实际走的是"从服务端获取配置 → 自行下载安装包 → 打开/解压安装"的自定义路径。`electron-updater` 的 `autoUpdater.checkForUpdates()` 在渲染进程中**从未被调用**。

### 3.2 更新检查触发时机

更新检查在以下场景触发：

#### 场景 1：登录成功后立即检查

```javascript
// galaxy/src/entries/menu/App.js（约第 878 行）
handleUpdate({
    username: user,
    majorVersion: getGlobalData('clientVersion'),
});
```

#### 场景 2：定时轮询检查

```javascript
// galaxy/src/entries/menu/App.js
const UPDATE_INTERVAL = 600000;      // 10 分钟
const handleUpdateTime = 24 * 60 * 60 * 1000;  // 24 小时

const checkDailyUpdate = () => {
    const { delayUpdateTime, lastHandleTime } = getGlobalData('updateConf') || {};

    // 条件1：距上次检查超过 24 小时
    if (lastHandleTime && lastHandleTime + handleUpdateTime < Date.now()) {
        handleUpdate({ username, majorVersion });
    }
    // 条件2：到了预定更新时间
    else if (delayUpdateTime && new Date(delayUpdateTime).getTime() < Date.now()) {
        handleUpdate({ username, majorVersion });
    }
    // 条件3：普通检查（但标记为 isCheckUpdate，不弹窗）
    else {
        handleUpdate({ username, majorVersion }, false, true);
    }

    // 10 分钟后再次检查
    updateTimer = setTimeout(checkDailyUpdate, UPDATE_INTERVAL);
};
```

#### 场景 3：用户手动检查

```javascript
// galaxy/src/entries/menu/App.js
const checkUpdateVersion = (isLatestVersion = false) => {
    handleUpdate({ username, majorVersion }, isLatestVersion);
};
```

### 3.3 handleUpdate() —— 更新决策核心

这是整个更新系统最关键的函数，执行以下步骤：

```
handleUpdate()
    │
    ├─ 1. 调用后端 API：checkUpdate()
    │     GET /tongbao/sales/tool/app/updateConf
    │     → 获取 clientBaseInfo（当前客户端基本信息）
    │
    ├─ 2. 获取 Apollo 远程配置
    │     getWechatAssistApollo('client.wxzs.update.conf')
    │     → 获取 updateVersionConf（版本更新策略）
    │
    ├─ 3. 版本比对
    │     ├─ 解析 versionConf 中的可用版本列表
    │     ├─ findNearestMatchingVersion() 或 getLatestVersion()
    │     ├─ 判断当前版本是否已是最新
    │     └─ 检查是否需要回退（revertConf）
    │
    ├─ 4. 提取更新配置
    │     updateConf = revertConf[currentVersion]
    │                  || sourceConf[currentVersion]
    │                  || sourceConf.default
    │     → { tipsContent, tipsTitle, isForceUpdate, updateUrl,
    │        silentUpdate, isReverseFile, activeUpdatePrompt }
    │
    ├─ 5. 保存更新状态到 globalData
    │     setGlobalData('updateConf', { silentUpdate, isReverseFile, ... })
    │
    └─ 6. 执行更新策略
         │
         ├─ [静默更新] silentUpdate && !isReverseFile
         │   → ipc.send('checkDownloadFile', updateUrl)
         │   → 用户无感知，后台下载
         │
         ├─ [弹窗更新] !silentUpdate
         │   → Modal.confirm() 显示更新弹窗
         │   → 用户点击"立即更新" → ipc.send('checkDownloadFile', updateUrl)
         │   → 用户点击"稍后更新" → 关闭弹窗
         │
         └─ [强制更新] isForceUpdate = true
             → 弹窗无"稍后更新"按钮
             → 或到达 lastUpdateTime 后自动设为强制
```

### 3.4 下载流程详解

#### 3.4.1 主进程接收下载请求

```javascript
// galaxy-client/src/event/updater.js
ipc.on('checkDownloadFile', (event, downloadUrl, reStartType) => {
    new DownloadFile(BrowserWindow.fromWebContents(event.sender), downloadUrl).start();
});
```

#### 3.4.2 DownloadFile 类执行流程

```
DownloadFile.start()
    │
    ├─ 1. 通知渲染进程：updateStart
    │
    ├─ 2. 检查历史下载文件
    │     stat(historyFilePath)
    │     ├─ .exe 文件已存在 → 直接发送 update-downloaded（跳过下载）
    │     └─ .zip 文件已存在 → 删除后重新下载
    │
    ├─ 3. 执行下载
    │     mainWindow.webContents.downloadURL(downloadUrl)
    │
    ├─ 4. 监听 will-download 事件
    │     ├─ item.setSavePath(filePath)
    │     ├─ item.on('updated') → 发送 downloadProgress（百分比）
    │     └─ item.once('done')
    │         ├─ completed:
    │         │   ├─ .zip → stopBsInject() + 10秒等待 + extractZip()
    │         │   │         → 发送 update-downloaded
    │         │   └─ .exe → 直接发送 update-downloaded
    │         └─ interrupted → 发送 updateError
    │
    └─ 5. 错误处理 → 发送 updateError
```

#### 3.4.3 下载进度反馈

```javascript
// 主进程 → 渲染进程
item.on('updated', (event, state) => {
    if (state === 'progressing') {
        const percent = (item.getReceivedBytes() / item.getTotalBytes() * 100).toFixed(0);
        utils.sendToRenderMsg('downloadProgress', percent);
    }
});
```

```javascript
// 渲染进程接收进度
ipc.on('downloadProgress', (event, updatePercent) => {
    setPercent(updatePercent);  // 更新进度条
    if (updatePercent >= 100) {
        reportLog({ name: '自动更新成功' });
        message.success('更新成功，准备安装最新版本');
    }
});
```

### 3.5 安装/应用更新流程

下载完成后，渲染进程收到 `update-downloaded` 事件：

```javascript
// galaxy/src/entries/menu/App.js
ipc.on('update-downloaded', (event, data) => {
    const isFullUpdate = data.filePath.indexOf('.exe') > -1;
    const isWxUrl = isWeChatFamilyInstaller(data.filePath);

    // .exe 文件：用系统默认方式打开（运行安装程序）
    if (isFullUpdate) {
        ipc.send('open-file', data.filePath);
    }

    setUpdateVisible(false);

    const updateConf = getGlobalData('updateConf');

    // 静默更新 + 自动重载：直接重启应用
    if (updateConf && updateConf.silentUpdate && updateConf.autoReload) {
        ipc.send('app-relaunch');
        return;
    }

    // 微信/企微安装包：刷新页面
    if (isWxUrl) {
        window.location.reload();
        return;
    }

    // 停止 BasicService 进程
    ipc.send('stopBs64Inject');
    ipc.send('stopBsInject');

    // .exe 全量更新：退出应用（用户手动安装）
    if (isFullUpdate) {
        ipc.send('app-exit');
    }
    // .zip 增量更新：重启应用（解压后直接生效）
    else {
        ipc.send('app-relaunch');
    }
});
```

#### 安装方式对照

| 更新包类型 | 安装方式 | 用户操作 |
|-----------|---------|---------|
| `.exe` 全量安装包 | `shell.openPath()` 启动安装程序 → `app.quit()` | 用户需完成安装向导 |
| `.zip` 增量更新包 | `extractZip()` 解压覆盖 → `app.relaunch()` | 自动完成，应用重启 |
| 微信/企微安装包 | `shell.openPath()` 启动安装程序 | 刷新页面，用户手动安装 |
| 静默 + autoReload | 后台下载 → `app.relaunch()` | 完全无感知 |

## 四、更新 URL 配置链路

### 4.1 build.yml 中的 publish 配置（未生效）

```yaml
# galaxy-client/config/weixinzhushou/build.yml
publish:
  provider: generic
  url: http://127.0.0.1   # ⚠️ localhost，electron-updater 无法使用
  channel: latest
```

这个配置是为 `electron-updater` 的标准流程准备的，但由于 URL 为 `127.0.0.1`，实际上**无法工作**。

### 4.2 urls.js 中的更新 URL 映射

```javascript
// galaxy-client/src/common/urls.js
updateUrlMap: {
    test: 'oss://gh-fe/wxzs-client/test',
    prod: 'oss://gh-fe/wxzs-client/prod',
    vt:   'http://172.16.35.27:9091/salestool/vt',
},
get updateUrl() {
    return this.updateUrlMap[this.type];
}
```

> 注意：`oss://` 协议格式并非标准 HTTP URL，这些是 OSS 存储路径，用于 `electron-updater` 的 `setFeedURL`。但由于 `setUpdateUrl` IPC 在渲染进程中从未被调用，这些配置**也未生效**。

### 4.3 实际生效的更新 URL 来源

```
Apollo 远程配置
  key: 'client.wxzs.update.conf'
      │
      ▼
  updateVersionConf.versionConf[targetVersion].sourceConf[currentVersion].updateUrl
      │
      ▼
  实际的下载 URL（如 https://xxx.oss-cn-beijing.aliyuncs.com/...）
```

**结论**：实际的更新下载地址完全由 Apollo 远程配置动态控制，`build.yml` 和 `urls.js` 中的更新 URL 配置均未参与实际更新流程。

## 五、electron-updater 标准流程分析（已注册但未使用）

### 5.1 注册的 IPC 事件

```javascript
// galaxy-client/src/event/updater.js

// 事件1：设置更新源（从未被渲染进程调用）
ipc.on('setUpdateUrl', (event, version) => {
    let feedURL = '';
    if (type === 'prod') {
        feedURL = `${updateUrl}/${version}`;
    } else {
        const match = version.match(/^\d+\.\d+\.\d+/);
        if (match) {
            feedURL = `${updateUrl}/${match[0]}/${version}/${appConfig.elfkey}`;
        }
    }
    autoUpdater.setFeedURL(feedURL);
});

// 事件2：执行检查（从未被渲染进程调用）
ipc.on('checkForUpdate', () => {
    autoUpdater.checkForUpdates();
});

// 事件3：检查进程状态并安装（被 wsKillJava 间接触发）
ipc.on('checkJava', (event, flag) => {
    Promise.all([
        utils.judgeProcessExist('javaw.exe'),
        utils.judgeProcessExist('weixin.exe')
    ]).then(result => {
        if (result[0].status === 0 && result[1].status === 0) {
            callbackFun();                // 设置 updateFlag = true
            autoUpdater.quitAndInstall(); // ⚠️ 前提：需要先 checkForUpdates 成功
        } else {
            flag === 'first'
                ? utils.sendToRenderMsg('wsKillJava')
                : utils.sendToRenderMsg('killJavaFail');
        }
    });
});
```

### 5.2 未使用原因分析

| 方面 | 说明 |
|-----|------|
| `setUpdateUrl` | 在 galaxy/src 的整个渲染进程代码中搜索不到调用 |
| `checkForUpdate` | 同上，从未被调用 |
| `autoUpdater` 事件 | 未监听 `update-available`、`download-progress`、`update-downloaded` 等标准事件 |
| `publish.url` | 配置为 `127.0.0.1`，即使调用也无法工作 |

> **推测**：项目早期可能计划使用 `electron-updater` 标准流程（配合 `latest.yml` 自动检查），但后来改为 Apollo 配置驱动的自定义下载方案。旧代码未清理。

### 5.3 checkJava 中的 quitAndInstall 问题

```javascript
// checkJava 中调用了 autoUpdater.quitAndInstall()
```

## 八、存在的问题与风险

### 8.1 架构问题

| 问题 | 严重程度 | 说明 |
|-----|---------|------|
| electron-updater 代码残留 | 🟡 中 | `setUpdateUrl`、`checkForUpdate`、`quitAndInstall` 注册但未使用，增加维护混乱 |
| 两套更新机制并存 | 🟡 中 | `electron-updater` 标准方案和自定义下载方案共存，职责不清 |
| build.yml publish 配置失效 | 🟡 中 | `url: http://127.0.0.1` 使 `latest.yml` 自动更新完全无效 |
| urls.js updateUrlMap 未使用 | 🟡 中 | `oss://` 协议的 URL 映射从未被实际调用 |

### 8.2 实现问题

| 问题 | 严重程度 | 说明 |
|-----|---------|------|
| .exe 缓存导致跳过下载 | 🔴 高 | 如果本地已存在同名 .exe 文件，直接触发 `update-downloaded`，可能安装旧版本 |
| 10 秒硬编码等待 | 🟡 中 | ZIP 解压前等待 BasicService 停止的时间不确定，可能不够 |
| `checkJava` 中 `quitAndInstall` 无效 | 🟡 中 | 未经 `checkForUpdates` 的 `quitAndInstall` 可能静默失败 |
| 下载错误未清理临时文件 | 🟡 中 | 下载中断后部分文件未删除，可能影响下次更新 |
| `log.error` 记录正常信息 | 🟢 低 | `downloadFile.js` 中用 `log.error` 记录下载路径 |
| `reStartType` 参数未使用 | 🟢 低 | `checkDownloadFile` 接收但忽略了重启类型参数 |

### 8.3 安全问题

| 问题 | 严重程度 | 说明 |
|-----|---------|------|
| 更新包未做签名验证 | 🔴 高 | 自定义下载的 .exe/.zip 没有验证数字签名，可能被中间人攻击 |
| 更新 URL 通过 HTTP 传输 | 🔴 高 | Apollo 返回的 `updateUrl` 如果是 HTTP 协议，存在篡改风险 |
| `build.yml` 的 `verifyUpdateCodeSignature: true` 未生效 | 🟡 中 | 标准 `electron-updater` 未使用，该配置形同虚设 |

## 九、优化建议

### 9.1 短期优化（低风险）

1. **清理 electron-updater 残留代码**
   - 移除 `setUpdateUrl`、`checkForUpdate` IPC 事件
   - 移除 `checkJava` 中的 `autoUpdater.quitAndInstall()` 调用
   - 或者明确标注为"预留接口"

2. **修复 .exe 缓存判断**
   ```javascript
   // 当前：同名文件存在就跳过下载
   // 建议：比对文件大小或 hash 值
   if (stats && this.historyFilePath.endsWith('.exe')) {
       // 应该检查文件完整性，而不是直接使用
       await remove(this.historyFilePath);
   }
   ```

3. **统一日志级别**
   - `downloadFile.js` 中的 `log.error` 改为 `log.info`

### 9.2 中期优化（中等风险）

1. **增加更新包签名验证**
   ```javascript
   // 下载完成后验证签名
   const isValid = await verifyFileSignature(filePath, expectedPublisher);
   if (!isValid) {
       utils.sendToRenderMsg('updateError', { reason: 'signature_invalid' });
       await remove(filePath);
       return;
   }
   ```

2. **改进进程等待逻辑**
   ```javascript
   // 替代硬编码 10 秒等待
   async function waitForProcessExit(processName, maxWait = 30000, interval = 1000) {
       const start = Date.now();
       while (Date.now() - start < maxWait) {
           const result = await judgeProcessExist(processName);
           if (result.status === 0) return true;
           await new Promise(r => setTimeout(r, interval));
       }
       return false;
   }
   ```

3. **清理 build.yml 和 urls.js 的无效配置**
   - `publish.url` 改为实际 URL 或移除
   - `updateUrlMap` 如果不用则移除

### 9.3 长期优化（高风险，需架构调整）

1. **统一更新方案**：要么完全使用 `electron-updater`（配置正确的 publish URL），要么完全使用自定义下载方案，消除二者并存的混乱
2. **实现差分更新**：当前 ZIP 增量方案是简单文件覆盖，可考虑使用 bsdiff 等差分更新算法减少下载量
3. **增加回滚机制**：更新失败时自动回滚到上一个可用版本
