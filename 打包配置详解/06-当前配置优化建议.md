## 二、逐项分析与优化方案

### P1：证书密码明文存储 🔴

**当前配置**：

```yaml
win:
  certificateFile: newSignCert.pfx
  certificatePassword: gaotuketang
```

**风险**：
- 配置文件通常会提交到代码仓库
- 任何有仓库访问权限的人都能获取证书密码
- 证书泄露可能被用于签名恶意软件

**优化方案**：

```yaml
# 方案 A（推荐）：完全移除，改用环境变量
win:
  certificateFile: newSignCert.pfx
  # 不再配置 certificatePassword
  # 通过环境变量 CSC_KEY_PASSWORD 传入
```

```bash
# CI/CD 中设置环境变量
export CSC_KEY_PASSWORD=your_password
# 或 Windows
set CSC_KEY_PASSWORD=your_password
```

```yaml
# 方案 B：证书文件也通过环境变量传入
win:
  sign: sign-script/sign.js
  # 在 sign.js 中从环境变量读取证书信息
```

```javascript
// sign-script/sign.js（改进版）
exports.default = async function(configuration) {
  const certFile = process.env.CSC_LINK || 'newSignCert.pfx';
  const certPassword = process.env.CSC_KEY_PASSWORD;
  
  if (!certPassword) {
    console.warn('CSC_KEY_PASSWORD not set, skipping signing');
    return;
  }
  
  // 执行签名逻辑...
}
```

**迁移步骤**：
1. 在 CI/CD 系统中配置 `CSC_KEY_PASSWORD` 环境变量（作为 Secret）
2. 从 `build.yml` 中删除 `certificatePassword` 行
3. 测试 CI/CD 构建流程是否正常
4. 确认后将证书文件（`.pfx`）也通过 Secret 管理（可选）

### P2：asar: false 导致源码可见 🔴

**当前配置**：

```yaml
asar: false
```

**风险**：
- 用户可以直接查看和修改 JavaScript 源码
- 商业逻辑、API 密钥、服务器地址暴露
- 可能被竞品分析或恶意利用

**为什么当前设为 false**：

项目使用了原生 DLL（`PipeCore.dll`、`BasicService.exe` 等），这些文件无法被包含在 asar 中。另外 `ffi-napi`、`sqlite3` 等 native 模块也需要在 asar 外运行。

**优化方案**：

```yaml
# 开启 asar，但将 native 模块解压
asar: true
asarUnpack:
  - "**/*.node"                       # 所有 native addon
  - "**/node_modules/sharp/**"        # 图片处理库
  - "**/node_modules/sqlite3/**"      # SQLite
  - "**/node_modules/ffi-napi/**"     # FFI
  - "**/node_modules/ref-napi/**"     # ref-napi
  - "**/node_modules/ref-struct-napi/**"
  - "**/node_modules/better-sqlite3/**"
  - "**/extraResources/**"            # 额外资源
```

**验证步骤**：
1. 修改配置后执行构建
2. 运行安装包，检查所有功能是否正常
3. 重点测试：IPC 通信、SQLite 读写、FFI 调用
4. 如果某个模块报错，将其加入 `asarUnpack`

**注意**：由于额外资源已通过 `extraResources` 复制到 `resources/` 目录，它们本身就在 asar 之外。只需要关注 `node_modules` 中的 native 模块。

### P3：appId 末尾多余的点号 🟡

**当前配置**：

```yaml
appId: com.baijiahulian.tqclient.
```

**问题**：
- 标准反向域名格式不应以点号结尾
- 可能导致某些系统 API 行为异常（如 Windows 注册表键名）
- 不符合 macOS CFBundleIdentifier 规范

**优化方案**：

```yaml
appId: com.baijiahulian.tqclient
```

**⚠️ 重大注意事项**：

修改 appId **会导致以下影响**：

| 影响 | 说明 | 后果 |
|------|------|------|
| 用户数据目录变化 | `%APPDATA%/com.baijiahulian.tqclient.` → `%APPDATA%/com.baijiahulian.tqclient` | 旧数据可能无法自动迁移 |
| 自动更新识别 | electron-updater 通过 appId 识别已安装应用 | 可能无法自动更新，需要手动安装 |
| Windows 注册表 | 卸载信息的注册表路径变化 | 旧版本无法被新版本自动覆盖安装 |

**安全迁移方案**：

```nsi
; installer.nsh 中添加迁移逻辑
!macro customInit
  ; 检查旧版本的注册表项
  ReadRegStr $0 HKLM "Software\Microsoft\Windows\CurrentVersion\Uninstall\com.baijiahulian.tqclient." "UninstallString"
  ${If} $0 != ""
    MessageBox MB_YESNO "检测到旧版本，是否先卸载？" IDYES uninstall IDNO skip
    uninstall:
      ExecWait '$0 /S'
    skip:
  ${EndIf}
!macroend
```

**建议**：如果当前版本已经大规模部署，修改 appId 的风险较高。可以保留现状或在下一个大版本中处理。

### P4：publish.url 是 localhost 🟡

**当前配置**：

```yaml
publish:
  provider: generic
  url: http://127.0.0.1
  channel: latest
```

**问题**：
- 自动更新功能无法正常工作
- `latest.yml` 中的下载地址指向 localhost

**优化方案**：

```yaml
# 方案 A：配置真实的更新服务器地址
publish:
  provider: generic
  url: https://your-cdn.example.com/qz-wxzs/prod/
  channel: latest

# 方案 B：如果通过运行时代码动态设置 feedURL，
# 至少配置一个有意义的占位值
publish:
  provider: generic
  url: https://placeholder.example.com/
  channel: latest
```

**如果项目确实通过运行时代码设置更新地址**：

```javascript
// 运行时代码示例
const { autoUpdater } = require('electron-updater');

autoUpdater.setFeedURL({
  provider: 'generic',
  url: getUpdateServerUrl()  // 从 runtime.yml 或 API 获取
});
```

那么 build.yml 中的 publish 配置主要影响 `latest.yml` 的生成。即使运行时覆盖了 URL，建议 build.yml 中也配置正确的地址，方便排查问题。

### P6：allowElevation 与 requestedExecutionLevel 语义不清 🟢

**当前配置**：

```yaml
nsis:
  allowElevation: false

win:
  requestedExecutionLevel: requireAdministrator
```

**分析**：

虽然不是真正的「冲突」（见 03-nsis 文档中的详细分析），但 `allowElevation: false` 的语义是「安装程序不自行请求提权」，而 `requireAdministrator` 已经在 exe 层面强制管理员权限。这会造成阅读配置时的困惑。

**优化方案**：

```yaml
nsis:
  allowElevation: true  # 允许安装程序在需要时请求提权（更符合语义）

win:
  requestedExecutionLevel: requireAdministrator  # 保持不变
```

### P7：files 过滤规则不一致 🟢

**当前配置中的不一致**：

```yaml
files:
  - '!**/.npmrc'           # 使用 **/ 前缀 — 递归排除所有目录
  - '!/.prettierrc*'       # 使用 / 前缀 — 仅排除根目录
  - '!**/.eslintrc*'       # 使用 **/ 前缀
  - '!/.gitignore'         # 使用 / 前缀
  - '!/yarn.lock'          # 使用 / 前缀
```

**问题**：
- `**/` 前缀：递归匹配所有目录中的文件
- `/` 前缀：仅匹配项目根目录下的文件
- 混用导致意图不明确

**优化方案**：

```yaml
files:
  - '**/*'
  
  # === 排除证书文件（全局）===
  - '!**/*.pfx'
  - '!**/*.cer'
  
  # === 排除开发目录（全局）===
  - '!**/config'
  - '!**/scripts'
  - '!**/sign-script'
  - '!**/tempFile'
  
  # === 排除根目录配置文件 ===
  - '!/.npmrc'
  - '!/.yarnrc'
  - '!/.env*'
  - '!/.eslintrc*'
  - '!/.prettierrc*'
  - '!/.babelrc*'
  - '!/.gitignore'
  - '!/tsconfig.json'
  - '!/yarn.lock'
  
  # === 排除根目录脚本文件 ===
  - '!/upload.js'
  - '!/changePkgVer.js'
  - '!/webpack.electron.config.js'
  
  # === 排除文档文件（全局）===
  - '!**/changeLog.md'
  - '!**/LICENSE.md'
  - '!**/README.md'
  
  # === 排除安装脚本 ===
  - '!/installer.nsh'
```

**改进点**：
1. 统一使用 `/` 前缀排除根目录文件
2. 使用 `**/` 前缀排除需要递归搜索的文件
3. 分组添加注释，便于维护

### P9：缺少 compression 配置 🟢

**当前状态**：未配置，使用默认值 `normal`。

**优化建议**：

```yaml
# 明确配置压缩级别
compression: normal

# 如果安装包体积是关键指标，可以改为：
compression: maximum
```

| 压缩级别 | 安装包大小 | 构建时间 | 推荐场景 |
|----------|-----------|---------|---------|
| `store` | 最大 | 最快 | 本地开发测试 |
| `normal` | 中等 | 中等 | 一般发布 |
| `maximum` | 最小 | 最慢 | 正式发布、带宽敏感 |

### 3.4 考虑 afterPack 钩子

可以添加 afterPack 钩子来注入构建信息：

```yaml
afterPack: scripts/afterPack.js
```

```javascript
// scripts/afterPack.js
const fs = require('fs-extra');
const path = require('path');

exports.default = async function(context) {
  const buildInfo = {
    buildTime: new Date().toISOString(),
    version: context.packager.appInfo.version,
    platform: context.electronPlatformName,
    commit: process.env.GIT_COMMIT || 'unknown'
  };

  await fs.writeJson(
    path.join(context.appOutDir, 'resources', 'build-info.json'),
    buildInfo,
    { spaces: 2 }
  );
};
```
