# 07 — 运行时配置加载机制与开关生效链路

> 本文档解析 `runtime.yml` / `runtime.dev.yml` 中的配置变量（如 `kibana: false`、`wsLogger: true`）是如何在最终代码中生效的。它们**不是**通过 Node 环境变量注入，而是通过 YAML 文件读取 + IPC 传递到渲染进程的。

---

## 一、配置文件

### 1.1 runtime.yml — 生产环境配置

```yaml
# config/weixinzhushou/runtime.yml（部分）
switch:
    report:
        kibana: true          # Kibana 日志上报（开启）
        sentry: true          # Sentry 错误上报（开启）
        starEyes: true        # 星眼数据上报（开启）
    debug:
        wsLogger: false       # WebSocket 日志（关闭）
```

### 1.2 runtime.dev.yml — 开发环境覆盖

```yaml
# config/weixinzhushou/runtime.dev.yml
switch:
  report:
    kibana: false             # 关闭 Kibana 日志上报
    sentry: false             # 关闭 Sentry 错误上报
    starEyes: false           # 关闭星眼数据上报
  debug:
    wsLogger: true            # 开启 WebSocket 日志
```

```
│  │     event.returnValue = appConfig;  // 同步返回          │          │
│  │ })                                                       │          │
│  └──────────────────────────────────────────────────────────┘          │
│                                                                         │
│  ④ 渲染进程使用配置（galaxy/src/common/project.js）                    │
│  ┌──────────────────────────────────────────────────────────┐          │
│  │ appConfig = ipc.sendSync('get-app-config');              │          │
│  │ appSwitch = appConfig.switch;                            │          │
```

## 三、关键代码解析

### 3.1 主进程：getAppConfig()

**文件**：`galaxy-client/src/utils.js`

```javascript
exports.getAppConfig = () => {
    if (!appConfig) {
        const configRoot = `../config/${this.getProduct()}`;
        // 生产环境读取 resources/app-config.yml
        // 开发环境读取 config/{product}/runtime.yml
        const configFilePath = path.resolve(
            __dirname,
            isDev ? `${configRoot}/runtime.yml` : '../../app-config.yml'
        );
        appConfig = this.getYamlConfig(configFilePath);

        // 开发环境：加载 runtime.dev.yml 并覆盖
        if (isDev) {
            const devConfigFilePath = path.resolve(
                __dirname,
                `${configRoot}/runtime.dev.yml`
            );
            const devConfig = this.getYamlConfig(devConfigFilePath);
            Object.assign(appConfig, devConfig);  // ⚠️ 浅合并
        }
    }
    return appConfig;
};
```

**核心要点**：

| 环境 | 配置来源 | 合并方式 |
|------|---------|---------|
| 开发环境 | `runtime.yml` + `runtime.dev.yml` | `Object.assign` 浅合并 |
| 生产环境 | `app-config.yml`（打包时从 runtime.yml 复制） | 无合并 |

### 3.2 IPC 传递配置

**文件**：`galaxy-client/src/event/ipc.js`

```javascript
ipc.on('get-app-config', (event) => {
    const appConfig = getAppConfig();
    const moreConfig = { devServerRoot, server, type, mode, version };
    if (Array.isArray(app.modules)) {
        Object.assign(appConfig, moreConfig, { modules: app.modules });
    } else {
        Object.assign(appConfig, moreConfig);
    }
    event.returnValue = appConfig;  // 同步返回给渲染进程
});
```

渲染进程通过**同步 IPC**（`ipc.sendSync`）获取完整配置对象，这意味着渲染进程在启动时就拥有了所有配置数据。

### 3.3 渲染进程：project.getSwitch()

**文件**：`galaxy/src/common/project.js`

```javascript
function getAppSwitch() {
    if (!appSwitch) {
        appSwitch = window.SWITCH = getAppConfig().switch;
    }
    return _.cloneDeep(appSwitch);
}

function getProperty(obj, str) {
    const routes = str.split('.');
    let result = obj;
    while ((route = routes.shift())) {
        result = result[route];
    }
    return result;
}

const project = {
    getSwitch(str) {
        return getProperty(getAppSwitch(), str);
    },
    getConfig(str) {
        return getProperty(getAppConfig(), str);
    },
};
```

`project.getSwitch('report.sentry')` 的执行过程：

```
1. getAppSwitch() → 返回 appConfig.switch 对象
2. getProperty(switchObj, 'report.sentry')
3. 'report.sentry'.split('.') → ['report', 'sentry']
4. switchObj['report'] → { kibana: true, sentry: true, starEyes: true }
5. { kibana: true, ... }['sentry'] → true
```

## 四、各开关的具体使用位置

### 4.1 switch.report.sentry — Sentry 错误上报

**生效位置**：galaxy 前端每个入口文件的 `Sentry.init()`

```javascript
// galaxy/src/entries/sub/index.js
// galaxy/src/entries/menu/index.js
// galaxy/src/entries/load/index.js
Sentry.init({
    enabled: project.getSwitch('report.sentry'),
    // ...
});
```

| 值 | 效果 |
|-----|------|
| `true`（生产） | Sentry 正常采集错误并上报 |
| `false`（开发） | Sentry 禁用，不上报任何错误 |

### 4.2 switch.report.kibana — Kibana 日志上报

**配置位置**：`galaxy/src/config/url.config.js` 中定义了 `kibanaUrlMap`

```javascript
kibanaUrlMap: {
    prod: 'http://monitor-log.umeng100.com/monitor/log/open/tongbao/pre/report.ajax',
    test: 'http://monitor-log.umeng100.com/monitor/log/open/tongbao/pre/report.ajax',
    vt: 'http://test-monitor-log.umeng100.com/monitor/log/open/tongbao/pre/report.ajax',
```

Kibana 日志上报与 habo（哈勃）上报系统集成，`report.js` 中的 `initHaboReport()` 和 `eventReport()` 函数负责数据上报。

> **注意**：在当前代码中，`report.kibana` 开关的显式检查调用没有在 galaxy 前端代码中直接找到。它可能是通过 habo SDK 内部机制使用，或者该开关当前处于预留状态。

### 4.4 switch.debug.wsLogger — WebSocket 日志

**生效位置**：`galaxy/src/common/log.js`

```javascript
return news => {

    // 过滤 pong、recvmsg 等噪音消息
```

## 五、⚠️ 浅合并的隐患

### 5.1 Object.assign 是浅合并

```javascript
Object.assign(appConfig, devConfig);
```

`Object.assign` 只做第一层属性的覆盖，不会递归合并嵌套对象。

**当前场景**：

```yaml
# runtime.yml 的 switch
switch:
    report:
        kibana: true
        sentry: true
        starEyes: true
    debug:
        wsLogger: false

# runtime.dev.yml 的 switch
switch:
  report:
    kibana: false
    sentry: false
    starEyes: false
  debug:
    wsLogger: true
```

合并过程：
```javascript
// Object.assign 会用 devConfig.switch 整体替换 appConfig.switch
appConfig.switch = devConfig.switch;
// 最终 switch = { report: { kibana: false, sentry: false, starEyes: false }, debug: { wsLogger: true } }
```

**当前没问题**：因为 `runtime.dev.yml` 的 `switch` 结构完整，替换后所有字段都在。

### 5.2 潜在风险

如果 `runtime.dev.yml` 只覆盖部分字段：

```yaml
# 假设 runtime.dev.yml 只写了
switch:
  debug:
    wsLogger: true
```

则合并后 `switch.report` 会**丢失**（因为整个 `switch` 被替换为只有 `debug` 的对象）。

**改进建议**：使用深合并代替浅合并：

```javascript
// 使用 lodash 的 merge 代替 Object.assign
const _ = require('lodash');
_.merge(appConfig, devConfig);  // 深层递归合并
```

## 六、浏览器模式的配置

galaxy 前端还有 browser 模式（不依赖 Electron），此时无法通过 IPC 获取配置：

**文件**：`galaxy/src/common/project.browser.js` / `project.browserDev.js`

```javascript
function getAppSwitch() {
    return {
        report: {
            kibana: false,
            sentry: process.env.NODE_ENV !== 'development',
            starEyes: false
        },
        debug: {
            wsLogger: true
        }
    }
}
```

浏览器模式下，开关值是**硬编码**的，不从 YAML 配置读取。

## 七、总结：不是环境变量，是 YAML 文件读取 + IPC 传递

```
┌─────────────────────────────────────────────────────────┐
│  配置传递方式对比                                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ❌ 不是：process.env.KIBANA = 'false'                 │
│  ❌ 不是：webpack DefinePlugin 注入                    │
│  ❌ 不是：编译时常量替换                               │
│                                                         │
│  ✅ 是：                                               │
│  1. YAML 文件 → js-yaml 解析为 JS 对象                │
│  2. JS 对象缓存在主进程内存中                          │
│  3. 渲染进程通过同步 IPC 获取完整配置对象              │
│  4. 渲染进程通过 project.getSwitch() 按路径读取值      │
│                                                         │
│  这是一种「运行时配置」机制，不是「编译时注入」。      │
│  配置值在应用启动时确定，运行中不会变化（有缓存）。    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```
