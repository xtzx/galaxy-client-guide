## 一、核心架构调整

### 1.1 链路合并：5条 → 3条


| 原链路                          | 调整方案                 |
| ---------------------------- | -------------------- |
| ① 云端MQTT → 客户端 → 逆向（下行任务）    | **保留**，作为链路①         |
| ② 逆向 → 客户端 → 云端MQTT（上行回报）    | **合并②③**，作为链路②       |
| ③ 逆向 → 客户端 → 前端（WebSocket推送） | 合并入链路②               |
| ④ 前端 → 客户端 → 逆向（前端指令下行）      | **保留**，作为链路③         |
| ⑤ 客户端 → 云端HTTP（数据上报）         | **取消独立链路**，作为链路②中的附注 |


合并后链路②的名称：**「逆向 → 客户端 → 云端/前端（上行回报与推送）」**

合并理由：链路②③在代码层面共享 80% 的路径（IPC轮询 → dispatchOutBound → msgHandleBase → businessHandler），唯一分叉点是 `businessHandler` 根据 `flowSource` 路由到 `cloudFlowOutBound`（云端）或 `frontFlowOutBound`（前端）。

### 1.2 平台合并：双平台 → 单平台+标签

- 删除平台切换Tab按钮
- 微信和企微合并为一份数据

- 标题改为「Galaxy Client 数据链路分析 — 微信4.0 / 企业微信」

### 1.3 去掉 ASCII 流程图

移除 Mermaid/ASCII 视图切换按钮，仅保留 Mermaid 图。

## 二、内容优化要求

**问题**：当前很多 MQTT 任务类型（如 REFRESH_TIMELINE:143）仅在常量文件中定义了 type 值，但 `WxConvertServiceList` / `WorkWxConvertServiceList` 中没有注册对应 handler，`filter()` 永远不会匹配到。

**要求**：

- 合并微信和企微任务清单到同一表格
- 增加「平台」列（微信/企微/共用）
- 增加「状态」列：**已实现**（有handler文件，已注册到ServiceList）/ **仅定义**（仅常量定义，无处理逻辑）
- 「仅定义」行灰显，handler列显示为「无处理器（仅常量定义）」
- 所有标记必须对照源码逐一验证

**已确认的「已实现」类型（微信）**：
CHATROOM_SEND_MSG(1), FRIEND_SEND_MSG(100), UPLOAD_FRIEND_LIST(103), CHATROOM_LEAVE(7/13), CHATROOM_INVITATION_ACCEPT(27), SCAN_QRCODE_JOIN_CHATROOM(155), CHATROOM_UPDATE_NOTICE(5), CHATROOM_UPDATE_NAME(6), CHATROOM_INVITE(2/1002/1102), CHATROOM_KICK_OUT(3/33), CHATROOM_PREPARE(11), REJECT_CHATROOM_ADD_FRIENDS(28), CHATROOM_UPDATE_QRCODE_REL(12), DELETE_FRIEND_BATCH(125), MODIFY_FRIEND_REMARK(116), DELETE_FRIEND(101), FRIEND_ACCEPT_REQUEST(104), ADD_CHATROOM_FRIEND(126), MARK_AS_READ(169), UPLOAD_ALL_LABEL_INFO(119), REVOKE_MSG(170), ADD_FRIEND_BY_PHONE(111), SEARCH_FRIEND_BY_PHONE(115), FRIEND_ADD_LABEL(105), FRIEND_REMOVE_LABEL(106), CREATE_LABEL(120), DELETE_LABEL(121), CHATROOM_ADD_ADMIN(17) 等

### 2.2 链路② 上行链路 — msgHandleBase 逻辑深化

**问题**：msgHandleBase.js 是全系统最复杂的文件（1656行），当前展示不够清晰。

**要求**：看网页即可了解全部代码逻辑，需拆解为多层可展开结构：

**第1层 — 完整调用链**：

```
outBoundMsg → [notDealSet过滤] → messageHandler → responseMsgHandler → businessHandler → [云端/前端/广播]
```

**第2层 — messageHandler 过滤链**（5道过滤）：

1. `isNotContains(sendType)` — 微信: 不在CALLBACK_SET中; 企微: 非recvmsg/msgreport/sendmsg_*
2. `isSysMsg(jsonObj)` — to是'newsapp'
3. `isNotSendMsg(jsonObj)` — recvmsg且sequence≠0
4. `isNotSelfSendMsg(jsonObj, wxid)` — [微信] from≠ownerWxId的recvmsg
5. 微信4.0媒体校验 — [微信] 文件无attachid / 视频无aeskey

**第3层 — 回执路径对比表**：


| 维度    | 微信4.0（两段式）                           | 企微（三段式）             | 直通路径                                                                                                                                                       |
| ----- | ------------------------------------ | ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 第一条   | sendmessage                          | recvmsg 或 sendmsg_* | N/A                                                                                                                                                        |
| 第二条   | recvmsg → 直接完成                       | sendmsg_* 或 recvmsg | N/A                                                                                                                                                        |
| 第三条   | 无需                                   | msgreport           | N/A                                                                                                                                                        |
| Key生成 | wx4-wxId-from-to-msgType-association | wxId_clientId       | N/A                                                                                                                                                        |
| 直通类型  | N/A                                  | N/A                 | HandleDelContact, recvmsgs, CreateChatRoomResponse, HandleChatroomMemberResp, AddChatRoomMemberResponse, ModContactRemarkResponse, recvmsg(status=2且非任务回执) |


**第4层 — businessHandler 路由表**：

- flowSource=1(FRONT) → frontFlowOutBound → WebSocket推送前端
- flowSource=2(CLOUD) → cloudFlowOutBound → convert-service/response → MQTT上报
- 无flowSource → 广播（前端+云端同时推送）
- 特殊: userlist/quickSend → 直接走 cloudFlowOutBound

**第5层 — convert-service/response 处理器清单**（合并微信+企微），标注HTTP上报行为

### 2.3 链路③ 前端下行 — cmdId 处理逻辑补全

**问题**：当前只列了 cmdId 映射，但每个分支的具体执行逻辑没有总结。

**要求**：为每个 cmdId 添加完整处理逻辑摘要，所有逻辑基于源码：


| cmdId              | 处理器                            | 核心逻辑                                                 |
| ------------------ | ------------------------------ | ---------------------------------------------------- |
| getAllConfig       | getAllconfigTask.js            | 调用 frontSendService.sendGetAllConfig() 推送所有已注册逆向状态配置 |
| upload             | UploadTask.js                  | 读取本地文件路径 → 上传阿里云OSS → 返回ossUrl给前端                    |
| **forward**        | forwardTask.js                 | **需展开5个内部分支**（见下方）                                   |
| frontLogout        | FrontLogoutTask.js             | frontConnection.login = false                        |
| frontLogin         | frontLoginTask.js              | 3秒防抖 + frontConnection.login = true                  |
| getMqttStatus      | GetMqttConnectionStatusTask.js | 遍历RegistryConfig返回每个机器人MQTT连接状态                      |
| killAll            | KillTask.js                    | 向所有逆向广播KILL_PROCESS指令，等待1秒                           |
| killJava           | killAppTask.js                 | 上报所有退出登录 + 关闭WebSocket                               |
| reportLogicWorking | ReportLogicWorkingTask.js      | 更新LogicWorkingCache + MQTT上报 + WORKING状态重发login      |


**forward 内部分支（最复杂）**：

- `SEND_MESSAGE`: HTTP/HTTPS链接自动下载到本地，替换content路径
- `KICK_OUT`: [企微]→WORK_KICK_OUT_MEMBER类型转换+数字ID; [微信]→校验群成员存在性后过滤
- `REMARK`(企微only): 构建WorkWxFriendBO→WORK_CHANGE_REMARK
- `CHATUSER_INFO_LIST`(企微only): 直接查DB返回群成员详情，不转发逆向
- `USER_LIST`: 防重复(START/INIT/SECOND状态拦截)→先推旧数据→180s超时重置
- 其他type: 直接转发到 dispatchInBound
