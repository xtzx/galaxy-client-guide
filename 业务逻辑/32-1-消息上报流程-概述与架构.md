# 企微与微信消息上报流程 - 概述与架构

> 本系列文档详细梳理了微信助手中**逆向 → 微信助手 → MQ（云端）**的消息上报完整流程。
> 涵盖企业微信和个人微信两种场景，包含 IPC 消息接收、出站调度、三段式回执匹配、
> 转换服务处理、MQTT 上报、超时检测等全部环节。

---

## 一、消息上报整体概述

### 1.1 什么是消息上报

消息上报是指：当逆向客户端（控制企微/微信进程的 DLL）执行完任务后，将执行结果
通过 IPC 管道返回给微信助手（Node.js 进程），微信助手经过一系列处理后，
最终通过 MQTT 将结果上报到云端服务器的完整过程。

### 1.2 上报消息的分类

上报消息主要分为以下几大类：

| 分类 | 说明 | 示例 |
|------|------|------|
| **任务回执消息** | 云端下发的群发任务执行结果 | 发送文本/图片/文件/视频等的回执 |
| **聊天记录消息** | 逆向监听到的收发消息 | recvmsg 类型的私聊/群聊消息 |
| **好友变更消息** | 好友列表、删除好友等变更 | friendupdate、deletefriend |
| **群变更消息** | 群成员变动、建群等 | chatroomMembers、createchatroom |
| **登录状态消息** | 登录、登出、心跳 | login、logout、pong |
| **系统事件消息** | 错误报告、崩溃通知等 | bugreport、error |

### 1.3 消息上报与消息下发的关系

## 二、消息上报完整架构图

### 2.1 总体数据流

    │

## 三、IPC 消息接收

### 3.1 asyncSelectTask 核心循环

文件：`src/msg-center/core/reverse/asyncSelectTask.js`

asyncSelectTask 是整个上报链路的**入口**，它运行一个持续的循环，不断从 IPC
命名管道读取逆向客户端返回的消息。

```javascript
// 核心接收循环(简化)
async function loop(pipeLineWrapper) {
    while (true) {
        try {
            // 1. 检查管道是否有数据
            const selectCode = Clibrary.IpcSelectCltChannel(pipeCode);

            if (selectCode > 0) {
                // 2. 读取IPC消息
                const message = Clibrary.IpcClientRecvMessage(pipeCode);

                // 3. 处理大数字精度问题
                const processedMsg = replaceLargeNumbers(message);

                // 4. 解析消息并获取类型
                const jsonObj = JSON.parse(processedMsg);
                const type = jsonObj.type;

                // 5. 记录日志
                logUtil.customLog(
                    `[接收逆向推送消息] messageType-${type} ` +
                    `channelId: ${channelId} [status-${status}] ` +
                    `wxId: [wxid-${wxId}] 内容: ${processedMsg}`
                );

                // 6. 分发到出站调度器
                dispatchOutBound(processedMsg, pipeLineWrapper);
            }
        } catch (error) {
            // 错误处理...
        }
    }
}
```

### 3.2 IPC 接收关键日志

| 日志模板 | 说明 |
|----------|------|
| `pipeCode:${pipeCode} processId:${processId} [wxid-${wxid}] selectCode:${selectCode} 处理时间为:${time} 接收到IPC消息:{...}` | IPC 原始接收日志，包含管道码、进程ID、处理时间 |
| `[接收逆向推送消息] messageType-${type} channelId: ${channelId} [status-${status}] wxId: [wxid-${wxId}] 内容: {...}` | 消息分类接收日志，包含消息类型和完整内容 |

### 3.3 IPC 消息的基本结构

逆向返回的 IPC 消息都是 JSON 字符串，基本结构如下：

```json
{
    "type": "recvmsg",           // 消息类型(核心字段)
    "client_id": "CIGABB...",    // 消息唯一标识(企微专用)
    "conversation_id": "R:xxx",  // 会话ID
    "content_type": 2,           // 内容类型编号
    "send_id": 1688857807622564, // 发送者ID
    "sendtime": 1773136881,      // 发送时间
    "sequence": 0,               // 序列号
    "server_id": 0,              // 服务端消息ID
    "msg_id": 0,                 // 消息ID
    "timer": 1773136874,         // 时间戳
    "reportId": "32436_3221",    // 上报ID
    "status": 0,                 // 状态码(0=成功)
    "flag": 6,                   // 标志位
    "textmsg": [...],            // 文本消息内容
    "filemsg": [...],            // 文件消息内容
    "linkmsg": [...],            // 链接消息内容
    "itemlist": [...]            // 企微消息项列表
}
```

**企业微信常见的 type 值：**

| type | 说明 |
|------|------|
| `recvmsg` | 接收消息回调（三段式第一条） |
| `sendmsg_sendtext` | 发送文本回执（三段式第二条） |
| `sendmsg_sendimage` | 发送图片回执（三段式第二条） |
| `sendmsg_sendfile` | 发送文件回执（三段式第二条） |
| `sendmsg_sendvideo` | 发送视频回执（三段式第二条） |
| `msgreport` | 消息上报确认（三段式第三条） |
| `AddOrUpdateFiles` | 文件更新通知 |
| `pong` | 心跳响应 |
| `login` | 登录成功 |
| `logout` | 登出通知 |
| `bugreport` | 错误报告 |

## 四、出站消息调度 (dispatchOutBound)

### 4.1 调度器职责

文件：`src/msg-center/dispatch-center/dispatchOutBound.js`

dispatchOutBound 是出站消息的**核心调度器**，负责：
1. 解析逆向返回的 JSON 消息
2. 过滤心跳(pong)消息
3. 处理 bugreport 和 error 类型（上报到哈勃监控）
4. 更新 Galaxy 版本缓存
5. **根据微信类型路由到对应的消息处理中心**

### 4.2 核心路由逻辑

```javascript
function dispatchOutBound(message, pipeLineWrapper) {
    try {
        let jsonObject = message ? JSON.parse(message) : {};

        // 过滤 pong 消息
        if (jsonObject.type !== 'pong') {
            logUtil.customLog(
                `[wxid-${pipeLineWrapper.wxid}][DispatchCenter]:` +
                `接收逆向type=${jsonObject.type} reportId=${jsonObject.reportId}`
            );
        }

        // 处理 bugreport
        if (jsonObject.type === 'bugreport') {
            reportLog({ name: 'REVERSE_BUG_REPORT', wxid: pipeLineWrapper.wxid });
        }

        // 处理 error 类型
        if (jsonObject.type?.indexOf('error') !== -1
            && jsonObject.type !== 'cdnonerror') {
            reportLog({
                name: 'REVERSE_ERROR',
                wxid: pipeLineWrapper.wxid,
                errorName: jsonObject.type,
                errorMessage: jsonObject.error,
            });
        }

        // 核心路由：根据微信类型分发
        const sendType = jsonObject.type;
        if (pipeLineWrapper.workWx) {
            // 企业微信 → WkMsgHandlerCenter
            WkMsgHandlerCenter.outBoundMsg(jsonObject, pipeLineWrapper, sendType);
        } else {
            // 个人微信处理
            // 失败的第三阶段任务延迟处理，给逆向更多响应时间
            if (CallBackClassify.THIRD_CALLBACKS.has(jsonObject.type)
                && jsonObject.status != 0) {
                setTimeout(() => {
                    WxMsgHandlerCenter.outBoundMsg(
                        jsonObject, pipeLineWrapper, sendType
                    );
                }, apolloConfig.taskFailWaitTime);
            } else {
                WxMsgHandlerCenter.outBoundMsg(
                    jsonObject, pipeLineWrapper, sendType
                );
            }
        }
    } catch (error) {
        logUtil.customLog(
            `[DispatchCenter]:[dispatchOutBoundError] error=${error.message}`
        );
    }
}
```

### 4.3 个人微信的延迟处理机制

对于个人微信，如果第三阶段的回调(recvmsg)状态不是成功(status != 0)，
会延迟 `apolloConfig.taskFailWaitTime` 毫秒后再处理。

这是因为在某些情况下，逆向可能先返回失败状态，但稍后会返回正确的结果。
延迟处理可以给逆向更多的响应时间，避免过早判定任务失败。

### 4.4 dispatchOutBound 关键日志

| 日志模板 | 说明 |
|----------|------|
| `[wxid-${wxid}][DispatchCenter]:接收逆向type=${type} reportId=${reportId}` | 接收到非 pong 消息 |
| `[wxid-${wxid}] [DispatchCenter]: [逆向返回error] message=${message}` | 逆向返回错误 |
| `[DispatchCenter]:[dispatchOutBoundError] error=${message}` | 调度器内部错误 |

## 五、消息处理中心入口 (outBoundMsg)

### 5.1 outBoundMsg 处理链路

文件：`src/msg-center/dispatch-center/handle/msgHandleBase.js`

outBoundMsg 是消息处理的总入口，执行三步链式处理：

```
outBoundMsg(jsonObj, wrapper, sendType)
    │
    │ Step 1: messageHandler()
    │         三段式回执匹配 + taskId 关联
    │         [日志] [DispatchCenter] [msgHandleBase]: outBoundMsg sendType=xxx
    ▼
    │ Step 2: responseMsgHandler()
    │         后置处理（用户列表响应等）
    │         [日志] [DispatchCenter] [msgHandleBase]: responseMsgHandler
    ▼
    │ Step 3: send() → businessHandler()
    │         根据 flowSource 路由到对应出站处理器
    │         [日志] [DispatchCenter] [msgHandleBase] businessHandler init
    ▼
    目标处理器 (cloudFlowOutBound / frontFlowOutBound / 广播)
```

### 5.2 businessHandler 路由规则

```javascript
const FLOW_HANDLER_MAP = {
    '1': frontFlowOutBound,   // FlowSourceEnum.FRONT  → 发送到前端
    '2': cloudFlowOutBound,   // FlowSourceEnum.CLOUND → 发送到云端
};

businessHandler(jsonObject, finalMessage, pipeLineWrapper) {
    // 1. user_list 和 quick_send 类型直接走云端
    if (jsonObject.type === 'user_list'
        || QUICK_SEND_TYPE.includes(jsonObject.type)) {
        cloudFlowOutBound(finalMessage, pipeLineWrapper);
        return;
    }

    // 2. 获取 flowSource
    let source = jsonObject.flowSource;

    // 3. 没有 source → 广播到所有处理器
    if (!source) {
        outBoundByBroadcast(msgStr, pipeLineWrapper);
        return;
    }

    // 4. 有 source → 路由到指定处理器
    if (FLOW_HANDLER_MAP[source]) {
        FLOW_HANDLER_MAP[source](msgStr, pipeLineWrapper);
    }
}
```

**路由规则总结：**

| flowSource | 来源 | 路由目标 | 说明 |
|:---:|------|---------|------|
| 无 | 逆向主动推送 | 广播(云端+前端) | 如好友变更、群变更等被动消息 |
| 1 | 前端发起 | frontFlowOutBound | 前端操作的回执 |
| 2 | 云端发起 | cloudFlowOutBound | 云端任务的回执(**群发消息走这条**) |

### 5.3 outBoundMsg 关键日志

| 日志模板 | 说明 |
|----------|------|
| `[DispatchCenter] [msgHandleBase]: outBoundMsg sendType = ${sendType},jsonObject: {...}` | 消息处理入口 |
| `[DispatchCenter] [msgHandleBase]: responseMsgHandler jsonObject.type = ${type}` | 后置处理完成 |
| `[DispatchCenter] [msgHandleBase] businessHandler init` | 开始路由分发 |
| `[DispatchCenter] [msgHandleBase] outBoundByBroadcast` | 广播消息 |
| `[DispatchCenter] [msgHandleBase] businessHandler-逆向消息处理,source=${source}` | 定向路由 |
