**重要**：SLS 日志中**只有子任务 taskId**，没有前端的 messageId。所以不能直接用 messageId 搜索日志，需要用发送目标 wxid 反查。

### 2.3 前端状态判断逻辑

每个发送对象可能收到多条消息（文字+图片+文件等），前端通过 `getOneStatus()` 综合判断：

| 优先级 | 条件 | 最终状态 |
|--------|------|---------|
| 1 | 任一消息处于「发送中」 | -1（发送中） |
| 2 | 任一消息处于「待发送」 | 2（待发送） |
| 3 | 全部发送成功 | 0（成功） |
| 4 | 全部发送完毕，有一条失败 | 1（失败） |
| 5 | 全部发送完毕，有一条超时 | -2（超时） |

**只要有一条消息失败/超时，整个对象就显示红色叉。**

## 三、客户端三段式回执机制

客户端判断一条消息是否发送成功采用**三段式校验**（`msgResNodeMap`）：

```
第一条（first）：逆向发送回执
    taskId, status:0 = 发出去了
    type: "sendmessage"

第二条（second）：微信 echo 回包
    status:2 = 微信服务器收到了
    type: "recvmsg"

第三条（third）：服务器最终确认
    content: 实际发送内容
    status:0 = 真正送达
```

三条都收齐 → 任务成功删除缓存  
超时未收齐 → `GalaxyTaskStatusTimer` 上报超时（status = OVER_TIME）

## 四、排查过程复盘

### 4.1 走的弯路

1. **直接搜 messageId (28800756)**：搜不到，因为日志里只有子任务 taskId
2. **搜 `通知任务结果`**：搜到了一条 taskId=1266762440482496972 的失败日志，但这不一定属于目标任务
3. **搜 `发文本任务失败`**：搜不到，因为失败的不是文本消息

### 4.2 正确路径

1. **搜 `zhouming AND wxid_c6w6hg2qwkie22`**，下载全量日志
2. 从 `MQTT收到消息` 日志中提取所有子任务的 taskId 和消息类型
3. 从 `msgResNodeMap` / `第三条消息` 日志中确认每条消息的三段式状态
4. **发现链接消息（49002）没有第三段确认** → 触发超时 → 前端显示失败

### 4.3 最终结论

| taskId | 消息类型 | 三段式状态 | 结果 |
|--------|---------|-----------|------|
| 1266762014244470961 | 文字 | ✅✅✅ 三段齐 | 成功 |
| 1266762061132865598 | 图片 | ✅✅✅ 三段齐 | 成功 |
| 1266762058947633465 | 文件 | ✅✅✅ 三段齐 | 成功 |
| 1266762099011625104 | 名片 | ✅✅✅ 三段齐 | 成功 |
| 1266762108771500107 | 链接 | ✅✅❓ 第三段缺失 | **超时/失败** |

因为5条消息中有1条失败，`getOneStatus()` 判定整体状态为失败，前端显示红色叉。

## 六、常见失败原因速查

| 错误码 | 含义 | 常见场景 |
|--------|------|---------|
| `-24` | `MM_ERR_SPAM` | 发送频繁被微信限制 |
| `-34` | `MM_ERR_FREQ_LIMITED` | 频率过高 |
| `-4` | `MM_ERR_NOUSER` | 对方已删除/不存在 |
| `-22` | `MM_ERR_BLACKLIST` | 被对方拉黑 |
| `-1` | `MM_ERR_SYS` | 微信系统错误 |
| `536870914` | 逆向管道异常 | 机器人登录状态异常 |
| `-200` | `MM_ERR_ACCOUNT_BAN` | 发送方账号被封 |
| `超时` | 三段式未收齐 | 网络问题/逆向异常 |
