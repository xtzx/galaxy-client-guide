## 1. 业务概述

### 1.1 功能说明

修改群公告（CHATROOM_UPDATE_NOTICE，任务类型 type=5）允许机器人修改指定群聊的公告内容。

**前提条件：** 机器人必须是群主或群管理员。

### 1.2 微信 vs 企业微信

| 维度 | 微信 | 企业微信 |
|------|------|----------|
| 任务处理 | `mqttGroupAnnounceService.js` | `mqttWorkWxGroupAnnounceService.js` |
| 响应处理 | `announcementResponse.js` | `workWxAnnouncementResponse.js` |
| type | `roomannouncement` | `modifynotice` |
| 群ID格式 | `xxx@chatroom` | `R:xxx` |
| 权限校验 | 无（逆向端校验） | 有（`judgeChatroomOwnerOrAdmin`） |
| 回调机制 | 双回调（第一次 + 真实结果） | 双回调（任务消息 + 结果消息） |

### 1.3 双回调机制说明

群公告修改与普通任务不同，采用**双回调机制**：

```
第一次回调（确认收到任务）
   type: "roomannouncement"（微信）/ "modifynotice"（企微）
   含义: 逆向确认收到任务，但尚未得到微信的真实执行结果

第二次回调（真实结果）
   type: "MM.SetChatRoomAnnouncementResponse"（微信）/ "modconversationnotice"（企微）
   含义: 微信/企微服务器返回的真实执行结果（成功/失败）
```

代码在收到第一次回调后，会等待 `announcementWaitTime`（默认 2 秒），然后从缓存中获取第二次回调的真实结果。如果 2 秒内第二次回调未到达，则判定为超时。

```
│        │  日志: "[接收mqtt任务] [messageType-5]"                             │
│        ▼                                                                    │

│        │  filter: serverTask.type === CHATROOM_UPDATE_NOTICE                │
│        │  构建 clientTask (type=roomannouncement)                           │

│        ▼                                                                    │
│  【阶段3】cloudFlowInBound.js 添加来源标记                                  │
│        │  添加 flowSource=2 (标记来源为云端)                                │

│        ▼                                                                    │
│  【阶段4】dispatchInBound.js 调度入站消息                                   │
│        │  解析消息、获取注册信息、路由到处理函数                               │
│        │  日志: "[dispatchInBound] 开始执行"                                │
│        ▼                                                                    │
│  【阶段5】reverseSend.js 发送给逆向                                         │
│        │  通过IPC管道将任务发送给逆向客户端                                   │
│        │  日志: "[发送消息给逆向] wxid: [wxid-xxx] type=[roomannouncement]"  │
│        │                                                                    │
│  ══════╪════════ 分界线：以上是任务下发，以下是响应处理 ══════════════════   │
│        │                                                                    │
│        ▼                                                                    │
│  【阶段6】逆向执行并返回第一次回调                                           │
│        │  asyncSelectTask.js 接收IPC消息                                    │
│        │  日志: "接收到IPC消息:{...}"                                       │
│        │  日志: "[接收逆向推送消息] messageType-roomannouncement"            │
│        ▼                                                                    │
│  【阶段7】dispatchOutBound.js → msgHandleBase → cloudFlowOutBound          │
│        │  日志: "[DispatchCenter] [msgHandleBase]: outBoundMsg"             │
│        │  日志: "[CloudFlowHandler]:outBound执行convert-service:..."       │
│        ▼                                                                    │
│  【阶段8】announcementResponse.js 处理群公告响应                            │
│        │  filter: GROUP_ANNOUNCEMENT 或 ANNOUNCEMENT                       │
│        │  日志: "修改群公告-任务上报 -clientMsg:[...]"                       │
│        │  日志: "等待修改群公告的真实结果返回信息taskId:[xxx]"                │
│        │                                                                    │
│        │  await sleep(2000);  // 等待第二次回调                             │
│        ▼                                                                    │
│  【阶段9】匹配真实结果                                                      │
│        │                                                                    │
│        ├──▶ 成功：roomMap[chatroom] 有值                                    │
│        │    日志: "修改群公告的真实结果准备上报:[...]"                        │
│        │    → taskResponseV2 上报任务结果                                   │
│        │    → MqttSendService.sendMessage 上报群公告消息                    │
│        │                                                                    │
│        ├──▶ 失败：realMsg.status != 0                                      │
│        │    → 上报失败状态，不上报群公告消息                                  │
│        │                                                                    │
│        └──▶ 超时：roomMap[chatroom] 无值                                    │
│             日志: "逆向返回修改群公告结果超时"                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 第二次回调的处理

```
逆向返回真实结果
   type: "MM.SetChatRoomAnnouncementResponse"
        │
        ▼
announcementResponse.js filter() 匹配
        │
        ▼
进入 operate()，走 else if 分支
   → 构建 ChangeAnnouncementBO
   → 存入 this.roomMap[chatroom]
        │
        ▼
阶段8 sleep(2000) 结束后从 roomMap 取出
```

## 5. 阿里云日志排查方案

### 5.1 前置信息收集

排查前需要从工单中获取以下信息：

| 必要信息 | 获取方式 | 示例 |
|----------|----------|------|
| wxId | 工单/业务系统 | `wxid_jaqj5up55d5322` |
| chatroom | 工单/业务系统 | `45265239246@chatroom` |
| taskId/msgId | 首条MQTT日志 | `1262015738903416941` |
| 失败时间 | 工单描述 | 2025-02-20 14:00 |
| 微信/企微 | 群ID格式判断 | `@chatroom`=微信，`R:`=企微 |

### 5.2 分阶段查询

**第一步：定位 taskId（如果只知道 wxId 和大致时间）**

```sql
"wxId的值" and "群公告" and "开始处理任务"
```

**第二步：用 taskId 查询完整链路（推荐首选）**

```sql
"taskId的值"
```

> 这是最高效的查询方式，一次可以拿到该任务的全部日志。

**第三步：分阶段精确查询**

| 阶段 | 查询语句 | 目的 |
|------|----------|------|
| 任务接收 | `"wxId值" and "开始处理任务" and "taskId值"` | 确认MQTT收到任务 |
| 任务处理 | `"wxId值" and "Group-Announcement - clientTask="` | 确认任务被处理 |
| 入站处理 | `"wxId值" and "[CloudFlowHandler]:inBound"` | 确认入站流程 |
| 调度执行 | `"wxId值" and "dispatchInBound"` | 确认调度器执行 |
| **发送逆向** | `"wxId值" and "[发送消息给逆向]" and "roomannouncement"` | **关键分界点** |
| 逆向返回 | `"wxId值" and "[CloudFlowHandler]:outBound"` | 逆向是否返回 |
| 响应处理 | `"wxId值" and "修改群公告-任务上报"` | 响应是否被处理 |
| 等待结果 | `"wxId值" and "等待修改群公告的真实结果返回信息"` | 是否进入等待 |
| 最终结果 | `"wxId值" and ("修改群公告的真实结果" or "逆向返回修改群公告结果超时")` | 成功还是超时 |
| 错误日志 | `"wxId值" and "codeError"` | 是否有异常 |
| 第二次回调 | `"wxId值" and "SetChatRoomAnnouncementResponse"` | 真实结果是否返回 |

**第四步：扩大时间范围**

由于群公告有 2 秒等待期，最终结果日志的时间戳会比前面的晚 2 秒以上。如果查不到最终结果，请将搜索时间范围扩大到任务时间 +15 秒。

## 6. 问题定位判断标准

### 6.1 根据最后一条日志判断问题归属

| 查到的最后一条日志 | 问题出在哪里 | 可能原因 |
|-------------------|-------------|----------|
| 只有"MQTT收到消息" | **galaxy-client** | mqttGroupAnnounceService 未匹配或 clientTask 构建失败（chatroom/content 为空） |
| 有"Group-Announcement - clientTask=" 但无后续 | **galaxy-client** | cloudFlowInBound 处理异常 |
| 有"[dispatchInBound] 开始执行" 但无发送逆向 | **galaxy-client** | 逆向注册信息不存在或 IPC 连接异常 |
| 有"[发送消息给逆向]" 但无逆向返回 | **逆向** | 逆向未执行任务或 IPC 通道断开 |
| 有"修改群公告-任务上报" 但无等待日志 | **galaxy-client** | 代码在解析/构建阶段异常 |
| 有"等待修改群公告的真实结果" 但无后续 | **galaxy-client 或进程异常** | 进程可能在 sleep 期间崩溃/重启 |
| 有"逆向返回修改群公告结果超时" | **逆向** | 逆向 2 秒内未返回真实结果（SetChatRoomAnnouncementResponse） |
| 有"修改群公告的真实结果准备上报" 且 status!=0 | **逆向/微信** | 逆向执行失败，检查 status 和 reason 字段 |
| 有"修改群公告的真实结果准备上报" 且 status=0 | **云端** | galaxy-client 已上报成功，问题可能在云端处理 |

### 6.2 快速判断流程图

```
有 "[发送消息给逆向]" 日志？
   │
   ├── 无 → 问题在 galaxy-client（任务下发阶段）
   │         → 继续往前追，看停在哪个阶段
   │
   └── 有 → 有逆向返回的日志？
              │
              ├── 无 → 问题在逆向端（逆向未返回任何消息）
              │
              └── 有 → 有 "等待修改群公告的真实结果返回信息" ？
                         │
                         ├── 无 → 问题在 galaxy-client（响应处理异常）
                         │
                         └── 有 → 有最终结果日志？
                                    │
                                    ├── 超时 → 问题在逆向（未返回第二次回调）
                                    ├── 成功 → 问题在云端（galaxy-client 已上报）
                                    └── 无 → 进程异常（sleep 期间崩溃/重启）
```

## 7. 排查实战案例

- **问题描述**：用户反馈修改群公告失败
- **wxId**：`wxid_jaqj5up55d5322`
- **chatroom**：`45265239246@chatroom`
- **taskId**：`1262015738903416941`
- **时间**：timestamp `1771473605`
- **类型**：微信群（`@chatroom` 格式）

### 7.2 排查过程

#### 步骤一：用 taskId 查询完整链路

| 11 | `[接收逆向推送消息] messageType-roomannouncement` | ✅ |

| 15 | `等待修改群公告的真实结果返回信息taskId:[1262015738903416941]` | ✅ |

#### 步骤三：查找最终结果

```sql

```

#### 步骤四：查找第二次回调

```sql
"wxid_jaqj5up55d5322" and "SetChatRoomAnnouncementResponse"
```

结果：**空**。逆向未返回真实结果。

#### 步骤五：查找错误日志

```sql
"wxid_jaqj5up55d5322" and "codeError"
```

结果：**空**。没有代码异常。

### 7.3 案例结论

| 分析项 | 结论 |
|--------|------|
| galaxy-client 任务下发 | ✅ 正常 |
| galaxy-client 发送给逆向 | ✅ 正常 |
| 逆向第一次回调 | ✅ 正常（status:0） |
| 逆向第二次回调 | ❌ **缺失** |
| 日志最终停在 | "等待修改群公告的真实结果返回信息" |
| 后续无超时/成功日志 | 说明 sleep(2000) 后进程可能异常，或阿里云日志时间范围需扩大 |

**结论**：galaxy-client 侧逻辑正常，问题出在**逆向端未返回第二次回调**（`MM.SetChatRoomAnnouncementResponse`），或者进程在 `sleep(2000)` 期间出现了异常。需要联系逆向团队排查为什么 `SetChatRoomAnnouncementResponse` 没有返回。

### 7.4 注意事项

- 此案例中发现了两个不同的 taskId（`1262021953091948909` 和 `1262015738903416941`），是同一个群的两次不同群公告修改任务，需要注意区分
- 所有日志时间戳相同（`1771473605`），最终结果的日志时间戳应该在 +2 秒之后，需要扩大搜索时间范围

## 10. 附录：常见失败原因

### 10.1 群公告相关

| 失败原因 | 现象 | 排查方法 |
|----------|------|----------|
| 权限不足 | 逆向执行失败，status!=0 | 确认机器人是群主或管理员 |
| 群公告内容为空 | clientTask 构建返回 null，无后续日志 | 检查 MQTT 消息中 content 字段 |
| chatroom 为空 | clientTask 构建返回 null | 检查 MQTT 消息中 chatroom 字段 |
| 逆向超时 | "逆向返回修改群公告结果超时" | 逆向未在 2 秒内返回 SetChatRoomAnnouncementResponse |
| IPC 连接断开 | "IPC尚未连接成功" 或 "sendMsgToReverseError" | 检查逆向连接状态 |
| 进程崩溃 | 等待日志后无任何后续 | 检查进程重启日志 |

### 10.2 通用失败原因

| 失败原因 | 日志关键字 | 说明 |
|----------|-----------|------|
| 逆向未注册 | `尚未和逆向建立注册信息` | 逆向客户端尚未连接 |
| IPC 发送失败 | `sendMsgToReverseError` | IPC 通道异常 |
| 重复任务被锁 | `taskId已经存在 已经被锁` | 5 秒内重复下发同一任务 |
| 参数错误 | `[发送给逆向时参数错误]` | wxId、channelId、message 缺失 |
| 任务状态锁异常 | `galaxyTaskLockError` | 任务缓存过程中异常 |
