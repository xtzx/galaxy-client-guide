# 30-文件上传完整链路与日志排查指南

## 一、概述

本文档详细梳理**个人微信**和**企业微信**两套场景下，图片、视频、文件、语音等多媒体消息从逆向接收到上报云端的完整文件上传链路。每个环节均标注了对应的日志关键词，用于阿里云日志快速排查。

### 架构差异总览

| 维度 | 个人微信 | 企业微信 |
|---|---|---|
| 消息入口 | `recvmsgs`(批量) / `recvmsg`(单条) | `RECV_MSG` |
| 类型字段 | `msgtype1` + `msgtype2` | `content_type` |
| 文件缓存机制 | `msgFileCache`(CDN URL → 消息对象) | `FILE_KEY_MAP`(weworkServerId → 消息对象) |
| 文件下载方式 | CDN 下载（逆向异步完成） | filepath 回调（逆向下载后回调路径） |
| 上传方式 | CDN 下载完成后更新 content | OSS 上传后更新 content |
| 处理文件 | `recvMsgsService.js` / `recvMsgService.js` | `workWxMsgRecordService.js` + `workWxFilePathService.js` |

## 二、个人微信文件上传链路

### 2.1 消息入口

所有个人微信消息通过两个入口进入：

| 入口 | 消息类型 | 说明 |
|---|---|---|
| `recvmsgs` | 批量接收消息 | 他人发送的消息，包含 `msgs` 数组 |
| `recvmsg` (status=2) | 单条消息 | 自己发送的消息 |
| `recvmsg` (status=3) | 文件下载凭证 | 文件消息第二阶段（含 CDN 信息） |

**入口日志关键词**：

```
[recvmsgs] 收到批量消息                    → recvmsgs 入口
进入processSingleMsg                       → 单条消息开始处理（含 msgtype1/msgtype2/from/to/content前50字符）
[recvmsg][status=3] 收到文件下载凭证       → 文件消息第二阶段
```

### 2.2 消息类型对照表

| msgtype1 | msgtype2 | 消息类型 | wechatMsgType | 处理方法 |
|---|---|---|---|---|
| 1 | 0 | 文本消息 | 1 | `processTextMsg` |
| 3 | 0 | 图片消息 | 3 | `processImageMsg` |
| 43 | 0 | 视频消息 | 43 | `processVideoMsg` |
| 34 | 0 | 语音消息 | 34 | `processVoiceMsg` |
| 49 | 5 | 卡片链接 | 62(TYPE__MSG_CARD) | `processFileOrCardMsg` |
| 49 | 6 | 文件消息 | 50(TYPE__MSG_CARD_FILE) | `processFileOrCardMsg` |
| 49 | 33 | 小程序 | 62 | `processFileOrCardMsg` |
| 49 | 51 | 视频号 | 62 | `processFileOrCardMsg` |
| 49 | 63 | 视频号直播 | 70 | `processFileOrCardMsg` |
| 49 | 74 | 文件上传令牌 | - | 跳过上报，等待 status=3 |

### 2.3 图片消息链路 (msgtype1=3)

```
逆向推送 recvmsgs
  └─ msgs[i]: { msgtype1: 3, content: "<msg><img cdnthumburl=.../>...</msg>" }
        │
        ▼
recvMsgsService.processImageMsg()
  ├─ 解析 content XML → 提取 <img> 标签的 cdnthumburl 属性
  ├─ recordInfo.content = ""                      ← content 先留空
  ├─ recordInfo.wechatMsgType = 3
  │
  ├─ [有 CDN] → msgFileCache.putFile2MsgCache(cdn, clientMsgCloud)
  │             等待逆向 CDN 下载完成后更新 content
  ├─ [无 CDN] → 打印警告日志
  │
  └─ MqttSendService.sendMessage()                ← 第一次上报（content=""）
        │
        ▼
  等待逆向下载完成 → 回调更新 content → 第二次上报（content=文件URL）
```

**日志排查关键词**：

| 步骤 | 关键词 | 说明 |
|---|---|---|
| 逆向原始数据 | `进入processSingleMsg` + `msgtype1=3` | 逆向推送的原始消息 |
| CDN 提取成功 | `图片消息放入缓存 cdn=` | CDN 地址已提取 |
| CDN 为空 | `图片消息CDN为空，跳过缓存` | content XML 中缺少 CDN |
| 处理异常 | `processImageMsgError` | 处理过程报错 |
| MQTT 上报 | `MQTT发往云端数据成功` | 最终上报结果 |

**阿里云查询语句**：

```sql
-- 查所有图片消息
* AND "进入processSingleMsg" AND "msgtype1=3"

-- 查指定账号的图片消息
* AND "wxid-你的微信ID" AND "图片消息放入缓存"

-- 查图片处理异常
* AND "processImageMsgError"
```

### 2.5 文件消息链路 (msgtype1=49, msgtype2=6)

文件消息采用**两阶段推送**，比图片/视频多一个阶段：

```
【阶段一】recvmsgs 收到 msgtype2=74（文件上传令牌）
  └─ 只有 fileuploadtoken，没有 CDN 地址
  └─ 代码直接 return，不上报（避免重复上报）

       │ （等待文件上传到 CDN）
       ▼

【阶段二】recvmsg (status=3) 收到 msgtype2=6（文件下载凭证）
  └─ recvMsgService.processFileDownloadCredential()
        │
        ├─ 解析 content XML → 提取 <appmsg><appattach><cdnattachurl> CDN 地址
        ├─ 提取 aeskey（解密密钥）、attachid、fileext、title 等
        ├─ recordInfo.content = ""                ← content 先留空
        ├─ recordInfo.wechatMsgType = 50 (TYPE__MSG_CARD_FILE)
        │
        ├─ msgFileCache.putFile2MsgCache(cdn, clientMsgCloud)
        │
        └─ MqttSendService.sendMessage()          ← 第一次上报（content=""）
              │
              ▼
        等待 CDN 下载完成 → 更新 content → 第二次上报
```

注：少数情况下 msgtype2=6 也可能出现在 recvmsgs 中（老版本协议或特殊场景），此时走 `processFileOrCardMsg`。

**日志排查关键词**：

| 步骤 | 关键词 | 说明 |
|---|---|---|
| 阶段一令牌 | `收到文件上传令牌(msgtype2=74)` | 第一阶段，无需关注 |
| 阶段二凭证到达 | `[recvmsg][status=3] 收到文件下载凭证` | 包含 title/ext/size/cdn |
| CDN 提取成功 | `文件消息放入缓存 cdn=` | CDN 地址已提取 |
| CDN 为空 | `CDN地址为空，无法下载文件` | 凭证中缺少 CDN |
| XML 解析失败 | `文件下载凭证XML解析失败` | content XML 格式异常 |
| 处理异常 | `processFileDownloadCredentialError` | 处理过程报错 |
| recvmsgs 中的文件 | `文件消息(msgtype2=6) cdn=` | recvmsgs 中出现的文件消息 |

**阿里云查询语句**：

```sql
-- 查文件下载凭证
* AND "wxid-你的微信ID" AND "收到文件下载凭证"

-- 查文件消息缓存
* AND "wxid-你的微信ID" AND "文件消息放入缓存"

-- 查文件处理异常
* AND "processFileDownloadCredentialError"
```

### 2.8 个人微信 CDN 缓存机制详解

个人微信使用 `msgFileCache` 模块管理文件缓存：

| 属性 | 值 | 说明 |
|---|---|---|
| 缓存容器 | `Map<string, {clientMsg, time}>` | CDN URL → 消息对象 |
| 最大容量 | 1000 条 | 超过后触发过期清理 |
| 过期时间 | 30 秒 | 超时自动淘汰 |

**工作流程**：

```
putFile2MsgCache(cdn, clientMsgCloud)    ← 消息处理时存入
     │
     │  （等待逆向 CDN 下载）
     ▼
popFile2MsgCache(cdn)                    ← CDN 下载完成后取出并删除
     │
     ├─ 更新 clientMsgCloud.recordInfo.content = 下载后的 URL
     └─ 再次 MqttSendService.sendMessage()  ← 二次上报
```

> **注意**: 如果 CDN 下载耗时超过 30 秒，缓存会被淘汰，导致 content 永远为空。

## 三、企业微信文件上传链路

### 3.1 消息入口

企业微信所有消息通过一个统一入口：

```
逆向推送 RECV_MSG → workWxMsgRecordService.operate()
```

**入口日志关键词**：

```
[MsgRecordServiceImpl-operate] - client收到逆向消息-私聊/群聊消息-[完整JSON数据]
```

> 该日志包含逆向给的**完整原始 JSON**，是最关键的排查起点。

### 3.2 消息类型对照表

| content_type | 消息类型 | wechatMsgType | 文件上传方式 |
|---|---|---|---|
| 0, 2 | 文本消息 | 1 | 无需上传 |
| 7, 14, 101 | 图片消息 | 3 | filepath 回调 + OSS |
| 123 | 特殊图片 | 3 | 无处理（content 留空） |
| 5, 23, 103 | 视频消息 | 43 | filepath 回调 + OSS |
| 8, 15, 102 | 文件消息 | 50 | filepath 回调 + OSS |
| 16 | 语音消息 | 34 | filepath 回调 + silk→mp3 + OSS |
| 29 | 表情消息 | 47 | 无需上传（直接提取 URL） |
| 13, 1011 | 链接消息 | 62 | 无需上传 |
| 41 | 名片消息 | 42 | 无需上传 |
| 78 | 小程序 | 68 | 无需上传 |
| 141 | 视频号卡片 | 69 | 无需上传 |
| 146 | 视频号直播 | 70 | 无需上传 |
| 53 | 语音/视频通话 | 特殊 | 独立处理 |
| 54 | 通话结束 | 特殊 | 独立处理 |
| 213 | 接龙消息 | 特殊 | 无需上传 |
| 1004 | 被移除群聊 | - | 事件处理 |
| 1005 | 主动退群 | - | 事件处理 |

### 3.3 发送方向判断

企业微信通过 `send_id` 和 `sequence` 区分消息方向：

| 条件 | 含义 | 文件处理方式 |
|---|---|---|
| `sendId === ownerWxId && sequence === 0` | **自己发送**的消息 | 同步上传：`filemsg[0].sendPath` → OSS → content = URL |
| 其他情况 | **接收**的消息 | 异步回调：`FILE_KEY_MAP` 缓存 → filepath 回调 → OSS → content = URL |

### 3.4 图片消息链路 (content_type=7/14/101)

#### 3.4.1 自己发送的图片（sequence=0）

```
逆向推送 RECV_MSG { content_type: 14, send_id: ownerWxId, sequence: 0 }
        │
        ▼
workWxMsgRecordService.operate()
  ├─ recordInfo.wechatMsgType = 3 (TYPE__IMAGE)
  ├─ recordInfo.content = ""
  ├─ filemsg[0].sendPath → getOssUrl() → AliyunOssManagerUtil.uploadFile()
  ├─ recordInfo.content = OSS URL                 ← content 直接有值
  │
  └─ MqttSendService.sendMessage()                ← 一次上报（content=URL）
```

#### 3.4.2 接收的图片（当前走 extJsonContent 路径）

```
逆向推送 RECV_MSG { content_type: 14, sequence: N }
        │
        ▼
workWxMsgRecordService.operate()
  ├─ recordInfo.wechatMsgType = 3 (TYPE__IMAGE)
  ├─ recordInfo.content = ""
  ├─ 存储扩展信息到 extJsonContent: { aesKey, md5, filesize, wxServerId, authKey, isHd, fileName }
  │   （当前 aboloStatus=false，不走 filepath 回调）
  │
  └─ MqttSendService.sendMessage()                ← 一次上报（content=""，附带 extJsonContent）
```

> **注意**: 接收的图片当前**不走 filepath 回调**（`aboloStatus = false`），而是将下载所需的 `aesKey`、`wxServerId` 等信息存入 `extJsonContent`，由云端自行下载。

**日志排查关键词**：

| 步骤 | 关键词 | 说明 |
|---|---|---|
| 逆向原始数据 | `MsgRecordServiceImpl-operate` + `client收到逆向消息` + `content_type":14` | 完整原始数据 |
| OSS 上传（自己发送） | `AliyunOssManagerUtil` + `uploadFile` | 上传文件到 OSS |
| OSS 上传成功 | `ossClientPutSuccess` | 包含最终 URL |
| OSS 上传失败 | `ossClientPutError` | 上传失败原因 |
| 最终上报 | `MsgRecordServiceImpl-operate` + `开始发送云端` | 上报到云端的数据 |

**阿里云查询语句**：

```sql
-- 查所有企微图片消息的逆向原始数据
* AND "MsgRecordServiceImpl-operate" AND "client收到逆向消息" AND ("content_type\":14" OR "content_type\":7" OR "content_type\":101")

-- 查指定账号
* AND "wxid-你的企微ID" AND "MsgRecordServiceImpl-operate" AND "client收到逆向消息" AND ("content_type\":14" OR "content_type\":7" OR "content_type\":101")

-- 查 OSS 上传结果
* AND "wxid-你的企微ID" AND "ossClientPutSuccess"
* AND "wxid-你的企微ID" AND "ossClientPutError"
```

### 3.5 视频消息链路 (content_type=5/23/103)

#### 3.5.1 自己发送的视频（sequence=0）

```
逆向推送 RECV_MSG { content_type: 5, send_id: ownerWxId, sequence: 0, filemsg: [{sendPath: "..."}] }
        │
        ▼
workWxMsgRecordService.operate()
  ├─ recordInfo.wechatMsgType = 43 (TYPE__VIDEO)
  ├─ recordInfo.content = ""
  ├─ filemsg[0].sendPath → getOssUrl() → AliyunOssManagerUtil.uploadFile()
  ├─ recordInfo.content = OSS URL                 ← content 直接有值
  │
  └─ MqttSendService.sendMessage()                ← 一次上报（content=URL）
```

#### 3.5.2 接收的视频（filepath 回调模式）

```
逆向推送 RECV_MSG { content_type: 5, videomsg: [{weworkServerId: "xxx", ...}] }
        │
        ▼
workWxMsgRecordService.operate()                   ← 日志①
  ├─ recordInfo.wechatMsgType = 43 (TYPE__VIDEO)
  ├─ recordInfo.content = ""                       ← content 先留空
  ├─ getFileKey(videomsg) → 提取 weworkServerId 作为 key
  ├─ FILE_KEY_MAP[key] = clientMsgCloud            ← 存入内存缓存
  │
  └─ MqttSendService.sendMessage()                 ← 日志②: 第一次上报（content=""）
        │
        │  （逆向异步下载视频到本地）
        ▼
逆向推送 filepath { type: "filepath", key: "weworkServerId", filename: "视频路径" }
        │
        ▼
workWxFilePathService.operate()                    ← 日志③
  ├─ 去除 key 的 _thumb 后缀
  ├─ getCacheCloudInfo(key)                        ← 从 FILE_KEY_MAP 获取，最多重试5次（每次1秒）
  │     └─ 日志④: 获取云端类数据文件处理耗时
  ├─ getWorkWxRealPath(filename)                   ← 相对路径 → 拼接 apppath 转绝对路径
  ├─ AliyunOssManagerUtil.uploadFile()             ← 上传文件到 OSS
  │     └─ 日志⑤: ossClientPutSuccess / ossClientPutError
  ├─ clientMsgCloud.recordInfo.content = OSS URL   ← content 更新为 URL
  ├─ MqttSendService.sendMessage()                 ← 日志⑥: 第二次上报（content=URL）
  └─ delete FILE_KEY_MAP[key]                      ← 清理缓存
        │
        ▼
  日志⑦: 文件消息处理完成 KEY:[xxx] URL:[https://...]
```

**日志排查关键词**：

| 步骤 | 关键词 | 说明 |
|---|---|---|
| ① 逆向原始数据 | `MsgRecordServiceImpl-operate` + `client收到逆向消息` + `content_type":5` | 完整原始 JSON |
| ② 第一次上报 | `MsgRecordServiceImpl-operate` + `开始发送云端` | content="" |
| ③ filepath 到达 | `WorkWxFilePathService` + `init name` | key 和 filename |
| ④ 缓存等待耗时 | `获取云端类数据文件处理耗时` | 超时=5秒可能失败 |
| ⑤ OSS 上传 | `ossClientPutSuccess` / `ossClientPutError` | 上传结果 |
| ⑥ 第二次上报 | `mqtt sendMessage clientMsg` + `wechatMsgType":43` | content=URL |
| ⑦ 处理完成 | `文件消息处理完成` + `KEY:` + `URL:` | 最终 URL |
| 异常 | `WorkWxFilePathServiceError` | filepath 处理报错 |

**阿里云查询语句**：

```sql
-- 查所有企微视频消息的逆向原始数据
* AND "MsgRecordServiceImpl-operate" AND "client收到逆向消息" AND ("content_type\":5" OR "content_type\":23" OR "content_type\":103")

-- 查指定账号视频消息
* AND "wxid-你的企微ID" AND "MsgRecordServiceImpl-operate" AND "client收到逆向消息" AND ("content_type\":5" OR "content_type\":23" OR "content_type\":103")

-- 查 filepath 回调是否到达
* AND "wxid-你的企微ID" AND "WorkWxFilePathService" AND "init name"

-- 查缓存匹配等待耗时
* AND "wxid-你的企微ID" AND "获取云端类数据文件处理耗时"

-- 查最终处理结果
* AND "wxid-你的企微ID" AND "文件消息处理完成"

-- 查异常
* AND "WorkWxFilePathServiceError"
```

### 3.6 文件消息链路 (content_type=8/15/102)

文件消息流程与视频消息**完全一致**，区别仅在于 `content_type` 和 `wechatMsgType`：

#### 3.6.1 自己发送的文件（sequence=0）

```
逆向推送 RECV_MSG { content_type: 8, send_id: ownerWxId, sequence: 0, filemsg: [{sendPath: "..."}] }
        │
        ▼
workWxMsgRecordService.operate()
  ├─ recordInfo.wechatMsgType = 50 (TYPE__MSG_CARD_FILE)
  ├─ recordInfo.content = ""
  ├─ filemsg[0].sendPath → getOssUrl() → AliyunOssManagerUtil.uploadFile()
  ├─ recordInfo.content = OSS URL
  │
  └─ MqttSendService.sendMessage()                ← 一次上报（content=URL）
```

#### 3.6.2 接收的文件（filepath 回调模式）

```
逆向推送 RECV_MSG { content_type: 8, filemsg: [{weworkServerId: "xxx", ...}] }
        │
        ▼
workWxMsgRecordService.operate()
  ├─ recordInfo.wechatMsgType = 50 (TYPE__MSG_CARD_FILE)
  ├─ recordInfo.content = ""
  ├─ getFileKey(filemsg) → 提取 weworkServerId
  ├─ FILE_KEY_MAP[key] = clientMsgCloud            ← 存入缓存
  │
  └─ MqttSendService.sendMessage()                 ← 第一次上报（content=""）
        │
        ▼
逆向推送 filepath { type: "filepath", key: "xxx", filename: "文件路径" }
        │
        ▼
workWxFilePathService.operate()                    ← 同视频消息流程
  ├─ getCacheCloudInfo(key) → OSS 上传 → content = URL → 第二次上报
  └─ delete FILE_KEY_MAP[key]
```

**日志排查关键词**（与视频一致，仅 content_type 不同）：

```sql
-- 查所有企微文件消息
* AND "MsgRecordServiceImpl-operate" AND "client收到逆向消息" AND ("content_type\":8" OR "content_type\":15" OR "content_type\":102")

-- filepath 和 OSS 上传日志同视频消息
```

### 3.7 语音消息链路 (content_type=16)

语音消息是最复杂的类型，涉及 **filepath 回调 + silk→mp3 转换 + 双重 OSS 上传**：

```
逆向推送 RECV_MSG { content_type: 16, filemsg: [{weworkServerId, voiceTime, aeskey, ...}] }
        │
        ▼
workWxMsgRecordService.operate()
  ├─ recordInfo.wechatMsgType = 34 (TYPE__VOICE)
  ├─ recordInfo.content = ""
  ├─ recordInfo.extContent = voiceTime（语音时长）
  ├─ recordInfo.extJsonContent = { aeskey, md5, weworkserverid, filesize, voicetime }
  ├─ FILE_KEY_MAP[key] = clientMsgCloud            ← 存入缓存
  │
  └─ MqttSendService.sendMessage()                 ← 第一次上报（content=""）
        │
        ▼
逆向推送 filepath { type: "filepath", key: "xxx", filename: "xxx.silk" }
        │
        ▼
workWxFilePathService.operate()
  ├─ getCacheCloudInfo(key) → 获取缓存的 clientMsgCloud
  ├─ getWorkWxRealPath(filename)
  │
  ├─ 检测到 wechatMsgType == TYPE__VOICE:
  │   ├─ 确保文件名有 .silk 后缀
  │   ├─ AliyunOssManagerUtil.uploadFileNoCache() → 上传 silk 文件到 voiceMsg/ 目录
  │   ├─ url = OSS silk URL
  │   │
  │   └─ [自己发送的语音] → silkToMp3():
  │       ├─ AudioConversionUtil.silk2mp3() → silk 转 mp3
  │       ├─ AliyunOssManagerUtil.uploadFileNoCache() → 上传 mp3 到 OSS
  │       └─ SendToFrontUtil.sendResponse2Front() → 推送 mp3 URL 给前端
  │
  ├─ recordInfo.content = silk OSS URL
  ├─ MqttSendService.sendMessage()                 ← 第二次上报（content=silk URL）
  └─ delete FILE_KEY_MAP[key]
```

**日志排查关键词**：

| 步骤 | 关键词 | 说明 |
|---|---|---|
| 逆向原始数据 | `client收到逆向消息` + `content_type":16` | 含 filemsg 完整数据 |
| filemsg 数据 | `newlog语音消息:fileMsg:` | filemsg 数组内容 |
| aeskey | `newlog语音消息:aeskey:` | aeskey 值 |
| 存入缓存 | `把语音消息放入缓存key:` | key 和缓存内容 |
| filepath 回调 | `WorkWxFilePathService` + `init name` | filepath 到达 |
| silk 上传 | `uploadFileNoCache` + `voiceMsg` | silk 文件上传 |
| silk→mp3 | `mp3 original mp3Url` | mp3 转换结果 |
| mp3 上传 | `mp3 original filepath:` | mp3 上传结果 |
| 语音处理失败 | `HANDLE_VOICE_ERROR` / `上报mp3文件失败` | 转换或上传失败 |
| 文件处理完成 | `文件消息处理完成` | 最终 URL |

**阿里云查询语句**：

```sql
-- 查所有企微语音消息
* AND "wxid-你的企微ID" AND "newlog语音消息:fileMsg:"

-- 查语音缓存
* AND "wxid-你的企微ID" AND "把语音消息放入缓存"

-- 查语音处理异常
* AND "wxid-你的企微ID" AND "HANDLE_VOICE_ERROR"
* AND "wxid-你的企微ID" AND "上报mp3文件失败"
```

### 3.8 企业微信 FILE_KEY_MAP 缓存机制详解

企业微信使用 `FILE_KEY_MAP`（内存中的普通对象）管理文件缓存：

| 属性 | 值 | 说明 |
|---|---|---|
| 缓存容器 | `Object`（普通 JS 对象） | weworkServerId → clientMsgCloud |
| 过期机制 | **无自动过期** | 依赖 filepath 回调来清除 |
| 重试机制 | 最多 5 次，每次间隔 1 秒 | filepath 可能先于 RECV_MSG 到达 |
| key 来源 | `weworkServerId > wxServerId > ServerId` | 优先级从高到低 |

**工作流程**：

```
RECV_MSG 处理时:
  getFileKey(videomsg/filemsg) → 提取 weworkServerId
  FILE_KEY_MAP[key] = clientMsgCloud               ← 存入

filepath 回调时:
  getCacheCloudInfo(key, {retryCount:0, maxRetryCount:5})
    ├─ 第1次: sleep(1s) → FILE_KEY_MAP[key]
    ├─ 第2次: sleep(1s) → FILE_KEY_MAP[key]
    ├─ ...最多重试5次
    └─ 返回 clientMsgCloud 或 undefined

  上传 OSS → 更新 content → 发送 MQTT → delete FILE_KEY_MAP[key]
```

> **风险点**: 如果 RECV_MSG 处理异常导致 key 未存入 FILE_KEY_MAP，filepath 回调会在 5 秒后超时返回 undefined，消息丢失。

## 四、公共上报链路

无论个人微信还是企业微信，最终上报都经过同一条链路：

```
MqttSendService.sendMessage(ownerWxId, clientMsgCloud)
        │
        ├─ 日志: "mqtt sendMessage clientMsg: {完整JSON}"
        ├─ 校验 clientMsg 非空
        ├─ 获取 registry → 判断企微/个微 → 设置 clientMsgSource (1=企微, 0=个微)
        ├─ 添加字段: username, javaClientVersion, clientType, clientVersion, clientPushTime
        ├─ 检查 taskId < 0 → 模拟任务不上报
        │
        └─ mqExcuteMsg(username, topic, clientMsg)
              │
              ├─ [MQTT 客户端未初始化]:
              │     ├─ 日志: "MQTT client未初始化完毕"
              │     ├─ MqttMakeUpManager.processSaveNotSend() → 保存到本地队列
              │     └─ 触发登录验证
              │
              ├─ [MQTT 发送成功]:
              │     └─ 日志: "MQTT发往云端数据成功 msgId: xxx, cost=xxx, messsage: {完整JSON}"
              │
              └─ [MQTT 发送异常]:
                    ├─ 日志: "MQTT发往云端数据异常"
                    ├─ MqttMakeUpManager.processSaveNotSend() → 保存待重发
                    └─ UploadMetricTask → 上报超时指标
```

**公共日志排查关键词**：

| 步骤 | 关键词 | 说明 |
|---|---|---|
| 发送入口 | `mqtt sendMessage clientMsg:` | 完整的消息 JSON |
| 模拟任务跳过 | `仿造的任务 不需要上报` | taskId 为负数 |
| MQTT 未初始化 | `MQTT client未初始化完毕` | 需要排查连接问题 |
| 发送成功 | `MQTT发往云端数据成功` | 含 msgId、cost（耗时）、完整 message |
| 发送异常 | `MQTT发往云端数据异常` | 发送失败 |
| wxId 为空 | `wxid is null 消息发送失败` | 参数异常 |

**阿里云查询语句**：

```sql
-- 查 MQTT 发送成功
* AND "wxid-你的ID" AND "MQTT发往云端数据成功"

-- 查 MQTT 发送失败
* AND "wxid-你的ID" AND ("MQTT发往云端数据异常" OR "MQTT client未初始化完毕")

-- 查特定 wechatMsgType 的上报
* AND "wxid-你的ID" AND "MQTT发往云端数据成功" AND "wechatMsgType\":43"
```

## 五、OSS 上传链路

文件上传到阿里云 OSS 的完整流程（企业微信专用，个人微信走 CDN）：

```
AliyunOssManagerUtil.uploadFile(filepath, dir, wxid)
  ├─ 日志: "uploadFile filepath:xxx dir:xxx"
  │
  ├─ [缓存命中]: fill2UrlMap.has(filepath) → 直接返回缓存 URL
  │
  └─ [缓存未命中]: uploadFileNoCache(filepath, dir, wxid)
        ├─ 日志: "uploadFileNoCache,originPath:xxx"
        ├─ 检查 STS 凭证是否过期 → 过期则 refreshAccess()
        ├─ 构建 OSS 路径: stormEarth-node/robotFiles/{dir}{wxid}/{timestamp}/{fileName}
        │
        ├─ [上传成功]: ossClient.put(originPath, filepath, {timeout: 1200000})
        │     ├─ 日志: "ossClientPutSuccess [URL] [filepath] [dir] [fileName]"
        │     └─ 存入缓存 fill2UrlMap
        │
        └─ [上传失败]:
              └─ 日志: "ossClientPutError [errorName] [errorMessage]"
```

**OSS 存储目录**：

| 目录 | 说明 |
|---|---|
| `qrCode/` | 二维码图片 |
| `file/` | 普通文件（文件/视频/图片-自己发送的） |
| `imgMsg/` | 图片消息 |
| `videoMsg/` | 视频消息 |
| `voiceMsg/` | 语音消息（silk 格式） |

**阿里云查询语句**：

```sql
-- 查 OSS 上传成功
* AND "wxid-你的ID" AND "ossClientPutSuccess"

-- 查 OSS 上传失败
* AND "ossClientPutError"

-- 查 OSS 凭证获取失败
* AND "获取oss秘钥失败"
```

## 七、常见问题速查

| 现象 | 可能原因 | 排查关键词 |
|---|---|---|
| content 一直为空 | CDN/filepath 回调未到达 | `WorkWxFilePathService init name` 或 `放入缓存 cdn=` |
| content 一直为空 | FILE_KEY_MAP 缓存超时（企微） | `获取云端类数据文件处理耗时` time > 5000 |
| content 一直为空 | msgFileCache 缓存过期（个微30秒） | CDN 下载耗时过长 |
| content 一直为空 | OSS 上传失败 | `ossClientPutError` |
| 消息未上报 | MQTT 未初始化 | `MQTT client未初始化完毕` |
| 消息未上报 | msgtype 不支持 | `暂不支持的消息类型` |
| 消息未上报 | 消息时效性过滤 | `checkMsgAvailable` 返回 false |
| 消息未上报 | 模拟任务被跳过 | `仿造的任务 不需要上报` |
| 语音消息无 mp3 | silk→mp3 转换失败 | `HANDLE_VOICE_ERROR` |
| 文件上报延迟 | filepath 回调延迟 | `获取云端类数据文件处理耗时` |
| OSS URL 无效 | STS 凭证过期未刷新 | `获取oss秘钥失败` |
