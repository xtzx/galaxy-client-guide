# 22-微信好友列表获取与上报链路

> 微信好友列表的完整获取、入库、发送前端、上报云端链路分析，含阿里云日志排查指南和真实案例

---

## 一、触发源头

好友列表获取有两种触发场景：

### 场景 1：前端页面主动请求（刷新好友列表）

前端通过 WebSocket 发送 `type="userlist"` 消息，由 `forwardTask` 处理。

**文件**: `src/msg-center/business/task-front/forwardTask.js` 第319-377行

```
前端点击"刷新好友"
    │
    ▼
forwardTask() 接收 WebSocket 消息
    │  clientTask.type = "getuserlist"
    │  先发送缓存中的旧数据给前端
    │  设置 180秒超时定时器
    │
    ▼
dispatchInBound() → reverseSend → IPC → 逆向
```

### 场景 2：微信登录成功后自动触发

登录成功后，逆向主动推送 `type="getuserlist"` 消息（包含好友/群 wxid 列表）。

## 二、完整处理链路

### 阶段 1：请求好友列表 → 逆向返回 wxid 列表

```
前端刷新 / 登录成功
    │
    │  type="getuserlist" 发给逆向 或 逆向主动推送
    ▼
逆向返回 type="getuserlist"
    │  内容: { type: "getuserlist", objarray: [wxid1, wxid2, ...] }
    │  含义: 只有好友/群的 wxid，没有详细信息
    ▼
asyncSelectTask.loop() → IPC接收
    │
    ▼
dispatchOutBound() → msgHandleBase.outBoundMsg()
    │
    ▼
responseMsgHandler()
    │  wxUserListResponseMsgHandler.isHandler(type, workWx) → true
    ▼
wxUserListResponseMsgHandler.execute()
    │  if (type === "getuserlist")
    ▼
handlerGetUserList()
    │  • 从 objarray 提取好友 wxid 列表
    │  • 过滤系统账号（EXCLUDE_WXID_SET）
    │  • 缓存到 wxUserAndChatRoomCache.initData()
    │  • 更新状态 → SECOND
    ▼
handleFetchRommMember() → fetchRoomMembers()
```

### 阶段 2：分页请求好友详情

```
fetchRoomMembers()
    │  按 userLimit=2000 分页
    │
    ▼  for 循环每一页
fetchUserListInfoByPage()
    │  sendMsgBo = {
    │      type: "userinfolist",
    │      data: { flag: 0, array: [wxid1, wxid2, ...] }
    │  }
    │  dispatchInBound() → 发给逆向
    │  await sleep(1500)  每页间隔1.5秒
    │
    ▼  逆向逐页返回
type="userinfolist" 响应
    │  array: [{ wxid, nickname, headimg, sex, ... }, ...]
```

### 阶段 3：处理好友详情 → 入库 → 发前端 → 上报

```
wxUserListResponseMsgHandler.execute()
    │  if (type === "userinfolist")
    ▼
handleRoomMembers()
    │  解析好友详情，构建 externalUserDTO
    │  缓存到 wxUserAndChatRoomCache.externalUserList
    │  检查所有分页是否返回完成
    │
    │  if (successCount >= totalNode) → 全部完成
    ▼
handleSendFriendToFront()
    │
    ├─① 入库 SQLite
    │    wkConversationsService.diffUpdate()  群列表
    │    wkExternalUserService.diffUpdate()   好友列表
    │
    ├─② 发送给前端（WebSocket）
    │    frontSendService.sendWxFrinedList(wrapper, null, externalUserList)
    │
    ├─③ 上报群列表到云端（HTTP）
    │    reportUploadChatroomList()
    │
    └─ clearData()  清除缓存
```

### 阶段 4：sendWxFrinedList 内部的上报

```
sendWxFrinedList()     ← frontSend.js
    │
    ├─ sendMessageToFront()   → 发给前端 UI
    │
    └─ asyncTask.uploadFriendInfo()  → HTTP 上报好友列表到云端
         │  注意: 此处没有 await，异步执行
         │
         ▼
       uploadFriendInfo()     ← asyncTask.js
         │  HTTP POST → friendsListUrl
         │  生产: http://qun-center.umeng100.com/window/task/friendInfos
         │  请求体: { username, friendInfos: [{headUrl, nickname, ...}] }
```

## 七、上报字段详解与已知缺陷

### 上报接口字段结构（FriendInfo BO）

文件：`src/msg-center/core/bo/cloud/upstream/friendInfo.js`

| 字段 | 类型 | 说明 |
|------|------|------|
| `username` | string | 微信内部 ID（`wxid_xxxxx`），在 `frontSend.js` 的 map 中由 `item.wxid` 赋值 |
| `alias` | string | 微信号（用户自设，如 `LYC3399lyc`） |
| `nickname` | string | 昵称 |
| `remark` | string | 备注 |
| `headUrl` | string | 头像 URL |
| `city` / `province` | string | 城市/省份 |
| `sex` | number | 性别（0未知/1男/2女） |
| `conRemark` | string | 个性签名 |

### ⚠️ 缺陷：alias（微信号）在主链路中丢失

**两个字段的区别**：

| 字段名 | 含义 | 示例 |
|--------|------|------|
| **内部 wxid** | 微信系统分配，唯一不变 | `wxid_9yln7xtqm63b22` |
| **微信号（alias）** | 用户自设，可修改 | `LYC3399lyc` |

逆向 `userinfolist` 返回的原始数据中，`username` 字段存放的是**微信号**（不是内部 wxid）。

**当前主链路（`handleRoomMembers`）对该字段的处理**：

```javascript
// wxUserListResponseMsgHandler.js 第630-643行
// 解构时没有取 username（微信号）字段 → 直接丢弃
const {
    wxid,       // 内部 ID
    remark, sex, nickname, headimg, bigheadimg,
    ownerwxid, contacttype, local_type,
    city, province, country,
    // ❌ username（微信号）没有被解构，已丢失
} = external;

const userDto = {
    wxid,
    nickname,
    remark,
    headimg,
    // ❌ alias 字段缺失
};
```

```javascript
// frontSend.js 第786-790行
// map 时 username 字段被内部 wxid 覆盖
asyncTask.uploadFriendInfo(accountId, externalUserList.map(item => ({
    ...item,
    headUrl: item.headimg,
    username: item.wxid  // ← 用内部 wxid 覆盖了 username，alias 完全不存在
})));
```

**各链路 alias 传递对比**：

| 链路 | 文件 | alias 是否传递 |
|------|------|----------------|
| **旧链路** `friendsListResponseService.js` | 第126行 `alias: data.username` | ✅ 有（取自逆向 username 字段） |
| **主链路** `wxUserListResponseMsgHandler.js` | `handleRoomMembers` | ❌ **无，已丢失** |
| 好友请求 `addFriendRequest.js` | 第138行 | ✅ 有 |
| 修改备注 `modContactRemarkResponseService.js` | 第97行 | ⚠️ 用 wxid 替代（注释说明无该字段） |

**修复方案**（如需补充 alias 字段）：

```javascript
// handleRoomMembers 解构中加上 username
const { wxid, username, remark, ... } = external;

// userDto 中加上 alias
const userDto = {
    wxid,
    alias: username,  // 微信号
    ...
};
```

## 八、handleRoomMembers 过滤规则（影响好友数量）

`handleRoomMembers` 内部有两个**静默过滤**条件，命中后直接 `continue` 跳过，**无任何日志输出**：

```javascript
// 过滤1：contacttype 位运算检查
// 服务号、公众号、企业号等特殊账号会被过滤
if (contacttype && ((parseInt(contacttype) & 8) != 0 || parseInt(contacttype) % 2 != 1)) {
    continue;  // ← 静默丢弃
}

// 过滤2：local_type 不等于 1
if (local_type && local_type != 1) {
    continue;  // ← 静默丢弃
}
```

**常见场景**：逆向返回 9547 个好友，最终上报 9417 个（少了 130 个），这 130 个通常是被上述过滤条件过滤的特殊账号，属于**正常行为**。

**数量漏斗各阶段日志**：

| 阶段 | 日志关键词 | 数量含义 |
|------|-----------|----------|
| getuserlist | `逆向返回好友&群列表总数:[xxx]` | 逆向返回的原始 wxid 总数 |
| fetchRoomMembers | `好友列表总页=x 好友数=xxx` | 过滤系统账号后、分页请求的数量 |
| handleRoomMembers | `curNode:x 好友总数：xxx`（每页） | 逆向返回的每页详情数量 |
| handleSendFriendToFront | `好友数(缓存):xxx` | 经 contacttype/local_type 过滤后的数量 |
| 取到的用户数据为 | `好友xxx个` | 最终上报的数量 |

## 九、数量差异排查方法（数量漏斗比对）

当发现好友数量不对时，按如下漏斗逐级比对：

```
逆向 getuserlist 返回总数（handlerGetUserList 日志）
    ↓ 过滤系统账号
fetchRoomMembers 请求总数（fetchRoomMembers 日志）
    ↓ 逆向逐页返回详情
每页 userinfolist 返回数量加总（execute 日志 curNode）
    ↓ contacttype / local_type 过滤
handleSendFriendToFront 缓存数量（handleSendFriendToFront 日志）
    ↓ sendWxFrinedList 上报
最终上报数量（取到的用户数据为 日志）
```

**哪个阶段数量掉了，问题就在哪里**：

| 第1步总数 | 第3步加总 | 第4步缓存数 | 第5步上报数 | 结论 |
|-----------|-----------|-------------|-------------|------|
| **少** | - | - | - | 逆向 getuserlist 阶段缺少该好友 |
| 正常 | **少** | - | - | 逆向 userinfolist 某页未完整返回 |
| 正常 | 正常 | **少** | - | contacttype / local_type 过滤导致（通常正常） |
| 正常 | 正常 | 正常 | **少** | sendWxFrinedList 内部数据库 pageSize 截断 |
| 正常 | 正常 | 正常 | 正常 | 客户端处理正常，问题在云端或其他 |

## 十、在用户机器上调试的方法

### 注意事项

1. **日志 level 必须用 `error`**：`electron-log` 默认只写 `error` 级别到本地文件，`warn`/`info` 不会写入
2. **不能用微信号作为搜索目标**：逆向返回的是内部 wxid（`wxid_xxxxx`），不是用户设置的微信号（如 `LYC3399lyc`），用微信号做条件判断永远匹配不到
3. **用昵称或备注查找内部 wxid**：先在阿里云搜 `"用户名" and "昵称"` 找到内部 wxid，再用内部 wxid 做后续判断

| `"lvzhibin" and "好友列表上传成功"` | ❌ 查不到 |
| `"lvzhibin" and 企业微信好友列表上传*` | ✅ 可以查到 |
| 仅 `"lvzhibin"` 后下载 JSON | ✅ 包含该条日志 |

### 根本原因

阿里云 SLS 日志服务的**分词索引机制**导致：

1. **中文感叹号 `！`**: 日志原文是 `企业微信好友列表上传成功！response=[...]`，`！` 是中文全角感叹号，SLS 默认将其作为分词分隔符，导致 `"成功"` 和 `！` 后面的内容被拆开索引
2. **精确短语匹配失效**: 双引号 `""` 在 SLS 中表示精确短语匹配，但分词后 token 被拆散，精确匹配找不到完整短语
3. **通配符绕过分词**: `*` 通配符不依赖精确短语匹配，它匹配以前缀开头的所有 token，因此 `好友列表上传*` 能够命中

### 最佳实践

| 场景 | 推荐做法 | 示例 |
|------|----------|------|
| 含中文标点的日志 | 用通配符 `*` | `好友列表上传*` |
| 长中文短语 | 拆分为多个短关键词 + AND | `"好友" and "上传" and "成功"` |
| 英文关键词 | 双引号精确匹配 | `"handleSendFriendToFront"` |
| 不确定完整文本 | 去掉引号用模糊匹配 | `好友列表上传成功` |
| 含特殊字符 | 截断到特殊字符前 + 通配符 | `企业微信好友列表上传*` |

### 常见分词陷阱字符

| 字符 | 说明 | 处理方式 |
|------|------|----------|
| `！` `？` `：` `，` | 中文全角标点 | 截断 + 通配符 |
| `[` `]` `(` `)` | 方括号/圆括号 | 截断 + 通配符 |
| `=` `:` | 键值分隔符 | 拆分为前后两部分 |
| 空格 | 默认分隔符 | 用双引号包裹或拆分查询 |
