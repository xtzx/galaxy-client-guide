## 一、功能概述

### 1.1 业务描述

「踢私加好友」功能的完整业务逻辑如下：

1. **监控阶段**：当前机器人号在群内被某个群成员私加好友
2. **检测来源群**：客户端识别到该好友来自于某个群（通过 `findChatroomsByMember`）
3. **上报云端**：将带有来源群信息（`chatroom` + `scene=14`）的 609 消息上报给云端
4. **云端决策**：云端判断为群内私加好友行为，检查群主号/管理员号是否在线
5. **下发踢人任务**：如果群主号/管理员号在线，云端向该号下发 `CHATROOM_KICK_OUT`（type=3）任务
6. **执行踢人**：群主号/管理员号客户端收到踢人任务，构建 `delchatmenber` 指令发给逆向执行
7. **全群踢出并拉黑**：群主号需要在自己所有管辖的群内踢出该用户并拉黑

### 1.2 涉及角色

| 角色 | 说明 |
|------|------|
| 监控号（机器人号） | 在群内被私加好友的号，负责监控和上报 |
| 群主号/管理员号 | 有踢人权限的号，执行实际的踢人操作 |
| 云端（Sun/Java后端） | 接收上报、做出踢人决策、下发踢人任务 |
| 逆向（DLL/IPC） | 与微信客户端交互，执行实际的微信操作 |
| 前端（Web UI） | 展示状态、手动触发踢人 |

### 1.3 模块配置

**文件**: `config/weixinzhushou/runtime.yml`

```yaml
modules:
    - autoKickout             # 自动踢人模块
    - privateAddFriends       # 私聊加好友模块
    - AddGroupFriend          # 群内加好友模块
```

> 注意：这些 module 名仅作为功能开关传递给服务端，在 galaxy-client 源码中不直接引用。
> 真正的业务逻辑分散在各个 convert-service / convert-response / task-mqtt 模块中。

### 4.2 通过好友回执处理（核心）

**文件**: `src/msg-center/business/convert-response/wx4AddFriendTaskResponse.js`

这是**私加好友检测的核心文件**，处理逆向返回的 `addfriend` 回执。

#### 4.2.1 过滤条件（第54-57行）

```javascript
filter(clientRecord) {
    return clientRecord.type === 'addfriend' &&
           !clientRecord.data?.roomid;  // 排除群内加好友（有 roomid 的）
}
```

#### 4.2.2 来源群检测函数 `findChatroomsByMember`（第66-86行）

```javascript
findChatroomsByMember(ownerWxId, memberWxid) {
    try {
        // 从群成员信息表中查找该成员所在的所有群
        const allMembers = chatroomMemberInfoService.findByOwnRobot(ownerWxId);
        const chatroomList = [];
        for (const member of allMembers) {
            if (member.username === memberWxid && member.chatroom) {
                chatroomList.push(member.chatroom);
            }
        }
        return chatroomList;
    } catch (error) {
        logUtil.customLog(
            `[codeError] [findChatroomsByMemberError] wxId:${ownerWxId}, memberWxid:${memberWxid}`,
            { level: 'error', errorKey: 'findChatroomsByMemberError' }
        );
        return [];
    }
}
```

**关键依赖**: `chatroomMemberInfoService.findByOwnRobot(ownerWxId)`

**数据来源**: `src/msg-center/business/dao-service/chatroomMemberinfoService.js`（第288-291行）

```javascript
findByOwnRobot(ownRobot) {
    const result = ChatroomMemberInfoModel.findByOwnRobot(ownRobot);
    return result || [];
}
```

> **⚠️ 关键问题**: 如果群成员缓存为空或过期，`findChatroomsByMember` 将返回空数组，
> 导致 `sourceChatroom` 为 null，609 不带来源群信息，云端无法判定为群内私加好友！

#### 4.2.3 主处理函数 `operate`（第218-315行）

核心处理流程：

```
Step 1: 解析回执消息 → 获取 status、friendWxid、taskId
Step 2: 如果有 taskId → 调用 this.taskResponse() 上报云端
Step 3: 如果 status === 0（成功）：
    3a. 从 FriendRelation 表获取缓存的好友申请信息
    3b. 保存基本好友信息到数据库
    3c. 调用 findChatroomsByMember() 检测来源群  ← 核心
    3d. 加入待上报609队列（addToPending609Queue）
    3e. 延迟2秒发送 userinfo 请求获取完整信息
Step 4: 如果 status !== 0 → 记录失败日志
```

关键代码（第265-298行）：
```javascript
// 3. 检测好友来源群
const sourceChatroomList = this.findChatroomsByMember(ownerWxId, friendWxid);
const sourceChatroom = sourceChatroomList.length > 0 ? sourceChatroomList[0] : null;

// 4. 加入待上报队列
this.addToPending609Queue(friendWxid, ownerWxId, timer, sourceChatroom, pipeLineWrapper);

// 5. 延迟2秒后发送 userinfo 请求
setTimeout(() => {
    const userinfoTask = {
        type: 'userinfo',
        data: { flag: 0, to: friendWxid, chatroom: "" }
    };
    if (sourceChatroom) {
        userinfoTask.data.chatroom = sourceChatroom;
    }
    ReverseSendService.sendMessage(ownerWxId, pipeLineWrapper.id, JSON.stringify(userinfoTask));
}, 2000);
```

**关键日志**：
```
[wxid-${ownerWxId}] [Wx4AddFriendTaskResponse] 通过好友申请${statusText}: friend=${friendWxid}
[wxid-${ownerWxId}] [Wx4AddFriendTaskResponse] 开始处理新好友: ${friendWxid}
[wxid-${ownerWxId}] [Wx4AddFriendTaskResponse] 已加入待上报队列: friend=${friendWxid}
```

#### 4.2.4 待上报队列与609上报

**队列管理** `addToPending609Queue`（第96-123行）：

```javascript
addToPending609Queue(friendWxid, ownerWxId, timer, sourceChatroom, pipeLineWrapper) {
    const taskInfo = { friendWxid, ownerWxId, timer, sourceChatroom, pipeLineWrapper, createTime: Date.now() };
    pending609ReportQueue.set(friendWxid, taskInfo);

        const task = pending609ReportQueue.get(friendWxid);
        if (task) {
            this.report609FromQueue(friendWxid);

}
```

## 五、阶段三：609上报云端与云端决策

> 本阶段是**私加好友踢人链路中最关键的衔接环节**。  
> 只有 609 上报成功、且携带了正确的 `chatroom` + `scene=14`，云端才会触发后续踢人流程。

### 5.0 如何判断到此步骤为止逻辑正常？

在客户端控制台中如果看到以下两条日志：

```
[wxid-wxid_xxx] [Wx4AddFriendTaskResponse] ⏰ 延迟1秒后发送 userinfo 请求（确保数据同步完成），等待响应后上报609
[wxid-wxid_xxx] [Wx4AddFriendTaskResponse] 从队列上报609: friend=wxid_yyy, nickname=Xxx, headUrl=有
```

说明以下环节**已正常完成**：

| 步骤 | 状态 | 说明 |
|------|------|------|
| 1. 逆向返回 `addfriend` 回执 | ✅ | `filter()` 匹配成功，`operate()` 被调用 |
| 2. 通过好友申请成功 | ✅ | `status === 0`，否则不会执行后续逻辑 |
| 3. 好友信息保存到本地数据库 | ✅ | `wkExternalUserService.saveExternalUsersOp('add', friendInfo)` |
| 4. `findChatroomsByMember` 执行 | ⚠️ **无法从日志确认结果** | 执行了但**没有打印** sourceChatroom |
| 5. 加入待上报队列 | ✅ | 队列中有数据，否则后续 report609 不会触发 |
| 6. 延迟2秒后发 userinfo 请求 | ✅ | 第一条日志证明 |
| 7. userinfo 响应后上报609 | ✅ | 第二条日志证明 report609FromQueue 被调用 |
| 8. 好友完整信息获取 | ✅ | nickname=Joy🔥, headUrl=有 |
| 9. MQTT 消息实际发送 | ⚠️ **需额外确认** | 需查看 MQTT 发送成功日志 |
| 10. 609 携带 chatroom + scene | ⚠️ **需额外确认** | 这是最关键的一点 |

**⚠️ 重点关注**：上述日志只能证明 `report609FromQueue` 函数被执行了，但**无法确认**：

1. **`sourceChatroom` 是否有值** — `report609FromQueue` 的日志没有打印 sourceChatroom
2. **MQTT 是否实际发送成功** — 需要看 `mqExcuteMsg` 的日志
3. **609 消息体是否包含 chatroom 和 scene=14** — 需要看 MQTT 发送的完整 JSON

#### 确认方法

在控制台搜索以下日志来完整验证：

```
# 第1步：确认 609 消息体内容（包含 chatroom 和 scene）
搜索: mqtt sendMessage clientMsg
在结果中找 "type":609 的消息，检查 acceptFriendInfo 中是否有：
  "chatroom": "xxx@chatroom"   ← 不为空才正确
  "scene": 14                   ← 必须为14

# 第2步：确认 MQTT 实际发送成功
搜索: MQTT发往云端数据成功
在结果中匹配对应的 msgId

# 第3步：如果 chatroom 为空，说明 findChatroomsByMember 未找到来源群
搜索: findChatroomsByMemberError   ← 检查是否有异常
```

### 5.1 609 上报协议：MQTT（非HTTP/WS）

**609 消息通过 MQTT 协议上报到云端，不是 HTTP 接口，也不是 WebSocket。**

#### 完整调用链


### 5.5 云端决策逻辑（服务端，非galaxy-client）

云端接收到 609 消息后的处理逻辑（推测）：

1. 检查 `acceptFriendInfo.scene` 是否为 14（群内加好友）
2. 检查 `acceptFriendInfo.chatroom` 是否非空
3. 如果满足条件 → 判定为群内私加好友
4. 查找该群的群主号/管理员号
5. 检查群主号/管理员号是否在线（通过心跳 611 和 MQTT 连接状态）
6. 如果在线 → 向群主号/管理员号下发 `CHATROOM_KICK_OUT`（type=3）MQTT 任务
7. 如果不在线 → 记录待处理，等上线后再下发

> 不会触发踢人流程。这是当前功能异常的最可能原因之一。

### 5.6 MQTT 协议补充说明

#### MQTT 与 HTTP/WS 的区别

| 特性 | MQTT | HTTP | WebSocket |
|------|------|------|-----------|
| 协议类型 | 发布/订阅 | 请求/响应 | 全双工 |
| 在 Network 面板可见 | ❌ 不可见 | ✅ 可见 | ✅ 可见 |
| 连接方式 | 长连接 | 短连接 | 长连接 |
| 本项目用途 | 消息上报(609)、任务下发(踢人) | — | 前端通信 |

#### 为什么在浏览器 Network 面板看不到？

MQTT 连接是在 Electron **主进程**（Node.js 环境）中建立的，使用的是 `mqtt` npm 包（基于 TCP/TLS），
不经过浏览器的网络栈，因此**无法在 Chrome DevTools 的 Network 面板中看到**。

#### 如何查看 MQTT 通信？

1. **Electron 主进程控制台**：所有 MQTT 收发日志都打印在主进程
2. **日志文件**：查看应用日志目录下的日志文件
3. **阿里云 SLS**：查询云端收到的 MQTT 消息（参考第十五节）

```javascript
    let {id: taskId, toUsernames, chatroom} = serverTask;

    // 1. 校验被踢用户列表
    if (!toUsernames || toUsernames.length === 0) {
        let reason = "用户不存在";
        this.send2Cloud(taskId, wxId, reason);
        return true;
    }

    // 2. 获取当前群成员列表
    let memberSetOld = ChatroomMemberInfoService.getMemberList(chatroom);
    let memberOldMap = new Map();
    for(let i = 0; i < memberSetOld.length; i++) {
        memberOldMap.set(memberSetOld[i], true);
```

## 十四、已知问题与潜在Bug分析

### 14.1 【高优先级】群成员缓存为空导致 findChatroomsByMember 失败

**问题描述**:
`findChatroomsByMember` 依赖 `chatroomMemberInfoService.findByOwnRobot()` 
获取所有群成员数据。如果缓存为空或未同步完成，会返回空数组。

**影响**:
- `sourceChatroom` 为 null
- 609 上报不带 `chatroom` 和 `scene=14`
- 云端无法识别为群内私加好友
- 不会触发踢人流程

**排查方法**:
```
日志搜索: [findChatroomsByMemberError]
日志搜索: findChatroomsByMember
检查: sourceChatroomList 是否为空数组
```

### 14.2 【高优先级】recvMsgsService 的609不带来源群信息

**问题描述**:
`recvMsgsService.processPrivateSystemMsg` 中的609上报路径没有调用
`findChatroomsByMember`，因此不包含来源群信息。

**文件**: `src/msg-center/business/convert-service/recvMsgsService.js`（第900-1091行）

**影响**: 如果该路径先于 `wx4AddFriendTaskResponse` 触发并上报609，
云端收到的是不带来源群信息的609，无法触发踢人。

**排查方法**:
```
日志搜索: [recvmsgs] 识别到对方通过好友申请
检查: 该609是否在 wx4AddFriendTaskResponse 之前触发
```

### 14.3 【中优先级】chatroomInfoCache.getMemberKickOut 存在bug

**文件**: `src/msg-center/core/cache/chatroomInfoCache.js`（第488-498行）

```javascript
getMemberKickOut(chatroomId, toUsername) {
    if (!chatroomId || !toUsername) {
        return null;
    }
    // BUG: 应该是 return，但这里用了赋值
    return (this.memberKickOutMap[chatroomId + toUsername] = null);
}
```

**问题**: `this.memberKickOutMap[chatroomId + toUsername] = null` 是赋值操作，
会将缓存值设为 null 并返回 null。正确实现应该是：
```javascript
return this.memberKickOutMap[chatroomId + toUsername];
```

### 14.4 【中优先级】forwardTask 中 sendKickOutFailFront 缺少 request 参数

**文件**: `src/msg-center/business/task-front/forwardTask.js`（第259行）

```javascript
sendKickOutFailFront(clientTask, "用户已经不在群内或群不存在");
// 缺少第三个参数 request，导致 sendToFront 中 channelId 和 wxid 为 undefined
```

### 14.5 【低优先级】MqttKickOutService 中过滤逻辑未实际使用

**文件**: `src/msg-center/business/task-mqtt/mqttKickOutService.js`（第80-90行）

```javascript
let memberListKickOut = toUsernames.filter(wxid => memberOldMap.has(wxid));
// memberListKickOut 被计算但未使用
// 实际构建 clientTask 时使用的是 serverTask.toUsernames（原始列表）
```

**影响**: 即使用户已不在群内，仍会尝试踢出，导致逆向执行失败。

### 14.6 【高优先级】609上报时序问题

**问题描述**:
通过好友后，存在两条并行的609上报路径：
1. `wx4AddFriendTaskResponse.operate` → addfriend 回执 → 延迟2秒 → userinfo → 上报609
2. `recvMsgsService.processPrivateSystemMsg` → 系统消息 → 延迟5秒 → 上报609

虽然路径2有去重检查（检查 pending609ReportQueue），但如果路径1异常或延迟，
路径2可能先触发，导致上报不带来源群信息的609。

## 十八、排查Checklist

### 18.1 排查步骤

当「踢私加好友」功能不工作时，按以下步骤排查：

- [ ] **Step 1**: 确认监控号收到了好友申请
  - 搜索日志: `Wx4AddFriendRequest AND wxid-监控号`
  - 预期看到: "收到疑似好友申请" 和 "确认为新好友申请"

- [ ] **Step 2**: 确认608已上报
  - 搜索日志: `"好友申请已上报云端" AND wxid-监控号`

- [ ] **Step 3**: 确认云端下发了通过好友任务
  - 搜索日志: `"messageType-104" AND wxid-监控号`

- [ ] **Step 4**: 确认通过好友成功
  - 搜索日志: `Wx4AddFriendTaskResponse AND "通过好友申请成功" AND wxid-监控号`

- [ ] **Step 5**: 确认来源群检测成功
  - 搜索日志: `Wx4AddFriendTaskResponse AND wxid-监控号`
  - 检查: sourceChatroomList 是否有值

- [ ] **Step 6**: 确认609已上报且包含来源群信息
  - 搜索日志: `"已上报609" AND wxid-监控号`
  - 关键: 确认日志中有 chatroom 字段

- [ ] **Step 7**: 确认群主号/管理员号在线
  - 搜索日志: `HEART_BEAT AND wxid-群主号`
  - 确认: 最近有心跳上报

- [ ] **Step 8**: 确认群主号收到了踢人任务
  - 搜索日志: `"kick-out" AND wxid-群主号`
  - 或: `"messageType-3" AND wxid-群主号`

- [ ] **Step 9**: 确认踢人执行成功
  - 搜索日志: `"kickOutTaskResponseImpl" AND wxid-群主号`

- [ ] **Step 10**: 确认逆向连接正常
  - 搜索日志: `"dispatchInBound" AND wxid-群主号`
  - 排除: "尚未和逆向建立注册信息"

### 18.2 常见问题对照表

| 症状 | 可能原因 | 排查方法 |
|------|----------|----------|
| 609 不带 chatroom | 群成员缓存为空 | 检查 findChatroomsByMember 日志 |
| 609 不带 chatroom | recvMsgsService 路径先触发 | 检查两个609上报路径的时序 |
