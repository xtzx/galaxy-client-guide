# 26-微信助手工单排查通用方法论

> 面对各类微信助手工单问题的通用排查思路、消息发送失败专项排查路径、以及从海量日志中快速定位问题的方法论

---

## 一、快速排查命令（SLS 查询语句）

> 拿到工单后，按顺序执行以下查询即可快速定位问题。

### 1.1 Step 1：获取目标 wxid

工单中通常只提供微信昵称、微信号或备注名，但日志排查需要的是 `wxid_xxx` 格式的微信 ID。

> **首选：在运营后台直接查询**。后台的好友列表/群列表中直接展示了 wxid 与昵称/微信号的对应关系，无需翻日志。

如果后台查不到，再用以下 SLS 日志查询：

#### SLS 中文查询注意事项（必读）

SLS 查询中文经常返回空结果，三个坑都要注意：


| 问题     | 说明                                                     | 解决方法                       |
| ------ | ------------------------------------------------------ | -------------------------- |
| 运算符大小写 | `and`/`or`/`not` 被当作普通搜索词                              | 改用全大写 `AND` / `OR` / `NOT` |
| 分词索引   | SLS 按分隔符切分 token，不支持子串匹配                               | 搜**完整 token**或用 `LIKE` 语法  |
| 全角标点   | 日志中的全角冒号 `：`（U+FF1A）不在 SLS 默认分隔符列表中，会和前面的中文粘连成一个 token | 搜索时**带上全角冒号**              |


SLS 默认分隔符（空格、`]`、`{`、`"` 等）**不包含全角冒号 `：。`**



**正确做法（从优到劣）：**

```bash
 # ✅ 推荐方案，从结果中找用户的 wxid
[用户名称] | WHERE  message LIKE '%[用户昵称或者其他信息]%'

# 客户端每次同步好友列表时，会把完整的 friendList JSON 打入日志，包含所有好友数据，客户端启动后同步好友列表，或前端主动请求好友列表刷新。是全量数据，一定能查到所有好友。
[用户名称] AND friendList

# 搜完整 token（带上全角冒号，与实际索引 token 完全一致，支持下载）
[用户名称] AND 发送前端WS消息：

# ⚠️ SQL LIKE 语法（绕过分词索引，走全文扫描子串匹配）
[用户名称] | WHERE message LIKE '%发送前端WS消息%'
```

> **经验总结**：遇到中文搜不到时，优先找日志里的等效**英文字段名**（如 `friendList`、`updateFriend`、`sendMsgs` 等）。如果必须搜中文，用 `| WHERE message LIKE '%中文%'` 做子串匹配最保险。

**LIKE 语法的两个注意事项：**

1. **结果可能过多**：`发送前端WS消息` 日志在**每条** WS 消息下发时都会打印（包括 `getAllConfig`、`userlist`、聊天消息转发等），如果只用 `LIKE '%发送前端WS消息%'` 会命中所有类型的 WS 消息。需要叠加条件精确过滤：

```bash
# LIKE 叠加多条件，只匹配含 friendList 的好友列表日志
[用户名称] | WHERE message LIKE '%发送前端WS消息%' AND message LIKE '%friendList%'
```

1. **不支持下载**：LIKE 语法走全表扫描（scan 模式），SLS 会提示「暂不支持 scan 模式数据下载」。如果需要导出日志文件，必须换成索引查询：


| 查询方式                          | 是否支持下载                           |
| ----------------------------- | -------------------------------- |
| `[用户名称] AND friendList`（索引查询） | ✅ 支持                             |
| `[用户名称] AND 发送前端WS消息：`（索引查询）  | ✅ 支持                             |
| `[用户名称]                       | WHERE message LIKE '%..%'`（扫描查询） |


---

#### 通过 `updateFriend` 查单个好友（最精确，推荐）

当好友信息发生变更时（如修改备注、新增好友），客户端会触发**单个好友**的 `updateFriend` 事件，日志里只包含这一个好友的完整信息，无需从列表中翻找：

```bash
# 搜单个好友变更事件，再加昵称/备注/微信号定位
[用户名称] AND updateFriend AND [昵称/备注/微信号]

# 只加 updateFriend 过滤，结果少，人工翻
[用户名称] AND updateFriend
```

> 命中的日志结构如下（只有这一个好友的数据）：
>
> ```json
> "updateFriend":{
>   "wxid":"wxid_xxx",
>   "nickname":"对方昵称",
>   "remark":"备注",
>   "alias":"对方微信号",
>   "headimg":"..."
> }
> ```

**触发场景**：好友信息变更（修改备注、昵称变动等）、新增好友通过验证、修改备注任务成功后。由 `friendUpdateService` 和 `changRemarkMsgStrategy` 等模块产生。

**局限**：只有近期有过信息变更的好友才会有 `updateFriend` 日志；如果该好友一直没发生过变更，则找不到。

### 1.2 Step 2：获取任务 ID（taskId）

拿到目标 wxid 后，下一步是找到对应的 MQTT 任务。客户端每收到一条 MQTT 任务，`mqttClientBase.js` 会打印一条包含完整 payload 的日志，从中可以读取 `id`（即 taskId）和 `type`（任务类型，参见 2.3 枚举表）。

```bash
# 查该用户收到的、指向某目标好友的 MQTT 任务（精确，结果少）
[用户名称] AND [目标wxid] | WHERE message LIKE '%[mqttClientBase]:MQTT收到消息%'

# 不知道目标 wxid 时，查全部 MQTT 任务（结果较多，按 type 筛选）
[用户名称] | WHERE message LIKE '%[mqttClientBase]:MQTT收到消息%'
```

> 从命中的日志 payload 中提取：
> - `id` → 即 taskId，后续所有环节的日志都用此 ID 追踪
> - `type` → 任务类型（100=私聊群发, 1=群内发消息, 126=群内加好友, ...）
> - `toUsernames` → 操作目标列表

### 1.3 Step 3：追踪任务全链路

确定 taskId 后，搜该任务在客户端的完整生命周期日志（通常 5-15 条，数据量小，可直接给 AI 分析）：

```bash
# 搜该任务的全部链路日志
[用户名称] AND [taskId]

# 只看三段式回执结果
[用户名称] AND [taskId] AND msgResNodeMap

# 看最终上报结果（status 和 reason）
[用户名称] AND [taskId] AND 通知任务结果

# 看是否超时
[用户名称] AND [taskId] AND 设置任务超时返回
```

> 一个 taskId 通常产生 5-15 条日志，涵盖接收、分发、逆向执行、回执、上报全流程。各环节的日志关键字详见 2.5 节。

### 1.4 高效排查流程（结合 AI 工具）

```
① 获取目标 wxid（见 1.1）
   │
   ▼
② SLS 搜索：[用户名称] AND [目标wxid] | WHERE message LIKE '%[mqttClientBase]:MQTT收到消息%'
   → 从 payload 的 type 筛选目标任务，提取 id 字段 = taskId（见 1.2）
   │
   ▼
③ SLS 搜索：[用户名称] AND [taskId]
   → 得到 5-15 条日志（数据量很小）（见 1.3）
   │
   ▼
④ 全选日志 → 丢给 AI 工具分析
   AI 提示词参考：
   "以下是微信助手客户端的日志，请分析这个任务是否成功。
    如果失败，请告诉我：
    1. 任务类型是什么
    2. 在哪个环节失败的
    3. 失败原因是什么
    4. 三段式回执是否完整"
```

### 1.5 注意：不要直接搜前端 messageId

前端页面显示的「任务id」（如 `28800756`）是后端业务层的 messageId，MQTT 下发给客户端时会拆分成多个子任务，每个子任务有独立的 taskId（payload 中的 `id` 字段，如 `1266762014244470961`）。客户端日志只记录子任务 taskId，不记录父任务 messageId。

**正确做法**：用「用户名 + 目标 wxid + 时间范围」反查，或先搜全部 MQTT 任务再按 type 筛选。

## 二、MQTT 日志信息解析

### 2.1 日志格式

客户端每收到一条 MQTT 下发的任务，`mqttClientBase.js` 会打印如下日志：

```
[用户名称] [GID] [sonVersion] [版本号] [号数量: X] wxId=[wxid-机器人wxid][mqttClientBase]:MQTT收到消息，msgId: [taskId],payload: {JSON}
```

### 2.2 payload 字段详解

以一条真实日志的 payload 为例：

```json
{
  "elfKey": "tongbao",
  "id": "1266770393398509632",
  "traceId": "ss-robot-facade.544.17726072073230512",
  "logicId": 10293466,
  "type": 126,
  "createTime": 1772607207,
  "chatroom": "45227153858@chatroom",
  "toUsernames": ["wxid_6wxkb7xnsmb921"],
  "content": "我是la Lune",
  "sourceDesc": "tb-sales-tool",
  "weChatMsgType": 1
}
```

**每个字段的含义和作用**：


| 字段              | 含义                     | 作用                                                      | 示例                          |
| --------------- | ---------------------- | ------------------------------------------------------- | --------------------------- |
| `id`            | **任务唯一标识（taskId）**     | 后续所有日志都用此 ID 追踪。代码中 `clientTask.taskId = serverTask.id` | `"1266770393398509632"`     |
| `type`          | **任务大类型（SunTaskType）** | 决定这个任务属于哪个功能模块，**是判断任务类型的核心字段**                         | `126`（群内加好友）                |
| `elfKey`        | 业务标识                   | 固定值 `"tongbao"`，标识来源系统                                  | `"tongbao"`                 |
| `traceId`       | 链路追踪 ID                | 可用于在后端服务中追踪请求来源                                         | `"ss-robot-facade.544.xxx"` |
| `logicId`       | 后端业务 ID                | 在运营后台管理系统中可查询此 ID                                       | `10293466`                  |
| `createTime`    | 任务创建时间                 | Unix 时间戳（秒），用于过期判断（超过一定时间会被丢弃）                          | `1772607207`                |
| `toUsernames`   | 操作目标列表                 | 发送/操作的目标微信号，数组格式                                        | `["wxid_6wxkb7xnsmb921"]`   |
| `chatroom`      | 群号                     | 群内任务才有，格式为 `数字@chatroom`                                | `"45227153858@chatroom"`    |
| `content`       | 消息/操作内容                | 文本消息内容、验证消息、或媒体文件元信息                                    | `"我是la Lune"`               |
| `sourceDesc`    | 来源描述                   | 标识哪个服务触发的任务                                             | `"tb-sales-tool"`           |
| `weChatMsgType` | 微信消息子类型                | **仅 type=1 和 type=100 的发消息任务才有**，决定具体消息格式               | `1`（文字）                     |
| `ext`           | 扩展信息                   | 部分任务携带的额外信息（如引用消息）                                      | JSON 对象                     |
| `mediaUrl`      | 媒体文件地址                 | 图片/视频/文件消息的 OSS 下载地址                                    | URL 字符串                     |
| `thumbUrl`      | 缩略图地址                  | 图片/视频的缩略图地址                                             | URL 字符串                     |

### 2.4 type 和 weChatMsgType 的关系

`type` 和 `weChatMsgType` 是两个不同层级的类型：

```
type（SunTaskType）= 任务大类型 → 决定「做什么操作」
    │
    ├─ type=100（私聊发消息）
    │   └─ weChatMsgType 决定「发什么类型的消息」：
    │       ├─ 1 = 文字
    │       ├─ 3 = 图片
    │       ├─ 42 = 名片
    │       ├─ 43 = 视频
    │       ├─ 49001 = 文件
    │       └─ 49002 = 链接
    │
    ├─ type=1（群内发消息）
    │   └─ 同上，也用 weChatMsgType 细分
    │
    ├─ type=126（群内加好友）→ 没有 weChatMsgType，不是发消息任务
    ├─ type=2（邀请入群）→ 没有 weChatMsgType
    └─ ...其他类型同理
```

**总结**：`weChatMsgType` 只是「发消息」类任务（type=1 和 type=100）的消息格式细分，其他类型的任务（加好友、踢人、建群等）没有此字段。

### 2.5 taskId 全链路追踪

确定 taskId 后，用 `[用户名称] AND [taskId]` 可以搜到该任务的全部链路日志。

代码中通过 `clientTask.taskId = serverTask.id` 将 MQTT payload 的 `id` 传递到客户端任务对象，后续所有环节的日志都会携带这个 taskId：


| 环节      | 日志关键字              | taskId 位置                 |
| ------- | ------------------ | ------------------------- |
| MQTT 接收 | `接收mqtt任务`         | `taskId= xxx`             |
| MQTT 详情 | `MQTT收到消息`         | `msgId: xxx` 和 payload.id |
| 任务过期    | `消息过期`             | `taskId=xxx`              |
| 任务去重    | `消息重复`             | `taskId=xxx`              |
| 缓存建立    | `TASK_MSGTYPE_MAP` | 缓存 key 中                  |
| 逆向发送    | `发送消息给逆向`          | 消息体中                      |
| 三段式回执   | `msgResNodeMap`    | 日志内容中                     |
| 失败上报    | `任务失败上报任务结果`       | `taskId:[xxx]`            |
| 超时上报    | `任务超时上报任务`         | `taskId:[xxx]`            |
| 结果上报    | `通知任务结果`           | `taskId:xxx`              |
| MQTT 上行 | `MQTT发往云端数据成功`     | 消息体中                      |


一个 taskId 通常产生 5-15 条日志，数据量很小，可以直接给 AI 工具分析。

## 三、消息发送失败专项排查路径

### 3.1 排查流程图

```
开始排查
    │
    ▼
① 收集基本信息
    ├─ 登录账号（CAS 用户名）
    ├─ 机器人微信号（wxid_xxx）
    ├─ 发送目标（可能是昵称/微信号 → 需先反查 wxid，见第一节）
    ├─ 前端任务 ID（messageId）← 仅供参考，日志中搜不到
    └─ 大致发生时间
    │
    ▼
② SLS 查询该用户收到的 MQTT 任务（见 1.2）
    [用户名称] AND [目标wxid] | WHERE message LIKE '%[mqttClientBase]:MQTT收到消息%'
    时间范围：故障发生前后 1 小时
    → 得到该用户客户端收到的指向目标的 MQTT 任务列表
    │
    ▼
③ 从 payload 中识别目标任务
    ├─ 通过 type 字段判断任务类型（100=私聊群发, 1=群内发消息, ...）
    ├─ 通过 toUsernames 确认发送目标
    └─ 提取 id 字段 → 这就是 taskId
    │
    ▼
④ 用 taskId 搜索该任务的完整生命周期日志
    [用户名称] AND [taskId]
    → 任务链路上的所有日志都会携带这个 taskId
    │
    ▼
⑤ 分析三段式回执状态
    找「msgResNodeMap」日志：
    ├─ first.status:0  → 逆向已发送 ✅
    ├─ second.status:2 → 微信已收到 ✅
    └─ third 有内容     → 服务器确认 ✅
    │
    ├─ 三段齐全 → 成功
    ├─ 缺失第三段 → 超时（GalaxyTaskStatusTimer 上报）
    └─ first.status≠0 → 发送失败
    │
    ▼
⑥ 查看最终上报结果
    找「通知任务结果」日志
    查看 status 和 reason 字段 → 具体失败原因
    参考 wxFailReasonConstant.js 错误码表
    │
    ▼
结束
```

## 四、特定场景排查命令

> 以下是各类工单场景的专用 SLS 查询命令。通用的三步流程（获取 wxid → 获取 taskId → 追踪链路）见第一节。

| 场景 | 需要信息 | SLS 查询命令 |
|---|---|---|
| **消息发送失败** | 用户名 + 目标 wxid | `[用户名称] AND [目标wxid] \| WHERE message LIKE '%[mqttClientBase]:MQTT收到消息%'` |
| **群发全部失败** | 用户名 + 机器人 wxid | `[用户名称] AND [机器人wxid] AND 通知任务结果` |
| **好友列表异常** | 用户名 | `[用户名称] AND friendList` |
| **邀请入群失败** | 用户名 + 群号 | `[用户名称] AND [群号]@chatroom \| WHERE message LIKE '%[mqttClientBase]:MQTT收到消息%'` |
| **登录问题** | 用户名 | `[用户名称] AND (登录 OR login OR 二维码)` |
| **MQTT 连接问题** | 用户名 | `[用户名称] AND (mqttOnError OR mqttOnClose OR mqttOfflie)` |
| **客户端离线/无响应** | 用户名 | `[用户名称] AND (心跳 OR offline OR close)` |
| **任务不下发** | 用户名 + 机器人 wxid | 先检查 MQTT 连接状态，再查后端是否下发 |

## 五、通用工单排查方法论

### 5.1 分层排查

从入口到出口，逐层检查：

```
层级1：MQTT 通道
    └─ 搜索 "MQTT收到消息" → 任务是否到达客户端？
         ├─ 没到达 → MQTT 连接问题 / 云端未下发
         └─ 到达 → 继续

层级2：任务分发
    └─ 搜索 "接收mqtt任务" → 是否被正确识别？
         ├─ 被去重锁拦截 → "消息重复，直接终止"
         └─ 正常分发 → 继续

层级3：逆向执行
    └─ 搜索 "发送消息给逆向" → 是否发给了微信？
         ├─ 没发 → 任务参数校验失败 / 文件下载失败
         └─ 已发 → 继续

层级4：回执上报
    └─ 搜索 "msgResNodeMap" → 微信是否返回结果？
         ├─ 三段齐全 → 成功
         ├─ 缺失 → 超时
         └─ status≠0 → 失败，看错误码

层级5：云端存储
    └─ 搜索 "MQTT发往云端数据成功" → 结果是否上报？
         ├─ 已上报 → 检查后端数据库
         └─ 未上报 → MQTT 上行通道问题
```

### 5.2 从海量日志中获取有效信息

#### 策略一：先搜全部 MQTT 任务，再按 taskId 精确查（推荐）

1. **先搜目标 MQTT 任务**：`[用户名称] AND [目标wxid] | WHERE message LIKE '%[mqttClientBase]:MQTT收到消息%'`
2. **从 payload 的 `type` 字段确定目标任务**，提取 `id`（即 taskId）
3. **用 taskId 精确搜索**：`[用户名称] AND [taskId]`
4. 拿到的日志量很小（5-15 条），直接交给 AI 分析

#### 策略二：缩小范围后下载全量分析

SLS 在线搜索有局限性（分词问题、长字段截断），对于复杂排查场景：

1. 先用宽泛条件搜索：`[用户名称] AND [目标wxid]`
2. 设定精确时间范围（前后 30 分钟）
3. **下载 JSON 格式全量日志**
4. 用脚本/文本编辑器/AI 工具分析

#### 策略三：利用日志关键字分层过滤

每个处理环节都有固定的日志关键字，按顺序搜索：


| 顺序  | 搜索关键字                      | 含义                       |
| --- | -------------------------- | ------------------------ |
| 1   | `接收mqtt任务`                 | 任务是否被识别（含 taskId 和 type） |
| 2   | `MQTT收到消息`                 | 任务详细 payload             |
| 3   | `CloudFlowHandler:inBound` | 是否进入处理流程                 |
| 4   | `下任务时往缓存里放任务`              | 三段式缓存是否建立                |
| 5   | `发送消息给逆向`                  | 是否发给微信                   |
| 6   | `接收逆向推送消息`                 | 微信是否回包                   |
| 7   | `msgResNodeMap`            | 三段式状态                    |
| 8   | `第三条消息`                    | 最终确认                     |
| 9   | `通知任务结果`                   | 结果上报                     |
| 10  | `MQTT发往云端数据成功`             | 是否上报成功                   |
| 11  | `任务超时id`                   | 超时检测                     |
| 12  | `设置任务超时返回`                 | 超时上报                     |


#### 策略四：利用时间线还原事件

日志的 `__time__` 字段是 Unix 时间戳，同一个 taskId 的日志会在很短时间内（通常几秒）连续出现。按时间排序，可以还原完整的事件链。

#### 策略五：对比成功和失败的日志

同一个群发任务中，有的对象成功有的失败。对比成功和失败的日志差异（消息类型、时间间隔、回包内容），可以快速定位根因。
