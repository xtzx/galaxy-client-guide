# 企微与微信消息上报流程 - 三段式回执机制

> 本文档详细说明三段式回执机制的完整实现，包括企业微信、旧版个人微信、微信4.0
> 三种不同的回执顺序，消息节点管理、taskId 匹配、以及 errorOrder 兜底逻辑。

---

## 一、三段式回执机制概述

### 1.1 为什么需要三段式回执

逆向客户端执行一次发送消息操作，不是一次性返回结果，而是**分多次异步返回不同阶段
的回调消息**。微信助手需要将这些分散的回调消息**关联**起来，才能最终确认任务
是否执行成功并拿到 taskId。

### 1.2 三种微信类型的回执顺序

```
┌─────────────────────────────────────────────────────────────┐
│  企业微信三段式回执                                           │
│                                                              │
│  第一条: recvmsg          ← 逆向确认收到消息                  │
│  第二条: sendmsg_*        ← 逆向执行回执(携带taskId!)         │
│  第三条: msgreport        ← 服务端消息确认(携带msg_id等)       │
│                                                              │
│  关联键: wxId_clientId (如 "1688857807622564_CIGABBDq...")   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  旧版个人微信三段式回执                                       │
│                                                              │
│  第一条: msgreport        ← 服务端消息确认                    │
│  第二条: sendmessage      ← 发送回执(携带taskId)              │
│  第三条: recvmsg          ← 接收消息确认                      │
│                                                              │
│  关联键: wxId_toUser_timer (如 "wxid_abc_wxid_def_17731")    │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  微信4.0两段式回执                                            │
│                                                              │
│  第一条: sendmessage      ← 发送回执(携带taskId)              │
│  第二条: recvmsg          ← 接收消息确认(status=2)            │
│                                                              │
│  关联键: wx4-wxId-from-to-msgType-contentHash                │
│  (基于发送人、接收人、消息类型、内容特征的哈希)                  │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 三段式回执对比表

| 特性 | 企业微信 | 旧版个人微信 | 微信4.0 |
|------|---------|------------|---------|
| 阶段数 | 3 | 3 | 2 |
| 第一条类型 | recvmsg 或 sendmsg_* | msgreport | sendmessage |
| 第二条类型 | sendmsg_* 或 recvmsg | sendmessage | recvmsg |
| 第三条类型 | msgreport | recvmsg | 无 |
| taskId 来源 | 第二条(sendmsg_*)携带 | 第二条(sendmessage)携带 | 第一条(sendmessage)携带 |
| 关联键方式 | client_id | timer + toUser | 内容哈希 |
| 关联键生成 | getQywxKey() | getWxMsgKey() | getWx4AssociationKey() |

## 二、企业微信三段式回执详解

### 2.1 workWxMsgHandle.js 实现

文件：`src/msg-center/dispatch-center/handle/workWxMsgHandle.js`

```javascript
const WkMsgHandlerCenter = {
    ...MsgHandlerCenterBase,
    msgResNodeMap: {},

    // 判断是否需要重置消息节点
    // 企业微信：已有first节点时又收到recvmsg → 上一条消息结束
    isReset(sendType, node) {
        return (node && node.first
            && GalaxyCallBackType.RECVMSG.includes(sendType));
    },

    // 判断消息类型是否不在处理范围
    // 企业微信只处理 recvmsg、msgreport、sendmsg_* 三类
    isNotContains(sendType) {
        return !(GalaxyCallBackType.RECVMSG.includes(sendType)
            || GalaxyCallBackType.MSG_REPORT.includes(sendType)
            || sendType?.startsWith(GalaxyCallBackType.WORK_WX_SEND_PREFIX));
    },

    // 第一条消息判断
    // 条件：无现有节点 && (recvmsg 或 sendmsg_* 前缀)
    isFirstMsg(node, sendType) {
        return !node
            && (GalaxyCallBackType.RECVMSG.includes(sendType)
                || sendType.startsWith(GalaxyCallBackType.WORK_WX_SEND_PREFIX));
    },

    // 第二条消息判断
    // 条件：有first无second && (sendmsg_* 前缀 或 recvmsg)
    isSecondMsg(node, sendType) {
        return node && node.first && !node.second
            && (sendType.startsWith(GalaxyCallBackType.WORK_WX_SEND_PREFIX)
                || GalaxyCallBackType.RECVMSG.includes(sendType));
    },

    // 第三条消息判断
    // 条件：有first有second && msgreport 类型
    isThirdMsg(node, sendType) {
        return node && node.first && node.second
            && GalaxyCallBackType.MSG_REPORT === sendType;
    }
};
```

### 2.2 企业微信关联键 (getQywxKey)

企业微信使用 `client_id` 作为关联键，格式为 `${wxId}_${clientId}`。

```javascript
getQywxKey(jsonObj, wxId) {
    const clientId = jsonObj.client_id;
    return `${wxId}_${clientId}`;
}
```

`client_id` 是逆向为每次消息操作生成的唯一标识，同一操作的三条回执消息
共享同一个 `client_id`，从而实现关联。

**示例：**
```
// 同一任务的三条消息共享 client_id
recvmsg:          client_id = "CIGABBDq17/NBhiko8zNnYCAAyDlAg=="
sendmsg_sendtext: client_id = "CIGABBDq17/NBhiko8zNnYCAAyDlAg=="
msgreport:        client_id = "CIGABBDq17/NBhiko8zNnYCAAyDlAg=="

// 关联键
key = "1688857807622564_CIGABBDq17/NBhiko8zNnYCAAyDlAg=="
```

## 三、消息节点管理 (msgResNodeMap)

### 3.1 MsgResNode 数据结构

```javascript
const MsgResNode = {
    first: null,       // 第一条消息摘要
    second: null,      // 第二条消息摘要
    third: null,       // 第三条消息完整内容(用于最终返回)
    expireTime: 0,     // 过期时间戳
    thirdTime: 0       // 第三条消息处理计数
};

// first/second 的摘要结构 (getDefaultMsgResNode)
{
    taskId: jsonObj.taskId,      // 任务ID(可能为空)
    hash: jsonObj.hash,          // 哈希值(个微使用)
    status: jsonObj.status,      // 状态码
    client_id: jsonObj.client_id,// 客户端ID(企微使用)
    type: jsonObj.type           // 消息类型
}
```

### 3.2 节点存储方式

```javascript
// 普通消息：直接存储
msgResNodeMap[key] = msgResNode;

// 视频消息：数组存储(支持多条视频关联)
msgResNodeMap[videoKey] = [msgResNode1, msgResNode2, ...];
```

### 3.3 节点过期清理

每 10 秒执行一次过期检查：

```javascript
removeExpireNode() {
    const now = Date.now();
    Object.keys(this.msgResNodeMap).forEach((key) => {
        const value = this.msgResNodeMap[key];

        if (key.startsWith('video_') && Array.isArray(value)) {
            // 视频消息：过滤过期节点
            const validNodes = value.filter(n => n.expireTime >= now);
            if (validNodes.length === 0) {
                delete this.msgResNodeMap[key];
            } else {
                this.msgResNodeMap[key] = validNodes;
            }
        } else {
            // 普通消息：直接删除过期节点
            if (value.expireTime < now) {
                delete this.msgResNodeMap[key];
            }
        }
    });
}
```

### 3.4 关联键生成方式汇总

| 方法 | 格式 | 适用场景 |
|------|------|---------|
| `getQywxKey()` | `${wxId}_${clientId}` | 企业微信 |
| `getWxMsgKey()` | `${wxId}_${toUser}_${timer}` | 旧版个人微信 |
| `getWxHashKey()` | `${hash}` | 旧版个人微信(第二阶段后换key) |
| `getWx4AssociationKey()` | `wx4-${wxId}-${from}-${to}-${msgType}-${contentHash}` | 微信4.0 |
| `getVideoMsgKey()` | `video_${fromUser}_${toUser}_${msgType}` | 视频消息特殊处理 |

### 4.4 第三条消息处理 (thirdMsgHandler)

```javascript
thirdMsgHandler(msgResNode, msgKey) {
    // 1. 递增处理计数
    msgResNode.thirdTime = msgResNode.thirdTime + 1;

    // 2. 获取taskId (从first或second)
    const {third} = msgResNode;
    let taskId = this.getTaskId(msgResNode);

    // 3. 缓存 client_id → taskId 映射
    if (third.client_id) {
        MsgTaskInfoCache.msgIdToTaskIdInfo[third.client_id] = taskId;
    }

    // 4. 将taskId赋给third消息
    third.taskId = taskId;

    // 5. 更新 GALAXY_TASK_STATUS_MAP (三个状态都设为已接收)
    const galaxyTaskStatus = GalaxyTaskCache.GALAXY_TASK_STATUS_MAP[taskId];
    if (galaxyTaskStatus) {
        galaxyTaskStatus.firstMessageStatus = RECEIVE_STATUS;   // 1
        galaxyTaskStatus.secondMessageStatus = RECEIVE_STATUS;  // 1
        galaxyTaskStatus.thirdMessageStatus = RECEIVE_STATUS;   // 1
    }

    // 6. 清理节点
    this.remove(msgKey);

    // 7. 清理 TaskMsgTypeMap
    GalaxyTaskCache.removeTaskMsgTypeMapToTaskId(taskId);

    logUtil.customLog(
        `${msgKey} [msgConvert]返回的第三条消息内容：${JSON.stringify(third)}`
    );

    // 8. 返回带taskId的third消息 → 继续走 businessHandler 路由
    return third;
}
```

### 4.5 getTaskId 方法

```javascript
getTaskId(node) {
    let taskId = null;
    if (!node) return null;

    // 优先从 first 获取
    if (node.first) {
        taskId = node.first.taskId;
    }
    // first 没有则从 second 获取
    if (node.second && !taskId) {
        taskId = node.second.taskId;
    }
    return taskId;
}
```

**企业微信的 taskId 流转路径：**
```
1. 云端下发任务 → dispatchInBound → 发送给逆向(消息体携带taskId)
2. 逆向收到后，在 sendmsg_* 回执中携带原始 taskId
3. messageHandler 处理第二条(sendmsg_*)时:
   - taskIdMap[qywxKey] 已存有 taskId (dispatchInBound 时缓存的)
   - 将 taskIdMap[qywxKey] 赋给 node.second.taskId
4. thirdMsgHandler 中 getTaskId() 从 node.second 获取 taskId
5. 最终 taskId 被赋给 third 消息，传递给后续上报环节
```

## 五、errorOrder 兜底机制

### 5.1 什么时候进入 errorOrder

当消息**不匹配任何三段式阶段**时，进入 errorOrder 兜底处理：
- node 为 null（没有对应的第一条消息）
- node 有 first 但 isSecondMsg 和 isThirdMsg 都不匹配
- 消息顺序异常（如第二条缺失，第三条直接到达）

### 5.2 errorOrder 实现

```javascript
errorOrder(node, sendType, jsonObject, isWorkWx, key, wxId) {
    logUtil.customLog(
        `[msgConvert]收到消息res: errorOrder ${JSON.stringify(jsonObject)}, ` +
        `sendType-${sendType} node ${JSON.stringify(node)}`
    );

    // 尝试1：从node的first/second获取taskId
    let taskId = this.getTaskId(node);
    if (taskId) {
        delete this.taskIdMap[key];
        return this.thirdMsgHandler(
            {third: jsonObject, first: {taskId}}, key
        );
    }

    // 尝试2：通过内容匹配从缓存中获取taskId
    taskId = this.getCacheTaskId(jsonObject, isWorkWx, wxId);
    if (taskId) {
        return this.thirdMsgHandler(
            {third: jsonObject, first: {taskId}}, key
        );
    }

    // 都失败：无法关联taskId
    logUtil.customLog(
        `wxid-${wxId} [没有匹配到taskId] errorOrder ` +
        `${JSON.stringify(jsonObject)}, sendType-${sendType} ` +
        `node ${JSON.stringify(node)}`
    );

    return jsonObject;
}
```

### 5.3 getCacheTaskId 内容匹配

当 errorOrder 的第一种方式（从 node 获取 taskId）失败时，
会尝试通过消息**内容匹配**来找到 taskId：

```javascript
getCacheTaskId(msgObj, isWorkWx, wxId) {
    // 1. 获取当前消息的内容
    const currentContent = this.extractMessageContent(msgObj);
    const processedContent = this.processContentByType(
        currentContent, wechatMsgType, msgObj
    );

    // 2. 从 GalaxyTaskCache 获取该用户的任务列表
    const taskMsgTypeMap = GalaxyTaskCache.getTaskMsgTypeMap(
        ownerWxId, toUsername
    );

    // 3. 遍历任务列表，尝试类型+内容双重匹配
    taskMsgTypeMap.find((value) => {
        const [taskIdStr, msgType, sendTime, cachedContent] = value.split('-');
        if (wechatMsgTypes.includes(msgType)) {
            if (this.isContentMatch(processedContent, cachedContent, ...)) {
                taskId = taskIdStr;
                return true;
            }
        }
    });

    return taskId;
}
```

| 日志模板 | 说明 |
|----------|------|
| `[msgConvert]收到消息res: errorOrder ${json}, sendType-${type} node ${node}` | 进入 errorOrder |

```
  client_id = "CIGABBCE2L/NBhiko8zNnYCAAyDmAg==" (同一个!)
  content_type = 14, status = 0
  → isThirdMsg 检查:
    node.first = ✅ 存在
    node.second = ❌ 不存在 (因为第二条缺失!)
    → isThirdMsg = false ← 判断失败!

  → 进入 errorOrder:
    → getTaskId(node) → null (first无taskId, second不存在)
    → getCacheTaskId() → null (内容匹配也失败)
    → [没有匹配到taskId] ← 关联彻底失败!

  → msgreport 被返回，但 taskId 为 null
  → [msgConvert][wkTaskResponse]taskId is null ← 无法上报任务结果

结果:
  → GALAXY_TASK_STATUS_MAP[taskId] 状态始终为 {0, 0, 0}
  → GalaxyTaskStatusTimer 超时检测触发
  → [设置任务超时返回]:1268992150411157568
  → 上报 status=6 (OVER_TIME_TASK)
```

### 6.3 根本原因分析

```
                     成功案例(文本)              失败案例(图片)
                     ──────────────              ──────────────
第一条 recvmsg:      ✅ 正常                     ✅ 正常
第二条 sendmsg_*:    ✅ 携带taskId               ❌ 缺失(逆向未返回)
第三条 msgreport:    ✅ 三段式完成               ❌ 进入errorOrder
taskId匹配:          ✅ from node.second          ❌ 无法匹配
最终结果:             ✅ 成功上报(status=3)        ❌ 超时上报(status=6)
```

**结论：逆向对 `sendmsg_sendimage` 任务没有返回第二条回执，导致三段式
匹配链条断裂。图片实际已发送成功（msgreport status=0），但微信助手
无法确认这一点。**
