# 23-企微群发消息排查指南

> 企业微信群发消息的完整链路解析、常见异常诊断与阿里云日志查询实践

---

## 一、企微群发消息的完整处理链路

### 1.1 链路概览

```
云端（运营后台）配置群发任务
         ↓ MQTT 下发
mqttClientBase.js 接收消息
         ↓
mqttWorkWxChatService.js 处理（work-wx-friend-chat）
         ↓
cloudFlowInBound.js（CloudFlowHandler:inBound）
         ↓
dispatchInBound.js → reverseSend.js
         ↓ IPC 管道
逆向 DLL 执行发送
         ↓
三段式回执上报（MQTT 上报云端）
```

### 1.2 关键日志节点（按代码执行顺序）

| 顺序 | 日志关键字 | 打印位置 | 说明 |
|------|-----------|---------|------|
| ① | `[接收mqtt任务] [messageType-100], taskId= xxx [mqttTaskMapLock]` | `mqttClientBase.js` L388 | 最早的入口日志，含去重锁数量 |
| ② | `[mqttClientBase]:MQTT收到消息，msgId: xxx` | `mqttClientBase.js` L442 | 提取 wxId 后 |
| ③ | `work-wx-friend-chat - serverTask=[...]` | `mqttWorkWxChatService.js` L57 | 企微群发服务处理中 |
| ④ | `[CloudFlowHandler]:inBound, wxId=xxx` | `cloudFlowInBound.js` L46 | 转为客户端任务、打来源标记 |
| ⑤ | `[DispatchCenter]:下任务时往缓存里放任务` | `dispatchInBound.js` | 任务放入三段式缓存 |
| ⑥ | `ReverseSendService message` | `reverseSend.js` | 即将写入 IPC 管道 |
| ⑦ | `[发送消息给逆向] wxid: [...] type=[sendmsg_xxx]` | `reverseSend.js` | 已写入管道 |
| ⑧ | `[开始处理任务1]:MQTT收到消息，msgId: xxx` | `mqttClientBase.js` L459 | 在 execute() 调用完成后打印（**晚于 ④**） |
| ⑨ | `任务超时id xxx :[xxx]的超时时间为:[240000]` | 定时器 | 每 10s 一次，持续约 4 分钟 |
| ⑩ | `MQTT发往云端数据成功` | `mqExcuteMsg.js` L118 | 三段式回执上报成功 |

> ⚠️ **注意**：`[开始处理任务1]` 在代码中是 `execute()` 调用后才打印，实际执行顺序晚于 `[CloudFlowHandler]:inBound`，不要被日志文字顺序误导。

### 1.3 两类日志是同一链路的不同节点

```
同一个 taskId（如 1265071821720387667）会依次产生：

[接收mqtt任务] taskId= 1265071821720387667    ← mqttClientBase.js（入口）
[mqttClientBase]:MQTT收到消息，msgId: 1265071821720387667
work-wx-friend-chat - serverTask=[{"id":"1265071821720387667",...}]
[CloudFlowHandler]:inBound, ... taskId":"1265071821720387667"
[发送消息给逆向] ... taskId":"1265071821720387667"
[开始处理任务1]:MQTT收到消息，msgId: 1265071821720387667   ← 代码里在 execute() 后
```

它们共用同一个 `msgId`/`taskId`，是**同一条消息**在不同处理阶段的日志。

## 二、企微群发消息类型说明

| `wechatMsgType` | 发送类型 | 逆向指令 |
|----------------|---------|---------|
| `1` | 纯文本消息 | `sendmsg_sendtext` |
| `43` | 视频消息 | `sendmsg_sendvideo` |
| `49` | 小程序 / 链接 | `sendmsg_sendapplet` |
| `69` | 视频号（finder）| `sendmsg_sendfinder` |
| `602` | 企微小程序 | `sendmsg_sendapplet` |

`type=100` 是群发消息的统一任务类型，`wechatMsgType` 区分具体发什么内容。

## 三、真实案例分析：用户反馈收到两条消息

### 3.1 问题描述

用户 `huanglimin02` 的微信助手在群发时，学员反馈收到了**两条消息**（一条文本 + 一条视频号卡片），实际上只应该收到一条。

截图显示：
- 气泡消息：「💢宝贝3年级思维黄金期分水岭，能力要求更高，重思维培养，开放性题目更多！」
- 视频卡片：途途课堂视频号内容

### 3.2 排查日志文件

下载的时间段日志（`c801f768-deef-4fc4-b3fd-71629c8ad48e.json`）共 3054 行，包含约 40 条 `CloudFlowHandler:inBound` 记录。

### 3.3 分析结论

**根本原因：云端本次配置了两个独立的群发任务，客户端忠实执行了所有任务。**

| 消息 | taskId | msgType | 内容 | 云端任务创建时间 |
|------|--------|---------|------|----------------|
| 文本消息 | `1265071790997111025` | `1`（文本） | 💢宝贝3年级思维黄金期分水岭... | 1772202225 |
| 视频号 | `1265071821720387667` | `69`（finder）| 途途课堂视频 | 1772202232 |

**本次群发整体分布**（发给约 40 个用户）：

| 任务类型 | 下发条数 | 覆盖用户数 |
|---------|---------|----------|
| 文本模板A（"家长们都反馈..."）| 12 | 12 人 |
| 文本模板B（"宝贝3年级..."）| 13 | 13 人 |
| 视频号（finder）| 15 | 15 人 |

目标学员 `7881299543032161` 同时在三个任务列表中，收到了**3 条**消息。

### 3.4 发现的额外问题：任务超时未返回回执

两条任务发送给逆向后，逆向**没有返回回执**，客户端定时器每 10 秒报一次超时，持续了约 4 分钟：

```
任务超时id 1265071821720387667 :[1265071821720387667]的超时时间为:[240000]
任务超时id 1265071790997111025 :[1265071790997111025]的超时时间为:[240000]
```

消息实际已发出（截图可见），但三段式回执机制没有收到逆向的第二段确认。

### 3.5 问题定位结论

| 维度 | 结论 |
|------|------|
| 是否客户端 bug | **否**，客户端执行了云端下发的所有任务 |
| 是否重复发送 | 从客户端视角是两个不同 taskId 的独立任务 |
| 根本原因 | **云端运营后台配置了多类型任务**（文本 + 视频号），且目标用户列表存在重叠 |
| 逆向回执 | 任务超时，逆向未返回确认，但消息实际已发出 |
| 排查建议 | 检查云端任务配置，确认是否有意同时发文本和视频号 |

## 三-B、真实案例分析（续）：同一 taskId 被收到 3 次

### 3B.1 问题描述

同一用户 `huanglimin02`，同一条 taskId `1265071790997111025` 在客户端被接收了 3 次，但后端日志仅有 1 条发送记录（且为前一天晚上）。

### 3B.2 日志对比

| 来源 | 时间 | 次数 | 日志来源 |
|------|------|------|---------|
| 后端 MQBaseProducer | Feb 27 22:23:45 | **1 次** | 后端日志平台 |
| 客户端 [接收mqtt任务] | Feb 28 14:14 / 15:18 / 15:50 | **3 次** | 客户端阿里云日志 |

### 3B.3 分析结论

**根本原因：MQTT Broker QoS 1 自动补发，而非后端业务重发。**

机制分析：
1. 客户端 MQTT 连接使用 `QoS 1` + `clean: false`（持久会话）
2. `mqtt.js` 库自动 ACK（收到消息后立即发 PUBACK，不等任务完成）
3. 但 PUBACK 通过 TCP 发送，如果**连接在 PUBACK 到达 Broker 之前断开**，Broker 收不到 PUBACK
4. 下次重连后 Broker 重新投递同一条消息

```
Feb 27 22:23   后端发送 1 次
               ↓
Feb 28 14:14   客户端收到第 1 次 → PUBACK 发出 → TCP 断开 → Broker 没收到
Feb 28 15:18   客户端重连收到第 2 次 → PUBACK 发出 → TCP 又断开 → Broker 没收到
Feb 28 15:50   客户端重连收到第 3 次 → PUBACK 成功到达 → Broker 删除消息
```

### 3B.4 为什么去重锁没生效？

`mqttTaskMapLock`（去重锁）是纯内存的 `Map`，存在清理定时器：
- 第 1 次 14:14 → 加锁
- 第 2 次 15:18 → 距第 1 次 64 分钟，锁可能已被清理 → 视为新消息
- 第 3 次 15:50 → 距第 2 次 32 分钟，锁可能又被清理 → 视为新消息

所有 3 次接收都通过了过期检查（`createTime` 为前一天，超过 3 小时），被过期检查拦截丢弃。

### 3B.5 排查要点

判断重复类型的关键：

| 判断依据 | Broker QoS 1 补发 | 后端业务重发 |
|---------|------------------|-------------|
| 后端 send 次数 | **1 次** | **多次** |
| 客户端 taskId | 完全相同 | 完全相同 |
| 间隔 | 不规则（取决于重连时机） | 规则（云端重试策略，通常 1h） |

> 详细 MQTT 协议分析参考 [24-MQTT消息机制深度解析](./24-MQTT消息机制深度解析.md)

## 四、阿里云日志查询方案

### 4.1 按处理阶段查询

```
# 第一阶段：MQTT 接收（最早入口，即使后续失败也有此日志）
"huanglimin02" and 接收mqtt任务

# 第二阶段：企微群发服务处理
"huanglimin02" and work-wx-friend-chat

# 第三阶段：进入 cloudFlow（已构建客户端任务）
"huanglimin02" and "CloudFlowHandler" and "inBound"

# 第四阶段：已发送给逆向
"huanglimin02" and "发送消息给逆向"

# 查询任务超时情况
"huanglimin02" and 任务超时id

# 查询三段式回执上报
"huanglimin02" and "MQTT发往云端数据成功"
```

### 4.2 按目标用户查询（定位特定学员收到的消息）

```
# 方法1：用目标用户 ID 查（ID 出现在 conversationid、toUsernames 中）
"huanglimin02" and "7881299543032161"

# 方法2：结合消息处理阶段
"huanglimin02" and "CloudFlowHandler" and "7881299543032161"

# 方法3：查该用户的所有发送记录
"huanglimin02" and "发送消息给逆向" and "7881299543032161"
```

### 4.3 按 taskId 查询

> ⚠️ 详细原因见[阿里云日志查询指南 §1.7](../阿里云日志查询指南/12-常见问题与解决方案.md)

```
# 推荐方式1：不加引号直接用数字
"huanglimin02" and 1265071790997111025

# 推荐方式2：前 15 位 + 通配符（最保险）
"huanglimin02" and 126507179099711*

# 不推荐：精确短语（19 位长数字可能被分词拆分）
"1265071790997111025"   ← 可能查不到
```

### 4.4 完整排查流程（推荐顺序）

```
第一步：确认有没有接收到任务
"用户名" and 接收mqtt任务
（看 taskId 数量 = 云端下发了几个任务）

第二步：确认是否发给了逆向
"用户名" and "发送消息给逆向"
（看 type=[sendmsg_xxx] 确认消息类型）

第三步：确认逆向有没有回执
"用户名" and "MQTT发往云端数据成功"
（如果没有，再查：）

第四步：确认是否超时
"用户名" and 任务超时id

第五步：定位特定用户的任务
"用户名" and "目标用户ID"
```

## 六-A、MQTT 消息洪泛（reconnect 后消息积压）

### 6A.1 现象描述

分析 `huanglimin02` 的完整日志（`2bcf6050-...-.json`），发现客户端在 15:49-15:50 重连后 40 秒内收到 **368 条** `messageType-100` 的 MQTT 任务消息。

### 6A.2 数据分析

| 指标 | 数值 |
|------|------|
| 总消息数 | 368 条 |
| `createTime` 跨度 | 18 小时（昨晚 21:32 ~ 当天 15:34） |
| 投递耗时 | ~40 秒（15:49:20 ~ 15:49:57） |
| `mqttTaskMapLock` 峰值 | 328 条同时在锁中 |
| 过期消息（>3h） | 322 条（87%） |
| 有效消息 | 46 条（13%） |

### 6A.3 原因

- `clean: false`（持久会话）→ 客户端断线期间 Broker 持续缓存消息
- `reconnectPeriod = 0`（禁用自动重连）→ 重连不及时，消息持续积压
- 重连后 Broker 一次性投递所有积压消息

### 6A.4 客户端现有防护

1. **过期检查**：`mqttTaskExpireHours`（默认 3h），过期直接丢弃
2. **去重锁**：`mqttTaskMapLock` 防止同一 taskId 重复执行
3. **延迟队列**：`executeWithDelay` 对 type=100 任务做 200ms 间隔控制

> 详细分析参考 [24-MQTT消息机制深度解析](./24-MQTT消息机制深度解析.md)
