# 问题排查指南：好友数量不对

> 本文档提供"好友数量不对"问题的系统性排查方案，适用于用户反馈好友数多了或少了的场景

---

## 一、问题描述

### 1.1 典型现象

- 用户反馈前端显示的好友数与微信实际好友数**不一致**
- 可能的表现：好友数偏少（丢失部分好友）或偏多（包含了非好友）
- 该问题不同于"好友列表空白"（参见 [13-问题排查指南-好友列表空白.md](./13-问题排查指南-好友列表空白.md)），空白是完全无数据，本问题是数据有但数量不对

### 1.2 可能的原因分类

| 原因类别 | 具体场景 | 影响 |
|---------|---------|-----|
| **数据源问题** | 逆向返回的 `getuserlist` 总列表就不完整 | 少了好友 |
| **过滤问题** | `EXCLUDE_WXID_SET` 过滤了不该过滤的 wxid | 少了好友 |
| **分页丢失** | 某些分页请求失败或超时未响应 | 少了好友 |
| **contacttype 过滤** | `contacttype` 位运算过滤了本应保留的好友 | 少了好友 |
| **local_type 过滤** | `local_type != 1` 过滤了部分好友 | 少了好友 |
| **nickname 过滤** | `formatChatroomOrUserToList` 中 `item.nickname` 为空被过滤 | 少了好友 |
| **数据库 diffUpdate** | 差量更新时逻辑异常，删除了不该删的好友 | 少了好友 |
| **缓存未清理** | 多次请求导致缓存数据叠加 | 多了好友 |
| **限频拦截** | 2 小时内重复请求被限频，使用了旧数据 | 数量过时 |
| **超时处理** | `userinfolistWait` 超时触发异常流程 | 少了好友 |

### 2.2 每步数量变化追踪

```
步骤②  getuserlist 返回总数             例: 500
          │
          ▼
步骤③  EXCLUDE_WXID_SET 过滤后          例: 495 (过滤了 filehelper 等5个系统号)
          │
          ▼
步骤④  分页请求                          例: 1 页 (495 < 2000)
          │
          ▼
步骤⑥  formatChatroomOrUserToList 后    例: 群 20 个 + 好友 475 个 = 495
          │                                 ⚠️ 如果 nickname 为空会丢失
          ▼
步骤⑦  handleRoomMembers 过滤后         例: 好友 460 个
          │                                 ⚠️ contacttype/local_type 过滤了 15 个
          ▼
步骤⑧  最终发送给前端                    例: 好友 460 个
```

**关键数量检查点**：每一步的数量应该能对得上，如果某步数量突变就是问题所在。

## 三、排查步骤

> 以下查询均在阿里云 SLS 执行
> 项目: `gaotu-new` / 日志库: `gaotu-wxzs-client-node-log`
> **注意**: 将 `{username}` 替换为实际用户名，调整好时间范围

### 3.1 第一步：确认初始总数

**目的**: 确认逆向返回的原始好友+群总数

**查询**:

```sql
* and {username} and handlerGetUserList and "逆向返回" | select message
```

**预期日志**:

```
[初始化] [handlerGetUserList] [wxid-xxx] [channelId] 逆向返回好友&群列表总数:[500] 过滤后数: 495
```

**分析要点**:
- `总数` vs `过滤后数`：差值就是被 `EXCLUDE_WXID_SET` 过滤的系统账号数
- 如果 `过滤后数` 本身就比预期少 → 问题在逆向端（数据源不完整）
- 如果 `总数` 显示为 0 或很小 → 逆向获取好友列表失败

---

### 3.2 第二步：确认分页请求

**目的**: 确认分页数量和每页好友数

**查询**:

```sql
* and {username} and fetchRoomMembers | select message order by __time__ asc
```

**预期日志**:

```
[wxid-xxx] [fetchRoomMembers] 好友列表总页=1 好友数=495
[fetchRoomMembers] [forItem] [wxid-xxx] [channelId] [495] pageNum=1
```

**分析要点**:
- `好友列表总页` × `userLimit(2000)` ≈ 过滤后数
- 每个 `pageNum` 是否都有对应的 `forItem` 日志
- 如果某页的 `forItem` 日志缺失 → 该页请求可能未发出

---

### 3.3 第三步：确认分页响应和分类结果

**目的**: 确认每页数据是否正常返回，群和好友的分类是否正确

**查询**:

```sql
* and {username} and wxUserListResponseMsgHandler and "群总数" | select message order by __time__ asc
```

**预期日志**:

```
wxUserListResponseMsgHandler [userinfolist] [wxid-xxx]，curNode:1 群总数：20 好友总数：475
```

**分析要点**:
- `群总数 + 好友总数` 应该等于该页请求的好友数
- 如果 `好友总数：0` → `formatChatroomOrUserToList` 过滤出了问题（参见 [13-问题排查指南-好友列表空白.md](./13-问题排查指南-好友列表空白.md)）
- 如果某页的 `curNode` 日志缺失 → 该页响应丢失
- 多页情况下，累加所有页的好友总数，看是否与预期一致

---

### 3.4 第四步：确认好友处理结果

**目的**: 确认 `handleRoomMembers` 处理后的好友数量和完成状态

**查询**:

```sql
* and {username} and handleRoomMembers | select message order by __time__ asc
```

**预期日志**:

```
[wxid-xxx] WxUserListResponseMsgHandler handleRoomMembers wxId: xxx,总页：1,当前页：1
[wxid-xxx] WxUserListResponseMsgHandler handleRoomMembers wxId: xxx [好友列表全部获取成功]
```

**分析要点**:
- 是否出现 `[好友列表全部获取成功]` → 没有的话说明有分页响应丢失
- `总页` 和 `当前页` 是否匹配 → `successCount >= totalNode` 才会触发完成
- 如果缺少此日志 → 检查是否有超时通知 (`onUserInfoListNotReturn`)

---

### 3.5 第五步：确认最终发送数量

**目的**: 确认入库和发送给前端的好友数量

**查询**:

```sql
* and {username} and handleSendFriendToFront | select message order by __time__ asc
```

**预期日志**:

```
[handleSendFriendToFront] [数据开始入库] [wxid-xxx]
[wxid-xxx] diffUpdateUserListFinish 成功 数据更新:true 耗时:XXms 长度:460
[wxid-xxx] [handleSendFriendToFront] 发送数据到前端 hasChange:true 群数(缓存):20 好友数(缓存):460
```

**分析要点**:
- `好友数(缓存)` 就是最终发送到前端的好友数 → 与用户看到的数量对比
- `diffUpdateUserListFinish` 的 `长度` → 实际入库的好友数
- `数据更新:false` 且 `无数据需要发送到前端` → 数据没有变化或为空
- 如果 `好友数(缓存)` 比步骤三的好友总数少 → 被 `contacttype` 或 `local_type` 过滤

---

### 3.6 第六步：确认前端实际接收数据

**目的**: 确认前端 WebSocket 收到的数据量

**查询**:

```sql
* and {username} and "取到的用户数据为" | select message order by __time__ desc
```

**预期日志**:

```
[wxid_xxx][取到的用户数据为]：好友460个， 群数20
```

**分析要点**:
- 此数量应与步骤五的 `好友数(缓存)` 一致
- 如果不一致 → 数据库查询环节有问题

---

### 3.7 第七步：检查数量丢失原因

根据前面步骤定位到数量在哪一步减少后，针对性排查：

#### 7a. contacttype 过滤排查

**说明**: `handleRoomMembers` 中的过滤规则：

```javascript
// contacttype 位运算过滤（第660行）
if (contacttype && ((parseInt(contacttype) & 8) != 0 || parseInt(contacttype) % 2 != 1)) {
    continue; // 跳过非好友
}
// local_type 过滤（第663行）
if (local_type && local_type != 1) {
    continue;
}
```

过滤条件说明：
- `contacttype & 8 != 0` → 该联系人被标记为"已拉黑"
- `contacttype % 2 != 1` → 该联系人不是"好友"关系（最低位不为1）
- `local_type != 1` → 非本地好友类型

**排查方法**: 如果此步数量大幅减少，需要查看逆向返回的原始数据中 `contacttype` 的值：

```sql
* and {username} and "发送前端WS消息" and userinfolist | select message
```

#### 7b. nickname 为空排查

**说明**: `formatChatroomOrUserToList` 中的过滤规则（第604行）：

```javascript
if (item.nickname) {
    friendIdList.push(item);
}
```

`nickname` 为空/undefined 的好友会被直接丢弃。

#### 7c. 分页超时排查

**查询**:

```sql
* and {username} and "onUserInfoListNotReturn" | select message
```

或：

```sql
* and {username} and "fetchRoomMembersRetry" | select message
```

**分析**: 如果有此日志说明某些分页超时未返回，触发了重试机制。

---

### 3.8 第八步：检查限频和重复请求

**目的**: 确认是否被限频拦截

**查询**:

```sql
* and {username} and "两个小时内" | select message order by __time__ desc
```

**分析**: 如果有此日志，说明在 2 小时内重复请求好友列表，第二次请求被拦截，使用的是第一次的数据。

---

### 3.9 第九步：检查错误日志

**目的**: 确认处理过程中是否有异常

**查询**:

```sql
* and {username} and (codeError or wxUserListResponseMsgHandlerError or handlerGetUserListError or fetchRoomMembersError or handleRoomMembersError) | select message order by __time__ desc
```

## 五、常见问题场景

### 5.1 场景一：好友少了，contacttype 过滤过多

**特征**:
- ⑥ 好友总数(分类后) 正常
- ⑦ 好友数(过滤后) 明显减少
- 差异在 `handleRoomMembers` 中的 contacttype/local_type 过滤

**原因**: 逆向返回的 `contacttype` 值可能不准确，或微信版本升级导致字段含义变化

**处理方案**: 联系逆向开发确认 `contacttype` 字段的正确性

### 5.2 场景二：好友少了，分页响应丢失

**特征**:
- ③ 分页请求数 > ④ 分页响应数
- 某些 `curNode` 的响应日志缺失
- 没有 `[好友列表全部获取成功]` 日志

**原因**: 
- `fetchUserListInfoByPage` 的 `sleep(1500)` 固定延时导致请求堆积
- 逆向客户端处理能力不足，丢弃了部分请求
- IPC 管道通信异常

**处理方案**: 
- 检查逆向客户端是否正常运行
- 确认 IPC 连接状态
- 考虑增大 `friendWait` 值或改为事件驱动模式

### 5.3 场景三：好友少了，nickname 为空被过滤

**特征**:
- ② 过滤后数正常
- ⑥ 好友总数(分类后) 偏少
- `群总数 + 好友总数` < 该页应有数量

**原因**: 逆向返回的 `userinfolist` 中某些好友的 `nickname` 字段为空

**处理方案**: 
- 这是 `formatChatroomOrUserToList` 的已知过滤行为
- 建议：无 nickname 的好友不应直接丢弃，可以使用 wxid 作为 fallback

### 5.4 场景四：好友数量过时（限频拦截）

**特征**:
- 有 `不允许在两个小时内再次请求好友列表` 日志
- 好友数据是老数据

**原因**: `FriendListRateLimiter` 限制了 2 小时内的重复请求

**处理方案**: 
- 等待限频窗口过后重新请求
- 或需要在限频机制中增加强制刷新选项

### 5.5 场景五：diffUpdate 差量更新异常

**特征**:
- ⑦ 好友数(入库前) 正常
- ⑧ 好友数(前端接收) 偏少
- `diffUpdateUserListFinish` 日志中 `数据更新:true` 但数量不对

**原因**: `wkExternalUserService.diffUpdate` 的差量对比逻辑有误，可能误删了有效好友

**处理方案**: 
- 检查 diffUpdate 的实现逻辑
- 确认是否是新旧数据格式不一致导致的误判

## 八、fetchUserListInfoByPage 已知设计缺陷

### 8.1 当前实现

```javascript
// wxUserListResponseMsgHandler.js 第 268-289 行
async fetchUserListInfoByPage(payload) {
    const { totalNode, curNode, wxId, channelId, friendIds } = payload;
    let sendMsgBo = { ... };
    dispatchInBound(channelId, wxId, JSON.stringify(sendMsgBo));  // 同步，fire-and-forget
    await sleep(friendWait);  // friendWait = 1500ms 固定延时
}
```

### 8.2 缺陷说明

| 缺陷 | 说明 | 风险 |
|-----|------|-----|
| 固定延时 | `friendWait=1500ms` 硬编码，不根据数据量或响应时间调整 | 好友数多时总耗时过长；逆向慢时请求堆积 |
| 盲发模式 | 不等待上一页响应就发下一页 | 逆向来不及处理导致丢包 |
| 无错误反馈 | `dispatchInBound` 失败后仍继续发下一页 | 某页丢失无法感知 |
| 无重试保障 | 依赖外部 `userinfolistWait` 超时检测，但只通知不重试 | 丢失的分页无法补回 |

### 8.3 对好友数量的影响

当分页请求堆积或丢失时，会导致：
- `handleRoomMembers` 中的 `successCount` 无法达到 `totalNode`
- 不会触发 `handleSendFriendToFront`
- 20 秒后超时触发 `onUserInfoListNotReturn` 通知
- 好友数据不完整或完全未入库
