# 好友群列表同步与群成员缓存链路分析

## 一、背景

微信助手中多个功能依赖本地群成员缓存数据，例如：

- **踢私加好友**：通过 `findChatroomsByMember` 查本地缓存判断新好友是否来自群聊
- **群管理**：群成员列表展示
- **群发消息**：群内@功能等

群成员缓存的填充完全依赖 `GET_USER_LIST` 流程的触发。如果该流程未执行，缓存为空，上述功能将全部失效。

本文档详细梳理了从前端触发到群成员缓存落库的**完整链路**。

## 三、前端触发（galaxy-web）

### 3.1 触发消息格式

```js
// galaxy-web/src/enums/msg.js（第12-14行）
export const USERLIST = {
    type: 'userlist'
}
```

前端通过 WebSocket（ipc.sendToHost）发送 `{type: "userlist"}` 消息给客户端。

### 3.2 触发时机（共 3 处）

#### 触发点一：账号登录成功（自动触发）

**文件**：`galaxy-web/src/entries/sub/store/base/thunks.js` 第 784-796 行

```js
// handleLoginNews 函数内
if (lastLoginStats !== 0) {
    dispatch(setUserListState({ userlistState: true }));
    setTimeout(() => {
        const { wxid } = getState().base.userInfo;
        setGlobalData('wxid', wxid);
        sendMessage(USERLIST);   // ← 发送 {type: "userlist"}
    }, 1500);
}
lastLoginStats = newsBody.status;  // 记录本次状态
```

**触发条件**：`lastLoginStats !== 0`

| `lastLoginStats` 值 | 含义 | 是否触发 |
|---|---|---|
| `NaN`（初始值） | 应用刚启动，首次收到登录状态 | ✅ `NaN !== 0` 为 true |
| `非0数字`（如 1、4） | 上次状态为未登录/异常 | ✅ 触发 |
| `0` | 上次已是登录成功 | ❌ 不触发（防重复拉取） |

**关键点**：
- `lastLoginStats` 是**模块级变量**（`let lastLoginStats = NaN`，第 52 行），不持久化
- 每次 Electron webview 刷新或重建，该变量重置为 `NaN`
- 客户端周期性发送 login 消息到前端（心跳），如果账号已登录，第一次会触发（`NaN !== 0`），后续不再触发（`0 !== 0` 为 false）

#### 触发点二：顶部导航栏「同步数据」按钮（手动触发）

#### 触发点三：群管理页「同步」操作（手动触发）

## 四、客户端接收与处理（galaxy-client）

### 4.1 入口：frontFlowInBound

**文件**：`galaxy-client/src/msg-center/dispatch-center/dispatch/frontFlowInBound.js`

```js
// 第 63-66 行：路由映射

    [CmdIdConstant.FORWARD]: forwardTask,  // cmdId='forward' → forwardTask

async function frontFlowInBound(message) {
    const msgObj = JSON.parse(message);
    const request = { ...WsRequest, ...msgObj };
    let frontTask = frontHandlerMap[request?.cmdId];  // → forwardTask

```

### 4.2 forwardTask 处理

**文件**：`galaxy-client/src/msg-center/business/task-front/forwardTask.js` 第 319-378 行

```js
// 第 319 行：匹配 type === 'userlist'
if (GalaxyTaskType.USER_LIST === clientTask.type) {
    // 第 327 行：防重复请求
    if (registry.wxid && (msgStatus === MsgStatus.START 
        || msgStatus === MsgStatus.INIT 
        || msgStatus === MsgStatus.SECOND)) {
        return;  // 正在获取中，不允许重复请求
    }

    // 第 332 行：类型转换 "userlist" → "getuserlist"
    clientTask.type = GalaxyTaskType.GET_USER_LIST;

    // 第 339 行：先发送缓存中的旧数据给前端（快速响应）
    frontSendService.sendWxFrinedList(registry.pipeLineWrapper);

    // 第 348 行：更新状态为 START
    wxUserAndChatRoomCache.updateUserMsgStatus(registry.wxid, MsgStatus.START);
    await sleep(1000);

    // 第 358 行：发送到逆向
    dispatchInBound(registry.id, registry.wxid, JSON.stringify(clientTask));

    // 第 362-374 行：设置 180 秒超时定时器
    setTimeout(() => {
        if (userInfoCache.userMsgStatus.get(registry.wxid) !== MsgStatus.FINISH) {
            userInfoCache.updateUserMsgStatus(registry.wxid, null);
            notify.onGetUserListTimeout(registry.wxid);
        }
    }, 180000);
}
```

**关键行为**：
1. 将 `type` 从 `"userlist"` 改为 `"getuserlist"`
2. 先返回缓存旧数据（前端快速看到列表）
3. 再发送到逆向获取最新数据
4. 设置 3 分钟超时保护

### 4.3 dispatchInBound 转发到逆向

**文件**：`galaxy-client/src/msg-center/dispatch-center/dispatchInBound.js` 第 221-253 行

### 5.1 消息路由

逆向返回的数据通过 TCP 连接到达客户端，经 `msgHandleBase.js` 路由到对应 handler。

**文件**：`galaxy-client/src/msg-center/dispatch-center/handle/msgHandleBase.js` 第 1641-1643 行


`isHandler` 匹配条件（第 81-87 行）：

```js
isHandler(type, isWorkWx) {
    return (
        [GalaxyTaskType.GET_USER_LIST, GalaxyTaskType.USER_INFO_LIST].includes(type) 
        && !isWorkWx
    );
}
```

### 5.2 execute 主入口

**文件**：`galaxy-client/src/msg-center/dispatch-center/handle/wxUserListResponseMsgHandler.js` 第 119-208 行

```
execute(jsonObject, wrapper)
  ├── type === "userinfolist" && contextId    → return（跳过，由 UserInfoListService 处理）
  ├── type === "userinfolist" && cacheKey     → updateSingleChatroomInfoCache
  ├── type === "getuserlist"                  → handlerGetUserList()
  └── type === "userinfolist" && status=SECOND → handleConversationInfos() + handleRoomMembers()
```

### 5.3 状态机流转

```
状态           触发条件                          下一步
─────────────────────────────────────────────────────────
null/NaN  →  forwardTask 收到 USERLIST           → START
START     →  收到 GET_USER_LIST 响应（总列表）    → SECOND
SECOND    →  所有 USER_INFO_LIST 分页返回完毕     → FINISH
FINISH    →  数据发送到前端                       → 清理缓存
```

### 5.4 handlerGetUserList（处理总好友列表）

**文件**：第 237-266 行

```js
async handlerGetUserList(jsonObject, wrapper) {
    const { objarray, channelId } = jsonObject;
    // objarray 包含所有好友和群的 wxid 列表（如 ["wxid_xxx", "12345@chatroom", ...])
    const userIds = objarray?.filter(wxid => !EXCLUDE_WXID_SET.has(wxid));

    // 初始化缓存数据
    wxUserAndChatRoomCache.initData(wrapper.wxid, { userIds, channelId });
    // 状态 START → SECOND
    wxUserAndChatRoomCache.updateUserMsgStatus(wrapper.wxid, MsgStatus.SECOND);
    // 开始分页获取详情
    await this.handleFetchRommMember(wrapper);
}
```

### 5.5 fetchRoomMembers（分页获取详情）

**文件**：第 292-345 行

```js
async fetchRoomMembers(payload) {
    // 将 userIds 按 2000 个一页分组
    let friendList = Lists.partition(friends, userLimit);
    for (let friendIds of friendList) {
        // 发送 {type: "userinfolist", data: {array: friendIds}} 到逆向
        await this.fetchUserListInfoByPage({ totalNode, curNode, wxId, channelId, friendIds });
    }

```

### 5.6 handleConversationInfos（处理群列表，保存群成员缓存）

**文件**：第 384-519 行

每个分页的 `USER_INFO_LIST` 返回后，包含好友和群的详细信息。此方法处理群的部分：

1. 遍历群列表，保存群基础信息（群名、群主、头像）到 SQLite
2. 对于有 `simplelist`（成员列表字符串）的群，直接保存群成员到 `chatroomMemberinfo` 表
3. **调用 `fetchChatroomMemberInfo`**（第 515 行）获取群成员详情

### 5.7 fetchChatroomMemberInfo（获取群成员详情 ⭐ 关键函数）

**文件**：第 722-811 行

```js
async fetchChatroomMemberInfo(chatroomIds, wxid) {
    // 逐个群发送请求（1秒间隔）
    for (const chatroomId of chatroomIds) {
        // 发送 chatuserlistex 请求（获取成员列表和管理员信息）
        ChatroomInfoCache.sendChatroomDetailListRequest({ chatroomId, cacheKey, wxid });
        // 同时发送 chatuserinfo 请求（获取群成员的群昵称）
        setTimeout(() => {
            memberDetailTaskManager.sendChatuserinfoTask(wxid, chatroomId);
        }, 500);
        await sleep(1000);
    }
    // 等待 3 秒让响应返回并缓存
    await sleep(3000);
    // 60 秒后检查并重试失败的群
    setTimeout(() => this.retryFailedChatrooms(wxid, cacheKeyPrefix), 60000);
}
```

**这个函数写入的缓存数据，就是 `findChatroomsByMember`（踢私加好友）查询的数据源。**

### 5.8 handleSendFriendToFront（发送到前端，标记完成）

**文件**：第 976-1063 行

所有分页返回后，执行最终操作：
1. 将好友列表和群列表 diffUpdate 入库
2. 状态更新为 `FINISH`
3. 通过 WebSocket 发送最新列表到前端渲染
4. 上报群列表到云端

## 六、场景隔离：UserInfoListService 与 wxUserListResponseMsgHandler

### 6.1 问题发现背景

排查「踢私加好友」功能失效时，发现 `fetchChatroomMemberInfo` 从未被调用。日志显示：

```
[wxUserListResponseMsgHandler] 跳过处理有 contextId 的消息（由 UserInfoListService 处理）
```

### 6.2 两条路径对比

| 维度 | wxUserListResponseMsgHandler | UserInfoListService |
|---|---|---|
| **文件** | `dispatch-center/handle/wxUserListResponseMsgHandler.js` | `business/convert-service/userInfoListService.js` |
| **匹配条件** | `type=USER_INFO_LIST && !contextId` | `type=USER_INFO_LIST && contextId` |
| **使用场景** | 登录后全量好友/群列表同步 | 邀请入群等特定场景的用户信息回调 |
| **是否调用 fetchChatroomMemberInfo** | ✅ 是 | ❌ 否 |
| **是否更新群成员缓存** | ✅ 是（SQLite 入库） | ❌ 否（只转发给 memberDetailTaskManager） |
| **触发频率** | 登录后一次 + 手动同步 | 每次邀请入群操作都会触发 |

### 6.3 隔离逻辑代码

**wxUserListResponseMsgHandler.js 第 121-132 行**：

```js
if (GalaxyTaskType.USER_INFO_LIST === jsonObject.type && jsonObject.contextId) {
    logUtil.customLog(
        `[wxid-${wrapper.wxid}] [wxUserListResponseMsgHandler] 跳过处理有 contextId 的消息（由 UserInfoListService 处理）`
    );
    return jsonObject;  // 跳过，不处理
}
```

**userInfoListService.js filter 方法**：

```js
filter(clientRecord) {
    if (clientRecord.type !== GalaxyTaskType.USER_INFO_LIST) return false;
    if (clientRecord.contextId) return true;   // 有 contextId → 此处理器处理
    return false;                               // 无 contextId → 走老路径
}
```

## 七、实际案例：「踢私加好友」失效根因

### 7.1 排查过程

针对用户 `peifangjun`（wxid: `wxid_1c7u3kr5o20022`），逐步查询 SLS 日志：

| 查询语句 | 结果 | 含义 |
|---|---|---|
| `peifangjun AND Wx4AddFriendTaskResponse` | 有日志，609 无 chatroom 字段 | 群成员缓存查询为空 |
| `peifangjun AND fetchChatroomMemberInfo` | **空** | 从未调用群成员获取 |
| `peifangjun AND handleConversationInfos` | **空** | 群列表处理从未执行 |
| `peifangjun AND handlerGetUserList` | **空** | 总列表处理从未执行 |
| `peifangjun AND GET_USER_LIST` | **空** | 前端从未发起好友列表请求 |
| `peifangjun AND UserInfoListService` | **48 条**，全部 scene=member_invite | 只有邀请入群场景的消息 |
| `peifangjun AND WxUserListResponseMsgHandler` | "跳过处理有 contextId 的消息" | 被场景隔离拦截 |

### 7.2 因果链

```
前端未触发 sendMessage(USERLIST)
    ↓
GET_USER_LIST 从未发送到逆向
    ↓
handlerGetUserList 从未执行
    ↓
fetchRoomMembers 从未执行
    ↓
handleConversationInfos 从未执行
    ↓
fetchChatroomMemberInfo 从未执行
    ↓
chatroomMemberinfo 表为空
    ↓
findChatroomsByMember() 返回空数组
    ↓
609 上报无 chatroom 和 scene:14
    ↓
云端无法判断「群内加好友」场景
    ↓
不下发 type=3 踢人任务
    ↓
踢私加好友完全失效
```

### 7.3 为什么前端没触发？

该账号是 `5.5.0-test` 版本，在 SLS 日志时间窗口内：
- **从未收到 `lastLoginStats !== 0` 的登录状态变化**：说明账号在该时间段内一直保持登录状态（`lastLoginStats` 始终为 `0`），前端 webview 没有重建
- **用户没有点击「同步数据」按钮**：没有手动触发

## 九、解决方案

### 短期（运营侧）

让用户在前端页面右上角点击「**同步数据**」按钮，立即触发完整的 `USERLIST → GET_USER_LIST → fetchChatroomMemberInfo` 链路。

### 长期（代码改进建议）

| 方案 | 说明 | 改动位置 |
|---|---|---|
| 登录后自动触发 | 客户端检测到登录成功且缓存为空时，主动发起 GET_USER_LIST | `galaxy-client` 登录回调 |
| 缓存为空时按需触发 | `findChatroomsByMember` 查询为空时标记需刷新，异步触发 | `wx4AddFriendTaskResponse.js` |
| 定时刷新 | 每隔 N 小时自动触发一次全量同步 | `galaxy-client` 定时器 |
| UserInfoListService 补充缓存 | 邀请入群时顺带更新群成员缓存 | `userInfoListService.js` |
