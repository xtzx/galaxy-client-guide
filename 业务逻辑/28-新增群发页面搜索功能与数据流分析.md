# 28-新增群发页面搜索功能与数据流分析

> 「新增群发」Drawer 页面的组件结构、搜索条件到结果渲染的完整数据流、名称模糊搜索误匹配问题分析

## 一、页面结构与文件映射

### 1.1 截图对应的入口文件

「新增群发」页面是一个 Ant Design `Drawer` 抽屉组件，位于 **galaxy（前端 Web）** 的 `sub` 模块中：

```
galaxy/src/entries/sub/component/sendSetting/index.js  ← SendSetting 组件
```

Drawer 的标题逻辑：

```javascript
const getDrawerTitle = () => (
    <div>
        {isSendModal ? '新增公告' : '新增群发'}

        </span>
    </div>

```

### 1.2 三栏布局组件映射

页面内部采用 `Row` + `Col` 三栏布局：

```
<Row>
  <Col span={8}>  → Condition（搜索条件表单）
  <Col span={6}>  → FilterResult（搜索结果列表）
  <Col span={10}> → Content（发送内容 + 时间设置 + 发送设置）
</Row>
```

| 区域 | 对应截图 | 组件 | 文件路径 |
|------|---------|------|---------|
| 左栏（span=8） | 名称/交友时间/群内好友/选择标签/用户行为/群专用条件/确定搜索 | `Condition` | `sub/component/sendSetting/component/condition.js` |
| 中栏（span=6） | "此次搜索出 7 个结果" + 外部群(3) + 客户(4) | `FilterResult` | `sub/component/sendSetting/component/filterResult.js` |
| 右栏（span=10） | 发送内容（文字/图片/文件/链接等）+ 时间设置 + 发送设置 | `Content` | `sub/component/sendSetting/component/content.js` |

### 1.3 注意区分两套"新增群发"

项目中存在**两套**新增群发实现，入口不同、代码不同：

| 模块 | 入口 | 页面组件 | 搜索方式 | 适用场景 |
|------|------|---------|---------|---------|
| `entries/sub/` | Electron 子窗口（broadCast 模块） | `sendSetting/index.js` Drawer | 前端过滤 + PinyinMatch | 客户端内嵌页面 |
| `entries/menu/staging/` | Web 端菜单页 | `groupMessage/form.js` 独立页面 | 前端过滤 + indexOf + API | Web 管理后台 |

本文分析的是 **`entries/sub/`** 模块下的 Drawer 版本（截图所示）。

## 二、搜索条件表单（Condition 组件）

| 群主和管理员 | `isMaster` | `Select` | 全部(2)/是(1)/否(0) |

### 2.2 搜索触发入口

```javascript
// 用户点击"确定搜索"
const onFinish = async (values) => {
    handleConditionInfo({ conditionInfo: { ...conditionInfo, ...values } }); // → Redux
    await handle({ ...conditionInfo, filterTagList, filterRule, ...values,
        filterConditionList: values.userActions ? values.userActions : userActions }, 1);
};
```

注意：**标签搜索有特殊分支**。当 `filterTagList.length > 0` 时，`onFinish` 会提前 `return`，不直接调用 `handle()`，而是通过 WebSocket 发送 `getlabelcustomer` 消息获取标签成员后，再由 `useEffect` 监听 `labelListFriend` 变化来触发搜索：

```javascript
// 标签分支的异步触发
useEffect(() => {
    handle({ ...conditionInfo, filterTagList, filterRule,
        filterConditionList: userActions }, 1, () => {} );
    setSearchBtnLoading(false);
}, [store.getState().base.labelListFriend]);
```

## 三、搜索核心逻辑（handleCondition thunk）

文件：`sub/broadCast/store/thunks.js` → `handleCondition()`

### 3.1 数据源

搜索的数据源是 Redux 中的 `base.userList`，这是应用启动时通过 WebSocket 从逆向端加载的**全量好友+群列表**。搜索过程是**纯前端过滤**（少数条件需调后端 API 辅助）。

```javascript
const { userList = [], userInfo = {}, chatroomList } = base;
const result = filterUserList(userList);
const { friendsArr, groupsArr, qyGroupsArr } = result;
```

`filterUserList()` 将 `userList` 按类型拆分为：

| 变量 | 含义 |
|------|------|
| `friendsArr` | 好友列表（截图中的"客户"） |
| `groupsArr` | 外部群列表（截图中的"外部群"） |
| `qyGroupsArr` | 企业群列表 |

### 3.2 过滤链（串行执行，逐步缩小结果集）

```
全量数据 (userList)
  │
  │ filterUserList() 分类
  ▼
┌────────────────────────────────────────────────────────┐
│ friendsArr (好友)         groupsArr (外部群)             │
│ qyGroupsArr (企业群)                                    │
└────────────────────────────────────────────────────────┘
  │                              │
  │ ① 群主/管理员过滤              │ ① 群主/管理员过滤
  │    (isMaster)                │    (isMaster)
  │                              │
  │ ② 名称 + 群人数过滤            │ ② 名称 + 群人数过滤
  │    PinyinMatch.match()       │    PinyinMatch.match()
  │                              │
  │ ③ 地区过滤                    │（群无此步骤）
  │    handleAreaSearch()        │
  │                              │
  │ ④ 群内好友过滤                 │（群无此步骤）
  │    → API: queryUserList      │
  │      InChatroomList          │
  │                              │
  │ ⑤ 交友时间过滤                 │（群无此步骤）
  │    → API: queryByBecome      │
  │      FriendTime              │
  │                              │
  │ ⑥ 标签过滤                    │（群无此步骤）
  │    WebSocket 获取标签成员       │
  │                              │
  │ ⑦ 用户行为过滤                 │（群无此步骤）
  │    → API: filterFriends      │
  │      ByCondition             │
  ▼                              ▼
┌────────────────────────────────────────────────────────┐
│ dispatch(setFilterFriend({ filterFriend: resultFriend }))│

└────────────────────────────────────────────────────────┘
```

### 3.3 各步骤详解

#### ① 群主/管理员过滤（群专用）

```javascript
if (Number(isMaster) === 1) {
    // 群发场景：群主 OR 管理员
    resultGroup = groupsArr.filter((ele) => {
        return ele.ownerwxid === userInfo.wxid
            || (ele.managerIds || []).indexOf(getGlobalData('wxid')) !== -1;
    });
}
if (Number(isMaster) === 0) {
    // 既不是群主也不是管理员
    resultGroup = groupsArr.filter((ele) => {
        return ele.ownerwxid !== userInfo.wxid
            && (ele.managerIds || []).indexOf(getGlobalData('wxid')) == -1;
    });
}
```

#### ② 名称搜索（PinyinMatch 模糊匹配）

**外部群 — 只匹配 `nickname` 一个字段：**

```javascript
if (rangeStart || rangeEnd || nickname) {
    resultGroup = resultGroup.filter((ele) => {
        return (
            (rangeStart ? ele.number >= rangeStart : true) &&
            (rangeEnd ? ele.number <= rangeEnd : true) &&
            (nickname && ele.nickname
                ? PinyinMatch.match(ele.nickname, nickname)
                : nickname ? false : true)
        );
    });
}
```

**好友/客户 — 匹配 `remark`、`nickname`、`username` 三个字段（任一命中即保留）：**

```javascript
if (nickname) {
    const tempArr = resultFriend.filter((ele) => {
        return (
            (ele.remark && PinyinMatch.match(ele.remark, nickname)) ||
            (ele.nickname && PinyinMatch.match(ele.nickname, nickname)) ||
            (ele.username && PinyinMatch.match(ele.username, nickname))
        );
    });
    resultFriend = [...tempArr];
}
```

#### ③ 地区过滤（好友专用）

```javascript
if (Array.isArray(area) && area.length > 0) {
    resultFriend = handleAreaSearch(area, resultFriend);
}
```

#### ④ 群内好友过滤（好友专用，需调后端 API）

```javascript
if (grouping) {
    const { data: { data: members }} = await queryUserListInChatroomList({
        chatroomIds: grouping === 'not-any'
            ? combineGroupArr.map(item => item.wxid) : [grouping],
    });
    // 'not-any' → 不在任何群中的好友
    // 具体群ID → 在该群中的好友
}
```

#### ⑤ 交友时间过滤（好友专用，需调后端 API）

```javascript
if (time) {
    const res = await queryByBecomeFriendTime({
        logicWechatId: userInfo.wxid,
        becomeFriendStartTime: +new Date(time[0]),
        becomeFriendEndTime: +new Date(time[1]),
    });
    // 后端返回符合时间范围的 wxid 列表，前端做交集
}
```

#### ⑥ 标签过滤（好友专用）

标签成员数据来自 WebSocket 消息 `getlabelcustomer`，存储在 `base.labelListFriend` 中。支持 OR（并集）和 AND（交集）两种规则：

```javascript
if (filterRule == 'OR') {
    // 取所有标签成员的并集
    filterResult = [...new Set(labelMemberList)];
} else if (filterRule == 'AND') {
    // 取所有标签成员的交集
    filterResult = intersectionOfArrays(data);
}
resultFriend = resultFriend.filter(item => filterResult.includes(item.wxid));
```

#### ⑦ 用户行为过滤（好友专用，需调后端 API）

```javascript
if (filterConditionList.length) {
    const res = await Promise.all(
        // 按 500 人一批分片请求
        new Array(requestNum).fill(true).map((_, index) =>
            filterFriendsByCondition({
                userIds: userIds.slice(index * LIMIT, (index + 1) * LIMIT),
                filterConditionList,
                wxId: getGlobalData('wxid'),
                chatRoomIds: chatroomList
            })
        )
    );
    // 后端返回符合行为条件的用户列表
}
```

## 五、名称搜索"误匹配"问题分析

### 5.1 问题现象

搜索"测试"后，结果中出现了名为"remarks5"的用户，看似不应该匹配。

### 5.2 原因分析

**原因一：多字段匹配 — 显示名 ≠ 搜索命中字段**

好友搜索同时匹配三个字段：

| 字段 | 含义 | 说明 |
|------|------|------|
| `ele.remark` | 备注名 | 界面上优先显示此字段 |
| `ele.nickname` | 微信昵称 | 底层昵称，不一定和备注名相同 |
| `ele.username` | 微信 ID | wxid_xxx 格式 |

截图中显示的"remarks5"是该用户的 **备注名（remark）**，但其底层的 **微信昵称（nickname）** 或 **微信 ID（username）** 可能包含与"测试"匹配的字符。例如该用户的 nickname 可能是"测试账号5"，只是被设置了备注名"remarks5"覆盖显示。

**原因二：PinyinMatch 拼音模糊匹配**

`PinyinMatch.match()` 库支持以下匹配方式：

| 匹配方式 | 示例 | 说明 |
|---------|------|------|
| 中文精确匹配 | "测试" → "测试账号" | 子串包含 |
| 拼音全拼匹配 | "ceshi" → "测试账号" | 拼音子串 |
| 拼音首字母匹配 | "cs" → "测试账号" | 首字母缩写 |
| 混合匹配 | "测s" → "测试账号" | 中文+拼音混合 |

如果某用户的 nickname 或 username 中包含的字符序列恰好能被 PinyinMatch 匹配到搜索词的拼音，就会产生看似不相关的命中。

**原因三：潜在的竞态条件（标签搜索分支）**

当选择了标签时，`onFinish` 提前 return，搜索由 `useEffect` 异步触发。此时 `handle()` 使用的 `conditionInfo` 来自 Redux props，如果此刻 Redux 还未更新（React 批量渲染），可能导致搜索条件丢失：

```javascript
// 标签分支：提前 return，不直接调用 handle
if (filterTagList.length) {
    handleConditionInfo({ conditionInfo: { ...conditionInfo, ...values } });
    sendMessage({ type: 'getlabelcustomer', ids: filterTagList });
    setSearchBtnLoading(true);
    return;  // ← 不执行 handle()
}

// 后续由 useEffect 触发，但 conditionInfo 可能还是旧值
useEffect(() => {
    handle({ ...conditionInfo, ... }, 1, () => {} );
}, [store.getState().base.labelListFriend]);
```

### 5.3 排查建议

1. **确认该用户的真实 nickname/username**：在运营后台或通过日志查看该用户的完整信息（remark、nickname、username 三个字段），确认是否有隐含匹配
2. **在控制台测试 PinyinMatch**：
   ```javascript
   import PinyinMatch from 'pinyin-match';
   PinyinMatch.match("remarks5", "测试");   // 预期返回 false
   PinyinMatch.match(userNickname, "测试"); // 检查 nickname 的实际值
   ```
3. **检查是否触发了标签搜索分支**：如果同时选了标签，可能触发竞态条件

| 过滤结果（企业群） | handleCondition 计算 | Redux `broadcast.filterQyGroup` |
| 选中项 | 用户勾选 | Redux `broadcast.filterResultInfo` |

---

## 九、关键实现特点总结

1. **搜索以前端过滤为主**：数据源 `userList` 在应用启动时一次性加载到 Redux，后续搜索全部在浏览器端 JavaScript 中完成，无需每次请求后端
2. **PinyinMatch 支持拼音模糊匹配**：用户输入中文、拼音全拼、拼音首字母均可匹配，比普通 `indexOf` 功能更强但也更容易产生意料之外的匹配
3. **好友匹配三字段**：`remark`（备注名）、`nickname`（微信昵称）、`username`（微信ID），界面显示的是优先级最高的字段（通常是 remark），但搜索会命中底层其他字段
4. **过滤链串行执行**：每个条件依次缩小结果集，不会扩大范围
5. **少数条件需调后端**：交友时间、群内好友、用户行为需要后端 API 辅助（客户端本地无此数据）
6. **标签搜索走异步分支**：标签数据通过 WebSocket 获取，存在潜在竞态风险
