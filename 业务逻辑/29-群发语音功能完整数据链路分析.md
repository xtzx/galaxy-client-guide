# 群发语音功能完整数据链路分析

> 问题背景：企微群发功能 → 添加语音 → 手机发送语音 → 识别成功 → 取消 → 重新添加语音 → 一直"获取音频中"

---

## 一、整体架构

群发语音功能的数据流确实是 **逆向 → galaxy-client → galaxy** 的链路：

```
手机企微发送语音给自己
    ↓
[逆向层] 拦截到 RECV_MSG（content_type=16 语音消息）
    ↓
[逆向层] 发送 filepath 回调（语音文件本地路径）
    ↓
[galaxy-client] workWxMsgRecordService 处理语音消息 → 缓存到 FILE_KEY_MAP
    ↓
[galaxy-client] workWxFilePathService 处理文件路径 → 上传 silk 到 OSS
    ↓
[galaxy-client] 判断"发送给自己" → silkToMp3 转码 → 上传 mp3 到 OSS
    ↓
[galaxy-client] 通过 WebSocket 发送 uploadVoice 命令给前端
    ↓
[galaxy-client] 同时通过 MQTT 上报消息到云端
    ↓
[galaxy 前端] 接收 uploadVoice → 更新 Redux voiceInfo / API 可查
    ↓
[galaxy 前端] VoiceModal 检测到语音 → 显示语音
```

---

## 二、详细数据链路（6 个关键环节）

### 环节 1：逆向 → galaxy-client（RECV_MSG 语音消息）

**处理文件**：`src/msg-center/business/convert-service/workwx/workWxMsgRecordService.js`

**触发条件**：逆向发来 `type: 'RECV_MSG'`，`content_type: 16`（语音消息）

**核心逻辑**（第 782-856 行）：
```javascript
if (content_type == 16) {
    recordInfo.wechatMsgType = EMessage.TYPE__VOICE;
    recordInfo.content = "";
    let fileMsg = clientMsgBO.filemsg;
    // ...
    // 将语音消息缓存到 FILE_KEY_MAP，等待 filepath 回调
    FILE_KEY_MAP[keyValue] = clientMsgCloud;
}
```

**关键点**：
- 语音消息不直接处理，而是**缓存到 `FILE_KEY_MAP`**，等待逆向发回文件路径
- `key` 取自 `filemsg[i].weworkServerId` 或 `wxServerId`
- 如果 `voiceTime` 为空，消息会被跳过（`return`）

**日志关键词**：
```
[wxid-{wxid}] newlog语音消息:fileMsg:
[wxid-{wxid}] 把语音消息放入缓存key:
```

---

### 环节 2：逆向 → galaxy-client（filepath 文件路径回调）

**处理文件**：`src/msg-center/business/convert-service/workwx/workWxFilePathService.js`

**触发条件**：逆向发来 `type: 'filepath'`

**核心逻辑**（第 164-278 行）：
```javascript
// 1. 从缓存获取之前存入的消息对象（带重试，最多5次，每次1秒）
const clientMsgCloud = await this.getCacheCloudInfo(key, retryInfo);

// 2. 上传 silk 文件到 OSS
url = await AliyunOssManagerUtil.uploadFileNoCache(pathStr, ...);

// 3. 关键判断：是否是"发送给自己"的语音
if (
    recordInfo.fromUsername &&
    recordInfo?.toUsernames?.[0] === recordInfo.fromUsername
) {
    // 只有发送给自己的语音才会走 silkToMp3 → uploadVoice 推送前端
    this.silkToMp3(pathStr, pipeLineWrapper, fileName, url, recordInfo);
}

// 4. 上报云端
MqttSendService.sendMessage(ownWxId, clientMsgCloud);
```

**关键点**：
- **`FILE_KEY_MAP` 匹配**：用 `key`（weworkServerId）从缓存取出之前的消息对象，如果取不到（重试5次），整个流程中断
- **"发送给自己"判断**（⚠️ 极为关键）：只有 `fromUsername === toUsernames[0]` 时，才会触发 `silkToMp3()` 和 `uploadVoice` 推送。如果判断失败，语音虽然上传了 OSS 和云端，但**不会推送给前端**
- **silk → mp3 转码**：调用 ffmpeg 进行转码，可能因路径/权限/依赖问题失败

**日志关键词**：
```
wxid-{wxid} [WorkWxFilePathService] init name:[{filename}] key:[{key}]
[获取云端类数据文件处理耗时] time: {ms}, [{key}]
wxid-{wxid} [文件消息处理完成] time: {ms}, KEY:[{key}] URL: [{url}]
[WorkWxFilePathService] 空链接 url:
[codeError] wxid-{wxid} [WorkWxFilePathServiceError]
```

---

### 环节 3：galaxy-client silk → mp3 转码 + 推送前端

**处理文件**：`workWxFilePathService.js` → `silkToMp3()` 方法（第 323-385 行）

**核心逻辑**：
```javascript
// 1. silk 转 mp3
let mp3Url = await AudioConversionUtil.silk2mp3(pathStr, fileName);

// 2. 上传 mp3 到 OSS
let url = await AliyunOssManagerUtil.uploadFileNoCache(mp3Url, ...);

// 3. 构造前端响应
let response = {
    cmdId: "uploadVoice",
    channelId: pipeLineWrapper.id,
    wxid: pipeLineWrapper.wxid,
    body: extJsonContent,  // 包含 url, path, silkUrl, voicetime, aesKey 等
};

// 4. 推送给前端 ⭐⭐⭐
SendToFrontUtil.sendResponse2Front("uploadVoice", response);
```

**关键点**：
- 如果 `silk2mp3` 转码失败，整个推送流程中断，前端永远收不到语音
- mp3 上传 OSS 失败也会中断
- `response.body` 中需要包含 `url` 字段，前端才能识别为有效语音

**日志关键词**：
```
wxid-{wxid} pathStr:[{path}] fileName:[{name}] mp3 original mp3Url:
wxid-{wxid} pathStr:[{path}] fileName:[{name}] mp3 original filepath:
上报mp3文件失败 pathStr:
HANDLE_VOICE_ERROR
```

## 三、日志排查指南

### 排查顺序（沿数据流逐步检查）

#### Step 1：确认逆向是否发送了消息

搜索关键词：
```
[wxid-{wxid}] [MsgRecordServiceImpl-operate] - client收到逆向消息
```

期望看到 `content_type` 为 16 的语音消息。**如果没有**，说明逆向没有把第二条语音传递过来。

#### Step 2：确认语音是否进入 FILE_KEY_MAP 缓存

搜索关键词：
```
[wxid-{wxid}] newlog语音消息:fileMsg:
[wxid-{wxid}] 把语音消息放入缓存key:
```

期望看到 `voiceTime` 非空，且成功放入缓存。**如果 `voiceTime` 为 null**，消息会被跳过（workWxMsgRecordService 第 826 行 `if (voiceTime != null)` 判断）。

#### Step 3：确认 filepath 回调是否到达

搜索关键词：
```
wxid-{wxid} [WorkWxFilePathService] init name:
```

期望看到对应 key 的 filepath 消息。**如果没有**，说明逆向没有返回文件路径回调。

#### Step 4：确认 FILE_KEY_MAP 是否匹配成功

搜索关键词：
```
[获取云端类数据文件处理耗时]
```

如果耗时超过 5 秒（5次重试 × 1秒），说明缓存匹配失败。**可能原因：key 不一致，或第一步的消息没有正确缓存**。

#### Step 5：确认"发送给自己"判断是否通过

这步**没有直接日志**，需要从 Step 2 的日志中提取 `fromUsername` 和 `toUsernames`，确认两者是否一致。

如果 `fromUsername !== toUsernames[0]`，则 `silkToMp3()` 不会被调用，`uploadVoice` 不会推送给前端。

#### Step 6：确认 silk → mp3 转码是否成功

搜索关键词：
```
wxid-{wxid} pathStr: fileName: mp3 original mp3Url:
wxid-{wxid} pathStr: fileName: mp3 original filepath:
HANDLE_VOICE_ERROR
上报mp3文件失败
```

**如果看到 `HANDLE_VOICE_ERROR`**，说明转码或上传失败。

#### Step 7：确认 uploadVoice 是否推送成功

搜索关键词（galaxy 前端日志/阿里云日志）：
```
uploadVoice
```

在 galaxy sub 的 thunks 中有 `uqunLogUpdateReport`：
```
type: 'uploadVoice', sendStatus: 'hanleStart'
```

## 四、常见故障点总结

| # | 环节 | 可能故障 | 表现 | 检查方法 |
|---|------|---------|------|---------|
| 1 | 逆向层 | 第二条语音未送达 galaxy-client | 无 RECV_MSG 日志 | 搜索 `MsgRecordServiceImpl-operate` |
| 2 | MsgRecord | voiceTime 为空导致跳过 | 有 fileMsg 日志但无"放入缓存"日志 | 搜索 `newlog语音消息` |
| 3 | FilePath | filepath 回调未到达 | 无 WorkWxFilePathService init 日志 | 搜索 `WorkWxFilePathService init` |
| 4 | FilePath | FILE_KEY_MAP 匹配失败 | 超时未匹配到消息 | 搜索 `获取云端类数据文件处理耗时` |
| 5 | FilePath | "发送给自己"判断失败 | 无 mp3 相关日志 | 对比 fromUsername 和 toUsernames |
| 6 | silkToMp3 | silk 转码失败 | `HANDLE_VOICE_ERROR` | 搜索 `HANDLE_VOICE_ERROR` |
| 7 | silkToMp3 | mp3 上传 OSS 失败 | `上报mp3文件失败` | 搜索 `上报mp3文件失败` |
| 8 | WebSocket | uploadVoice 推送失败 | 前端无日志 | 搜索 `uploadVoice` |
| 9 | 前端 Redux | voiceInfo 未更新/引用未变 | 弹窗无反应 | 检查 Redux DevTools |

## 五、针对"取消后重新发送"场景的特别分析

取消后重新发送语音时，关键问题在于：

1. **前端 VoiceModal 关闭时不会清理 Redux voiceInfo**（sub 模块）
   - 只清了本地 state `setVocieItem({})`
   - Redux `voiceInfo` 保持旧值
   - 如果新语音的 body 结构与旧的完全一致（引用未变），`useEffect([voiceInfo])` 不会重新触发

2. **galaxy-client 的 FILE_KEY_MAP 是否有残留**
   - 正常情况下，处理完成后会 `delete this.FILE_KEY_MAP[key]`
   - 但如果上次处理异常中断，可能残留脏数据

3. **逆向层可能对"发给自己"的重复消息做了去重**
   - 如果短时间内发了两条类似的语音，逆向可能只通知一次
