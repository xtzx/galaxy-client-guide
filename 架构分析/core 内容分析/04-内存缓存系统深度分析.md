# 内存缓存系统深度分析

> 分析范围：`galaxy-client/src/msg-center/core/cache/` 目录全部文件（含子目录 `common/`、`wx/`、`workwx/`）
> 关联模块：`dispatch-center/dispatchInBound.js`、`core/data-config/galaxyTaskType.js`、`init/log.js`

---

## 一、模块概述

### 1.1 功能定位

内存缓存系统是 Galaxy 客户端中**跨模块共享数据**的核心基础设施。它通过在内存中维护各种 Map、Array、Object 结构，为业务逻辑提供高速的中间数据存取能力。

缓存覆盖了以下几大类数据：

- **群聊信息**：群详情、群成员列表、群主信息、新入群成员等
- **任务状态**：下发任务的回执跟踪、消息类型映射、超时状态管理
- **用户数据**：好友列表、群列表的分页拉取状态和数据聚合
- **好友关系**：新好友通知、好友申请信息
- **文件映射**：CDN 地址到消息对象的映射
- **系统状态**：管道码映射、逆向版本号、工作状态、备注任务状态

### 1.2 为什么需要内存缓存

Galaxy 客户端的业务流程中存在大量**异步交互场景**，主进程需要频繁地与微信进程、云端服务器、渲染进程交换数据。内存缓存在以下场景中不可或缺：

- **请求-响应模式的数据桥接**：主进程向逆向发送获取群成员列表的请求后，响应是异步返回的。缓存作为中间存储，将请求方和响应方桥接起来。请求方发出请求后通过轮询缓存等待数据就绪
- **分页数据聚合**：好友列表和群列表数据量可能很大，逆向分多次返回。缓存负责将分页数据逐步拼接成完整列表
- **任务回执跟踪**：云端下发的任务可能有多个阶段的回执（如发送确认、消息已读回执等），缓存跟踪每个任务在各阶段的状态
- **防重复请求**：某些操作（如获取群信息）可能被多处业务代码触发，缓存记录已发出的请求，避免重复向逆向发送相同请求
- **临时数据交换**：不同业务模块之间通过缓存传递临时数据，避免深层函数调用传参

### 1.3 在整体架构中的位置

缓存层位于核心层（core）的中间位置，被几乎所有业务模块引用：

- **写入方**：逆向消息处理器（handle/）、MQTT 任务处理器（task-mqtt/）、调度中心
- **读取方**：业务转换服务（convert-service/）、云端上报模块、前端推送模块
- **自身也是消费者**：部分缓存（如 chatroomInfoCache）会主动向调度中心发送请求获取数据

## 三、核心缓存深度分析

### 3.1 chatroomInfoCache —— 群聊信息缓存中心

这是整个缓存系统中最复杂的模块，包含 **7 个独立子缓存**和多个工具方法。

#### 3.1.1 子缓存一览

| 子缓存名称 | 数据结构 | 键格式 | 值类型 | 用途 |
|-----------|----------|--------|--------|------|
| `chatroomKeywordInfoCache` | Map | `chatroomId_wxid` | `{chatroomAccessType, chatroomName, headimg}` | 群关键信息（验证方式、群名、头像） |
| `newChatroomMembersCache` | Map | `chatroomId+时间戳` | `Array<member>` | 临时群成员列表 |
| `singleChatroomInfoCache` | Map | `chatroomId_wxid` | `Object` | 单个群的详情信息（群主等） |
| `isGetSingleChatroomInfoCache` | Map | `chatroomId_wxid` | `boolean` | 是否已发出获取群详情的请求 |
| `newMemberCache` | Map | `accountId` | `{chatroomId: Array<member>}` | 新入群成员缓存 |
| `chatroomMemberListCache` | Object | `chatroomId` | `Array<member>` | 群成员列表缓存 |
| `memberKickOutMap` | Object | `chatroomId+toUsername` | `fromUsername` | 踢人记录 |
| `inviteTalkList` | Map | `chatroomId` | `Array<string>` | 邀请入群话术缓存 |
| `chatroomNewMemberList` | Array | — | `Array<Object>` | 新入群成员组列表 |
| `userInfoListRecord` | Array | — | `Array<Array<string>>` | 已请求过的用户信息列表记录 |

#### 3.1.2 异步轮询获取模式

`chatroomInfoCache` 中有一个非常重要的设计模式：**发出请求 + 轮询缓存等待响应**。

以获取群关键信息为例（`getChatroomKeywordInfoCache`）：

1. 某业务模块需要获取某个群的信息
2. 通过另一处逻辑向逆向发送 `userinfolist` 请求
3. 请求是异步的，响应由逆向消息处理器接收后写入缓存（`setChatroomKeywordInfoCache`）
4. 业务模块调用 `getChatroomKeywordInfoCache(key, retryInfo)` 轮询缓存
5. 如果缓存中没有数据，休眠 300ms 后重试，最多重试 5 次
6. 如果在 5 次重试（约 1.5 秒）内数据到达缓存，返回数据
7. 如果超过重试次数仍未获取到，返回 undefined

同样的模式也用于 `getNewChatroomMembersCache`（1 秒间隔，最多重试由调用方决定）和 `getSingleChatroomInfoCache`（1 秒间隔，最多 4 次）。

#### 3.1.3 主动请求能力

`chatroomInfoCache` 不仅是被动存储，还具备**主动向逆向发送请求**的能力：

- `getChatroomInfoListRequest`：向逆向发送获取群详情（userinfolist）的请求
- `updateChatroomInfoListRequest`：向逆向批量获取群详情（chatuserinfolist）
- `sendChatroomDetailRequest`：向逆向获取群成员详情（chatuserinfo）
- `sendChatroomDetailListRequest`：向逆向获取群成员详情列表（chatuserlistex）

这些方法通过 `dispatchInBound` 向调度中心发送消息，最终经逆向 IPC 传递给微信进程。

#### 3.1.4 邀请话术缓存的过期清理

`putInviteTalk` 方法在存储邀请话术时，会自动清理超过 10 分钟的旧话术：

1. 以群 ID 为维度获取已有话术列表
2. 将新话术拼接为固定格式（ownerId + 分隔符 + chatroomId + 分隔符 + talk + 分隔符 + createTime）
3. 过滤掉创建时间超过 10 分钟的旧话术
4. 更新缓存

这是整个缓存系统中为数不多的**自动过期清理机制**。

#### 3.1.5 群成员列表工具方法

- `filterChatroomMemberList`：去重和过滤空成员，支持字符串格式和对象格式的成员数据
- `getChatroomMemberList`：解析 `^G` 分隔的群成员字符串为数组
- `isContainsInChatroom`：检查某人是否在某群中
- `putChatroomMemberList`：模拟方法重载（1 个参数调用版本 1，2 个参数调用版本 2）

### 3.2 chatroomFetchStateManager —— 群信息获取状态管理器

这是一个设计相对完善的**状态机 + 任务队列**系统，专门管理群信息的批量获取流程。

#### 3.2.1 状态机设计

每个群信息获取任务有 4 种状态：

```
pending（待获取）→ fetching（获取中）→ success（成功）
                                      → failed（失败）→ pending（重试）
```

状态存储在 `fetchStateMap` 中，键为 `${wxid}_${chatroomId}`，值包含完整的状态上下文：

- `chatroomId`：群 ID
- `wxid`：微信 ID
- `status`：当前状态（pending/fetching/success/failed）
- `requestTime`：请求发出时间
- `retryCount`：已重试次数
- `maxRetryCount`：最大重试次数（默认 3）
- `timeout`：超时时间（默认 10 分钟）

#### 3.2.2 队列消费机制

`pendingQueue` 是一个 FIFO 数组，存储待获取的群信息任务。消费流程：

1. 调用 `addToQueue(wxid, chatroomIds)` 批量添加群到队列
2. 对每个群 ID，检查是否已在获取中或已成功，跳过重复
3. 初始化状态为 `pending`，推入队列
4. 触发 `startProcessQueue()` 开始消费
5. 消费循环从队列头部取出任务，更新状态为 `fetching`
6. 调用 `fetchChatroomDetail` 向逆向发送获取请求
7. 每个请求之间间隔 1 秒（`fetchInterval`）
8. 队列消费完毕后设置 `isProcessing = false`

通过 `isProcessing` 标记防止并发消费。

#### 3.2.3 超时与重试机制

- 构造函数中启动了一个**每 60 秒执行一次**的定时检查器（`startTimeoutChecker`）
- 检查器遍历所有 `fetching` 状态的任务，如果请求时间超过 10 分钟则视为超时
- 超时的任务：如果重试次数未达上限（3 次），重新入队并重置为 `pending` 状态
- 达到上限的任务标记为 `failed`，不再重试

#### 3.2.4 外部接口

- `markSuccess(wxid, chatroomId)`：由逆向消息处理器在成功接收到群信息后调用
- `markFailed(wxid, chatroomId)`：获取失败时调用
- `getState(wxid, chatroomId)`：查询某个群的获取状态
- `clearState(wxid, chatroomId)`：清理某个群的状态记录
- `getStats()`：获取统计信息（各状态数量、队列长度）

### 3.3 galaxyTaskCache —— 任务状态缓存

#### 3.3.1 核心数据结构

维护三个关键数据结构：

**GALAXY_TASK_STATUS_MAP**（Object）：
- 键：taskId
- 值：任务状态对象，包含各阶段的消息接收状态、微信 ID、创建时间等
- 用途：跟踪从云端下发的任务在客户端的执行状态

**TASK_MSGTYPE_MAP**（Map）：
- 键：`${sendUsername}-${toUsername}`（发送者-接收者）
- 值：`Array<string>`，每项格式为 `taskId-msgType-sendTime-content`
- 用途：根据发送者和接收者快速查找关联的任务信息，主要用于群发消息的漏消息补偿

**TASK_INFO_MAP**（Map）：
- 键：taskId
- 值：`${sendUsername}-${toUsername}`
- 用途：通过 taskId 反查对应的发送者-接收者键，实现双向查找

#### 3.3.2 消息内容处理逻辑

`processContentByType` 方法根据微信消息类型对内容进行差异化处理：

- 文本消息（type 1）：直接保留，截断到 500 字符
- 图片消息（type 3）：提取文件路径或 URL
- 视频消息（type 43）：提取文件路径或 URL
- 文件消息（type 49001）：提取文件路径或 URL
- 链接消息（type 49002）：提取 URL
- 名片消息（type 42）：提取 wxid
- 表情/语音（type 47/34）：保留原始内容
- 其他类型：截断到 500 字符

#### 3.3.3 任务清理逻辑

`removeTaskMsgTypeMapToTaskId` 方法实现了通过 taskId 清理关联缓存的完整流程：

1. 从 `TASK_INFO_MAP` 中通过 taskId 找到对应的 key（`sendUsername-toUsername`）
2. 从 `TASK_MSGTYPE_MAP` 中获取该 key 下的所有 value 数组
3. 过滤掉 taskId 匹配的项
4. 如果过滤后数组为空，删除整个 key
5. 清理 `TASK_INFO_MAP` 中的 taskId 记录

### 3.4 用户与群聊数据缓存（wx/ 和 workwx/）

微信和企微各有一套用户与群聊数据缓存，结构高度相似。

#### 3.4.1 共同的状态机

两者都定义了 `MsgStatus` 枚举：

| 状态 | 值 | 说明 |
|------|-----|------|
| START | -1 | 开始 |
| INIT | 0 | 初始化 |
| SECOND | 1 | 第二阶段 |
| FINISH | 3 | 完成 |

用于跟踪用户列表获取的整体进度。

#### 3.4.2 共同的核心缓存

| 缓存名称 | 类型 | 说明 |
|----------|------|------|
| `userMsgStatus` | Map | 用户信息请求状态（MsgStatus） |
| `userInfoInitData` | Map | userlist 初始化数据 |
| `conversationList` | Map | 群列表（逐页追加） |
| `externalUserList` | Map | 好友列表（逐页追加） |
| `userPageMap` | Map | 好友列表分页请求数据 |
| `usersRequestMap` | Map | 好友列表请求成功/失败统计 |
| `timeoutTimerMap` | Map | 超时定时器映射 |
| `reqTimes` | Object | 请求次数统计 |

#### 3.4.3 企微额外的缓存

企微版本（wkUserAndChatRoomCache）比微信版本多出两个缓存：

- `conversationPageMap`：群列表的分页请求数据（微信没有这个，因为微信的群列表在好友列表请求中一起返回）
- `conversationRequestMap`：群列表请求的成功/失败统计

#### 3.4.4 分页数据聚合模式

好友列表和群列表都采用分页拉取模式：

1. `initData` 初始化所有缓存，设置状态为 INIT
2. 每页数据返回时调用 `updateConversationList` 或 `updateExternalUserList`，通过 `concat` 将新页追加到已有数据
3. 同时通过 `updateExternalUserPageReqList` 记录每页请求的原始数据（用于重试判断）
4. 通过 `updateExternalUserRequest` 统计成功和失败次数
5. 所有页返回后，通过检查成功次数判断是否完整

### 3.5 msgFileCache —— 消息文件缓存

这是唯一一个实现了**完整的容量限制和过期清理**的缓存模块。

#### 3.5.1 缓存设计

- **容量上限**：1000 条
- **过期时间**：30 秒
- **数据结构**：`Map<cdn_url, {clientMsg, time}>`

#### 3.5.2 写入时的清理策略

当缓存大小超过 1000 时，在写入前遍历整个 Map，清除所有超过 30 秒的过期条目。这是一种「懒清理」策略：只在新数据需要写入且超出容量时才触发清理。

#### 3.5.3 读取模式

提供两种读取方式：

- `getFile2MsgCache(cdn)`：只读取，不删除
- `popFile2MsgCache(cdn)`：读取并删除（类似队列的 pop 操作）

### 3.6 newFriendCache —— 新好友缓存

存储添加好友时的系统话术信息。使用数组的 `unshift` 方法将新数据插入头部（最新的排在最前面）。

`removeAll` 方法存在一个逻辑 Bug（详见问题分析部分）。

### 3.7 极简缓存模块

以下模块仅声明了数据结构，没有任何业务逻辑，由外部直接读写属性：

- **quickSendCache**：`TASK_MAP` (Map) —— 极速群发任务的重试数据
- **remarkTaskCache**：`REMARK_TASK_MAP` (Object) —— 备注任务数据
- **LogicWorkingCache**：`LOGIC_WORKING_MAP` (Object) —— 微信实例工作状态
- **msgTaskInfoCache**：`msgIdToTaskIdInfo` (Object) —— 消息 ID 到任务 ID 的映射
- **galaxyVersionCache**：`galaxyVersion` (String) —— 逆向版本号
- **roomMemberWait4ReportMap**：`map` (Map) —— 群成员待上报数据
- **workwx/workWxNewFriendApplyCache**：空对象 + todo 注释

## 四、缓存生命周期管理

### 4.1 写入时机

| 缓存 | 写入触发 |
|------|----------|
| chatroomKeywordInfoCache | 逆向返回 userinfolist 数据时 |
| newChatroomMembersCache | 逆向返回群成员列表时 |
| singleChatroomInfoCache | 逆向返回单个群详情时 |
| GALAXY_TASK_STATUS_MAP | 调度中心下发任务到逆向时 |
| TASK_MSGTYPE_MAP | 调度中心下发任务到逆向时 |
| conversationList / externalUserList | 逆向分页返回好友和群数据时 |
| file2MsgCache | 文件下载任务创建时 |
| newFriendMsgList | 逆向推送新好友系统消息时 |

### 4.2 清理时机

| 缓存 | 清理方式 |
|------|----------|
| chatroomKeywordInfoCache | 手动调用 remove 方法，**无自动过期** |
| newChatroomMembersCache | 读取后 10ms 延时自动删除 |
| singleChatroomInfoCache | 读取后 10ms 延时自动删除 |
| GALAXY_TASK_STATUS_MAP | 任务完成时手动删除 |
| TASK_MSGTYPE_MAP | 任务完成时通过 taskId 反查删除 |
| conversationList / externalUserList | initData 时重置 |
| file2MsgCache | 容量超限时清理过期条目 / popFile2MsgCache 读取后删除 |
| inviteTalkList | putInviteTalk 时自动清理 10 分钟前的旧数据 |
| fetchStateMap | 手动 clearState / 超时后标记 failed |

### 4.3 缺少清理机制的缓存

以下缓存**没有任何自动清理机制**，随着运行时间增长会持续占用内存：

- `chatroomKeywordInfoCache`：群关键信息永不过期
- `chatroomMemberListCache`：群成员列表永不过期
- `memberKickOutMap`：踢人记录永不过期
- `chatroomNewMemberList`：新入群成员列表永不过期（removeAll 方法有 Bug）
- `userInfoListRecord`：已请求记录只增不删
- `GALAXY_TASK_STATUS_MAP`：如果任务完成逻辑未触发则永不清理
- `newFriendMsgList`：新好友话术永不过期
- `workWxNewFriendApplyInfoList`：好友申请信息永不过期
- `LOGIC_WORKING_MAP`、`REMARK_TASK_MAP`、`msgIdToTaskIdInfo`：外部负责清理

## 五、业务场景映射

### 5.1 场景一：获取群主信息（异步轮询模式）

**完整流程**：

1. 业务模块需要知道某个群的群主是谁
2. 调用 `chatroomInfoCache.getSingleChatroomInfoData(chatroom, ownerWxId)`
3. 该方法内部调用 `getChatroomInfoListRequest`，构造 `userinfolist` 类型的消息
4. 消息通过 `dispatchInBound` → `reverseSend` → 逆向 IPC 发送到微信进程
5. 同时设置 `isGetSingleChatroomInfoCache` 标记为 true，防止重复请求
6. 然后开始轮询 `getSingleChatroomInfoCache`，每 1 秒检查一次
7. 微信进程处理请求后通过管道返回群详情数据
8. 逆向消息处理器接收到响应，调用 `updateSingleChatroomInfoCache` 写入缓存
9. 下一轮轮询检测到数据已就绪，返回群详情
10. 读取后 10ms 延时自动清理缓存条目

### 5.2 场景二：好友列表分页拉取与聚合

**完整流程**：

1. 微信登录成功后触发好友列表拉取
2. 调用 `wxUserAndChatRoomCache.initData(accountId, initData)` 重置缓存
3. 向逆向发送 `getuserlist` 请求
4. 逆向分多页返回数据（每页可能 200-500 个好友）
5. 每页返回时：
   - 调用 `updateConversationList` 追加群列表数据
   - 调用 `updateExternalUserList` 追加好友数据
   - 调用 `updateExternalUserPageReqList` 记录该页请求数据
   - 调用 `updateExternalUserRequest(accountId, true)` 记录成功
6. 如果某页请求失败：
   - 调用 `updateExternalUserRequest(accountId, false)` 记录失败
   - 通过 `updateExternalUserRetryCount` 增加重试计数
   - 重新发送该页请求
7. 所有页返回后，`conversationList` 和 `externalUserList` 中就是完整数据
8. 业务模块从缓存中读取完整列表，推送到前端展示并上报云端

### 5.3 场景三：群发消息的漏消息补偿

**完整流程**：

1. 调度中心下发群发消息任务到逆向
2. `dispatchInBound` 在下发时调用 `galaxyTaskCache.setTaskMsgTypeMap(jsonObject)`
3. 缓存记录：`sendUsername-toUsername` → `[taskId-msgType-sendTime-content]`
4. 同时记录：`taskId` → `sendUsername-toUsername`（反向索引）
5. 微信进程发送消息成功后，回报 `msgreport` 类型的消息
6. 消息处理器通过 `getTaskMsgTypeMap(sendUsername, toUsername)` 查找关联任务
7. 如果找到对应的任务记录，确认消息发送成功，清理缓存
8. 如果超时未收到回报，任务超时检查器发现该任务未完成，上报超时状态
9. 任务完成后调用 `removeTaskMsgTypeMapToTaskId(taskId)` 清理所有关联缓存

### 5.4 场景四：批量群信息获取的流控

**完整流程**：

1. 登录后需要获取所有群的详细信息
2. 调用 `chatroomFetchStateManager.getInstance().addToQueue(wxid, chatroomIds)`
3. 状态管理器为每个群创建 `pending` 状态记录
4. 消费循环逐个处理：取出 → 设为 `fetching` → 发送请求 → 等待 1 秒
5. 逆向返回群信息后，业务模块调用 `markSuccess` 标记为成功
6. 如果 10 分钟内未返回，超时检查器将其重新入队重试（最多 3 次）
7. 达到最大重试次数后标记为 `failed` 放弃

## 六、问题分析与优化建议

### 6.1 严重问题

#### 6.1.1 多数缓存无过期和容量限制（内存泄漏风险）

16 个缓存模块中，只有 `msgFileCache` 实现了完整的容量限制（1000）和过期清理（30 秒）。大部分缓存（如 `chatroomKeywordInfoCache`、`chatroomMemberListCache`、`memberKickOutMap`、`newFriendMsgList` 等）没有任何过期和容量限制机制。

在长时间运行的场景下（Galaxy 客户端通常 24 小时不间断运行），这些缓存会持续增长，最终可能导致内存溢出。

特别是 `chatroomKeywordInfoCache` 和 `GALAXY_TASK_STATUS_MAP`，在每天处理大量任务的生产环境中增长速度最快。

**建议**：
- 为所有 Map 类型的缓存设置容量上限和 TTL（生存时间）
- 可以考虑使用统一的缓存基类，内置 LRU 淘汰和 TTL 过期机制
- 或者引入 `node-cache` 等成熟的内存缓存库

#### 6.1.2 newFriendCache.removeAll 方法存在逻辑 Bug

`removeAll` 方法的实现有严重错误：

1. 方法接收一个要删除的元素数组
2. 创建一个 Set 用于快速查找
3. 对 `newFriendMsgList` 执行 `map`，返回 `m.has(item)` 的布尔值
4. 将 `map` 的结果（布尔值数组）赋值回 `newFriendMsgList`

这意味着调用 `removeAll` 后，`newFriendMsgList` 中原本的对象数据被替换成了 `[true, false, true, ...]` 这样的布尔值数组，完全丧失了原始数据。

正确的实现应该使用 `filter` 而非 `map`，且条件应该取反（保留不在删除列表中的元素）。

**影响**：新好友缓存的批量删除功能完全损坏。

#### 6.1.3 chatroomInfoCache.getMemberKickOut 赋值而非取值

`getMemberKickOut` 方法的 return 语句中使用了赋值运算符 `=` 而非仅读取：

```
return (this.memberKickOutMap[chatroomId + toUsername] = null);
```

这行代码做了两件事：先将该键的值设为 null，然后返回 null。也就是说每次调用 `getMemberKickOut` 不仅不会返回踢人记录，还会**清除**该记录。

**影响**：踢人记录查询功能完全失效，且会破坏已有数据。

#### 6.1.4 chatroomInfoCache.removeAll 方法的空 filter 回调

`removeAll` 方法中的 `filter` 回调函数体为空（`(item) => {}`），这意味着回调永远返回 `undefined`（falsy），`filter` 的结果将是一个空数组。调用此方法会**清空整个列表**而非按条件删除。

### 6.2 设计问题

#### 6.2.1 异步轮询获取的效率问题

`chatroomInfoCache` 中的 `getChatroomKeywordInfoCache`、`getNewChatroomMembersCache`、`getSingleChatroomInfoCache` 都使用递归 + sleep 的方式轮询缓存。这种模式存在以下问题：

- **固定间隔**：不管数据是否已经到达，都要等完整的间隔时间才检查
- **递归实现**：高频调用时有栈增长的风险
- **缺少超时通知**：如果超时返回 undefined/空数组，调用方不知道是数据确实不存在还是请求超时

**建议**：使用 Promise + EventEmitter 模式替代轮询，数据到达时主动 resolve Promise，无需轮询等待。

#### 6.2.2 缓存与业务逻辑混合

`chatroomInfoCache` 不仅是缓存模块，还包含了向逆向发送请求的业务逻辑（`getChatroomInfoListRequest`、`sendChatroomDetailRequest` 等）。这违反了单一职责原则，使得缓存模块依赖了调度中心（`dispatchInBound`），增加了模块间的耦合。

**建议**：将主动请求逻辑移到独立的 service 模块中，缓存只负责数据存取。

#### 6.2.3 微信和企微缓存的代码重复

`wx/wxUserAndChatRoomCache.js` 和 `workwx/wkUserAndChatRoomCache.js` 有约 80% 的代码完全相同（`initData`、`clearData`、`updateUserMsgStatus`、`updateConversationList`、`updateExternalUserList` 等方法签名和实现一致）。企微版本仅多了群列表分页相关的方法。

**建议**：提取公共基类或 mixin，减少代码重复。

#### 6.2.4 chatroomFetchStateManager 的定时器永不停止

构造函数中启动的 `setInterval`（每 60 秒检查超时）没有提供停止机制。即使所有群信息都已获取完成（fetchStateMap 为空），定时器仍然会持续运行。

**建议**：在 fetchStateMap 为空时自动停止定时器，有新任务入队时重新启动。

#### 6.2.5 缺少缓存监控能力

除了 `chatroomFetchStateManager` 提供了 `getStats()` 方法外，其他缓存模块都没有提供查看当前缓存大小、命中率等监控信息的能力。在排查内存问题时，无法快速了解各缓存的占用情况。

**建议**：为所有缓存添加 `getSize()` 和 `getStats()` 方法，并定期上报到监控系统。

### 6.3 代码质量问题

#### 6.3.1 putChatroomMemberList 的方法重载实现错误

`putChatroomMemberList` 尝试模拟方法重载，但逻辑有误：

- 当 `args.length === 1` 时调用 `putChatroomMemberList1(args[0], args[1])`
- 但 `args.length === 1` 意味着只有一个参数，`args[1]` 是 `undefined`
- 而 `putChatroomMemberList1` 需要两个参数（chatroom, memberList）
- 应该是 `args.length === 2` 时调用 `putChatroomMemberList1`，`args.length === 1` 时调用 `putChatroomMemberList2`

**影响**：以 2 个参数调用时实际走了批量写入的逻辑，以 1 个参数调用时写入了 undefined 值。

#### 6.3.2 isRequestedUserInfoList 的返回值逻辑错误

`isRequestedUserInfoList` 方法用于检查某个 wxid 列表是否已经被请求过。但其中的 `filter` 结果始终是一个数组（可能为空数组），而 `if (!collect)` 对空数组的判断是 `false`（空数组是 truthy 值）。

这意味着 `if (!collect)` 分支**永远不会执行**，`this.userInfoListRecord.push(wxidList)` 永远不会被调用，方法永远返回 `true`。

**影响**：防重复请求机制完全失效。

#### 6.3.3 数据结构混用

同一个模块中混用了 Map、Object（{}）和 Array 来存储类似的缓存数据：

- `chatroomKeywordInfoCache`：使用 Map
- `chatroomMemberListCache`：使用 Object
- `memberKickOutMap`：使用 Object
- `inviteTalkList`：使用 Map
- `chatroomNewMemberList`：使用 Array

Object 作为缓存容器存在键类型限制（只能是字符串）、无法直接获取大小（需要 `Object.keys().length`）、没有 `has` 方法等问题。

**建议**：统一使用 Map 作为缓存容器。

#### 6.3.4 workwx 目录下存在两个不同版本的同名文件

- `cache/workWxNewFriendApplyCache.js`（16 行，有完整实现）
- `cache/workwx/workWxNewFriendApplyCache.js`（5 行，空对象 + todo）

两者功能重叠，容易在引用时产生混淆。

**建议**：确定使用哪个版本，删除另一个。

### 6.4 架构问题

#### 6.4.1 缺少统一的缓存抽象

16 个缓存模块各自独立实现，没有统一的基类或接口规范。每个缓存的 API 风格、生命周期管理方式都不同。这导致：

- 新增缓存时没有参考模板，容易写出不一致的代码
- 无法统一监控所有缓存的状态
- 无法统一设置容量限制和过期策略

**建议**：实现一个 `CacheBase` 类，提供：
- `get(key)` / `set(key, value, ttl)` / `delete(key)` 标准接口
- 内置 LRU 淘汰和 TTL 过期
- `getStats()` 监控接口
- 容量限制配置

#### 6.4.2 全局可变状态带来的并发风险

所有缓存都是模块级的单例对象，被多个异步操作同时读写。虽然 Node.js 是单线程的，但由于 async/await 的存在，以下场景可能出现竞态：

1. A 操作读取了 `chatroomKeywordInfoCache`，开始处理
2. 在 A 的 await 期间，B 操作修改了同一个缓存条目
3. A 继续执行时使用的数据已经不是最新的

对于大部分场景这不是问题，但对于需要原子操作的场景（如 `compareAndSet` 模式的 CAS 操作），可能导致逻辑错误。

## 八、总结

### 8.1 模块评价

内存缓存系统是 Galaxy 客户端中文件数最多、覆盖面最广的模块。它为异步消息驱动的架构提供了关键的中间数据存储能力，尤其是「发请求 + 轮询等待」模式在群聊信息获取等场景中是不可替代的。`chatroomFetchStateManager` 的设计质量相对较高，具备完整的状态机、队列、超时、重试机制。

### 8.2 核心优点

- **chatroomFetchStateManager 设计完善**：状态机 + 队列 + 超时重试 + 统计监控，是全系统中设计最规范的模块
- **msgFileCache 的容量和过期管理**：是唯一正确实现了容量限制和过期清理的缓存
- **galaxyTaskCache 的双 Map 关联查找**：通过 TASK_INFO_MAP 实现 taskId 到 TASK_MSGTYPE_MAP 的反向索引，查找效率高
- **分页数据聚合模式**：通过 concat 追加实现分页数据的渐进式聚合，设计简洁
- **inviteTalkList 的自动过期**：在写入时清理 10 分钟前的旧数据

### 8.3 核心待改进项

| 问题 | 风险等级 | 影响范围 |
|------|----------|----------|
| 大部分缓存无过期/容量限制 | 高 | 长期运行内存泄漏 |
| getMemberKickOut 赋值 Bug | 高 | 踢人记录查询破坏数据 |
| newFriendCache.removeAll Bug | 高 | 缓存数据被替换为布尔值 |
| chatroomInfoCache.removeAll 空回调 | 高 | 清空整个列表 |
| putChatroomMemberList 重载实现错误 | 中 | 写入 undefined 值 |
| isRequestedUserInfoList 逻辑永真 | 中 | 防重复请求失效 |
| 缓存与业务逻辑混合 | 中 | 模块耦合度高 |
| 微信/企微缓存代码重复 80% | 低 | 维护成本高 |
| workwx 目录重复文件 | 低 | 引用混淆 |
| fetchStateManager 定时器永不停止 | 低 | CPU 空转 |
| 缺少统一缓存抽象 | 中 | 新增缓存无规范 |
