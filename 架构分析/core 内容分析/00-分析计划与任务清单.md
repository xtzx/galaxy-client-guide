## 二、分析规范

### 2.1 文档要求

- **篇幅**：每份文档 800 - 1500 行
- **不展示代码**：通过文字描述即可理解代码执行逻辑
- **务必结合代码**：所有分析基于实际代码，不推测、不省略
- **问题与优化**：分析代码中存在的问题、潜在 Bug、不合理的库选择、可优化的逻辑等
- **功能背景**：说明该功能为什么需要存在，解决了什么业务问题
- **业务场景**：结合实际业务场景，说明功能具体做了哪些事情

### 2.2 每份文档的标准结构

每份分析文档将按以下结构组织：

```

   - 功能定位与作用
   - 为什么需要这个模块（业务背景）
   - 在整体架构中的位置

二、文件职责清单
   - 每个文件的职责说明
   - 文件之间的依赖关系

三、核心数据结构与状态
   - 关键变量、Map、队列等数据结构说明
   - 状态流转

四、核心逻辑详解
   - 初始化流程
   - 主要执行链路（逐步骤拆解）
   - 关键分支与异常处理
   - 与其他模块的交互

五、业务场景映射
   - 具体业务场景下的执行流程
   - 数据流向

六、问题分析与优化建议
   - 现有代码的问题点
   - 潜在的 Bug 或风险
   - 库选择是否合理
   - 性能或架构优化建议

```

## 四、模块依赖关系总览

以下是 8 个模块之间的核心依赖关系，便于理解分析顺序：

```
                    ┌────────────────────┐
                    │  registry-config   │ ← 所有模块的「中央注册表」
                    │  （微信实例注册表） │
                    └────┬───────┬───────┘
                         │       │
              ┌──────────┘       └──────────┐
              ▼                              ▼
     ┌────────────────┐            ┌─────────────────┐
     │     mq/        │            │    reverse/      │
     │  MQTT 通信     │            │  逆向 IPC 通信   │
     │ (服务端→客户端)│            │ (客户端→微信进程) │
     └───────┬────────┘            └────────┬─────────┘
             │                              │
             ▼                              ▼
     ┌────────────────┐            ┌─────────────────┐
     │    queue/       │            │  worker-threads/ │
     │  内存队列       │            │  Worker 线程池   │
     │ (任务缓冲/流控)│            │  (并行任务处理)  │
     └───────┬────────┘            └────────┬─────────┘
             │                              │
             │         ┌──────────┐         │
             └────────►│  cache/  │◄────────┘
                       │ 内存缓存 │
                       └────┬─────┘
                            │
                            ▼
                    ┌────────────────┐
                    │    front/      │
                    │ WebSocket 推送 │
                    │ (→ 渲染进程)   │
                    └────────────────┘

                    ┌────────────────┐
                    │    pool/       │
                    │  对象池管理    │
                    │ (封装 Worker)  │
                    └────────────────┘
```

**数据流向总结**：
1. 服务端通过 **MQTT** 下发任务指令 → 客户端接收并解密
2. 任务可能进入 **内存队列** 进行缓冲和流控
3. 任务执行时通过 **逆向 IPC**（命名管道）与微信进程通信
4. 逆向通信的重 CPU 操作通过 **Worker 线程池** 并行处理
5. 所有微信实例的状态统一由 **注册表** 管理
6. 执行过程中产生的中间数据存入 **内存缓存**
7. 最终结果通过 **WebSocket** 推送到前端渲染进程展示
