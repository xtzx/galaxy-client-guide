## 1. 设计思路概览

### 1.1 为什么需要心跳方案

Electron 客户端在 Windows 上运行时，可能因为以下原因崩溃：

- 主进程 Node.js 未捕获异常（`uncaughtException`）
- 渲染进程崩溃（`render-process-gone`）
- 系统级崩溃（内存溢出、操作系统杀死进程）
- DLL 调用导致的段错误

对于最后两类情况，应用层无法在崩溃前执行任何清理代码。因此项目采用了一种「**心跳 + 启动判断**」的方案：

1. **运行时持续写入心跳**：每 10 秒往 `electron-store` 写入 `activityInfo`，标记 `status=0`（正在运行）
2. **正常退出时标记**：用户主动关闭窗口时，写入 `status=1`（正常退出）
3. **下次启动时判断**：读取上次的 `activityInfo`，如果 `status=0` 且距今超过 30 秒，则判定为崩溃

### 1.2 核心数据流

```
┌─────────────────┐    每10秒      ┌──────────────────┐
│  schedual.js    │ ───────────▶  │ setRecordInfo(0) │
│  (node-cron)    │               │ 写入 store       │
└─────────────────┘               └──────────────────┘
                                           │
                                           ▼
                                  ┌──────────────────┐
                                  │ electron-store   │
                                  │ activityInfo     │
                                  │ {status,time,...}│
                                  └──────────────────┘
                                           │
        ┌──────────────────────────────────┼────────────────────────┐
        │ 正常退出                         │                        │ 下次启动
        ▼                                  │                        ▼
┌──────────────────┐                       │             ┌───────────────────────┐
│ window.js close  │                       │             │ judgeCrashAndReport()  │
│ setRecordInfo(1) │                       │             │ 读取 store            │
└──────────────────┘                       │             │ status=0 & >30s?      │
                                           │             │   → 上报 CRASH        │
                                           │             └───────────────────────┘
```

## 2. 核心文件 recordActivityInfo.js 全量解析

**文件路径**：`galaxy-client/src/common/recordActivityInfo.js`

```javascript
const registryList = require('../msg-center/core/registry-config/registryList').getInstance();
const { clientVersion } = require('./urls');
const store = require('./store');
const {formatTimestamp} = require('./format');
const logUtil = require('../init/log')
const {reportLog} = require('../init/habo');

const judgeCrashAndReport = () => {
    const lastRecord = store.getActivityInfo() || {};
    // 上次是正常记录 & 30s内没有新的记录 & 先后客户端版本一致
    if (lastRecord.status === 0
        && Date.now() - lastRecord.time > 30000
        && lastRecord.clientVersion === clientVersion) {
        // 上报客户端崩溃
        logUtil.customLog(`客户端崩溃了: ${JSON.stringify(lastRecord)}`);
        reportLog({
            name: 'CRASH',
            ...lastRecord
        });
    }
}

// status 0：定时器记录；1：正常退出;
const setRecordInfo = status => {
    const memoryInfo = global.memoryInfo || {};
    const lastRecord = store.getActivityInfo() || {};
    if (Date.now() - lastRecord.time < 6000 && status === 0) {
        return;
    }
    const registrys = registryList.getRegistryList()
    const wxIdStr = registrys.map(registry => registry?.wxid).join(',');
    const casUserInfo = store.getUserInfo() || {};
    const recordInfo = {
        ...memoryInfo,
        status,
        time: Date.now(),
        timeStr: formatTimestamp(Date.now()),
        appStartTime: global.startTime,
        appStartTimeStr: formatTimestamp(global.startTime),
        casId: casUserInfo.user,
        clientVersion,
        wxIdStr,
    }
    store.setActivityInfo(recordInfo);
}

module.exports = {
    setRecordInfo,
    judgeCrashAndReport
}
```

### 2.1 模块依赖关系

| 依赖 | 来源 | 用途 |
|------|------|------|
| `registryList` | `registry-config/registryList.js` | 获取当前所有微信实例的 wxid |
| `clientVersion` | `common/urls.js` | 客户端版本号 |
| `store` | `common/store.js` | 读写 `activityInfo` 持久化数据 |
| `formatTimestamp` | `common/format.js` | 时间戳格式化为可读字符串 |
| `logUtil` | `init/log.js` | 写入 SLS + electron-log |
| `reportLog` | `init/habo.js` | 上报到 Habo 埋点系统 |

## 3. 心跳写入 setRecordInfo

### 3.1 函数签名

```javascript
const setRecordInfo = status => { ... }
```

| 参数 | 类型 | 含义 |
|------|------|------|
| `status` | `number` | `0` = 定时器记录（运行中）；`1` = 正常退出 |

### 3.2 6 秒节流保护

```javascript
if (Date.now() - lastRecord.time < 6000 && status === 0) {
    return;
}
```

当 `status=0`（定时心跳）时，如果距上次写入不足 6 秒，直接跳过。这是一个节流机制，避免 `electron-store` 频繁写文件影响磁盘 I/O。

> **注意**：当 `status=1`（正常退出）时，不受此限制，确保退出标记一定被写入。

### 3.3 写入的数据结构

```javascript
const recordInfo = {
    ...memoryInfo,           // 从 global.memoryInfo 扩展（CPU/内存等系统信息）
    status,                  // 0 或 1
    time: Date.now(),        // 当前时间戳
    timeStr: formatTimestamp(Date.now()),  // 可读时间
    appStartTime: global.startTime,       // 应用启动时间戳
    appStartTimeStr: formatTimestamp(global.startTime),
    casId: casUserInfo.user, // CAS 登录用户 ID
    clientVersion,           // 客户端版本号
    wxIdStr,                 // 所有连接的微信 wxid，逗号分隔
}
```

### 3.4 activityInfo 在 store.js 中的实现

**文件路径**：`galaxy-client/src/common/store.js`

```javascript
getActivityInfo() {
    return store.get('activityInfo');
},
setActivityInfo(activityInfo) {
    store.set('activityInfo', activityInfo);
},
```

数据最终持久化到 `electron-store` 管理的 JSON 文件中（位于 `app.getPath('userData')` 目录）。

### 3.5 global.memoryInfo 的来源

`global.memoryInfo` 由 `processUsageReport.js` 中的 `ProcessReport.reportAll()` 方法定期写入：

```javascript
// processUsageReport.js 中
global.memoryInfo = memoryInfo;
```

写入内容包括：`systemMemoryFree`、`systemMemoryTotal`、`systemCpuUsage`、`appUsedCpuTotal`、`appUsedMemoryTotal` 等系统资源信息。

## 4. 启动时崩溃判断 judgeCrashAndReport

### 4.1 判断逻辑

```javascript
const judgeCrashAndReport = () => {
    const lastRecord = store.getActivityInfo() || {};
    if (lastRecord.status === 0
        && Date.now() - lastRecord.time > 30000
        && lastRecord.clientVersion === clientVersion) {
        logUtil.customLog(`客户端崩溃了: ${JSON.stringify(lastRecord)}`);
        reportLog({
            name: 'CRASH',
            ...lastRecord
        });
    }
}
```

三个条件必须同时满足：

| # | 条件 | 含义 |
|---|------|------|
| 1 | `lastRecord.status === 0` | 上次记录是定时心跳（非正常退出） |
| 2 | `Date.now() - lastRecord.time > 30000` | 距上次心跳已超过 30 秒 |
| 3 | `lastRecord.clientVersion === clientVersion` | 版本号一致（排除升级导致的重启） |

### 4.2 为什么需要版本号校验

如果客户端进行了版本更新，旧版本的 `activityInfo` 记录的版本号与新版本不同。此时 `status=0` 并不意味着崩溃，而是因为更新导致的正常重启。通过版本号比对可以避免误报。

### 4.3 调用时机

在 `appStart.js` 中，应用启动后立即调用：

**文件路径**：`galaxy-client/src/msg-center/start/appStart.js`

```javascript
const AppMain = {
    run() {
        global.startTime = Date.now();
        FrontStart.run();
        ReverseStart.run();
        scheduleRun();
        MemoryQueueApplication.run();
        judgeCrashAndReport();  // ← 启动时判断上次是否崩溃
    }
}
```

## 5. 定时任务中的心跳触发

**文件路径**：`galaxy-client/src/msg-center/start/schedual.js`

```javascript
const cron = require('node-cron');
const { setRecordInfo } = require('../../common/recordActivityInfo');

function scheduleRun() {
    cron.schedule('*/10 * * * * *', () => {
        HeartTimer.heartBeat();
        delaySendFrontmsgTimer.start();
        PingTimer.ping();
        ProcessMakeUpTaskTimer.taskMakeUpTask();
        GalaxyTaskStatusTimer.checkGalaxyTaskStatus();

        // 记录当前活动信息
        setRecordInfo(0);  // ← 每 10 秒写入心跳
    });
}
```

### 5.1 心跳频率分析

- **cron 触发频率**：每 10 秒
- **节流保护**：6 秒内不重复写入
- **实际写入频率**：约每 10 秒一次

### 5.2 心跳与崩溃检测的时间窗口

| 场景 | 时间线 |
|------|--------|
| 正常运行 | 每 10 秒更新 `time` 字段 |
| 崩溃发生 | 心跳停止写入 |
| 下次启动 | `Date.now() - lastRecord.time > 30000` |
| **最短检测延迟** | 30 秒（崩溃刚好发生在心跳写入之后） |
| **最长检测延迟** | 40 秒（崩溃发生在下一次心跳写入之前） |

## 6. 正常退出时的状态标记

**文件路径**：`galaxy-client/src/init/window.js`

```javascript
mainWindow.on('close', e => {
    e.preventDefault();
    if (updateFlag) {
        // 更新导致的关闭
        setTimeout(() => {
            app.mainWindow = null;
            app.quit();
        }, 1000);
    }
    else {
        dialog
            .showMessageBox(app.mainWindow, {
                type: 'warning',
                title: '关闭',
                message: '是否确认关闭？',
                buttons: ['取消', '确定'],
            })
            .then(index => {
                if (index.response === 1) {
                    store.clearAutoCompleteInfo();
                    sendToRenderMsg('ws-kill-java-only');
                    sendToRenderMsg('logout-notify');
                    logUtil.customLog('AppQuitConfirm')
                    setRecordInfo(1);  // ← 正常退出标记
                    setTimeout(() => {
                        app.mainWindow = null;
                        app.quit();
                    }, 2000);
                }
            });
    }
});
```

### 6.1 退出流程时序

1. 用户点击关闭按钮 → 触发 `close` 事件
2. `event.preventDefault()` 阻止默认关闭
3. 弹出确认对话框
4. 用户点击「确定」
5. 清理自动填充信息
6. 通知渲染进程关闭 WebSocket
7. **`setRecordInfo(1)` 写入正常退出标记**
8. 延迟 2 秒后执行 `app.quit()`

### 6.2 更新导致的关闭

当 `updateFlag = true` 时（自动更新触发），直接退出不写入 `status=1`。此时下次启动是新版本，版本号校验会排除崩溃误判。

## 10. 进程异常捕获 monitor.js

**文件路径**：`galaxy-client/src/common/monitor.js`

### 10.1 主进程未捕获异常

```javascript
function processErrorMonitor() {
    process.on('uncaughtException', error => {
        crashOrErrorReport('UncaughtException', null, error);
    });
    process.on('unhandledRejection', (reason, promise) => {
        crashOrErrorReport('UnhandledRejection', null, reason);
    });

```

### 10.2 异常上报处理

```javascript
async function crashOrErrorReport(name, event, error) {
    if (error) {
        logUtil.customLog(
            `[${name}], ${error.name} ${error.message} ${error.stack}`,
            { level: 'error', errorKey: 'UncaughtException' }
        );
        reportLog({
            name: 'UNCAUGHT_EXCEPTION',
            errorName: name,
            errorMessage: error.message,
            errorStack: error.stack,
            errorStr: JSON.stringify(error),
        })
    } else if (event) {
        logUtil.customLog(
            `[${name}]`,
            { level: 'error', errorKey: 'RendererProcessCrashed' }
        );
    }
}
```

### 10.3 Sentry 的历史（已弃用）

代码中可以看到大段被注释的 Sentry 集成代码（`@sentry/electron`），说明项目曾经使用 Sentry 做异常监控，后来切换到了 Habo + SLS 的方案。

## 12. 资源使用上报 processUsageReport.js

**文件路径**：`galaxy-client/src/common/processUsageReport.js`

`ProcessReport` 类负责收集和上报系统资源使用情况：

### 12.1 采集的指标

| 类别 | 采集方式 | 指标 |
|------|----------|------|
| 系统整体 | `systeminformation` | CPU 使用率、总内存、空闲内存 |
| 主进程 | `app.getAppMetrics()` | Browser/GPU/Utility/Tab 各进程的 CPU 和内存 |
| 微信/企微进程 | `pidusage` + `GetSubProcPhysicalMmSize` | 各微信实例的 CPU 和内存 |

### 12.2 超限告警

```javascript
if ((memInfo.used / memInfo.total) * 100 > 99 || cpuInfo.currentLoad > 99) {
    notify.onMemoryLimit(registryList, cpuPercent, memoryPercent);
}
```

当系统内存使用率或 CPU 使用率超过 99% 时触发灵犀告警。

### 12.3 与心跳的关联

```javascript
global.memoryInfo = memoryInfo;
```

资源信息写入 `global.memoryInfo` 后，被 `setRecordInfo` 扩展到 `activityInfo` 中。崩溃时这些数据会随 `CRASH` 事件一起上报，帮助定位崩溃是否由资源耗尽导致。

## 14. 完整流程时序图

```
时间线 ───────────────────────────────────────────────────▶

[启动阶段]
  │
  ├── electron.js 入口
  │     └── initLog() → 日志初始化
  │
  ├── AppStart.run()
  │     ├── FrontStart.run()
  │     ├── ReverseStart.run()

  │     ├── MemoryQueueApplication.run()
  │     └── judgeCrashAndReport()   ← 检查上次是否崩溃
  │           │
  │           ├── [status=0 & >30s & 版本一致]
  │           │     ├── logUtil.customLog('客户端崩溃了')
  │           │     └── reportLog({ name: 'CRASH', ... })
  │           │           ├── eventReport() → Habo
  │           │           └── notify.crash() → 灵犀告警
  │           │
  │           └── [否则：正常启动，不上报]
  │
[运行阶段]
  │
  ├── 每10秒: schedual.js cron
  │     ├── HeartBeatTimer.heartBeat()
  │     ├── PingTimer.ping()
  │     ├── ...其他定时任务
  │     └── setRecordInfo(0)        ← 写入心跳
  │           └── store.setActivityInfo({ status:0, time:now, ... })
  │
  ├── ProcessReport.reportAll()     ← 定期采集资源
  │     └── global.memoryInfo = {...}
  │
[退出阶段]
  │
  ├── 用户点击关闭 → dialog确认
  │     ├── setRecordInfo(1)        ← 标记正常退出
  │     └── app.quit()
  │
  └── 崩溃（无法执行任何代码）
        └── activityInfo 停留在 status=0
              → 下次启动被 judgeCrashAndReport 检测到
```

## 15. 局限性分析与改进方向

### 15.1 当前方案的局限

| 局限 | 说明 |
|------|------|
| 30 秒检测窗口 | 崩溃必须等到下次启动才能检测，且判断阈值为 30 秒 |
| 更新场景需要版本校验 | `updateFlag=true` 时不写 `status=1`，依赖版本号比对排除 |
| 无法区分崩溃类型 | 系统 kill、OOM、DLL 段错误等都表现为相同的 `status=0` |
| 单次写入无冗余 | `electron-store` 写入是单文件操作，写入过程中崩溃可能导致数据损坏 |
| 无实时性 | 不像 Sentry 能在崩溃时立即发送（minidump），必须等下次启动 |

### 15.2 与 Electron crashReporter 的关系

项目**未使用** Electron 内置的 `crashReporter` 模块。`crashReporter` 可以在原生层面捕获崩溃并生成 minidump，但需要配套的收集服务器。当前项目选择了更轻量的心跳方案。
