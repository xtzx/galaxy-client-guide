# 12 — 内存队列执行器设计（MemoryQueue）

> **文档定位**：深入分析 `MemoryQueueExecute.js` 的串行队列设计，包括并发控制（CAS）、超时机制、背压策略，  
> 以及附属的 `mqTask.js` 延迟队列。  
> 理解这两个队列对于分析消息处理的流控和执行顺序至关重要。

---

## 1. 设计背景

### 1.1 为什么需要内存队列

在 galaxy-client 的消息处理流程中，某些操作需要**严格串行化**执行：

- **企微极速群发**：多条群发任务需要按顺序逐条执行，不能并发（否则会触发企微反作弊机制）
- **MQTT 消息入站处理**：来自云端的指令需要排队执行，防止逆向 IPC 同时发送多条指令导致 Named Pipe 阻塞

### 1.2 为什么不用现有方案

| 方案 | 不适用的原因 |
|------|------------|
| `stream.Transform` | 面向流数据，不适合任务调度 |
| `p-queue` | 第三方依赖，项目倾向自实现 |
| `Bull`/`BeeQueue` | 依赖 Redis，Electron 桌面应用不需要分布式队列 |
| 直接 `async/await` 串行 | 无法从外部动态添加任务 |

最终选择**自实现轻量内存队列**，满足以下需求：
- 动态入队
- 串行消费
- 超时保护
- 背压丢弃
- 简单 CAS 并发保护

## 3. MemoryQueueExecute 完整解析

**文件**：`galaxy-client/src/msg-center/core/queue/MemoryQueueExecute.js`

### 3.1 类定义与构造

```javascript
class MemoryQueueExecute {
    constructor(timeOut, sleepTime, taskSize) {
        this.queue = [];           // 任务队列
        this.isInterrupted = false;// 停止标志
        this.beforeTime = Date.now();
        this.number = 0;           // 已处理任务计数
        this.timeOut = timeOut;    // 单任务超时时间（ms）
        this.sleepTime = sleepTime;// 轮询间隔（ms）
        this.taskSize = taskSize;  // 最大队列容量
        this.compareAndSet = 0;    // CAS 标志（0=空闲, 1=执行中）
    }
}
```

### 3.2 put() — 入队

```javascript
put(workTask) {
    logUtil.customLog(`[MemoryQueueExecute] put workTask：${JSON.stringify(workTask)}`);
    if (this.queue.length >= this.taskSize) {
        logUtil.customLog('[MemoryQueueExecute] queue is full');
        return false;
    }
    this.queue.push(workTask);
    return true;
}
```

**背压策略**：队列满（≥ 30）时直接返回 `false`，新任务被丢弃。

| 情况 | 处理 | 返回值 |
|------|------|--------|
| 队列未满 | 入队成功 | `true` |
| 队列已满 | 丢弃任务 | `false` |

注意：被丢弃的任务不会重试或进入补偿队列。调用方需要根据返回值决定是否采取其他策略。

### 3.3 start() — 异步消费循环

```javascript
async start() {
    this.beforeTime = Date.now();
    while(!this.isInterrupted) {
        let workTaskHandler = this.queue[0];
        
        // 条件1：队列有任务 + CAS=0（空闲）
        if (workTaskHandler && this.compareAndSet === 0) {
            this.compareAndSet = 1;        // 标记为执行中
            workTaskHandler = this.queue.shift();  // 出队
            if (workTaskHandler) {
                this.beforeTime = Date.now();
                this.number++;
                workTaskHandler.handler((err) => {
                    if (err) {
                        logUtil.customLog(`WorkTaskHandler invoke error`);
                    } else {
                        logUtil.customLog('WorkTaskHandler invoke success');
                    }
                });
            }
        }
        
        await sleep(this.sleepTime);  // 等待 30 秒
        
        // 条件2：超时保护 — 上一个任务执行超过 timeOut
        if (workTaskHandler && Date.now() - this.beforeTime > this.timeOut) {
            workTaskHandler = this.queue.shift();  // 强制取下一个任务
            if (workTaskHandler) {
                this.beforeTime = Date.now();
                this.number++;
                workTaskHandler.handler((err) => { ... });
            }
        }
        
        if (!workTaskHandler) {
            logUtil.customLog('[MemoryQueueExecute] no task sleep');
        }
    }
}
```

### 3.4 执行流程图

```
start() 进入无限循环
    │
    ▼
┌─────────────────────────┐
│ queue[0] 有任务？         │
│ compareAndSet === 0？    │
├───────────┬─────────────┤
│ 是        │ 否           │
▼           ▼             │
设置 CAS=1  ┌────────┐    │
出队执行     │ 跳过   │    │
            └────────┘    │
    │                     │
    ▼                     │

    │                     │
    ▼                     │
┌─────────────────────────┐
│ 超时判定                  │
│ Date.now() - beforeTime  │
│ > timeOut (180s)?        │
├───────────┬─────────────┤
│ 是        │ 否           │
▼           ▼             │

    ▼                     │

```

### 3.5 reset() — 重置 CAS 标志

```javascript
reset() {
    this.compareAndSet = 0;
}
```

**关键机制**：`compareAndSet` 被设为 1 后，需要外部调用 `reset()` 才能恢复为 0。这意味着：

1. 任务执行后，**不会自动重置 CAS**
2. 必须由任务完成的回调或外部逻辑调用 `reset()`
3. 如果 `reset()` 不被调用，队列会在下次 `sleep` 后通过**超时保护**机制强制取下一个任务

当前调用 `reset()` 的位置：

| 文件 | 场景 |
|------|------|
| `workWxQuickMsgRecordService.js` | 收到极速群发的第二阶段消息时调用 `reset()` |

### 3.6 stop() — 停止消费

```javascript
stop() {
    this.isInterrupted = true;
}
```

设置中断标志，消费循环在下一次 `while` 判断时退出。

### 3.7 CAS 并发保护分析

在 Node.js 单线程环境下，传统意义上不存在多线程竞争，但这里的 CAS 机制有其特定用途：

```javascript
this.compareAndSet = 0;  // 空闲状态

// 执行任务时
if (workTaskHandler && this.compareAndSet === 0) {
    this.compareAndSet = 1;  // 标记执行中
    // ... 执行任务
}
```

**实际保护的场景**：

- `handler()` 是一个**回调式**异步操作，不返回 Promise
- 在 `handler()` 执行期间（可能跨越多个事件循环），`start()` 的 `while` 循环会继续运行
- `compareAndSet === 1` 防止在 handler 未完成时取出下一个任务执行

这是一种**协作式并发控制**，而非真正的原子 CAS 操作。

### 3.8 超时保护详解

```javascript
if (workTaskHandler && Date.now() - this.beforeTime > this.timeOut) {
    workTaskHandler = this.queue.shift();
    // 执行下一个任务（不等待上一个完成）
}
```

**设计意图**：

- 如果上一个任务执行时间超过 `timeOut`（180 秒），强制跳过等待
- 直接取下一个任务执行
- 注意：**上一个任务并没有被取消**，只是不再等待它完成
- 这可能导致短暂的"并行执行"（上一个任务还在运行，下一个已经开始）

## 4. MemoryQueueApplication — 单例管理器

**文件**：`galaxy-client/src/msg-center/core/queue/MemoryQueueApplication.js`

```javascript
const ApplicationConfigEnv = require('../../core/application-config/index');
const MemoryQueueExecute = require('./MemoryQueueExecute');

const memoryExecuteTimeOut = ApplicationConfigEnv['memory.queue.time.out'];
const memoryExecuteSleep = ApplicationConfigEnv['memory.queue.time.sleep'];
const taskSize = ApplicationConfigEnv['memory.queue.task.size'];

const MemoryQueueApplication = {
    run() {
        MemoryQueueExecute.init(memoryExecuteTimeOut, memoryExecuteSleep, taskSize).start();
    },
    destroy() {
        MemoryQueueExecute.getInstance().stop();
    }
};
```

### 4.1 配置参数

| 参数 | Key | 值 | 说明 |
|------|-----|---|------|
| 任务超时 | `memory.queue.time.out` | 180000ms (3min) | 单个任务最大执行时间 |
| 轮询间隔 | `memory.queue.time.sleep` | 30000ms (30s) | 队列消费循环的 sleep 时间 |
| 队列容量 | `memory.queue.task.size` | 30 | 最大任务数 |

三个环境（dev/qa/prod）使用相同的值。

### 4.2 单例模式

```javascript
const MemoryQueue = {
    singletonInstance: null,
    init(timeOut, sleepTime, taskSize) {
        if (!this.singletonInstance) {
            this.singletonInstance = new MemoryQueueExecute(timeOut, sleepTime, taskSize);
        }
        return this.singletonInstance;
    },
    getInstance() {
        if (!this.singletonInstance) {
            throw new Error("[MemoryQueueExecute] has not init");
        }
        return this.singletonInstance;
    },
};
```

- `init()` 保证只创建一次实例
- `getInstance()` 在未初始化时抛异常
- 全局唯一的消费循环

### 4.3 启动时机

## 6. mqTask.js — 延迟任务队列

**文件**：`galaxy-client/src/msg-center/core/queue/mqTask.js`

与 `MemoryQueueExecute` 不同，`mqTask.js` 实现了一个**按 wxId 隔离、带最小执行间隔**的任务队列。

### 6.1 完整实现

```javascript
const apolloConfig = require('../utils/getApolloConfig');

class TaskQueue {
    constructor() {
        this.queues = new Map();            // wxId → task[]
        this.lastExecutionTimes = new Map(); // wxId → timestamp
    }

    addTask(wxId, task) {
        if (!this.queues.has(wxId)) {
            this.queues.set(wxId, []);
        }
        const queue = this.queues.get(wxId);
        queue.push(task);
        if (queue.length === 1) {
            this.runNextTask(wxId);  // 队列为空时立即启动消费
        }
    }

    async runNextTask(wxId) {
        const queue = this.queues.get(wxId);
        if (!queue || queue.length === 0) return;

        // 最小间隔控制
        const now = Date.now();
        const lastExecutionTime = this.lastExecutionTimes.get(wxId) || 0;
        const timeSinceLastExecution = now - lastExecutionTime;
        if (timeSinceLastExecution < apolloConfig.taskDefaultWaitTime) {
            await new Promise(resolve => 
                setTimeout(resolve, apolloConfig.taskDefaultWaitTime - timeSinceLastExecution)
            );
        }

        const task = queue.shift();
        await task();

        this.lastExecutionTimes.set(wxId, Date.now());

        if (queue.length > 0) {
            this.runNextTask(wxId);  // 递归处理下一个
        } else {
            this.queues.delete(wxId);  // 清空
        }
    }
}

function executeWithDelay(wxId, task) {
    taskQueue.addTask(wxId, task);
}
module.exports = executeWithDelay;
```

### 6.2 与 MemoryQueueExecute 的对比

| 维度 | MemoryQueueExecute | mqTask (TaskQueue) |
|------|--------------------|--------------------|
| 队列数量 | 全局单一队列 | 每个 wxId 独立队列 |
| 并发控制 | CAS 标志 | 递归串行 |
| 执行间隔 | 固定 sleepTime | 动态 `apolloConfig.taskDefaultWaitTime` |
| 超时保护 | 有（timeOut） | 无 |
| 背压策略 | 丢弃（返回 false） | 无限入队 |
| 任务格式 | `{ handler(cb) }` | `async function` |
| 使用场景 | 极速群发 | 一般消息入站处理 |

### 6.3 间隔控制

`apolloConfig.taskDefaultWaitTime` 由 Apollo 动态配置，默认值 200ms。

```
任务A执行完毕 → 距离上次执行 < 200ms？
    ├─ 是 → 等待剩余时间
    └─ 否 → 立即执行下一个
```

这保证了每个微信实例的任务执行间隔至少为 200ms，避免发送频率过高。

## 7. 队列深度与延迟分析

### 7.1 MemoryQueueExecute 延迟模型

最坏情况下的消息处理延迟：

```
队列满载时（30 个任务）：
- 每个任务的最大执行时间 = min(实际执行时间, timeOut=180s)
- 轮询间隔 = sleepTime = 30s
- 最大延迟 ≈ 30 × (180 + 30) = 6300s ≈ 105分钟
```

实际情况通常远好于此：
- 大多数任务在几秒内完成
- 队列深度通常很浅（1-3 个任务）
- 极速群发场景下，通过 `reset()` 主动驱动消费

### 7.2 mqTask 延迟模型

```
N 个任务排队，每个任务执行时间 T：
- 每两个任务之间至少 200ms 间隔
- 第 K 个任务的延迟 = K × max(T, 200ms)
```

由于 `taskDefaultWaitTime` 可通过 Apollo 动态调整，运维可以在紧急情况下调低间隔加速处理。

## 8. 并发安全性分析

### 8.1 Node.js 单线程模型下的"伪并发"

Node.js 是单线程事件循环，不存在真正的多线程竞争。但以下场景仍需要保护：

1. **异步回调交叉**：`handler()` 内部的异步操作（如网络请求）可能跨越多个事件循环 tick
2. **多处 `put()`**：多个消息处理流程同时调用 `put()`，Array.push 在单线程下是安全的
3. **`reset()` 时机**：如果 `reset()` 在 `start()` 的 `sleep` 期间被调用，下一次循环会立即消费新任务

### 8.2 潜在问题

**问题 1：任务执行后 CAS 不自动重置**

```javascript
workTaskHandler.handler((err) => {
    // callback 被调用，但 compareAndSet 仍然是 1
    // 需要外部调用 reset()
});
```

如果外部忘记调用 `reset()`，队列会依赖超时机制（180 秒后）才能继续消费。

**问题 2：超时后的并行执行**

超时强制取下一个任务时，上一个任务可能仍在异步执行，导致短暂的并行。对于需要严格串行的场景（如极速群发），这可能是一个潜在风险。

### 9.1 日志记录

| 事件 | 日志内容 |
|------|---------|
| 入队 | `[MemoryQueueExecute] put workTask：{...}` |

| CAS 重置 | `[MemoryQueueExecute] reset：{value}` |
| 空闲等待 | `[MemoryQueueExecute] no task sleep` |
| 任务计数 | `[MemoryQueueExecute] current number = {N}` |

### 9.2 队列满的风险

当日志中频繁出现 `queue is full`，说明：
- 任务产生速度 > 消费速度
- 可能需要增加 `taskSize` 或减少 `sleepTime`
- 或优化 `handler` 的执行效率
