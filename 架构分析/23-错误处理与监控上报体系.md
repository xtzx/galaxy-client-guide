## 1. 监控上报体系全景

Galaxy 项目的监控体系分布在两个仓库中，使用不同的技术栈：

```
┌──────────────────────────────────────────────────────────────┐

│                                                              │

│  │       ▼         │  │      ▼       │  │      ▼         │  │
│  │ electron-log    │  │   Habo HTTP  │  │  Ali SLS       │  │
│  │ + SLS + Habo    │  │   + 灵犀告警  │  │                │  │

│                                                              │
│  Sentry: ❌ 已弃用（代码注释保留）                              │

┌──────────────────────────────────────────────────────────────┐
```

| 维度 | galaxy-client 主进程 | galaxy 前端渲染进程 |

## 2. 主进程未捕获异常处理（galaxy-client）

**文件路径**：`galaxy-client/src/common/monitor.js`

### 2.1 全局异常监听

```javascript
function processErrorMonitor() {
    process.on('uncaughtException', error => {
        crashOrErrorReport('UncaughtException', null, error);
    });
    process.on('unhandledRejection', (reason, promise) => {
        crashOrErrorReport('UnhandledRejection', null, reason);
    });
}
```

| 事件 | 触发条件 | 严重程度 |
|------|----------|----------|
| `uncaughtException` | 同步代码中未捕获的 throw | 最高（可能导致进程不稳定） |
| `unhandledRejection` | Promise reject 未被 catch | 高（异步错误） |

### 2.2 异常上报 crashOrErrorReport

```javascript
async function crashOrErrorReport(name, event, error) {
    if (error) {
        logUtil.customLog(
            `[${name}], ${error.name} ${error.message} ${error.stack}`,
            { level: 'error', errorKey: 'UncaughtException' }
        );
        reportLog({
            name: 'UNCAUGHT_EXCEPTION',
            errorName: name,
            errorMessage: error.message,
            errorStack: error.stack,
            errorStr: JSON.stringify(error),
        });
    } else if (event) {
        logUtil.customLog(`[${name}]`, {
            level: 'error',
            errorKey: 'RendererProcessCrashed'
        });
    }
}
```

异常上报包含完整的错误链：

| 字段 | 说明 |
|------|------|
| `errorName` | 错误分类（UncaughtException / UnhandledRejection） |
| `errorMessage` | 错误消息 |
| `errorStack` | 调用栈 |
| `errorStr` | 完整错误对象的 JSON 序列化 |

### 2.3 上报路径

```
uncaughtException
    │
    ├──▶ logUtil.customLog() ──▶ electron-log (本地 error 日志)
    │                          ──▶ SLS (阿里云远程日志)
    │
    └──▶ reportLog() ──▶ Habo HTTP 上报
                       ──▶ 灵犀告警（仅 CRASH/EXEC_ERROR 等）
```

### 2.4 Sentry 的弃用

`monitor.js` 中保留了大段被注释的 `@sentry/electron` 代码，说明 galaxy-client 曾使用 Sentry，现已切换到 Habo + SLS 方案。

## 4. 前端 React ErrorBoundary

**文件路径**：`galaxy/src/entries/menu/component/ErrorBoundary/index.js`

```javascript
class ErrorBoundary extends React.Component {
    constructor(props) {
        super(props);
        this.state = { eventId: null, hasError: false, error: null };
    }

    componentDidCatch(error, errorInfo) {
        const traceId = randomString(24);
        let errorMsg = error.message;
        if (errorMsg && typeof errorMsg === 'string' && error.message.length > 200) {
            errorMsg = error.message.substring(0, 200);
        }
        notification['error']({
            message: '抱歉...出现了点小故障，可以截图发给666服务台哦',
            duration: 20,
            placement: 'bottomRight',
            description: `${errorMsg}，traceId:${traceId}`
        });
        console.report('ComponentDidCatch', error.message, { error, errorInfo, traceId });
    }

    render() {
        if (this.state.hasError) {
            return ( /* 错误降级 UI */ );
        }
        return this.props.children;
    }
}
```

### 4.1 ErrorBoundary 的作用

| 功能 | 实现方式 |
|------|----------|
| 捕获渲染错误 | `componentDidCatch(error, errorInfo)` |
| 用户提示 | antd `notification` 组件，显示错误摘要和 traceId |
| 上报 Sentry | `console.report()` —— 由 `sentry.js` 注入的全局函数 |
| 降级渲染 | 当 `hasError = true` 时显示错误详情页面 |

## 5. Sentry 异常监控（galaxy 前端）

**文件路径**：`galaxy/src/common/sentry.js`

### 5.1 初始化配置

```javascript
import * as Sentry from '@sentry/browser';
import { CaptureConsole } from '@sentry/integrations';

function init({ enabled, getContext }) {
    Sentry.init({
        release: window.__WEB_COMMIT_ID__,
        enabled,
        ignoreErrors: ['ResizeObserver loop limit exceeded'],
        dsn: urlConfig.sentryUrl,
        environment: urlConfig.type,
        integrations: [
            new CaptureConsole({ levels: ['info', 'error'] }),
        ],
        beforeSend(event, hint) {
            if (typeof getContext === 'function') {
                setContext(event, getContext());
            }
            return event;
        },
        beforeBreadcrumb(breadcrumb) {
            if (['console', 'fetch'].includes(breadcrumb.category)) {
                return null;
            }
            return breadcrumb;
        },
    });
}
```

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `release` | `window.__WEB_COMMIT_ID__` | Git Commit ID 作为版本号 |
| `ignoreErrors` | `['ResizeObserver loop...']` | 忽略浏览器无害错误 |
| `environment` | `urlConfig.type` | `prod` / `test` / `vt` |
| `CaptureConsole` | `levels: ['info', 'error']` | 捕获 console.info 和 error |
| `beforeBreadcrumb` | 过滤 console/fetch | 减少面包屑噪音 |

### 5.2 console.report 全局函数

```javascript
console.report = function (type, msg, data) {
    if (enabled) {
        if (data) {
            Sentry.withScope(function (scope) {
                scope.setExtra('data', data);
                if (data.traceId) {
                    scope.setTag('traceId', data.traceId);
                }
                Sentry.captureMessage(`${type} ${msg}`);
            });
        } else {
            Sentry.captureMessage(`${type} ${msg}`);
        }
    }
};
```

`console.report` 被注入到全局 `console` 对象上，供 ErrorBoundary 和其他组件使用。

### 5.3 console.sentry — HTTP 错误上报

```javascript
console.sentry = function (originReportParams) {
    Sentry.captureEvent(setHttpContext(originReportParams));
};
```

`console.sentry` 用于结构化上报 HTTP 请求错误，包含请求信息、错误级别、用户信息等。

### 5.4 上报分类

| 函数 | 用途 | 调用场景 |
|------|------|----------|
| `console.report(type, msg, data)` | 通用消息上报 | ErrorBoundary、业务逻辑 |
| `console.sentry(params)` | HTTP 错误上报 | ajax.js 拦截器 |
| `Sentry.captureException(error)` | 异常对象上报 | 直接捕获的异常 |

## 6. Sentry 主进程监控（galaxy 旧版）

**文件路径**：`galaxy/electron/common/monitor.js`

```javascript
const Sentry = require('@sentry/electron');

function initSentry({ enabled, getContext }) {
    const commitId = git.short();
    Sentry.init({
        release: commitId,
        enabled,
        dsn: sentryUrl,
        environment: type,
        beforeSend(event) { ... },
        beforeBreadcrumb(breadcrumb) { ... },
    });
}

function processErrorMonitor() {
    process.on('uncaughtException', error => {
        crashOrErrorReport('UncaughtException', null, error);
    });
}

async function crashOrErrorReport(name, event, error) {
    if (error) {
        log.error(name, error);
    } else if (event) {
        // 渲染进程崩溃：弹窗询问是否刷新
        if (event.sender) {
            const { response } = await dialog.showMessageBox({
                message: '客户端已经失去响应',
                title: '是否刷新？',
                buttons: ['确定', '取消'],
            });
            if (response === 0) {
                event.sender.reload();
            }
        }
    }
    // 通过 IPC 获取 wxid 并发送 crash-report 到前端
    utils.sendToRenderMsg('global-wxid');
}
```

galaxy 旧版主进程使用 `@sentry/electron`，且渲染进程崩溃时会弹出对话框询问用户是否刷新。而 galaxy-client 已弃用 Sentry，改用 Habo + SLS。

## 9. 灵犀机器人告警

**文件路径**：`galaxy-client/src/common/notify.js`

### 9.1 条件触发

```javascript
const snedNotifyReport = (data) => {
    if (data.name === 'REVERSE_BUG_REPORT') {
        notify.bugreportCrash(data.wxid);
    } else if (data.name === 'EXEC_ERROR') {
        notify.execError();
    } else if (data.name === 'CRASH') {
        notify.crash();
    }
};
```

只有三种严重事件会触发灵犀告警。

### 9.2 所有告警类型

| 方法 | 触发场景 | 来源 |
|------|----------|------|
| `crash()` | 客户端崩溃 | `recordActivityInfo.js` |
| `bugreportCrash(wxid)` | 逆向 BUG 上报 | `dispatchOutBound.js` |
| `execError()` | 执行异常 | `inject.js` |
| `onMemoryLimit()` | CPU/内存超限 | `processUsageReport.js` |
| `onPongTimeout()` | 心跳 pong 超时 | 定时检测 |

```javascript
if ((memInfo.used / memInfo.total) * 100 > 99 || cpuInfo.currentLoad > 99) {
```

### 11.3 数据关联

```javascript

```

资源信息写入 `global.memoryInfo`，被心跳模块 `setRecordInfo` 扩展到 `activityInfo`。崩溃时这些数据随 `CRASH` 事件一起上报，帮助判断崩溃是否由资源耗尽导致。

## 12. 崩溃检测心跳（见文档13）

galaxy-client 通过「心跳写入 + 启动判断」实现崩溃检测。详细设计见文档13。

核心流程：
1. 每 10 秒写入 `activityInfo`（status=0）
2. 正常退出时写入 status=1
3. 下次启动时判断 status=0 且 >30 秒 → 判定崩溃
