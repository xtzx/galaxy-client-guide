# 通信机制总览与架构图

> 本文档是 galaxy-client 项目通信机制系列文档的总览篇，旨在从全局视角描述项目中所有通信机制的设计思路、协作关系和整体架构。

---

## 一、通信机制概述

### 1.1 五大通信机制

galaxy-client 作为一个基于 Electron 的桌面客户端，承担着"前端 UI 层 ↔ 业务逻辑层 ↔ 逆向进程层 ↔ 云端服务层"之间的桥梁角色。为了实现这一多层通信需求，项目采用了五大通信机制：

| 序号 | 通信机制 | 通信双方 | 传输协议 | 传输方向 | 核心用途 |
|------|---------|---------|---------|---------|---------|
| 1 | Electron IPC | 主进程 ↔ 渲染进程 | Electron 内部通道 | 双向 | UI 交互、配置获取、文件操作 |
| 2 | WebSocket | 主进程（服务端） ↔ 前端 Web（客户端） | ws:// | 双向 | 业务数据实时推送与指令接收 |
| 3 | MQTT | 客户端 ↔ 云端 | tcp:// | 双向 | 云端任务下发与结果上报 |
| 4 | 逆向 IPC（Named Pipe） | Electron ↔ 微信/企微进程 | Windows Named Pipe | 双向 | 控制微信/企微、收发消息 |
| 5 | HTTP/REST | 客户端 ↔ 后端 API | https:// / http:// | 请求-响应 | 数据同步、配置获取、鉴权 |

### 1.2 为什么需要五种通信机制

这五种通信机制并非冗余设计，而是各自服务于不同的技术场景和业务需求：

**Electron IPC** 是 Electron 框架原生提供的进程间通信方式。在 Electron 架构中，主进程和渲染进程是隔离的两个进程，只能通过 IPC 通道进行数据交换。这是 Electron 应用的基础通信层，负责 UI 层与本地系统能力之间的交互。

**WebSocket** 的引入有其特殊背景。galaxy-client 的前端 UI 并非直接由渲染进程的本地代码渲染，而是加载了一个远程 Web 应用（通宝系统）。由于远程 Web 页面无法直接使用 Electron IPC（contextBridge 限制），因此在主进程中启动了一个 WebSocket 服务器，让远程 Web 页面通过 WebSocket 与主进程通信。这是架构设计中最巧妙的一环——它将远程 Web 应用与本地 Electron 能力桥接起来。

**MQTT** 用于客户端与云端的异步消息通信。项目使用阿里云消息队列（MQ）作为 MQTT Broker。云端通过 MQTT 向客户端下发任务（如发消息、加好友、改备注等），客户端通过 MQTT 上报任务执行结果、登录状态、心跳等信息。MQTT 的发布/订阅模型非常适合这种"一对一"的任务下发场景。

**逆向 IPC（Named Pipe）** 是项目最核心也最底层的通信机制。galaxy-client 需要与微信或企业微信的进程进行通信，实现消息收发、好友管理、群聊管理等功能。这种通信通过 Windows Named Pipe 实现——Electron 主进程通过 FFI 调用 PipeCore.dll，建立与微信/企微进程的管道连接，以 JSON 格式收发数据。

**HTTP/REST** 作为最传统的通信方式，用于客户端与后端 REST API 之间的数据同步。包括好友列表上报、群成员同步、配置获取、鉴权令牌获取、日志上报等场景。HTTP 采用请求-响应模式，适合一次性的数据查询和同步操作。

### 1.3 通信机制的层次关系

五大通信机制形成了清晰的层次结构：

```
层级 4（最外层）：用户 UI 层
    ↕ [Electron IPC + WebSocket]
层级 3：Electron 主进程（业务逻辑层）
    ↕ [MQTT + HTTP/REST]
层级 2：云端服务层
    
层级 3：Electron 主进程（业务逻辑层）
    ↕ [逆向 IPC / Named Pipe]
层级 1（最底层）：微信/企微进程层
```

主进程是整个通信架构的枢纽——它向上对接 UI 层（通过 Electron IPC 和 WebSocket），向下对接微信/企微进程（通过 Named Pipe），向外对接云端服务（通过 MQTT 和 HTTP）。

## 五、消息调度中心：五大通信的枢纽

### 5.1 调度中心概述

消息调度中心（`msg-center/dispatch-center`）是五大通信机制的汇聚点。它负责：

- 接收来自 WebSocket（前端）的请求，路由到逆向 IPC
- 接收来自 MQTT（云端）的任务，路由到逆向 IPC
- 接收来自逆向 IPC（微信/企微）的响应，分发到 WebSocket（前端）和 MQTT（云端）
- 管理任务状态、消息匹配、并发控制

### 5.2 调度中心模块结构

```
dispatch-center/
│
├── dispatchInBound.js        入站调度（前端/云端 → 逆向）
├── dispatchOutBound.js       出站调度（逆向 → 前端/云端）
│
├── dispatch/
│   ├── frontFlowInBound.js   前端入站：WebSocket 消息路由
│   ├── frontFlowOutBound.js  前端出站：推送到 WebSocket
│   ├── cloudFlowInBound.js   云端入站：MQTT 任务 → 逆向
│   ├── cloudFlowOutBound.js  云端出站：逆向结果 → MQTT
│   └── sendToFront.js        发送到前端的通用封装
│
├── reverseSend.js            逆向发送：通过 Named Pipe 发消息
├── mqttSend.js               MQTT 发送：上报到云端
├── frontSend.js              前端发送：推送到 WebSocket
│
└── handle/
    ├── msgHandleBase.js      消息处理基类（三段式匹配）
    ├── wxMsgHandle.js        微信消息处理器
    ├── workWxMsgHandle.js    企微消息处理器
    └── *ResponseMsgHandler   特定消息响应处理器
```

### 5.3 入站流（Inbound）：请求到达逆向

入站流有两个入口：

**入口 A：前端 WebSocket 请求**
```
前端 wsSend({cmdId: 'forward', body: '...', channelId: 'xxx'})
    → WebSocket connection.on('message')
    → frontFlowInBound
    → frontHandlerMap[cmdId]
    → forwardTask → dispatchInBound → reverseSend
```

**入口 B：云端 MQTT 任务**
```
MQTT on('message')
    → mqttClientBase.execute()
    → task-mqtt/mqttXxxService.operate()
    → cloudFlowInBound → dispatchInBound → reverseSend
```

两条入站路径最终都汇聚到 `dispatchInBound`，再通过 `reverseSend` 发往微信/企微进程。

### 5.4 出站流（Outbound）：逆向结果分发

出站流有一个入口：

```
逆向 Named Pipe → asyncSelectTask.loop()
    → IpcClientRecvMessage()
    → dispatchOutBound
    → WxMsgHandle / WkMsgHandle
    → 三段式消息匹配
    → businessHandler
        ├─ frontFlowOutBound → sendToFront → WebSocket 推送前端
        └─ cloudFlowOutBound → convert-service/response/boost → mqttSend → MQTT 上报云端
```

### 5.5 三段式消息匹配机制

逆向返回的消息往往不是一条完整的结果，而是分多条返回。微信的消息处理采用"三段式"匹配：

- **第一条消息**：`sendmessage` 类型，表示微信已接收指令
- **第二条消息**：`recvmsg` / `msgreport` 类型，表示消息已发送或收到回执
- **第三条消息**：`MM.*` 等类型，表示最终操作结果（如 CDN 上传完成）

调度中心通过 `MsgResNode` 缓存这些消息片段，使用 `msgResNodeMap` 以 `taskId-type` 为 key 进行匹配。当三条消息都收齐后，才触发最终的 `businessHandler` 进行结果处理。

这种设计是因为微信底层的操作是异步的——发送一条图片消息可能先返回"已收到指令"，再返回"已上传 CDN"，最后返回"对方已收到"。三段式机制确保了对完整消息链路的追踪。

## 六、注册表（Registry）：多账号管理的核心

### 6.1 Registry 的角色

在 galaxy-client 中，一台电脑可能同时管理多个微信/企微账号。每个账号在系统中对应一个 Registry 条目，存储该账号的所有连接信息：

- **pipeCode**：与该微信进程的 Named Pipe 连接码
- **mqttClient**：该账号对应的 MQTT 客户端实例
- **wxInfo**：微信号信息（wxId、昵称、头像等）
- **channelId**：前端标识该账号的通道 ID
- **workWx**：是否为企业微信
- **lastPongTime**：最后一次心跳时间

### 6.2 Registry 在通信中的作用

```
Registry 条目 = {
    key: wxId,                    ← 微信 ID 作为唯一标识
    pipeCode: 12345,              ← 用于 Named Pipe 通信
    mqttClient: MqttClient,       ← 用于 MQTT 通信
    channelId: "channel_xxx",     ← 用于 WebSocket 通信
    processId: 67890,             ← 微信进程 PID
    workWx: false,                ← 账号类型
    ipcClientStatus: "AVAILABLE", ← 连接状态
    cloudSendStatus: "SENT",      ← 云端同步状态
    lastPongTime: 1710000000000,  ← 心跳时间戳
    wxInfo: { ... }               ← 账号详情
}
```

当一条消息需要从前端发送到微信时，调度中心通过 `channelId` 或 `wxId` 在 Registry 中查找对应条目，然后使用 `pipeCode` 通过 Named Pipe 发送。当逆向返回结果需要上报云端时，从 Registry 中取出 `mqttClient` 进行 MQTT 发布。

### 6.3 Registry 生命周期

```
进程检测 → IPC 连接 → 登录成功
    │
    ├─ RegistryConfig.add()     ← 添加条目
    ├─ MQTT 连接绑定             ← mqttClient 写入
    ├─ 心跳维护                  ← lastPongTime 更新
    │
    ├─ 登出 / 连接断开
    │   └─ RegistryConfig.remove() ← 删除条目
    │
    └─ 应用退出
        └─ RegistryConfig.removeAll() ← 清空所有
```

## 七、错误处理与容错机制

### 7.1 各通信机制的容错策略

| 通信机制 | 错误场景 | 容错策略 |
|---------|---------|---------|
| Electron IPC | 渲染进程崩溃 | `render-process-gone` 事件重启、`crashOrErrorReport` 上报 |
| WebSocket | 端口被占用 | 端口范围 13323-13423，逐个尝试 |
| WebSocket | 连接断开 | 前端无自动重连（架构缺陷） |
| MQTT | 连接失败 | 关闭自动重连（`reconnectPeriod: 0`），由 `PingTimer` 定期检测并手动重连 |
| MQTT | 发送失败 | `MqttMakeUpManager` 缓存未发送消息（最多 100 条），每 10 秒补发 |
| MQTT | 消息超时 | 消息创建时间超过 3 小时（可配置）视为过期，丢弃 |
| 逆向 IPC | 管道断开 | `selectCode < 0` 触发 `closeIpcConnect`，上报登出并清理 Registry |
| 逆向 IPC | 进程消失 | `initIpcTask` 每 5 秒扫描进程，发现新进程自动建连 |
| HTTP | 请求失败 | 各调用点自行 catch，通常记录日志 |
| 通用 | 未捕获异常 | `process.on('uncaughtException')` + `unhandledRejection` → 日志 + 告警 |

### 7.2 心跳与健康检查

系统通过多种定时任务维持通信健康：

| 定时任务 | 周期 | 检查内容 |
|---------|------|---------|
| HeartBeatTimer | 10 秒 | 检查逆向 IPC 心跳（lastPongTime），超时置不可用 |
| PingTimer | 10 秒 | 向逆向发送 Ping，检测 MQTT 连接，断开则重连 |
| ProcessMakeUpTaskTimer | 10 秒 | 补发 MQTT 未上报消息 |
| GalaxyTaskStatusTimer | 10 秒 | 检查任务执行超时 |
| delaySendFrontmsgTimer | 10 秒 | 延迟合并推送前端消息（群更新、好友更新） |
| GetCloudConfigTimer | 按配置 | 刷新云端配置 |
| GetOssAccessKeyTimer | 按配置 | 刷新 OSS 上传凭证 |

## 九、已知问题与优化建议

### 9.1 架构层面

1. **WebSocket 端口硬编码**：`index.html` 中 WebSocket 连接地址硬编码为 `ws://localhost:13323`，但实际端口可能在 13323-13423 范围内变化。应通过 `get-ws-port` IPC 动态获取。

2. **WebSocket 无重连机制**：前端 WebSocket 客户端在连接断开后没有自动重连逻辑，这在网络抖动或主进程重启时会导致前端与主进程失联。

3. **MQTT 关闭自动重连**：`reconnectPeriod: 0` 关闭了 MQTT 库的自动重连，改由 `PingTimer` 手动管理。这增加了复杂度，但提供了更精细的控制。

4. **逆向 IPC 轮询模式**：Named Pipe 的消息接收使用循环轮询（`IpcSelectCltChannel`），而非事件驱动。在多账号场景下可能消耗较多 CPU。

### 9.2 可靠性层面

1. **MQTT 消息补发队列上限**：未发送消息最多缓存 100 条，超过则丢弃。在长时间断网场景下可能丢失重要上报。

2. **三段式消息匹配超时**：如果三条消息中的某一条丢失，缓存的消息片段可能长时间占用内存。`GalaxyTaskStatusTimer` 负责清理超时任务，但清理策略可进一步优化。

3. **单 WebSocket 连接**：`frontConnection.channel` 只保存最后一个连接。如果前端页面刷新或多窗口打开，前一个连接会被覆盖，可能导致消息丢失。

## 十二、通信机制性能基准

### 12.1 延迟对比

| 通信机制 | 典型延迟 | 影响因素 |
|---------|---------|---------|
| Electron IPC | < 1ms | 进程内通信，几乎无延迟 |
| WebSocket | 1-5ms | localhost，TCP 连接 |
| Named Pipe | 1-10ms | 进程间通信，依赖系统调度 |
| MQTT | 50-500ms | 公网传输，依赖网络质量和 Broker 性能 |
| HTTP | 100ms-5s | 公网传输，依赖服务器响应时间 |

### 12.2 吞吐量限制

| 通信机制 | 理论吞吐 | 实际瓶颈 |
|---------|---------|---------|
| Electron IPC | 极高 | 序列化/反序列化开销 |
| WebSocket | 高 | 单连接，消息序列化 |
| Named Pipe | 高 | DLL 调用开销，轮询频率 |
| MQTT | 中 | 阿里云实例配额，网络带宽 |
| HTTP | 低-中 | 请求-响应开销，服务器处理能力 |

### 12.3 可靠性对比

| 通信机制 | 消息可靠性 | 容错能力 |
|---------|-----------|---------|
| Electron IPC | 极高（进程内） | 进程崩溃则全部丢失 |
| WebSocket | 中（无持久化） | 无重连，断开后丢失 |
| Named Pipe | 高（操作系统保证） | 进程退出则断开 |
| MQTT | 高（QoS 1 + 补偿） | 有离线消息和补偿队列 |
| HTTP | 高（请求-响应确认） | 失败可重试 |

## 十三、通信链路追踪

### 13.1 如何追踪一条消息的完整链路

当需要调试某个功能（如"发送文本消息"）时，可以按以下顺序追踪消息经过的每个通信节点：

**从前端发起的操作**：

```
1. 前端 JavaScript → ws.send(JSON)
     ↓ [WebSocket]
2. frontServer.connection.on('message')
     ↓
3. frontFlowInBound → frontHandlerMap[cmdId]
     ↓
4. forwardTask.execute()
     ↓
5. dispatchInBound(channelId, wxId, message)
     ↓
6. reverseSend.sendMessage()
     ↓ [Named Pipe]
7. Clibrary.IpcClientSendMessage(pipeCode, JSON)
     ↓
8. 微信进程执行
     ↓ [Named Pipe 返回]
9. asyncSelectTask → IpcClientRecvMessage()
     ↓
10. dispatchOutBound(message, wrapper)
      ↓
11. WxMsgHandle.outBoundMsg()
      ↓ (三段式匹配)
12. businessHandler
      ├─ frontFlowOutBound → sendToFront [WebSocket → 前端]
      └─ cloudFlowOutBound → convert → mqttSend [MQTT → 云端]
```

**从云端发起的操作**：

```
1. 云端 publish 到 MQTT Topic
     ↓ [MQTT]
2. mqttClientBase.on('message')
     ↓
3. execute() → task-mqtt/Service.filter() → operate()
     ↓ (可能有 HTTP 文件下载)
4. cloudFlowInBound(channelId, wxId, message)
     ↓
5. dispatchInBound → reverseSend
     ↓ [Named Pipe]
6. 微信执行
     ↓ [Named Pipe 返回]
7. asyncSelectTask → dispatchOutBound
     ↓
8. WxMsgHandle → businessHandler
     ├─ frontFlowOutBound [WebSocket → 前端]
     └─ cloudFlowOutBound → mqttSend [MQTT → 云端]
```

### 13.2 日志追踪关键字

按消息流转顺序，搜索以下日志关键字可以追踪完整链路：

| 阶段 | 关键字 | 说明 |
|------|--------|------|
| WebSocket 接收 | `frontFlowInBound` | 前端请求进入 |
| MQTT 接收 | `mqttClientBase`, `execute` | 云端任务进入 |
| 任务处理 | `forwardTask`, `mqttXxxService` | 任务解析和转换 |
| 入站调度 | `dispatchInBound` | 消息路由到逆向 |
| 逆向发送 | `reverseSend`, `IpcClientSendMessage` | 消息发送到微信 |
| 逆向接收 | `asyncSelectTask`, `IpcClientRecvMessage` | 微信返回消息 |
| 出站调度 | `dispatchOutBound` | 消息从逆向进入处理 |
| 消息匹配 | `msgHandleBase`, `thirdMsgHandler` | 三段式匹配 |
| 前端推送 | `sendToFront`, `sendResponse2Front` | 推送到前端 |
| 云端上报 | `mqttSend`, `mqExcuteMsg` | 上报到云端 |
| HTTP 请求 | `httpFetch` | 辅助 HTTP 操作 |

## 十四、版本兼容性说明

### 14.1 微信版本兼容

逆向 IPC 的工作依赖于注入的 DLL 与微信版本的兼容性。当微信更新时：
- DLL 中的内存偏移量可能需要调整
- 消息类型和数据格式可能变化（如微信 4.0 引入新的消息格式）
- 项目中通过 `galaxyTypeExtEnum.js` 映射任务类型，支持新旧版本兼容

### 14.2 Electron 版本

当前使用 Electron 20.0.2，部分 API 可能在新版本中有变化：
- `electron-better-ipc` 需要与 Electron 版本兼容
- `contextBridge` API 在不同 Electron 版本中行为可能略有不同
- `net` 模块的 API 在 Electron 新版本中可能有调整

### 14.3 Node.js 模块

- `ffi-napi` 需要与 Node.js 版本和 Electron 的 Node.js 版本兼容
- `ref-napi` 同上
- 这些原生模块需要针对 Electron 的 Node.js ABI 重新编译
