## 1. 两仓库分工关系

### 1.1 架构角色划分

```
┌─────────────────────────────────────────────────────────┐
│                   开发时运行架构                          │
│                                                         │
│  ┌──────────────────────┐  ┌──────────────────────────┐ │
│  │   galaxy-client       │  │   galaxy                  │ │
│  │   (Electron 主进程)    │  │   (React 前端)            │ │
│  │                       │  │                           │ │
│  │  src/electron.js      │  │  CRA dev server           │ │
│  │       │               │  │  (localhost:3000)          │ │
│  │       ▼               │  │       │                   │ │
│  │  BrowserWindow        │  │  load.html                │ │
│  │  .loadURL(            │──│  menu.html                │ │
│  │   localhost:3000/     │  │  sub.html                 │ │
│  │   menu.html)          │  │  vpn.html                 │ │
│  │                       │  │                           │ │
│  │  Named Pipe / MQTT    │  │  WebSocket Client         │ │
│  │  WebSocket Server     │  │  (连接 galaxy-client)      │ │
│  └──────────────────────┘  └──────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 1.2 职责说明

| 角色 | 仓库 | 开发时入口 | 说明 |
|------|------|-----------|------|
| Electron 主进程 | `galaxy-client` | `src/electron.js` | 直接由 Electron 运行，不需要 webpack 打包（dev 模式） |
| React 前端渲染端 | `galaxy` | `yarn start` → CRA dev server | 通过 `react-app-rewired` 定制 Webpack，提供多入口 HTML |

### 1.3 关键前提

- `galaxy` 仓库的 `electron/` 目录代码**已废弃**，Electron 主进程完全由 `galaxy-client` 承担
- `galaxy` 前端通过 WebSocket 连接 `galaxy-client` 的 `frontServer.js`（Node.js WS Server），**不再连接旧 Java 服务**
- 两个仓库需要同时运行才能完成完整的开发调试

## 2. galaxy 前端侧：yarn start 启动流程

### 2.1 package.json scripts（galaxy 仓库）

```json
{
  "scripts": {
    "start": "react-app-rewired start",
    "build": "react-app-rewired build",
    "build:weixinzhushou": "cross-env BUILD_PATH=weixinzhushou react-app-rewired build",
    "build:browser": "cross-env BUILD_PATH=browser react-app-rewired build"
  }
}
```

执行 `yarn start` 时实际调用 `react-app-rewired start`，它会加载 `config-overrides.js` 覆盖 CRA 默认的 Webpack 配置。

### 2.2 config-overrides.js 核心定制

**文件路径**：`galaxy/config-overrides.js`

#### 2.2.1 多入口 HTML 生成

```javascript
const multipleEntry = require('react-app-rewire-multiple-entry')([
    { entry: 'src/entries/load/index.js',   template: 'public/load.html',   outPath: '/load.html' },
    { entry: 'src/entries/menu/index.js',   template: 'public/menu.html',   outPath: '/menu.html' },
    { entry: 'src/entries/sub/index.js',    template: 'public/sub.html',    outPath: '/sub.html'  },
    { entry: 'src/entries/vpn/index.js',    template: 'public/vpn.html',    outPath: '/vpn.html'  },
]);
```

每个入口对应独立的 `HtmlWebpackPlugin` 实例，开发时可通过以下 URL 访问：

| 入口 | 开发 URL | 职责 |
|------|---------|------|
| `load.html` | `http://localhost:3000/load.html` | 启动加载页 |
| `menu.html` | `http://localhost:3000/menu.html` | 主控制面板 |
| `sub.html` | `http://localhost:3000/sub.html` | 功能子页面 |
| `vpn.html` | `http://localhost:3000/vpn.html` | VPN 登录页 |

#### 2.2.2 Webpack Alias 配置

```javascript
config.resolve.alias = {
    ...config.resolve.alias,
    '@': path.resolve(__dirname, 'src'),
    '@staging': path.resolve(__dirname, 'src/staging'),
    'electron': path.resolve(__dirname, 'src/alias/electron.js'),
};
```

关键点：`electron` 模块被别名映射到 `src/alias/electron.js`。

#### 2.2.3 electron 别名的作用

**`galaxy/src/alias/electron.js`**：

```javascript
module.exports = window.require('electron');
```

在 Electron 渲染进程中，`window.require` 是由 `nodeIntegration: true` 注入的 Node.js require 函数。通过 Webpack alias，前端代码中 `import electron from 'electron'` 实际会执行 `window.require('electron')`，绕过 Webpack 对 `electron` 模块的打包处理。

**`galaxy/src/alias/electron.browser.js`**（浏览器模式）：

```javascript
module.exports = {};
```

当 `BUILD_PATH=browser` 时，`conditional-loader.js` 会自动将 `electron.js` 替换为 `electron.browser.js`，返回空对象，使前端代码可以在纯浏览器中运行。

#### 2.2.4 条件编译 Loader

**`galaxy/conditional-loader.js`**：

```javascript
module.exports = function (source) {
    const callback = this.async();
    const isTargetBrowserPlatform = process.env.BUILD_PATH === 'browser';
    if (!isTargetBrowserPlatform) {
        return callback(null, source);
    }
    // 构造 .browser.js 路径
    const browserFilePath = originalFilePath.replace(fileExtension, browserFileExtension);
    if (fs.existsSync(browserFilePath)) {
        fs.readFile(browserFilePath, 'utf8', (readError, data) => {
            callback(null, data);
        });
    } else {
        callback(null, source);
    }
};
```

工作原理：
1. 当 `BUILD_PATH=browser` 时激活
2. 对每个 `.js`/`.ts` 文件检查是否存在同名的 `.browser.js`/`.browser.ts` 文件
3. 如果存在则替换源文件内容，实现条件编译

#### 2.2.5 IPC 适配层

**`galaxy/src/common/ipc.js`**（Electron 模式）：

```javascript
import ipc from 'electron-better-ipc/source/renderer';
export default ipc;
```

**`galaxy/src/common/ipc.browser.js`**（浏览器模式）：

```javascript
// 返回 Proxy，所有方法返回 Promise.resolve()
```

通过 `conditional-loader` 在浏览器模式下替换为空实现，避免 Electron 依赖。

### 2.3 CRA Dev Server 启动时序

```
yarn start
    │
    ▼
react-app-rewired start
    │
    ▼
加载 config-overrides.js
    │
    ├── 注册 multipleEntry（4个HTML入口）
    ├── 添加 Less loader
    ├── 设置 Webpack alias（@、@staging、electron）
    ├── 注入 conditional-loader
    └── 设置 DefinePlugin 环境变量
    │
    ▼
Webpack Dev Server 启动
    │
    ▼
监听 localhost:3000
    │
    ├── /load.html  → load 入口
    ├── /menu.html  → menu 入口
    ├── /sub.html   → sub 入口
    └── /vpn.html   → vpn 入口
```

## 3. galaxy-client 主进程侧：start:dev 命令

### 3.1 package.json scripts

**文件路径**：`galaxy-client/package.json`

```json
{
  "scripts": {
    "start:dev": "cross-env ELECTRON_NODE_ENV=dev PRODUCT=weixinzhushou electron --inspect=5678 ./src/electron.js",
    "start:prod": "cross-env ELECTRON_NODE_ENV=prod ENV_TYPE=prod PRODUCT=weixinzhushou electron --inspect=5678 ./src/electron.js"
  }
}
```

### 3.2 start:dev 命令拆解

```
npm run start:dev
    │
    ▼
cross-env ELECTRON_NODE_ENV=dev PRODUCT=weixinzhushou electron --inspect=5678 ./src/electron.js
```

| 部分 | 说明 |
|------|------|
| `cross-env` | 跨平台环境变量设置工具 |
| `ELECTRON_NODE_ENV=dev` | 设置运行环境为开发模式，影响配置加载和 URL 选择 |
| `PRODUCT=weixinzhushou` | 设置产品线为"微信助手"，影响配置目录选择 |
| `electron` | 使用 `devDependencies` 中的 Electron 20.0.2 |
| `--inspect=5678` | 开启 Node.js Inspector，调试端口 5678 |
| `./src/electron.js` | 主进程入口文件（**开发时直接运行源码，不需要 Webpack 打包**） |

### 3.3 开发模式 vs 生产模式的入口差异

| 模式 | 入口文件 | 说明 |
|------|---------|------|
| 开发 (`start:dev`) | `src/electron.js`（源码） | 直接运行，支持热调试 |
| 生产构建 | `public/electron.js`（打包产物） | 由 `webpack.electron.config.js` 打包生成 |

开发时不需要执行 `pre-build:test`（Webpack 打包主进程），因为 Electron 可以直接运行 Node.js 源码。

### 3.4 环境变量的作用域

```
ELECTRON_NODE_ENV=dev 的影响：

1. application-config/index.js
   │
   └── 根据 process.env.ELECTRON_NODE_ENV 选择配置：
       ├── 'dev' / 'test' → applicationQa.js（测试环境配置）
       ├── 'prod'          → applicationProd.js（生产环境配置）
       └── 其他            → 根据 urls.type 判断

2. urls.js
   │
   └── 决定 devServerRoot、loginUrl、updateUrl 等
       ├── isDev = true → 使用 localhost:3000 加载前端
       └── isDev = false → 使用打包产物路径

3. runtime.dev.yml
   │
   └── getAppConfig() 检测到开发环境后会合并 runtime.dev.yml 的覆盖配置

4. initWindow → webPreferences
   │
   └── 开发模式自动打开 DevTools
```

### 3.5 主进程启动时序（开发模式）

```
electron --inspect=5678 ./src/electron.js
    │
    ▼
1. global.readyStartTime = Date.now()
    │
    ▼
2. initLog()           — 初始化 electron-log + SLS
    │
    ▼
3. setupRemoteDebug()  — 检查 --remote-debug 参数
    │
    ▼
4. bootstrap()
    ├── addAppEvent()    — 注册 app 生命周期事件
    │   └── app.whenReady() → initWindow()
    │       └── BrowserWindow.loadURL('http://localhost:3000/...')
    ├── addIpcEvent()    — 注册 IPC 事件处理
    ├── addStoreEvent()  — 注册 store 变更监听
    └── AppStart.run()   — 启动业务子系统
        ├── FrontStart.run()        → WebSocket Server
        ├── ReverseStart.run()      → Named Pipe 扫描
        ├── scheduleRun()           → 定时任务
        ├── MemoryQueueApplication.run() → 内存队列
        └── judgeCrashAndReport()   → 崩溃检测
```

## 4. 开发时页面加载机制

### 4.1 loadUrl.js 的判断逻辑

**文件路径**：`galaxy-client/src/common/loadUrl.js`

```javascript
async function getLoadUrlAsync() {
    const isOnline = await net.check();
    if (isOnline) {
        const userId = store.getUserId();
        if (userId) {
            return successUrl;   // 已登录 → 加载主页面
        }
        return loginUrl;         // 未登录 → 加载 CAS 登录页
    }
    return offlineUrl;            // 无网络 → 加载离线页
}
```

### 4.2 开发时 URL 来源

**文件路径**：`galaxy-client/src/common/urls.js`

```javascript
// 关键判断
const isDev = process.env.ELECTRON_NODE_ENV === 'dev';

// 开发时使用 localhost
const devServerRoot = 'http://localhost:3000';

// loginUrl 在开发模式下指向测试环境 CAS
const loginUrlMap = {
    test: 'https://cas-test.xxx.com/login?service=...',
    prod: 'https://cas.xxx.com/login?service=...'
};

// successUrl 在开发模式下指向 localhost
const successUrlMap = {
    test: isDev ? `${devServerRoot}/menu.html` : 'menu.html',
    prod: isDev ? `${devServerRoot}/menu.html` : 'menu.html'
};
```

### 4.3 开发时加载流程

```
BrowserWindow 创建
    │
    ▼
getLoadUrlAsync()
    │
    ├── 网络检测 (net.check)
    │   └── 请求 CAS 登录页判断连通性
    │
    ├── 已登录 (userId 存在)
    │   └── loadURL('http://localhost:3000/menu.html')
    │       │
    │       ▼
    │   CRA Dev Server 返回 menu.html
    │   → React 应用启动
    │   → WebSocket 连接 galaxy-client 的 frontServer
    │
    └── 未登录
        └── loadURL('https://cas-test.xxx.com/login?...')
            │
            ▼
        CAS 登录页 → 登录成功后重定向到
        http://localhost:3000/menu.html?ticket=...
```

### 4.4 CAS 重定向处理

**文件路径**：`galaxy-client/src/init/window.js`

```javascript
// 监听 URL 变化，处理 CAS 重定向
mainWindow.webContents.on('did-navigate', (event, url) => {
    // 检测到 CAS ticket 参数时
    // 执行登录逻辑，将 ticket 换取用户信息
    // 登录成功后加载 successUrl
});
```

## 5. runtime.dev.yml vs runtime.yml 配置差异

### 5.1 配置加载机制

**文件路径**：`galaxy-client/src/utils.js` 中的 `getAppConfig()`

```javascript
function getAppConfig() {
    const runtimeConfig = getYamlConfig('runtime.yml');
    // 开发模式下合并 dev 配置
    if (isDev) {
        const devConfig = getYamlConfig('runtime.dev.yml');
        Object.assign(runtimeConfig, devConfig);
    }
    return runtimeConfig;
}
```

### 5.2 配置文件路径

| 文件 | 路径 | 用途 |
|------|------|------|
| `runtime.yml` | `config/weixinzhushou/runtime.yml` | 生产环境完整配置 |
| `runtime.dev.yml` | `config/weixinzhushou/runtime.dev.yml` | 开发环境覆盖配置 |

### 5.3 关键差异对比

| 配置项 | runtime.yml（生产） | runtime.dev.yml（开发） | 说明 |
|--------|-------------------|----------------------|------|
| `switch.kibana` | `true` | `false` | 关闭 Kibana 上报 |
| `switch.sentry` | `true` | `false` | 关闭 Sentry 错误上报 |
| `switch.starEyes` | `true` | `false` | 关闭星眸监控 |
| `switch.wsLogger` | `false` | `true` | 开启 WebSocket 调试日志 |

开发模式下的核心策略：**关闭所有远程上报，开启本地调试日志**。

### 5.4 runtime.yml 主要字段

```yaml
# 微信多开数量限制
multiple: 10
qyMultiple: 1

# 应用标识
elfkey: "weixinzhushou"
galaxyver: "5.5.0"

# 功能模块开关
modules:
  wechat: true
  wxwork: true

# 产品线标识
product: "weixinzhushou"

# 协议注册
protocol: "weixinzhushou"

# 支持的微信版本
SUPPORTED_WX_VERSIONS: ["3.9.x", "4.0.x"]
SUPPORTED_QYWX_VERSIONS: ["4.1.x"]

# JVM 配置（启动 Java 服务时使用）
Xms: "-Xms128m"
Xmx: "-Xmx256m"

# 上报开关
switch:
  kibana: true
  sentry: true
  starEyes: true
  wsLogger: false
```

## 6. 热更新与重启机制

### 6.1 前端代码热更新

| 层 | 机制 | 说明 |
|----|------|------|
| `galaxy` 前端 | CRA 内置 HMR（Hot Module Replacement） | 修改 React 组件后自动刷新，无需重启 |
| CSS 变更 | 样式注入 | 不刷新页面，直接替换样式 |

### 6.2 主进程代码重启

`galaxy-client` 的主进程代码**不支持热更新**。修改主进程代码后需要：

1. 手动终止 Electron 进程（`Ctrl+C`）
2. 重新执行 `npm run start:dev`

### 6.3 VSCode 调试配置

**文件路径**：`galaxy-client/.vscode/launch.json`

```json
{
    "name": "Debug client start",
    "type": "node",
    "request": "launch",
    "cwd": "${workspaceRoot}",
    "runtimeExecutable": "${workspaceRoot}/node_modules/.bin/electron",
    "windows": {
        "runtimeExecutable": "${workspaceRoot}/node_modules/.bin/electron.cmd"
    },
    "program": "${workspaceRoot}/src/electron.js",
    "runtimeArgs": ["--trace-warnings"],
    "protocol": "inspector"
}
```

使用方式：
1. 在 VSCode 中按 `F5` 启动调试
2. 支持断点、变量查看、调用栈等
3. 主进程代码修改后需要重启调试会话

### 6.4 nodemon 调试配置

```json
{
    "name": "nodemon start test",
    "program": "${workspaceFolder}/src/test/app-start.js",
    "runtimeExecutable": "nodemon",
    "restart": true,
    "type": "node"
}
```

针对单元测试场景，使用 `nodemon` 实现文件变更后自动重启。

## 7. 开发调试工具配置

### 7.1 主进程 Node.js Inspector

```bash
# start:dev 命令中已包含
electron --inspect=5678 ./src/electron.js
```

连接方式：
- Chrome: 访问 `chrome://inspect`，配置 `localhost:5678`
- VSCode: 使用 Attach 模式连接到 5678 端口

### 7.2 远程调试（remoteDebug）

**文件路径**：`galaxy-client/src/init/remoteDebug.js`

```javascript
// 通过命令行参数 --remote-debug 激活
function setupRemoteDebug() {
    if (process.argv.includes('--remote-debug')) {
        // 1. 启动 Node.js Inspector（主进程）
        inspector.open(9229, '0.0.0.0');

        // 2. 启动 Chrome DevTools Protocol（渲染进程）
        //    通过 TCP proxy 将 127.0.0.1 转发到 0.0.0.0

        // 3. 显示调试信息对话框
        dialog.showMessageBox({ message: debugInfo });
    }
}
```

远程调试允许从其他机器连接到运行中的 Electron 应用，适用于在部署机器上排查问题。

### 7.3 渲染进程 DevTools

开发模式下自动打开：

```javascript
// galaxy-client/src/init/window.js
if (isDev || isTest) {
    mainWindow.webContents.openDevTools();
}
```

### 7.4 WebSocket 调试日志

当 `runtime.dev.yml` 中 `switch.wsLogger: true` 时，`frontServer.js` 会记录所有 WebSocket 消息的收发详情，便于调试前端与主进程之间的实时通信。

## 8. DLL 文件的开发准备

### 8.1 为什么需要下载 DLL

`galaxy-client` 通过 `ffi-napi` 调用三个外部 DLL/EXE：

| 文件 | 用途 | 来源 |
|------|------|------|
| `PipeCore.dll` (Sirius) | Named Pipe 通信 | 内部 CI 服务器 |
| `BasicService.exe` | 进程注入桥接 | 内部 CI 服务器 |
| `Galaxy64.dll` | 进程工具库 | 内部 CI 服务器 |

这些文件不在 Git 仓库中，需要在开发前下载到 `extraResources/Inject/` 目录。

### 8.2 downloadDevFile.js

**文件路径**：`galaxy-client/scripts/downloadDevFile.js`

```bash
# 使用方式
node scripts/downloadDevFile.js 5.5.0-release01
# 或使用默认版本
node scripts/downloadDevFile.js
```

执行流程：
1. 读取 `scripts/versionConfig.json` 获取下载 URL
2. 清空 `extraResources/Inject/` 目录
3. 从内部 CI 服务器下载 ZIP 文件
4. 解压到 `extraResources/Inject/`
5. 过滤掉 `mock` 文件

### 8.3 versionConfig.json 结构

**文件路径**：`galaxy-client/scripts/versionConfig.json`

```json
{
    "5.5.0-release01": {
        "sirius_dll_path": "http://10.252.71.71:9091/Sirius/1040/.../Release.zip",
        "basicService_exe_path": "http://10.252.71.71:9091/BaseService/1013/.../Release.zip",
        "galaxy64_dll_path": "http://10.252.71.71:9091/Galaxy/2.0.2/.../Release.zip"
    },
    "default": {
        "sirius_dll_path": "http://10.252.71.71:9091/Sirius/1039/.../Release.zip",
        "basicService_exe_path": "http://10.252.71.71:9091/BaseService/1012/.../Release.zip",
        "galaxy64_dll_path": "http://10.252.71.71:9091/Galaxy/2.0.1/.../Release.zip"
    }
}
```

每个版本对应一组 DLL 下载地址，`default` 为兜底配置。

## 9. 常见开发报错及解决方式

### 9.1 DLL 加载失败

**报错**：`Error: Dynamic Linking Error: Win32 error 126`

**原因**：`extraResources/Inject/` 目录下缺少 DLL 文件

**解决**：
```bash
node scripts/downloadDevFile.js
```

### 9.2 前端页面白屏

**报错**：BrowserWindow 加载 `localhost:3000` 失败

**原因**：`galaxy` 前端的 CRA dev server 尚未启动

**解决**：确保先启动 `galaxy` 仓库的 `yarn start`，等待 dev server 就绪后再启动 `galaxy-client`

### 9.3 WebSocket 连接失败

**报错**：前端 console 显示 WebSocket 连接拒绝

**原因**：`galaxy-client` 主进程中的 `frontServer.js` 未启动，或端口被占用

**解决**：
1. 检查 `galaxy-client` 是否正常启动
2. 确认 13323-13423 端口范围内有可用端口
3. 查看主进程日志中 `WebSocket server started on` 信息

### 9.4 CAS 登录重定向失败

**报错**：登录后页面停留在 CAS 或显示 404

**原因**：CAS 回调 URL 与 dev server 地址不匹配

**解决**：确认 `urls.js` 中 `successUrlMap` 在开发模式下指向正确的 `localhost:3000` 地址

### 9.5 ffi-napi 编译报错

**报错**：`node-gyp rebuild` 失败

**原因**：`ffi-napi` 需要本地编译，依赖 Visual Studio Build Tools

**解决**：
1. 安装 Visual Studio Build Tools（C++ 桌面开发工作负载）
2. 安装 Python 3.x
3. 执行 `npm install` 重新编译原生模块

### 9.6 Electron 版本不匹配

**报错**：`The module was compiled against a different Node.js version`

**原因**：原生模块（sqlite3、ffi-napi 等）与 Electron 20.0.2 内置的 Node.js 版本不匹配

**解决**：
```bash
npx electron-rebuild
```
