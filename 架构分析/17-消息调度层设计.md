## 2. 入站调度（dispatchInBound）

### 2.1 职责

接收来自**云端 MQTT** 或**前端 WebSocket** 的指令，转发给目标**逆向 Named Pipe** 连接。

**文件路径**：`galaxy-client/src/msg-center/dispatch-center/dispatchInBound.js`

### 2.2 核心流程

```
入站消息 (channelId, wxId, message)
    │
    ▼
查找目标注册表 (RegistryConfig)
    │
    ├── 按 wxId 查找 → registry
    └── 按 channelId 查找 → registry
    │
    ▼
registry 不存在？ → 日志警告，返回
    │
    ▼
解析消息类型 (type)
    │
    ├── type 在 TaskCallBack 中？
    │   └── YES → 加锁处理 (async-lock)
    │       ├── 创建 GalaxyTaskStatus 缓存
    │       ├── 设置任务追踪状态
    │       └── inBoundAct() → reverseSend.sendMessage()
    │
    └── type 不在 TaskCallBack 中？
        └── 直接 inBoundAct() → reverseSend.sendMessage()
```

### 2.3 关键代码

```javascript
function dispatchInBound(channelId, wxId, message) {
    // 查找目标微信实例
    let registry = wxId
        ? RegistryConfig.getRegistryByKey(wxId, 'wxid')
        : RegistryConfig.getRegistryByKey(channelId, 'id');

    if (!registry) {
        logUtil.customLog(`尚未和逆向建立注册信息`);
        return;
    }

    const jsonObject = JSON.parse(message);
    const type = typeConvert(jsonObject.type || jsonObject.body.type);

    processInBoundTask({channelId, wxId, message, jsonObject, type, workWxFlag});
}
```

### 2.4 任务追踪（GalaxyTaskCache）

对于需要追踪结果的任务（如发送消息），在下发指令时就创建状态缓存：

```javascript
function processInBoundTask(params) {
    if (TaskCallBack.taskCallbackMap[type]) {
        lock.acquire(galaxyTaskLock, (done) => {
            const galaxyTaskStatus = {
                firstMessageStatus: 'NOT_RECEIVE',
                secondMessageStatus: 'NOT_RECEIVE',
                thirdMessageStatus: 'NOT_RECEIVE',
                username: wxId,
                taskId: jsonObject.taskId,
                createTime: Date.now()
            };
            GalaxyTaskCache.GALAXY_TASK_STATUS_MAP[taskId] = galaxyTaskStatus;
            GalaxyTaskCache.setTaskMsgTypeMap(jsonObject);
            inBoundAct(channelId, wxId, message);
            done();
        });
    } else {
        inBoundAct(channelId, wxId, message);
    }
}
```

### 2.5 typeConvert 适配

```javascript
function typeConvert(type) {
    // 企微消息类型统一前缀
    if (type.startsWith(GalaxyTaskType.WORK_WX_SEND_PREFIX)) {
        return GalaxyTaskType.WORK_WX_SEND_PREFIX;
    }
    return type;
}
```

企微的发送类型（如 `sendmsg_text`、`sendmsg_image`）统一转为 `sendmsg_` 前缀匹配。

## 3. 出站调度（dispatchOutBound）

### 3.1 职责

接收来自**逆向 Named Pipe** 的消息，经过处理后路由到**云端 MQTT 上报** 和/或**前端 WebSocket 推送**。

**文件路径**：`galaxy-client/src/msg-center/dispatch-center/dispatchOutBound.js`

### 3.2 核心流程

```
逆向消息 (message, pipeLineWrapper)
    │
    ▼
JSON.parse(message)
    │
    ├── type === 'bugreport' → 上报 REVERSE_BUG_REPORT
    ├── type 含 'error' → 上报 REVERSE_ERROR
    │
    ▼
判断微信/企微
    │
    ├── pipeLineWrapper.workWx === true
    │   └── WkMsgHandlerCenter.outBoundMsg(...)
    │
    └── pipeLineWrapper.workWx === false
        ├── type 在 THIRD_CALLBACKS 中且 status != 0
        │   └── 延迟执行 (apolloConfig.taskFailWaitTime)
        │       └── WxMsgHandlerCenter.outBoundMsg(...)
        └── 其他情况
            └── WxMsgHandlerCenter.outBoundMsg(...)
```

### 3.3 关键代码

```javascript
function dispatchOutBound(message, pipeLineWrapper) {
    let jsonObject = JSON.parse(message);

    // 错误消息上报
    if (jsonObject.type?.indexOf('error') !== -1) {
        reportLog({ name: 'REVERSE_ERROR', wxid: pipeLineWrapper.wxid, ... });
    }

    const sendType = jsonObject.type;

    if (pipeLineWrapper.workWx) {
        WkMsgHandlerCenter.outBoundMsg(jsonObject, pipeLineWrapper, sendType);
    } else {
        if (CallBackClassify.THIRD_CALLBACKS.has(sendType) && jsonObject.status != 0) {
            // 任务失败时延迟处理（等待可能的重试）
            setTimeout(() => {
                WxMsgHandlerCenter.outBoundMsg(jsonObject, pipeLineWrapper, sendType);
            }, apolloConfig.taskFailWaitTime);
        } else {
            WxMsgHandlerCenter.outBoundMsg(jsonObject, pipeLineWrapper, sendType);
        }
    }
}
```

### 3.4 延迟处理逻辑

当第三条回执消息（`THIRD_CALLBACKS`）的 `status != 0`（即任务失败）时，延迟 `apolloConfig.taskFailWaitTime` 毫秒再处理。这给了逆向程序可能重试的时间窗口。

## 4. 消息流来源标记（flowSource）

### 4.1 FlowSourceEnum

```javascript
const FlowSourceEnum = {
    FRONT: 1,    // 来自前端（WebSocket）
    CLOUND: 2,   // 来自云端（MQTT）
    OWNER: 3     // 来自逆向自身
};
```

### 4.2 标记写入时机

**cloudFlowInBound.js**：
```javascript
function cloudFlowInBound(channelId, wxId, message) {
    const jsonObject = JSON.parse(message);
    jsonObject[GalaxyConstant.FLOW_SOURCE_KEY] = FlowSourceEnum.CLOUND; // 标记来源为云端
    dispatchInBound(channelId, wxId, JSON.stringify(jsonObject));
}
```

### 4.3 标记读取时机

**msgHandleBase.js → businessHandler**：
```javascript
businessHandler(jsonObject, finalMessage, pipeLineWrapper) {
    let source = jsonObject[GalaxyConstant.FLOW_SOURCE_KEY];

    if (!source) {
        // 没有来源标记 → 广播到所有通道（前端 + 云端）
        this.outBoundByBroadcast(msgStr, pipeLineWrapper);
        return;
    }

    // 有来源标记 → 只路由到对应通道
    if (FLOW_HANDLER_MAP[source]) {
        FLOW_HANDLER_MAP[source](msgStr, pipeLineWrapper);
    }
}
```

### 4.4 路由映射

```javascript
const FLOW_HANDLER_MAP = {
    [FlowSourceEnum.FRONT]:  frontFlowOutBound,   // 1 → 推送到前端
    [FlowSourceEnum.CLOUND]: cloudFlowOutBound,   // 2 → 上报到云端
};
```

### 4.5 路由决策逻辑

```
逆向返回消息
    │
    ├── flowSource === 2 (来自云端的任务回执)
    │   └── 只路由到 cloudFlowOutBound → MQTT 上报
    │
    ├── flowSource === 1 (来自前端的指令回执)
    │   └── 只路由到 frontFlowOutBound → WS 推送
    │
    ├── flowSource === undefined (逆向主动推送)
    │   └── 广播到所有通道（前端 + 云端）
    │
    └── type === 'userlist' 或 QUICK_SEND_TYPE
        └── 强制路由到 cloudFlowOutBound
```

### 5.1 cloudFlowInBound — 云端下行入站


### 5.2 cloudFlowOutBound — 云端上行出站

**文件路径**：`galaxy-client/src/msg-center/dispatch-center/dispatch/cloudFlowOutBound.js`

核心结构：

```javascript
// 通用 Service 列表（登录/登出/心跳）
const commonConvertServiceList = [LoginService, LogoutService, PongService];

// 微信 Service 列表（66+ 个 Service）
const WxConvertServiceList = [...commonConvertServiceList, ...wxSpecificServices];

// 企微 Service 列表（30+ 个 Service）
const WorkWxConvertServiceList = [...commonConvertServiceList, ...wkSpecificServices];

function cloudFlowOutBound(message, pipeLineWrapper) {
    let msgObj = JSON.parse(message);
    let clientRecord = {...ClientMsgBO, ...msgObj};

    const serviceList = pipeLineWrapper.workWx
        ? WorkWxConvertServiceList
        : WxConvertServiceList;

    for (const service of serviceList) {
        if (service.filter(clientRecord)) {
            service.operate(message, pipeLineWrapper);
        }
    }
}
```

**设计模式**：责任链 + 过滤器模式

每个 Service 实现两个方法：
- `filter(clientRecord)` → 是否处理此消息
- `operate(message, pipeLineWrapper)` → 执行处理逻辑

消息会遍历整个 Service 列表，**所有**匹配 filter 的 Service 都会执行（非短路）。

## 6. handle 消息处理器详解

### 6.1 三段式消息聚合（msgHandleBase）

**文件路径**：`galaxy-client/src/msg-center/dispatch-center/handle/msgHandleBase.js`

这是整个消息处理的核心：将逆向返回的多条离散消息聚合为一个完整的任务结果。

#### 6.1.1 三段式消息模型

```
云端下发任务: sendmessage (taskId=123, type="sendTextMsg")
    │
    ▼
逆向接收并开始执行
    │
    ├── 第一条 (first): sendmessage 回执 → 确认逆向收到任务
    │
    ├── 第二条 (second): msgsendreport → 消息发送状态
    │
    └── 第三条 (third): msgreport → 最终发送结果
                          ↓
                    聚合完成 → taskId 传递 → 上报云端
```

#### 6.1.2 MsgResNode 数据结构

```javascript
const MsgResNode = {
    first: null,     // 第一条消息
    second: null,    // 第二条消息
    third: null,     // 第三条消息
    thirdTime: 0,    // 第三条消息处理次数
    expireTime: 0    // 过期时间
};
```

#### 6.1.3 消息聚合核心流程

```javascript
messageHandler(jsonObj, wrapper, sendType) {
    // 1. 过滤：非目标类型、系统消息、非群发消息、非本人消息
    if (this.isNotContains(sendType)) return jsonObj;
    if (this.isSysMsg(jsonObj)) return jsonObj;
    if (this.isNotSendMsg(jsonObj)) return jsonObj;

    // 2. 生成缓存 Key
    const key = wrapper.workWx
        ? this.getQywxKey(jsonObj, wxId)    // 企微: wxId_clientId
        : this.getWxMsgKey(jsonObj, wxId);  // 微信: wxId_toUser_timer

    // 3. 查找已缓存的 MsgResNode
    let msgResNode = this.get(key);

    // 4. 判断当前消息是第几条
    if (this.isFirstMsg(msgResNode, sendType)) {
        // 缓存第一条消息
        msgResNode = { first: { taskId, hash, status, type } };
        this.add(msgResNode, key);
        return jsonObj;
    }
    else if (this.isSecondMsg(msgResNode, sendType)) {
        // 缓存第二条消息
        msgResNode.second = { taskId, hash, status };
        return jsonObj;
    }
    else if (this.isThirdMsg(msgResNode, sendType, jsonObj)) {
        // 第三条 → 聚合完成 → 传递 taskId → 返回结果
        msgResNode.third = jsonObj;
        return this.thirdMsgHandler(msgResNode, key);
    }

    // 5. 异常情况处理
    return this.errorOrder(msgResNode, sendType, jsonObj, ...);
}
```

#### 6.1.4 微信 4.0 两段式模型

微信 4.0 版本简化为两段式：

```
第一条: sendmessage → 确认收到
第二条: recvmsg    → 直接作为最终结果（status=2 表示成功）
```

```javascript
// wxMsgHandle.js
isWx4RecvmsgComplete(node, sendType, msgObj) {
    if (node?.first
        && CallBackClassify.FIRST_CALLBACKS_NEW.has(node.first.type)
        && sendType === 'recvmsg') {
        // 状态转换：微信4.0 status=2 → 成功(0)，其他 → 失败(1)
        const convertedStatus = msgObj.status === 2 ? 0 : 1;
        msgObj.status = convertedStatus;
        // 继承第一条的 taskId
        msgObj.taskId = node.first.taskId;
        return true;
    }
    return false;
}
```

### 6.2 wxMsgHandle — 微信消息处理器

**文件路径**：`galaxy-client/src/msg-center/dispatch-center/handle/wxMsgHandle.js`

继承 `msgHandleBase`，覆写判断方法：

| 方法 | 实现逻辑 |
|------|---------|
| `isReset` | 4.0: 收到新 sendmessage 且已有 first → 重置；旧版: 三条齐全且收到新 msgreport |
| `isNotContains` | type 不在 `CALLBACK_SET` 中 |
| `isFirstMsg` | 无缓存 + type 在 `FIRST_CALLBACKS` 或 `FIRST_CALLBACKS_NEW` |
| `isSecondMsg` | 有 first + 无 second + type 在 `SECOND_CALLBACKS` |
| `isThirdMsg` | 有 first+second + type 在 `THIRD_CALLBACKS` |
| `isWx4RecvmsgComplete` | 微信4.0两段式回执完成判定 |

### 6.3 workWxMsgHandle — 企微消息处理器

| 方法 | 实现逻辑 |
|------|---------|

| `isNotContains` | 不是 RECVMSG / MSG_REPORT / WORK_WX_SEND_PREFIX |

| `isSecondMsg` | 有 first + 无 second + (sendmsg_ 前缀 或 RECVMSG) |

### 6.4 缓存过期清理

```javascript
// 每 10 秒清理一次过期的 MsgResNode
startCleanupTask() {
    this.cleanCacheIntervalId = setInterval(() => {
        this.removeExpireNode();   // 清理过期消息节点
        this.removeTaskIdMap();    // 清理过期 taskId 映射
    }, 10000);
}

removeExpireNode() {
    Object.keys(this.msgResNodeMap).forEach(key => {
        if (key.startsWith('video_') && Array.isArray(value)) {
            // 视频消息特殊处理：过滤有效节点
        } else if (value.expireTime < Date.now()) {
            delete this.msgResNodeMap[key];
        }
    });
}
```

```javascript
const SendToFrontUtil = {
    FILTER_SET: new Set(['pong', 'cdnonerror']),

    sendResponse2Front(taskType, response) {
        const message = JSON.stringify(response);
        if (!this.isFilter(JSON.stringify(response.body || response))) {
            return; // 过滤 pong/cdnonerror
        }
        if (frontConnection.channel) {
            frontConnection.channel.send(message, error => {
                if (error) {
                    reportLog({ name: 'SEND_FRONT_ERROR', ... });
                }
            });
        }
    },

    isFilter(message) {
        const { type } = JSON.parse(message);
        if (!type && this.FILTER_SET.has(type)) {
            return false; // 过滤心跳等消息
        }
        return true;
    }
};
```

### 9.4 MQTT 发送失败补偿

MQTT 发送失败时，消息存入补偿队列：

```
mqExcuteMsg() 发送失败
    │
    ▼
mqttMakeUpManager.addMessage(message)
    │
    ▼
ProcessMakeUpTaskTimer（每 10 秒检查）
    │
    ▼
重试发送 → 成功则移除，失败保留
    │
    ▼
队列满（100 条）→ 丢弃最旧消息
```

### 9.5 三段式聚合超时

```
MsgResNode 缓存
    │
    ▼
expireTime = genExpireTime(jsonObj)  // 默认 3 小时
    │
    ▼
removeExpireNode()（每 10 秒执行）
    │
    ├── 未过期 → 保留等待后续消息
    └── 已过期 → 删除缓存（任务状态为"未完成"）
```
