# 07-告警配置

> 基于日志数据实现自动化监控和告警

---

## 一、告警概述

### 1.1 告警的作用

- **及时发现问题**：自动检测异常情况
- **快速响应**：通过通知及时处理
- **减少人工监控**：7x24 小时自动监控

### 1.2 告警流程

```
日志数据 → 告警规则（查询+条件） → 触发判断 → 通知发送 → 人工处理
```

### 1.3 告警组件

| 组件 | 说明 |
|-----|-----|
| 告警规则 | 定义查询语句和触发条件 |
| 告警策略 | 定义通知方式和抑制规则 |
| 通知渠道 | 钉钉、邮件、短信等 |
| 告警记录 | 历史告警查看和统计 |

## 二、创建告警规则

### 2.1 创建步骤

1. 进入 SLS 控制台
2. 选择 Project → 告警
3. 点击「创建告警」
4. 配置告警规则
5. 配置通知方式
6. 保存并启用

### 2.2 基本配置

| 配置项 | 说明 | 示例 |
|-------|-----|-----|
| 规则名称 | 告警规则名称 | `错误率告警` |
| 规则描述 | 规则说明 | `当错误率超过5%时告警` |
| 所属 Project | 关联的项目 | `your-project` |
| 所属 Logstore | 关联的日志库 | `gaotu-wxzs-client-node-log` |

### 2.3 查询配置

```sql
-- 示例：统计最近5分钟的错误率
* | SELECT
        COUNT(*) AS total,
        SUM(CASE WHEN level = 'error' THEN 1 ELSE 0 END) AS errors,
        ROUND(SUM(CASE WHEN level = 'error' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS error_rate
```

### 2.4 触发条件

| 条件类型 | 说明 | 示例 |
|---------|-----|-----|
| 有数据 | 查询有结果就触发 | 错误日志存在 |
| 数值比较 | 比较查询结果数值 | `error_rate > 5` |
| 数据无变化 | 数据长时间不变 | 心跳检测 |
| 无数据 | 查询无结果触发 | 日志断流 |

**数值比较示例：**

```
error_rate > 5
```

```
errors > 100
```

```
total < 10
```

## 三、告警严重度

### 3.1 严重度级别

| 级别 | 说明 | 使用场景 |
|-----|-----|---------|
| **Critical** | 紧急 | 系统故障、数据丢失 |
| **High** | 高 | 错误率过高、核心功能异常 |
| **Medium** | 中 | 警告增多、性能下降 |
| **Low** | 低 | 信息提示、非紧急事项 |

### 3.2 设置严重度

根据触发条件设置不同严重度：

```
-- Critical: 错误率 > 10%
error_rate > 10 → Critical

-- High: 错误率 > 5%
error_rate > 5 → High

-- Medium: 错误率 > 2%
error_rate > 2 → Medium
```

## 四、通知渠道配置

### 4.1 钉钉机器人

**配置步骤：**

1. 在钉钉群中添加自定义机器人
2. 获取 Webhook 地址
3. 在 SLS 中添加钉钉通知渠道
4. 填入 Webhook 地址

**Webhook 格式：**

```
https://oapi.dingtalk.com/robot/send?access_token=xxx
```

**安全设置：**
- 关键词：添加 `告警` 等关键词
- 签名：使用签名验证

### 4.2 邮件

**配置内容：**
- 收件人邮箱（多个用逗号分隔）
- 邮件主题模板
- 邮件正文模板

### 4.3 短信

**配置内容：**
- 接收手机号
- 短信模板

### 4.4 Webhook

**配置内容：**
- Webhook URL
- 请求方法（POST）
- 请求头
- 请求体模板

**请求体示例：**

```json
{
    "alert_name": "${alert_name}",
    "severity": "${severity}",
    "message": "${alert_message}",
    "fire_time": "${fire_time}"
}
```

## 五、告警模板

### 5.1 可用变量

| 变量 | 说明 |
|-----|-----|
| `${alert_name}` | 告警规则名称 |
| `${severity}` | 严重度 |
| `${project}` | Project 名称 |
| `${logstore}` | Logstore 名称 |
| `${fire_time}` | 触发时间 |
| `${alert_message}` | 告警消息 |
| `${results}` | 查询结果 |
| `${query}` | 查询语句 |
| `${condition}` | 触发条件 |

### 5.2 钉钉消息模板

```
## ${severity} 告警

**告警名称**：${alert_name}
**触发时间**：${fire_time}
**Project**：${project}
**Logstore**：${logstore}

### 查询结果
${results}

### 查询语句
${query}
```

### 5.3 邮件模板

```html
<h2>SLS 告警通知</h2>

<table>
    <tr><td>告警名称</td><td>${alert_name}</td></tr>
    <tr><td>严重度</td><td>${severity}</td></tr>
    <tr><td>触发时间</td><td>${fire_time}</td></tr>
</table>

<h3>查询结果</h3>
<pre>${results}</pre>
```

## 六、告警抑制

### 6.1 为什么需要抑制

- 避免重复告警轰炸
- 减少告警疲劳
- 聚焦真正需要关注的问题

### 6.2 抑制策略

| 策略 | 说明 |
|-----|-----|
| 重复抑制 | 相同告警在一定时间内只发送一次 |
| 静默期 | 告警触发后的一段时间内不再告警 |
| 告警合并 | 多个相似告警合并为一条 |

### 6.3 配置示例

**重复抑制：**
- 抑制时间：30 分钟
- 同一告警 30 分钟内只发送一次

**静默期：**
- 告警触发后，1 小时内不再触发同一告警

## 七、告警恢复通知

### 7.1 恢复通知的作用

- 告知问题已解决
- 闭环告警处理流程

### 7.2 配置恢复通知

1. 在告警规则中启用「恢复通知」
2. 配置恢复通知模板

**恢复模板：**

```
## 告警恢复

**告警名称**：${alert_name}
**恢复时间**：${resolve_time}
**持续时间**：${duration}

问题已恢复正常。
```

## 八、Galaxy 项目推荐告警

### 8.1 错误率告警

**规则配置：**
- 名称：`Galaxy-错误率告警`
- 查询间隔：5 分钟
- 查询时间范围：5 分钟

```sql
* | SELECT
        COUNT(*) AS total,
        SUM(CASE WHEN level = 'error' THEN 1 ELSE 0 END) AS errors,
        ROUND(SUM(CASE WHEN level = 'error' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS error_rate
```

**触发条件：**
```
error_rate > 5
```

**严重度：**
- `error_rate > 10` → Critical
- `error_rate > 5` → High

### 8.2 错误数量告警

**规则配置：**
- 名称：`Galaxy-错误数量告警`
- 查询间隔：5 分钟

```sql
level:error | SELECT COUNT(*) AS error_count
```

**触发条件：**
```
error_count > 100
```

### 8.3 MQTT 断连告警

**规则配置：**
- 名称：`Galaxy-MQTT断连告警`
- 查询间隔：10 分钟

```sql
mqtt and (disconnect or 断开 or 断连 or error) | SELECT COUNT(*) AS mqtt_errors
```

**触发条件：**
```
mqtt_errors > 10
```

### 8.4 登录失败告警

**规则配置：**
- 名称：`Galaxy-登录失败告警`
- 查询间隔：15 分钟

```sql
(login or 登录) and (fail or 失败 or error) | SELECT COUNT(*) AS login_failures
```

**触发条件：**
```
login_failures > 20
```

### 8.5 日志断流告警

**规则配置：**
- 名称：`Galaxy-日志断流告警`
- 查询间隔：10 分钟
- 查询时间范围：10 分钟

```sql
* | SELECT COUNT(*) AS total
```

**触发条件：**
```
total < 10
```

这表示 10 分钟内日志少于 10 条，可能存在问题。

### 8.6 特定用户错误告警

**规则配置：**
- 名称：`Galaxy-VIP用户错误告警`
- 查询间隔：5 分钟

```sql
(vip_user1 or vip_user2) and level:error | SELECT COUNT(*) AS vip_errors
```

**触发条件：**
```
vip_errors > 0
```

### 8.7 数据同步异常告警

**规则配置：**
- 名称：`Galaxy-数据同步异常`
- 查询间隔：15 分钟

```sql
"handleSendFriendToFront" and "无数据" | SELECT COUNT(*) AS sync_errors
```

**触发条件：**
```
sync_errors > 5
```

## 十、最佳实践

### 10.1 告警设计原则

1. **精准**：避免误报和漏报
2. **及时**：快速发现和通知
3. **可操作**：告警信息足够定位问题
4. **分级**：区分严重程度

### 10.2 避免告警疲劳

1. 合理设置阈值，避免过于敏感
2. 使用告警抑制减少重复
3. 定期 review 和优化告警规则
4. 分级处理，非紧急告警可延迟通知

### 10.3 告警响应流程

1. **收到告警** → 确认告警内容
2. **初步判断** → 确定影响范围
3. **定位问题** → 查看详细日志
4. **处理问题** → 修复或临时规避
5. **确认恢复** → 验证问题解决
6. **记录总结** → 更新知识库

### 10.4 告警文档

为每个告警规则创建文档：
- 告警含义
- 可能原因
- 排查步骤
- 处理方法
