## 二、查询性能优化

### 2.1 缩小时间范围

**最重要的优化手段**

```sql
-- 较慢：查询一周
* | SELECT * LIMIT 100

-- 较快：查询一小时
* | SELECT * LIMIT 100
-- 在时间选择器中设置为「最近1小时」
```

### 2.2 使用搜索条件过滤

先用搜索条件过滤，再用 SQL 分析：

```sql
-- 较慢：在 SQL 中过滤
* | SELECT * WHERE level = 'error'

-- 较快：在搜索条件中过滤
level:error | SELECT *
```

### 2.3 避免 SELECT *

只选择需要的字段：

```sql
-- 较慢
* | SELECT *

-- 较快
* | SELECT message, __time__
```

### 2.4 使用索引字段

搜索和过滤时使用已建索引的字段：

```sql
-- 较快：level 是索引字段
level:error | SELECT COUNT(*)

-- 较慢：在 message 中搜索
* | SELECT COUNT(*) WHERE message LIKE '%error%'
```

### 2.5 限制返回数量

```sql
-- 始终加 LIMIT
* | SELECT * ORDER BY __time__ DESC LIMIT 100
```

### 2.6 避免复杂正则

```sql
-- 较慢：复杂正则
* | SELECT * WHERE REGEXP_LIKE(message, '.*error.*timeout.*')

-- 较快：简单条件
error and timeout | SELECT *
```

### 2.7 使用近似函数

```sql
-- 较慢：精确去重
* | SELECT COUNT(DISTINCT __source__)

-- 较快：近似去重
* | SELECT APPROX_DISTINCT(__source__)
```

### 2.8 分页查询优化

```sql
-- 不推荐：大 OFFSET
* | SELECT * LIMIT 100 OFFSET 10000

-- 推荐：基于时间分页
* | SELECT * WHERE __time__ < 1706000000 ORDER BY __time__ DESC LIMIT 100
```

## 三、索引优化

### 3.1 全文索引 vs 字段索引

| 类型 | 优点 | 缺点 |
|-----|-----|-----|
| 全文索引 | 可搜索所有内容 | 索引大，成本高 |
| 字段索引 | 精确查询，成本低 | 只能搜索指定字段 |

### 3.2 合理配置索引

只为需要搜索的字段建索引：

| 字段 | 是否需要索引 |
|-----|------------|
| `level` | 是（常用过滤） |
| `message` | 是（全文搜索） |
| `__time__` | 自动索引 |
| `__source__` | 视需求 |
| 大文本字段 | 考虑成本 |

### 3.3 分词器选择

| 分词器 | 适用场景 |
|-------|---------|
| 标准分词 | 英文为主 |
| 中文分词 | 中文内容 |
| 不分词 | 精确匹配 |

## 四、写入优化

### 4.1 批量写入

```javascript
// 不推荐：逐条写入
logs.forEach(log => client.info(log));

// 推荐：批量写入
const batch = logs.map(log => ({
    time: Math.floor(Date.now() / 1000),
    contents: { message: log }
}));
client.putLogs(batch);
```

### 4.2 合理的日志级别

避免将所有日志都上报到 SLS：

```javascript
// 生产环境：只上报 warn 和 error
if (level === 'warn' || level === 'error') {
    slsClient.log(message);
}
```

### 4.3 日志采样

高频日志进行采样：

```javascript
// 10% 采样
if (Math.random() < 0.1) {
    slsClient.info(message);
}
```

### 4.4 异步写入

不阻塞主业务流程：

```javascript
// 异步写入，不等待结果
setImmediate(() => {
    slsClient.info(message).catch(console.error);
});
```

### 4.5 本地缓冲

```javascript
const buffer = [];
const BUFFER_SIZE = 100;
const FLUSH_INTERVAL = 5000;

function log(message) {
    buffer.push(message);
    if (buffer.length >= BUFFER_SIZE) {
        flush();
    }
}

function flush() {
    if (buffer.length === 0) return;
    const logs = buffer.splice(0, buffer.length);
    slsClient.putLogs(logs);
}

setInterval(flush, FLUSH_INTERVAL);
```

## 五、成本控制

### 5.1 成本组成

| 项目 | 计费方式 |
|-----|---------|
| 存储费用 | 按存储量（GB/月） |
| 索引费用 | 按索引流量（GB） |
| 读写流量 | 按流量（GB） |
| 查询分析 | 按扫描数据量 |

### 5.2 降低存储成本

1. **缩短保留期**：根据实际需求设置
2. **热转冷存储**：老数据转为冷存储
3. **日志采样**：减少写入量

### 5.3 降低索引成本

1. **精简索引字段**：只索引必要字段
2. **减少分词**：简单场景不分词
3. **压缩日志**：精简日志内容

### 5.4 降低查询成本

1. **缩小时间范围**
2. **使用索引字段过滤**
3. **避免 SELECT *

### 5.5 成本估算

```
月存储成本 ≈ 日志量(GB) × 保留天数 × 单价
索引成本 ≈ 日志量(GB) × 索引膨胀系数 × 单价
```

## 七、查询超时处理

### 7.1 超时原因

- 时间范围过大
- 查询过于复杂
- 数据量过大
- 并发过高

### 7.2 解决方案

**方案一：缩小时间范围**

```sql
-- 分段查询
-- 第一段：最近1小时
* | SELECT * WHERE __time__ > now() - interval '1' hour

-- 第二段：1-2小时前
* | SELECT * WHERE __time__ BETWEEN now() - interval '2' hour AND now() - interval '1' hour
```

**方案二：简化查询**

```sql
-- 简化前
* | SELECT REGEXP_EXTRACT(message, '...') AS x, COUNT(*)
    GROUP BY REGEXP_EXTRACT(message, '...')
    ORDER BY COUNT(*) DESC

-- 简化后：先搜索过滤，再分析
error | SELECT SUBSTR(message, 1, 50) AS x, COUNT(*)
       GROUP BY SUBSTR(message, 1, 50)
       LIMIT 20
```

**方案三：使用近似函数**

```sql
-- 使用 APPROX_DISTINCT 替代 COUNT(DISTINCT)
* | SELECT APPROX_DISTINCT(user_id) AS unique_users
```

## 八、大数据量查询策略

### 8.1 分页查询

```javascript
async function queryAllLogs(query, timeRange) {
    const logs = [];
    let offset = 0;
    const limit = 100;

    while (true) {
        const result = await client.getLogs({
            query: `${query} | SELECT * LIMIT ${limit} OFFSET ${offset}`,
            ...timeRange
        });

        logs.push(...result.logs);

        if (result.logs.length < limit) break;
        offset += limit;

        // 避免请求过快
        await sleep(100);
    }

    return logs;
}
```

### 8.2 时间分片

```javascript
async function queryByTimeSlice(query, from, to, sliceHours = 1) {
    const logs = [];
    const sliceSeconds = sliceHours * 3600;

    for (let t = from; t < to; t += sliceSeconds) {
        const result = await client.getLogs({
            query,
            from: t,
            to: Math.min(t + sliceSeconds, to)
        });

        logs.push(...result.logs);
        await sleep(100);
    }

    return logs;
}
```

### 8.3 并行查询

```javascript
async function parallelQuery(queries) {
    // 控制并发数
    const concurrency = 5;
    const results = [];

    for (let i = 0; i < queries.length; i += concurrency) {
        const batch = queries.slice(i, i + concurrency);
        const batchResults = await Promise.all(
            batch.map(q => client.getLogs(q))
        );
        results.push(...batchResults);
    }

    return results;
}
```

## 九、常见性能问题

### 9.1 查询很慢

| 原因 | 解决方案 |
|-----|---------|
| 时间范围太大 | 缩小时间范围 |
| SELECT * | 只选择需要的字段 |
| 复杂正则 | 简化或使用字符串函数 |
| 大 OFFSET | 使用时间分页 |

### 9.2 查询超时

| 原因 | 解决方案 |
|-----|---------|
| 数据量过大 | 分段查询 |
| 复杂聚合 | 简化分析逻辑 |
| 并发过高 | 错峰查询 |

### 9.3 结果不全

| 原因 | 解决方案 |
|-----|---------|
| LIMIT 限制 | 增加 LIMIT 或分页 |
| 超时截断 | 缩小范围重试 |
| 索引问题 | 检查索引配置 |
