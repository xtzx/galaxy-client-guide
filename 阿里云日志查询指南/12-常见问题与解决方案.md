# 12-常见问题与解决方案

> SLS 使用中的常见问题和解决方案

---

## 一、查询问题

### 1.1 查不到日志

**问题**：执行查询后没有结果

**可能原因及解决方案：**

| 原因 | 解决方案 |
|-----|---------|
| 时间范围不对 | 调整为更大的时间范围 |
| 关键词错误 | 检查拼写和大小写 |
| 日志还未写入 | 等待几秒后重试 |
| 权限不足 | 联系管理员获取权限 |
| Logstore 错误 | 确认选择了正确的 Logstore |
| **分词导致精确匹配失败** | 使用通配符 `*` 或拆分关键词（详见 1.6） |

**排查步骤：**

```sql
-- 1. 先查询是否有任何日志
* | SELECT COUNT(*) AS total

-- 2. 如果有日志，检查关键词
* | SELECT message LIMIT 10

-- 3. 逐步添加过滤条件
关键词1 | SELECT COUNT(*)
关键词1 and 关键词2 | SELECT COUNT(*)
```

### 1.2 查询很慢

**问题**：查询执行时间很长

**解决方案：**

```sql
-- 1. 缩小时间范围
-- 在时间选择器中选择更小的范围

-- 2. 避免 SELECT *
-- 较慢
* | SELECT *
-- 较快
* | SELECT message, __time__

-- 3. 使用搜索条件过滤
-- 较慢
* | SELECT * WHERE message LIKE '%error%'
-- 较快
error | SELECT *

-- 4. 添加 LIMIT
* | SELECT * ORDER BY __time__ DESC LIMIT 100
```

### 1.3 查询超时

**问题**：查询超过 60 秒超时

**解决方案：**

1. **分段查询**：将大时间范围拆分为多个小范围
2. **简化查询**：减少复杂的正则和聚合
3. **使用近似函数**：`APPROX_DISTINCT` 替代 `COUNT(DISTINCT)`

### 1.4 结果不完整

**问题**：查询结果数量少于预期

**原因：**
- LIMIT 限制（默认 100）
- 查询超时被截断
- 索引未覆盖

**解决：**

```sql
-- 增加 LIMIT
* | SELECT * LIMIT 10000

-- 或使用分页
* | SELECT * ORDER BY __time__ DESC LIMIT 100 OFFSET 0
* | SELECT * ORDER BY __time__ DESC LIMIT 100 OFFSET 100
```

### 1.6 中文标点/特殊字符导致分词匹配失败

**问题**：日志中明确包含某个中文短语，用双引号精确匹配却查不到

**真实案例**：

日志原文：`企业微信好友列表上传成功！response=[{"data":9575,"code":200}]`

| 查询方式 | 结果 | 原因 |
|----------|------|------|
| `"企业微信好友列表上传成功"` | ❌ 查不到 | `成功` 和 `！` 被分词拆开 |
| `"好友列表上传成功"` | ❌ 查不到 | 同上 |
| `企业微信好友列表上传*` | ✅ 查到 | 通配符不依赖精确短语匹配 |

**根本原因**：

SLS 的分词索引会将中文全角标点（`！` `？` `：` `，` 等）作为分词分隔符。双引号 `""` 表示精确短语匹配，但分词后 token 被拆散，精确匹配找不到完整短语。

**常见分词陷阱字符**：

| 字符 | 说明 | 处理方式 |
|------|------|----------|
| `！` `？` `：` `，` | 中文全角标点 | 截断到标点前 + 通配符 `*` |
| `[` `]` `(` `)` | 方括号/圆括号 | 截断 + 通配符 |
| `=` `:` | 键值分隔符 | 拆分为前后两部分 |
| 空格 | 默认分隔符 | 用双引号包裹或拆分查询 |

**解决方案汇总**：

```
-- 方案1（推荐）：截断到特殊字符前 + 通配符
"用户名" and 企业微信好友列表上传*

-- 方案2：拆分为多个短关键词
"用户名" and "好友" and "上传" and "成功"

-- 方案3：去掉引号用模糊匹配
"用户名" and 好友列表上传成功

-- 方案4：英文关键词优先（不受分词影响）
"用户名" and "handleSendFriendToFront"
```

**最佳实践**：

1. **英文函数名/变量名** → 双引号精确匹配（如 `"handleSendFriendToFront"`）
2. **中文日志短语** → 通配符匹配（如 `好友列表上传*`）
3. **不确定时** → 先用短关键词搜到结果，再根据实际日志内容细化查询

### 1.7 超长数字 taskId 查不到

**问题**：已知某个 19 位的 taskId（如 `1265071790997111025`），用双引号精确匹配却查不到日志

**真实案例**：

已知 taskId=`1265071790997111025`，用以下方式查询均失败：

```
"1265071790997111025"
"huanglimin02" and "1265071790997111025"
```

**根本原因分析**：

SLS 分词索引对超长数字的处理存在不确定性。taskId 在日志中有多种上下文形式：

| 日志片段 | 上下文 | 分词结果 |
|---------|------|---------|
| `taskId= 1265071790997111025` | 等号 + 空格 + 数字 | 数字是独立 token |
| `"taskId\":\"1265071790997111025\""` | JSON 转义字符串 | 数字被 `\"` 包围，分词行为不确定 |
| `taskId:1265071790997111025` | 冒号紧跟数字 | 冒号是分隔符，数字可能是独立 token |

双引号强制精确短语匹配，但 19 位长数字可能因 SLS 内部索引问题导致匹配失败。

**解决方案**：

```
# 方案1（推荐）：不加引号，直接用数字作为关键词
"用户名" and 1265071790997111025

# 方案2（最保险）：取前 15 位 + 通配符
"用户名" and 126507179099711*

# 方案3：改用目标用户 ID 查（效果相同，更可靠）
"用户名" and "7881299543032161"

# 方案4：改用可读的英文关键词定位
"用户名" and "CloudFlowHandler" and "inBound"
```

**通用规律**：

| 查询内容 | 推荐方式 | 原因 |
|---------|---------|------|
| 19 位 taskId 数字 | 不加引号 or 前 15 位+通配符 | 长数字精确匹配不稳定 |
| 英文函数名 | 双引号精确匹配 | 不受分词影响 |
| 中文短语含标点 | 截断到标点前 + `*` | 中文标点是分词分隔符（见 §1.6） |
| 目标用户 ID（纯数字）| 双引号 or 不加引号均可 | 长度较短，通常无问题 |

### 1.5 语法错误

**常见错误及修复：**

| 错误 | 原因 | 修复 |
|-----|-----|-----|
| `Syntax error` | SQL 语法错误 | 检查关键字拼写 |
| `Column not found` | 字段不存在 | 检查字段名 |
| `Invalid time range` | 时间范围无效 | 检查时间设置 |

**常见语法错误示例：**

```sql
-- 错误：缺少 |
* SELECT * LIMIT 100
-- 正确
* | SELECT * LIMIT 100

-- 错误：字符串用双引号
* | SELECT * WHERE level = "error"
-- 正确：使用单引号
* | SELECT * WHERE level = 'error'

-- 错误：中文引号
* | SELECT * WHERE level = 'error'
-- 正确：英文引号
* | SELECT * WHERE level = 'error'
```

### 1.8 MQTT 重复投递日志排查

**问题**：用户反馈消息重复收到，需要确认是 Broker 补发还是后端重发。

**排查方案**：

```
# 步骤1：查客户端收到该 taskId 几次
"用户名" and "[接收mqtt任务]" and taskId前15位*

# 步骤2：查后端发送次数
# 在后端日志平台查 MQBaseProducer send msg + taskId

# 步骤3：查客户端去重日志
"用户名" and "消息重复"

# 步骤4：查客户端过期丢弃日志
"用户名" and "消息过期"

# 步骤5：查 MQTT 连接断开/重连日志
"用户名" and ("handleClose" or "mqtt init" or "HeartBeatTimer")
```

**判断结论**：

| 后端发送次数 | 客户端接收次数 | 结论 |
|------------|-------------|------|
| 1 次 | N 次 | **MQTT Broker QoS 1 补发**（连接不稳定，PUBACK 未到达） |
| N 次 | N 次 | **后端业务重发**（业务回执超时，云端重新下发） |

> 详细 MQTT 协议机制参考 [24-MQTT消息机制深度解析](../业务逻辑/24-MQTT消息机制深度解析.md)

## 二、数据问题

### 2.1 日志丢失

**问题**：期望有的日志没有写入 SLS

**排查步骤：**

1. **检查本地日志**：确认日志确实被生成
2. **检查网络**：SLS 端点是否可达
3. **检查 SDK 错误**：查看 SDK 的错误日志
4. **检查配额**：是否超过写入限制

**代码层面检查：**

```javascript
// 检查 SLS 客户端是否初始化
if (!slsClient) {
    console.error('SLS client not initialized');
}

// 检查写入是否有错误
try {
    await slsClient.info(message);
} catch (error) {
    console.error('SLS write error:', error);
}
```

### 2.2 日志延迟

**问题**：日志写入后延迟才能查到

**说明**：SLS 写入后通常有 **1-3 秒** 延迟才能查询到

**解决：**
- 等待几秒后重试查询
- 使用「实时查询」模式（如果可用）

### 2.3 中文乱码

**问题**：日志中的中文显示乱码

**原因**：编码问题

**解决：**
- 确保写入时使用 UTF-8 编码
- 检查 SDK 配置

### 2.4 字段不存在

**问题**：查询时提示字段不存在

**原因：**
- 字段未建索引
- 字段名拼写错误
- 使用了系统字段的错误写法

**解决：**

```sql
-- 检查可用字段
* | SELECT * LIMIT 1

-- 系统字段正确写法
__time__     -- 时间
__source__   -- 来源
__topic__    -- 主题
```

### 2.5 特殊字符问题

**问题**：搜索包含特殊字符的内容

**解决：**

```sql
-- 使用引号包裹
"[error]"
"key:value"

-- 或使用转义
\[error\]
```

## 三、权限问题

### 3.1 无法访问

**问题**：提示「没有权限」或「Access Denied」

**解决：**

1. 确认已登录正确的账号
2. 联系管理员授权
3. 检查 RAM 策略是否包含所需权限

**所需权限：**

```json
{
    "Action": [
        "log:Get*",
        "log:List*",
        "log:Query*"
    ],
    "Resource": "*"
}
```

### 3.2 AccessKey 失效

**问题**：使用 SDK 时提示 AccessKey 无效

**解决：**

1. 检查 AccessKey ID 和 Secret 是否正确
2. 检查 AccessKey 是否被禁用
3. 检查 AccessKey 是否过期（STS 临时凭证）
4. 重新创建 AccessKey

### 3.3 签名错误

**问题**：提示「SignatureNotMatch」

**原因：**
- AccessKey Secret 错误
- 请求被篡改
- 时间戳偏差过大

**解决：**
- 检查 AccessKey Secret
- 确保系统时间准确

## 四、告警问题

### 4.1 告警不触发

**问题**：满足条件但告警不发送

**排查：**

1. **检查告警规则是否启用**
2. **检查查询是否正确**：在查询页面验证
3. **检查触发条件**：条件表达式是否正确
4. **检查通知渠道**：是否配置正确

### 4.2 告警重复

**问题**：同一问题收到多次告警

**解决：**

1. **配置告警抑制**：设置抑制时间
2. **调整检测周期**：增加检测间隔
3. **优化查询条件**：使条件更精确

### 4.3 告警延迟

**问题**：告警发送有延迟

**原因：**
- 检测周期设置
- 日志写入延迟
- 通知渠道延迟

**解决：**
- 缩短检测周期（但会增加成本）
- 检查通知渠道配置

---

## 五、可视化问题

### 5.1 图表不显示

**问题**：配置了图表但不显示数据

**排查：**

1. **检查查询结果**：确认查询有数据
2. **检查字段映射**：X/Y 轴字段是否正确
3. **检查数据类型**：数值字段不要是字符串

### 5.2 图表数据不更新

**问题**：图表显示旧数据

**解决：**

1. **手动刷新**：点击刷新按钮
2. **检查时间范围**：是否选择了相对时间
3. **清除缓存**：刷新浏览器

### 5.3 仪表盘加载慢

**问题**：仪表盘打开很慢

**解决：**

1. **减少图表数量**：建议不超过 15 个
2. **优化查询**：简化复杂查询
3. **缩小时间范围**：使用更小的默认时间范围

## 六、中文搜索问题

### 6.1 中文搜索不到

**问题**：搜索中文关键词没有结果

**原因**：中文分词问题

**解决方案：**

```sql
-- 方案1：使用短语搜索
"群总数"

-- 方案2：使用通配符
*群总数*

-- 方案3：在 SQL 中使用 LIKE
* | SELECT * WHERE message LIKE '%群总数%'
```

### 6.2 中文分词不准

**问题**：中文被错误分词

**解决：**

1. **配置中文分词器**：在索引配置中启用
2. **使用短语搜索**：保持词组完整

## 七、SDK 问题

### 7.1 连接失败

**问题**：SDK 无法连接 SLS

**排查：**

```javascript
// 检查端点
const endpoint = 'cn-beijing.log.aliyuncs.com';  // 公网
// 或
const endpoint = 'cn-beijing-intranet.log.aliyuncs.com';  // 内网

// 检查网络
ping cn-beijing.log.aliyuncs.com
```

### 7.2 写入失败

**问题**：日志写入失败

**常见错误码：**

| 错误码 | 原因 | 解决 |
|-------|-----|-----|
| `WriteQuotaExceed` | 写入超限 | 降低写入频率 |
| `LogStoreNotExist` | Logstore 不存在 | 检查名称 |
| `InvalidAccessKeyId` | AccessKey 无效 | 检查凭证 |

### 7.3 SDK 性能问题

**问题**：SDK 写入影响应用性能

**解决：**

```javascript
// 1. 异步写入
setImmediate(() => {
    slsClient.info(message).catch(console.error);
});

// 2. 批量写入
const buffer = [];
function log(msg) {
    buffer.push(msg);
    if (buffer.length >= 100) flush();
}

// 3. 降低日志级别
if (level !== 'debug') {
    slsClient.info(message);
}
```

## 八、错误代码速查

| 错误码 | 说明 | 解决方案 |
|-------|-----|---------|
| `Unauthorized` | 认证失败 | 检查 AccessKey |
| `ProjectNotExist` | Project 不存在 | 检查 Project 名称 |
| `LogStoreNotExist` | Logstore 不存在 | 检查 Logstore 名称 |
| `InvalidAccessKeyId` | AccessKey ID 无效 | 重新获取 AccessKey |
| `SignatureNotMatch` | 签名不匹配 | 检查 AccessKey Secret |
| `WriteQuotaExceed` | 写入配额超限 | 降低写入频率或提升配额 |
| `ReadQuotaExceed` | 查询配额超限 | 降低查询频率 |
| `InvalidQuery` | 查询语法错误 | 检查查询语句 |
| `RequestTimeout` | 请求超时 | 简化查询或缩小范围 |
| `InternalServerError` | 服务端错误 | 重试或联系支持 |
